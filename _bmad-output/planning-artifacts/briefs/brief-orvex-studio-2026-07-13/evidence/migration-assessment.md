# Migration Assessment — "Orvex Wiki runs as/behind the Studio microservices"

**Date:** 2026-07-13 · **Author:** research subagent (adversarial, no-sugarcoat) · **Question:** how far has the wiki→microservices migration ACTUALLY progressed?

**Bottom line up front:** the satellites are *real code that is deployed and being debugged live* — not skeletons. But almost none of the **production** wiki runs through them. The whole end-to-end microservices path exists only on a **single standing dev cell**, where the engine's own seam modules were switched on for the first time **today (2026-07-13)**, and the integration gates that would prove it works end-to-end are mid-flight — including documented *fake-done* incidents on those exact gates. Progress is real and substantial as **build**; it is **not enough** to call the wiki "migrated," and it is essentially **zero in production**.

## 1. Target end-state ("migrated" per the split plan)

Per the Split Plan (`rBSZqx4gXE`), the program study (`qJojHbWJni`), and the wiki-api PRD/arch (`8jAiCBifDW` / `pG9s9OjqZA`): the fork's **77 fork-added controllers** collapse to a **thin vanilla-Docmost-v0.95.0 AGPL engine** that keeps only irreducible primitives (the block-write chokepoint + `ifVersion` CAS, ACL evaluation, page lifecycle + ratify/confirm tokens, session-mint, quota enforcement, the transactional **outbox** emit, tenant resolver, Yjs/Hocuspocus collab core, the React client bundle). *Everything else* — the verb grammar & DfM/block composition, RAG/hybrid search & the always-fresh knowledge sync, the MCP agent front-door, all AI/cited-ask, OIDC/SCIM/Clerk identity, billing/entitlements, audit/console — runs as **separate closed Studio satellites** that talk back over **pinned Apache-2.0 contracts** and a **Kafka-first event spine** fed from the engine outbox. The public wire never changes: a single ingress hides the split (zero client/CLI URL change). "Done" (per the Phase-1 memory) = the wiki **fully WORKS via every surface — api / mcp / cli / ai / rag / knowledge-sync — all as microservices**, and standalone ships the **full family stack** (no absent-satellite degradation path). Not "gates green" — a real end-to-end product-acceptance pass.

## 3. Honest headline

**Today, essentially none of the *production* wiki runs through microservices — and the full end-to-end path runs only on one standing dev cell that was first switched on 2026-07-13, with its proving E2E gates still red or fake-done.** Every satellite (wiki-api, knowledge, ai, identity, billing, mcp, contracts, cli) is **genuine, non-skeleton code that is deployed and being live-debugged** — that is real, meaningful progress and the hard extraction work is underway. But: (1) prod/`main` is deliberately **vanilla Docmost with the orvex modules OFF**; (2) the dev cell is mid-integration (ENG-1559 M5/M13/M14), with the outbox→Kafka→knowledge sync — the linchpin — only just emitting; (3) the very gates that would certify end-to-end (`TestM5KnowledgeE2E`) have a **documented fake-done / RED** history in `po-decisions/fake-done-forensics.md`; and (4) **no surface has a green, human-verified, real-data end-to-end acceptance pass.** Verdict: strong build progress, but **"the wiki runs behind the microservices" is true only on dev, unproven end-to-end, and false in production. The PO's suspicion is correct.**

## 4. What a real research-phase assessment MUST verify (exercise, don't trust labels)

For each surface, drive it against the **dev cell with a fresh tenant, a real identity-minted token, and real data**, observe the actual round-trip, and require ACL enforcement — a Linear "Done" or a gate journal record is **not** evidence (see the P1/P2 fake-done patterns):

- **api** — Through `orvex-wiki-api /v1`: `save` a new page, `get` it back, `edit` via block-patch with `ifVersion` (assert a stale write returns **409 VERSION_MISMATCH**, receipt `{url,id,version,persisted:true}`), `list`, and confirm the write actually lands in the engine ydoc via the chokepoint. Working = correct receipts + CAS + no partial writes.
- **mcp** — An agent over `/mcp` (Streamable-HTTP) invokes tools that fan out to wiki-api/knowledge/ai/identity as REST clients and round-trips a real read+write. Working = tool parity growing toward 73 and a real page mutation via a tool.
- **cli** — `orvex-cli` / `docmost-cli` reads and writes a page through the `/v1` grammar (not the legacy engine path). Working = a byte-faithful DfM round-trip write that survives (the edit-path corruption memory is the risk to disprove).
- **ai** — `orvex-studio-ai` cited-ask: real question → delegates retrieval to knowledge with the caller's scoped token → returns `{answer, citations, confidence, unanswered, gapNote, followups}`; an inline-edit writes back through the chokepoint carrying the delegated token (no elevated credential). Working = real citations to real pages + ACL respected.
- **rag** — `orvex-studio-knowledge` hybrid Turbopuffer query returns **tenant-namespaced, ACL ∩ token-scope filtered** results (two callers with different ACLs get different hits). Working = no cross-tenant/over-expose leak; `tsvector` genuinely gone.
- **knowledge-sync** — **THE linchpin, and the weakest link.** Mutate a page → assert one CloudEvent (`wiki.page.content_updated`) commits to the engine **transactional outbox** → relays to Kafka → knowledge indexes it → it becomes searchable within SLA. Working = end-to-end message trace (close the message-tracing-across-outbox gap), `TestM5KnowledgeE2E` genuinely **green on merged code**, no swallowed errors.
- **Cross-cutting seams to also exercise:** identity exchange-token → engine `session/exchange` → engine session mint (fail-closed on wrong key/iss/aud); billing caps read by the engine and enforced at the quota chokepoint returning the **402 QUOTA_EXCEEDED** contract on both REST and collab paths; tenant-move (still 501); and the **single-host ingress** proving zero client-URL change across the split. "Working" for every one = real data, real auth, ACL enforced, human-observed green on a fresh tenant — reproduced, not reported.
