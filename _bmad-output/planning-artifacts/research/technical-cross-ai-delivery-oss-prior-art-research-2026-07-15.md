# Build-vs-Buy: Open-Source Prior Art for the F1 Cross-AI Delivery Extension

## TL;DR

**FORK `supermemoryai/supermemory` (`apps/browser-extension`), backporting Gemini/Claude/Grok selector logic from the archived `mem0ai/mem0-chrome-extension` as a pattern donor. Do not adopt anything wholesale. Do not build the composer-injection layer from zero.**

Two genuine direct-prior-art candidates exist in this whole landscape — both MIT, both source-verified to WRITE into the provider's own compose box rather than scrape or render an own-UI panel:

- **`supermemoryai/supermemory`** (`apps/browser-extension`) — MIT, pushed **yesterday** (2026-07-14), confirmed by direct source read to write memory text into ChatGPT's `#prompt-textarea` and Claude's `contenteditable` div via `dataset` + `appendChild` before submit-capture — **but that `appendChild` is never followed by a synthetic `input`/`InputEvent` dispatch, so on live ChatGPT/Claude DOM the injected text plausibly never reaches what ProseMirror actually sends (see §4.1, §5.1)**. `host_permissions` is live for ChatGPT and Grok; Claude code exists but is not wired in (dormant, needs completion); Gemini is entirely absent. **Grok's only implemented behavior is the reverse of composer-injection**: a "Save to supermemory" button inside Grok's own native "Memory from your chats" dialog that scrapes that dialog's text and stores it into supermemory — `entrypoints/content/grok.ts` (re-fetched live 2026-07-15, 445 lines) contains no reference to Grok's prompt textarea, `contenteditable`, or any write-back-into-composer logic at all. Supermemory therefore has a genuine (if broken-as-written) composer-write for ChatGPT only.
- **`mem0ai/mem0-chrome-extension`** ("OpenMemory") — MIT, the single broadest provider match found (ChatGPT, Claude, Gemini, Grok, Perplexity, DeepSeek, Replit — the only candidate covering all 4 Orvex targets at once), with a clean, source-verified `value`/`innerHTML` write + `dispatchEvent(new Event('input', {bubbles:true}))` pattern. **Archived by its maintainer 2026-03-23** (~4 months ago) — a real discontinuation signal, not mere staleness.

Neither alone clears the rubric's **adopt** bar (neither is simultaneously active + 4-provider + AD-6/FR-D7-ready). Both clear **fork**: permissive license, workable composer-injection selectors and MV3 messaging for 1-3 providers each (mem0's writers are complete and correct as shipped; supermemory's need the missing input-event dispatch added before they actually work — see §4.1/§5.1), missing Orvex's memory-enrichment wiring, the FR-D7 breakage-canary, the AD-6 pre-send review pause, and the copy/paste degrade path — exactly the "code quality makes surgery cheaper than greenfield" case the rubric describes. Since both are MIT, combining patterns across them carries no licensing conflict. Everything else surveyed is either closed-SaaS with no forkable repo (MemoryPlugin, YourAIScroll, Sider, Monica, Merlin, MaxAI, AIPRM), a public repo with **no OSI license** despite looking open (Superpower ChatGPT, AI Prompt Genius, Open Prompt Manager — three separate instances of the "public repo ≠ open source" trap), copyleft-hostile (ChatHub, GPL-3.0), or architecturally mismatched (ChatALL, LibreChat, big-AGI — own-UI-over-API, not composer injection; chatgpt-exporter — DOM-scrape, wrong data direction).

**Net effect on ENG-2690 scope, stated plainly**: what's genuinely borrowable is modest — supermemory's WXT/MV3 scaffold and per-provider file layout, its (currently non-functional-as-written) ChatGPT composer-write skeleton, mem0's Gemini/Claude/Grok/Perplexity selector maps, and the ~10 low-level DOM/event-dispatch patterns in §5. That reduces the "stand up an MV3/WXT skeleton with per-provider selector maps and SPA-reinit handling" line item. It does **not** ship a working composer-injection feature, and it does **not** reduce the genuinely novel Orvex-specific work — which is most of F1: the AD-6 pre-send user-review pause (the interaction both candidates are architecturally opposite to — both silently auto-inject and auto-capture at submit, with no pause for user review at all), the per-provider GO/DEGRADE behavior split, the FR-D7 fail-loud breakage-canary with team alerting (no OSS prior art exists for this anywhere in the landscape — it must be built net-new), the copy/paste degrade UI, and Orvex's own memory-enrichment backend wiring (replacing `api.supermemory.ai` / `api.mem0.ai` calls). Practically, "fork supermemory" is close to "start from a clean WXT scaffold and transplant the §5 patterns + selector maps" — the timeline benefit should be attributed to the scaffold and patterns, not to inheriting an F1-shaped extension that already works.

**Correction to record**: two of the five research streams (licensing-trust, injection-libs) could not locate the Supermemory browser extension as a distinct, verifiable, actively-maintained repo and treated it as "reference-only, unconfirmed OSS status." The repo-verification stream — treated as ground truth per this report's sourcing rules — did a direct source read of `apps/browser-extension/entrypoints/content/chatgpt.ts` inside the `supermemoryai/supermemory` monorepo and confirmed genuine compose-box-write code under the root MIT LICENSE, pushed the day before this research. The correction stands: Supermemory's extension is real, open-source, and fork-eligible, not merely a closed-SaaS rival to benchmark against.

## 1. Recommendation

**FORK.** Primary base: `supermemoryai/supermemory` → `apps/browser-extension` (MIT, WXT/MV3, pushed 2026-07-14) — chosen for its scaffold, activity, and license, not because its composer-injection ships working out of the box (it currently doesn't; see §4.1). Pattern donor for the providers/behavior it lacks or gets wrong: `mem0ai/mem0-chrome-extension` (MIT, archived 2026-03-23, mine `src/gemini/content.ts`, `src/claude/content.ts`, and `src/grok/content.ts` for selector maps and — critically — the `dispatchEvent(new Event('input',{bubbles:true}))` pattern that supermemory's own ChatGPT/Claude writers omit; do not depend on the repo itself).

Rationale tied to the rubric:
- **License**: both candidates are MIT — clean for a Chrome-Web-Store-distributed, Orvex-backend-connected extension; only obligation is preserving copyright/license notice on borrowed files.
- **Activity**: supermemory is maintained today (pushed yesterday); mem0's extension is archived, so it is treated strictly as a one-time pattern-mine, never as a living dependency.
- **Mechanism fit**: both repos contain source-verified genuine compose-box-write code (not scrape, not own-UI) — the exact F1 mechanism family — but supermemory's own instances of it (ChatGPT, dormant-Claude) are missing the input-event dispatch that makes the write actually register with ProseMirror/React, whereas mem0's per-provider writers correctly dispatch it. The reusable, *working* primitive is mem0's, not supermemory's; supermemory's contribution here is the scaffold and file layout it should be transplanted into.
- **Provider coverage**: combined, borrowable code touches all 4 Orvex targets, but unevenly and in only one living repo for none of them — ChatGPT (supermemory, composer-write present but needs the dispatch fix; mem0, composer-write correct), Claude (supermemory, dormant + same missing-dispatch defect; mem0, composer-write correct), Gemini (mem0 only), Grok (mem0 only — **supermemory has no Grok composer-write at all**; its Grok integration only imports Grok's own native memory summary *into* supermemory, the opposite data direction).
- **Fork cost vs. greenfield**: the hard, provider-specific DOM/selector/event-dispatch work — the part that would otherwise take the most trial-and-error to get right — is already solved and license-clean, predominantly in mem0's per-provider files (§5 patterns 1-4). The genuinely novel Orvex requirements (AD-6 review-pause, FR-D7 canary, GO/DEGRADE split, memory backend) exist nowhere in the landscape and are 100% new work regardless of fork-vs-build-fresh, so forking only ever helps, never hurts, the timeline — but the help is closer to "clean scaffold + a well-tested pattern library to transplant into it" than "a working feature to inherit."

## 2. Decision Rubric & Scoring Method

Per the governing plan, a candidate is scored against four gates, all of which must hold for **adopt**, a subset for **fork**:

| Gate | Adopt requires | Fork requires |
|---|---|---|
| License | Permissive OSI (MIT/Apache-2.0/BSD) | Permissive or weak-copyleft |
| Activity | Commit within ~6mo, not archived | "Reasonably active" (softer bar; archived repos are treated as one-time pattern-mines, not fork bases, per this report's application of the rubric) |
| Mechanism fit | True MV3 content script writing into the provider's OWN composer | Same — must nail composer-injection selectors + MV3 messaging for at least 1-2 providers |
| Provider coverage | Already covers multiple of ChatGPT/Gemini/Claude/Grok | Partial is acceptable — gaps are expected |
| AD-6/FR-D7 reachability | Config/branding/memory-wiring changes only | Missing pieces (memory-enrichment, breakage-canary, degrade path, full coverage) are expected and are the reason to fork rather than adopt |

**Build-fresh** is the fallback only when every candidate is closed-SaaS with no repo, copyleft-hostile (GPL/AGPL) for a distributed extension, abandoned >12 months, or architecturally mismatched (scraper / own-UI-over-API). This landscape does not force build-fresh: two license-clean, mechanism-correct, partially-current candidates exist. The FORK path is chosen over build-fresh specifically because the composer-injection layer — the part of F1 with the most implementation risk and trial-and-error cost — is already solved in reusable, license-compatible code: concretely, that means mem0's per-provider writers (ChatGPT/Claude/Grok/Gemini, all confirmed to include the required input-event dispatch), not supermemory's own ChatGPT/Claude writers, which are present but currently miss that same step (§4.1/§5.1).

## 4. Top-Candidate Scorecards

### 4.1 supermemoryai/supermemory — `apps/browser-extension` (PRIMARY FORK BASE)

**What it is**: The browser-extension package inside Supermemory's main monorepo (a memory/context product for AI assistants). Built on WXT (a modern MV3 build framework) + React. Repo: https://github.com/supermemoryai/supermemory (accessed 2026-07-15).

**How it injects**: `entrypoints/content/chatgpt.ts` locates `document.getElementById("prompt-textarea")`, reads the user's in-progress draft to trigger a memory search, and on submit-intent (`capturePromptContent()`, fired from a capture-phase `click` listener on `#composer-submit-button` and a capture-phase `keydown` listener for Enter — i.e. it runs immediately *before* ChatGPT's own send handler, not after) sets `promptElement.dataset.supermemories = "..."` then does `promptTextarea.appendChild(document.createTextNode(storedMemories))` — a raw DOM mutation with **no following `dispatchEvent(new Event('input', {bubbles:true}))` anywhere in the file** (re-fetched live 2026-07-15, confirmed at the same line — 650 — as the locally-cached research copy). Pattern 1 in §5 states plainly that ChatGPT's ProseMirror-controlled composer "silently ignores a raw DOM mutation without a synthetic input event." That is exactly what this write is, so the injected memory text plausibly never becomes part of what ChatGPT actually sends — it needs re-verification against live DOM and, most likely, the missing dispatch added before it can be trusted at all. The near-identical pattern repeats for Claude (`contentEditableDiv.appendChild(document.createTextNode(storedMemories))`, `entrypoints/content/claude.ts`, same missing-dispatch defect, and dormant regardless per below). **There is no equivalent for Grok**: `entrypoints/content/grok.ts` (445 lines, re-fetched live) contains no reference to a prompt textarea, `contenteditable`, or any composer-write logic — its only function is `addSupermemoryButtonToGrokMemoryDialog()` / `saveGrokMemoriesToSupermemory()`, which adds a "Save to supermemory" button inside Grok's own native "Memory from your chats" settings dialog and scrapes *that* text into supermemory. This is the reverse of F1's injection direction and should not be counted as Grok composer-injection prior art at all.

**License/fork-cost**: MIT, root LICENSE confirmed. No copyleft obligation. Low fork cost — the codebase is clean, actively built, uses a modern toolchain (WXT), and its provider-selector-per-file layout (`entrypoints/content/<provider>.ts`) is directly analogous to what F1 needs.

**Maintenance**: 28,382 stars, 2,473 forks, 80 open issues, not archived, **pushed 2026-07-14** — the freshest of any candidate in the entire sweep.

**Fit vs. Orvex FR-D1..D7 + ToS-clean + breakage-canary**:
- Composer-write for ChatGPT: **present but not solved** — uses a raw `appendChild` without the input-event dispatch §5.1 itself identifies as required for ProseMirror; likely does not currently reach the sent message and must be replaced with the mem0 `dispatchEvent`/Context-Sync `execCommand` pattern (§5, patterns 1-2) and re-verified against live DOM before it can be trusted. It also auto-injects-then-auto-captures at submit with no pause, the opposite of AD-6's inject-then-let-the-user-review-then-send requirement — so even a dispatch fix only gets it to "wrong interaction, correct plumbing," not "solved." This tempers how much weight supermemory's ChatGPT primitive should carry as the reason it's the primary fork base — the WXT scaffold, activity, and file layout carry more of that argument than the composer-write code itself does.
- Grok: **not present as composer-write at all** — `host_permissions` includes `grok.com`, but the only Grok-specific code (`grok.ts`) adds a button to Grok's own memory-review dialog that scrapes and saves *Grok's* memory summary into supermemory; it does not write anything into Grok's prompt box. The only real Grok composer-injection reference in this landscape is mem0's (`src/grok/content.ts`, confirmed live: sets `.value` then `dispatchEvent(new Event('input', {bubbles:true}))`) — treat Grok as 100% pattern-mined from mem0, not partially inherited from supermemory.
- Claude: content-script code exists (with the same missing-dispatch defect as ChatGPT) but `host_permissions` in `wxt.config.ts` omits `claude.ai` — dormant, needs to be wired live, have the dispatch added, and re-verified.
- Gemini: absent entirely — net-new work (pattern-mine from mem0).
- FR-D7 breakage-canary: not present. It has a MutationObserver + URL-poll route-change re-init (`chatGPTRouteObserver`/`chatGPTUrlCheckInterval`) that silently re-attempts selector discovery on SPA navigation — useful seed infrastructure, but it fails silently rather than alerting; must be extended to fail loud.
- AD-6 (inject only into the user's own session): satisfied implicitly by construction (content script in the user's own logged-in tab) but not documented/tested as an explicit invariant.
- Permission scope: broader than F1 needs (`webRequest`, `tabs` beyond the minimum content-script + storage set) — prune on fork.
- Memory backend: wired to `api.supermemory.ai` — must be fully replaced with Orvex's own memory service.

### 4.2 mem0ai/mem0-chrome-extension ("OpenMemory") — SECONDARY / PATTERN DONOR

**What it is**: A standalone Chrome extension (separate repo from the `mem0ai/mem0` backend) that surfaced and injected mem0-stored memories into AI chat composers. Repo: https://github.com/mem0ai/mem0-chrome-extension (accessed 2026-07-15).

**How it injects**: `src/chatgpt/content.ts` (2,565 lines) locates `#prompt-textarea` / `div[contenteditable="true"]` / `textarea`, sets `.value`/`innerHTML`, then explicitly `inputElement.dispatchEvent(new Event('input', { bubbles: true }))` — the canonical technique for getting React/ProseMirror-controlled composers to register a programmatic write. The same pattern repeats per-provider across dedicated `src/{chatgpt,claude,gemini,grok,perplexity,deepseek,replit}/` folders — the broadest provider matrix of anything found in this research, and the **only candidate with a real, working Grok composer-injection reference**.

**License/fork-cost**: MIT, LICENSE file present; the archive banner itself states "You're welcome to fork this repo and build on it — the MIT license still applies." Fork cost is high as a *living dependency* (archived, zero upstream fixes going forward) but low as a *one-time pattern-mine* (clean, readable, per-provider file layout).

**Maintenance**: 679 stars, 108 forks, 0 open issues, **archived: true** as of 2026-03-23 — explicitly discontinued by the maintainer (README redirects users to the hosted mem0.ai product). This is a deliberate discontinuation signal, not mere staleness.

**Fit vs. Orvex needs**: Same gaps as supermemory (no AD-6 review-pause, no FR-D7 canary, no GO/DEGRADE tiering, memory backend wired to `api.mem0.ai`) plus the additional cost that its content script phones the user's live composer text to `api.mem0.ai` for search — that data-exfiltration-shaped call must be fully ripped out, not merely reconfigured, before any of its code ships under Orvex's name. **Recommended use**: mine `src/gemini/content.ts`, `src/claude/content.ts`, and `src/grok/content.ts` for selector maps and, critically, the `dispatchEvent('input', {bubbles:true})` pattern itself — needed not only to fill supermemory's Gemini gap and build Grok's composer-write from scratch (supermemory has none, see §4.1), but also to *fix* supermemory's ChatGPT writer and complete its dormant Claude wiring, both of which currently omit this dispatch. Do not import or depend on the repo itself.

### 4.3 Vineetpandey0/Context-Sync — pattern-level reference only

**What it is**: A small (14-star) MIT extension injecting saved context into ChatGPT/Claude/Gemini/DeepSeek composers. Repo: https://github.com/Vineetpandey0/Context-Sync.

**How it injects**: Per-provider techniques confirmed via source: `execCommand('insertText', ...)` for ChatGPT's ProseMirror composer; Quill `innerHTML` write + a deliberate "triple-fire" of Angular change-detection events for Gemini; native-setter + React synthetic events for DeepSeek.

**License/fork-cost**: MIT — clean, but the project is too immature (14 stars, 1 fork, no Grok coverage, no automated breakage detection — the maintainer explicitly asks users to file GitHub issues when selectors break, the opposite of FR-D7) to serve as a fork base.

**Fit**: Contributes exactly one thing supermemory+mem0 don't already cover well: the **Gemini "triple-fire" event-dispatch detail** — worth replicating as a road-tested specific rather than rediscovering by trial and error.

### 4.4 WebChatGPT / chatgpt-advanced — historical reference only

MIT, the most-starred true composer-write extension found (6,446★), and the cleanest textbook example of the F1 idiom (fetch context → compile enriched prompt → set `textarea.value` → dispatch synthetic submit). But functionally frozen since 2023-06-30 per its own README, single-provider, targets the legacy `chat.openai.com` domain rather than `chatgpt.com` (likely non-functional today), and its GitHub ownership silently transferred from the original author (`qunash`) to a different org (`interstellard`) with no fork-notice — a concrete, independently-verified instance of the "extension ownership transfer" trust risk named in Orvex canon. Useful only as proof the mechanism category is well-trodden; not a fork base.

### 4.5 AI Prompt Genius — integrity-trap scorecard

Public, actively committed (pushed 2026-07-09, six days before this research), 1,291 stars — looks like the strongest "active OSS prompt tool" candidate at a glance. Two disqualifiers found on direct inspection: (1) no LICENSE file exists in the repo (GitHub's license API returns null; secondary sources describe CC BY-NC-SA 4.0, which is non-OSI and explicitly NonCommercial either way — fatal for a commercial Chrome-Web-Store fork under either reading); (2) its actual mechanism, verified by reading `plugin/scripts/hotkey.js` end-to-end, is **clipboard copy** (`copyTextToClipboard()`), not composer-write — the seed-candidate list's assumption that it "inserts into the ChatGPT composer" does not hold up against the source. Retained only as a clean, real-world illustration of the **AD-6/FR-D7 degrade-to-clipboard fallback UX** (searchable prompt palette, hotkey trigger, variable-fill modal, clipboard write) — the pattern is worth harvesting even though the code and license are not.

## 5. Reusable Patterns to Steal

All patterns below are corroborated independently across two or more MIT-licensed source repos — that convergence is itself evidence these are the *correct* low-level techniques, not an artifact of one project's idiosyncrasy.

1. **React/ProseMirror-safe composer write** (`mem0ai/mem0-chrome-extension` `src/chatgpt/content.ts`, `src/claude/content.ts`, `src/grok/content.ts` — all confirmed via live re-fetch): set `.value`/`innerHTML`/`textContent`, then `element.dispatchEvent(new Event('input', { bubbles: true }))` (or an `InputEvent` with `inputType: 'insertText'`). Necessary because ChatGPT's and Claude's composers are ProseMirror/React-controlled and silently ignore a raw DOM mutation without a synthetic input event that their internal change-detection listens for. **`supermemoryai/supermemory`'s `entrypoints/content/chatgpt.ts` and `claude.ts` are a negative example of this exact rule, not a positive one**: both do the `appendChild`/write half of the pattern but omit the dispatch half entirely (verified live 2026-07-15) — the injected text is plausibly inert on real ChatGPT/Claude DOM. mem0's per-provider files are the correct, complete reference implementation of this pattern; supermemory's should be treated as needing the fix, not as a second working example of it.

2. **`execCommand('insertText', false, text)` for ProseMirror-family contenteditable editors** (`Vineetpandey0/Context-Sync`): focusing the editor and calling `document.execCommand('insertText', ...)` produces a native `beforeinput` event with `inputType: 'insertText'` — exactly what ProseMirror's/TipTap's/Slate's transaction pipeline listens for, behaving like a real keystroke rather than a raw mutation. Works from the content-script isolated world; no MAIN-world injection needed.

3. **`setNativeValue` trick for plain React-controlled `<textarea>`/`<input>`** (Context-Sync's DeepSeek injector; documented independently via the well-known Kent C. Dodds "fire change event that works with React" pattern): grab the native property setter via `Object.getOwnPropertyDescriptor(window.HTMLTextAreaElement.prototype, 'value').set`, call it directly to bypass React's internal `valueTracker`, then dispatch `new Event('input', { bubbles: true })`.

4. **Gemini's "triple-fire" event chain** (Context-Sync): writing into Gemini's Quill-based editor via `innerHTML` requires dispatching multiple synthetic events in sequence — a single `input` event was found insufficient to satisfy Angular's change-detection cycle in practice. A concrete, road-tested detail worth replicating rather than rediscovering by trial and error.

5. **Per-provider content-script file layout** (`src/<provider>/content.ts` in mem0; `entrypoints/content/<provider>.ts` in supermemory; per-provider functions in Context-Sync): clean isolation so one provider's selector drift doesn't crash the others — a natural seed for FR-D7's per-provider breakage isolation.

6. **Selector fallback chains** (mem0's Claude injector: `.ProseMirror` → generic `[contenteditable="true"]` → placeholder-`<p>` empty-state detection): necessary-but-not-sufficient plumbing for a breakage canary — when even the fallback chain is exhausted, *that* exhaustion event is exactly what should fire FR-D7's alert, rather than failing silently as every candidate currently does.

7. **MutationObserver + URL-poll SPA route-change re-init** (supermemory's `chatGPTRouteObserver` + `chatGPTUrlCheckInterval`): AI chat UIs are single-page apps that unmount/remount the composer on navigation (new chat, model switch); both leading candidates re-run selector discovery on route change. Good seed for FR-D7's self-healing logic — extend to alert-on-exhaustion rather than retry-forever.

8. **`data-*-initialized` idempotency guards**: used in both leading candidates to prevent double-injection of listeners across repeated observer fires — cheap, directly reusable.

9. **WXT (wxt.dev) as the MV3 build tool** (supermemory's extension): a modern MV3-native build framework (React + Vite + manifest generation) worth evaluating as F1's own build tool, independent of whether code is forked.

10. **Copy/paste degrade UX** (AI Prompt Genius, despite its license/mechanism disqualification): searchable palette + hotkey + variable-fill modal + `copyTextToClipboard()` — a clean reference for the AD-6/FR-D7 degrade-to-clipboard fallback path, even though the underlying code isn't reusable.

**No OSS prior art exists anywhere in the landscape for the FR-D7 breakage-canary itself** (self-detect breakage → notify user → fall back to copy/paste → alert team). Every candidate examined has retry/self-healing plumbing (patterns 6-7 above) but none closes the loop with a loud failure signal or team alert. This is standard SRE-style self-monitoring applied to DOM selectors, not something specific to any AI-extension OSS project — Orvex will be building this net-new regardless of fork-vs-build-fresh.

## 7. Trust & Security

**The malicious-extension precedent, independently verified 2026-07-15** (sharpens the two incidents already named in Orvex canon):

- **~900,000-user incident**: Two Chrome extensions impersonating the legitimate "AITOPIA" AI-sidebar extension — "Chat GPT for Chrome with GPT-5, Claude Sonnet & DeepSeek AI" (600,000+ installs) and "AI Sidebar with Deepseek, ChatGPT, Claude and more" (300,000+ installs) — secretly exfiltrated complete ChatGPT/DeepSeek conversation content plus browsing URLs, search queries, session tokens, and user IDs to attacker-controlled servers every 30 minutes, while their consent language claimed only "anonymous, non-identifiable analytics." Reported by OX Security 2025-12-29; Google confirmed under review 2025-12-30.
- **8-million-user incident**: A family of "free VPN" extensions led by Urban VPN Proxy (6M+ users on its own) embedded per-provider JavaScript executors (`chatgpt.js`, `claude.js`, `gemini.js`) that intercepted and exfiltrated AI conversations across ChatGPT, Claude, Gemini, Copilot, and Grok, sending data to `analytics.urban-vpn.com`. The same harvesting code was found in 7 sibling extensions from the same publisher. Discovered by Koi Security.
- **Adjacent context**: a "DarkSpectre" campaign family (ShadyPanda/GhostPoster/Zoom Stealer) reportedly affected 8.8M users over 7+ years across Chrome/Edge/Firefox — underscoring that AI-conversation-adjacent extension compromise is a pattern, not a one-off, materially raising the "install our extension" trust bar Orvex must clear.

**What this means for a forked F1**:
- Both leading fork-candidate repos themselves illustrate the exact failure shape to avoid: mem0-chrome-extension's content script legitimately phones the user's live composer text to `api.mem0.ai` for search — a real "extension reads your prompt and phones home" behavior baked into otherwise-good MIT code. **This call must be fully removed, not reconfigured**, before any of that code ships under Orvex's name; anything less repeats the pattern users are now primed to distrust.
- Permission-scope minimization is directly actionable from the two leading candidates' own manifests: both scope `host_permissions` per-provider domain rather than `<all_urls>` (contrast: `carlosguadian/universal-prompt-library` uses `<all_urls>`, a negative example to avoid). supermemory additionally requests `webRequest` and `tabs`, broader than a pure injector needs — prune these on fork to the minimum content-script + storage set.
- F1's structural answer (write-only into the user's own composer, no server-side scraping/storage of AI responses beyond what the user explicitly sends, stated ToS-clean/AD-6 posture) is the right inverse of the incidents above — but it must be *provable* (auditable, minimal-permission, no phone-home calls) to a skeptical user and to Chrome Web Store reviewers, not merely asserted in a privacy policy.
- Given the current climate, budget for CWS's "in-depth review" path (typically 1-2 weeks, occasionally 3-4) as a scheduling risk for ENG-2690, not just a technical one — content-script extensions touching AI-assistant sites are a known abuse vector reviewers are primed to scrutinize regardless of how minimal F1's actual permission set is.

## 8. Mechanism Fit & ToS Alignment

- AD-6 ("inject only into the user's own session; DEGRADE to copy/paste when a provider blocks") is satisfied *implicitly by construction* in both fork-candidate repos — they run as content scripts in the user's own logged-in tab, writing into the DOM the user already sees, with the user clicking send themselves — but **neither treats it as a documented, tested invariant**. Orvex's fork must make this explicit and testable (e.g. an assertion that the extension never operates outside the active tab's authenticated session), which is the difference between "coincidentally ToS-adjacent" and "deliberately ToS-clean by design."
- Neither candidate implements a **provider-tiered GO/DEGRADE split**. Both currently auto-inject into Claude the same way as ChatGPT, with no behavioral distinction for Anthropic's/xAI's "automated or non-human means" clauses. This confirms the DEGRADE decision for Claude/Grok is a deliberate **risk-posture choice**, not a technical limitation — mem0-chrome-extension's inclusion of working Claude and Grok content scripts is direct evidence the injection is technically feasible on those two providers; Orvex is choosing to constrain it for ToS reasons, which is a policy layer that must be built on top of the borrowed injection code, not found within it.
- Neither implements FR-D7's fail-loud breakage detection with user notification and team alerting (see §5) — this is the single largest gap between the prior art and Orvex's spec, and the one area with zero shortcut available anywhere in the landscape.

## 9. Mapping to the ENG-2689 Verdict & the ENG-2690 Pack

- ENG-2689 already fixed the mechanism (Chrome MV3 content script) and the per-provider GO/DEGRADE map (ChatGPT=GO, Gemini=GO, Claude=DEGRADE, Grok=DEGRADE). This research does not change either decision — if anything it reinforces the mechanism choice: every serious OSS attempt at this exact problem independently converges on MV3 content scripts using `execCommand`/native-setter/synthetic-event techniques, validating that F1's chosen approach is the well-trodden, working path, not a novel risk.
- **What forking reduces in ENG-2690's build plan, stated at the correct altitude**: the MV3/WXT scaffold and per-provider file layout (from supermemory, live and maintained), the provider content-script skeleton for ChatGPT (from supermemory — present but needs the missing input-event dispatch added and re-verified, not a working inherit-as-is), Gemini + Claude-completion + Grok selector logic and the *correct, complete* composer-write primitive with its dispatch step (from mem0, archived but license-clean to mine — this is the piece that actually makes ChatGPT/Claude/Grok injection work), SPA route-change re-init handling, and the copy/paste degrade UX pattern (from AI Prompt Genius, pattern-only). None of this ships F1's defining interaction (AD-6's review-pause) or a working composer-write for more than one provider without further fixing — the realistic savings are scaffold + selector maps + patterns, not a working feature.
- **What remains 100% net-new regardless of fork-vs-build-fresh**: the AD-6 pre-send review pause (both candidates auto-inject on submit; F1 needs inject-then-let-the-user-review-then-click-send), the provider-tiered GO/DEGRADE behavioral split, the FR-D7 fail-loud breakage-canary with team alerting (no OSS reference implementation exists anywhere), the copy/paste degrade trigger logic itself (only the UX shell is borrowable), and Orvex's own memory-enrichment backend wiring (replacing both candidates' third-party API calls).
- **Recommended engineering approach**: fork supermemory's `apps/browser-extension` as the living codebase; **add the missing input-event dispatch to its existing ChatGPT/Claude writers** (they do not work without it); port (not fork, not dependency-link) mem0's Gemini selector logic and Grok selector logic **built fresh from mem0** (supermemory has none) into it; strip supermemory's non-F1 scope (X/Twitter bookmarking, `webRequest`/`tabs` permissions, the Grok memory-import feature if not wanted); re-verify every borrowed selector and every composer-write (including the two supermemory claims to already have) against live provider DOM before shipping (both source repos are already selector-stale by weeks-to-months, and supermemory's writers are additionally unproven); build the AD-6 review-pause, GO/DEGRADE tiering, FR-D7 canary, and Orvex memory wiring as new Orvex-owned code on top.

## 10. Open Risks & Watch-list

- **Provider UI churn**: mem0-chrome-extension's archival is itself live evidence that maintaining composer-injection selectors against provider DOM churn is a real, recurring cost, not a one-time integration task — exactly the premise FR-D7 is designed to address. Any selector borrowed today (from either fork source) should be assumed partially stale and must be re-verified against live DOM before shipping, not trusted from repo code alone.
- **CWS policy update effective 2026-08-01**: referenced in the governing plan's outline; the licensing-trust stream found and cited a specific developer.chrome.com policy-update post landing around that date, enhancing user-privacy/data-collection-transparency requirements — this lands during or immediately after F1's likely build window. Other streams could not independently corroborate the exact date. **Action**: whoever owns the ENG-2690 CWS listing must independently re-verify current policy text before drafting permission justifications and privacy disclosures, rather than relying on this report's secondhand citation.
- **License drift on active candidates**: AI Prompt Genius and Open Prompt Manager are both real, actively-committed repos (pushed within the last ~5-6 weeks of this research) that could resolve their licensing ambiguity in either direction — worth a periodic re-check given Open Prompt Manager's otherwise-attractive 4-provider coverage, rather than a one-time dismissal.
- **mem0-chrome-extension abandonment**: archived means zero future upstream fixes; if F1 ever needs to point to the single most complete OSS precedent for "an MIT-licensed MV3 extension that writes memory-enriched context into ChatGPT/Claude/Gemini/Grok composers," this is it, but its abandonment should be re-verified if this research is reused more than a few months out (mem0ai could unarchive it, or a community fork could emerge).
- **Supermemory scope creep**: it's a general product (X/Twitter bookmarking + AI memory), not a single-purpose F1-shaped tool — forking the whole monorepo app pulls in unrelated capture/bookmark UI and broader permissions than F1 needs; must be pruned, not shipped as-is.
- **Supermemory's Claude coverage is dormant, not live**: `claude.ts` exists in the source tree but `host_permissions` in `wxt.config.ts` omits `claude.ai` — do not assume Claude support is production-ready without independently re-verifying and completing the wiring.
- **Supermemory's ChatGPT/Claude composer-write is unproven, and its Grok "coverage" is a different feature entirely** (found during this correction pass, live re-fetch 2026-07-15): `chatgpt.ts`/`claude.ts` write via `appendChild(document.createTextNode(...))` with no `dispatchEvent` anywhere in either file — the injected text may never reach the actual sent message on live ProseMirror-backed DOM, contradicting this report's earlier "solved" framing (§4.1). Separately, `grok.ts` contains no composer-write code at all; its only function imports Grok's own native memory summary *into* supermemory via a button in Grok's settings dialog — the opposite of F1's injection direction. Both defects must be independently re-verified against live product DOM (not just source) before any engineering estimate treats supermemory's composer-write as proven for any provider; mem0's per-provider writers (ChatGPT/Claude/Grok/Gemini, all confirmed to include the dispatch step) are the safer reference to build from.
- **Candidate abandonment as a category signal**: WebChatGPT (~2yr stale, functionally dead since 2023) and AI Prompt Genius/Superpower ChatGPT/Open Prompt Manager (licensing ambiguity) together indicate this niche has a short half-life and low licensing hygiene — nothing surveyed should be treated as a long-term-maintained dependency; the fork strategy here is explicitly a one-time code transplant, not an upstream-tracking relationship.
- **Cross-stream discrepancy on Supermemory, resolved**: two of the five research streams could not locate the Supermemory extension's dedicated repo and marked it "partial/unconfirmed." The repo-verification stream's direct source read (confirmed compose-box-write code, MIT LICENSE, monorepo path `apps/browser-extension`) is treated as ground truth in this report per the sourcing rule that live-verified primary-source reads outrank streams that failed to locate the artifact. If future research reuses this report, re-verify the repo still exists at the cited path before relying on the correction.
