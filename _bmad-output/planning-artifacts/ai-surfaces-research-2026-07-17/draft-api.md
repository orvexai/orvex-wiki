## Reading rules (the traps this list is written against)

1. **Code-at-HEAD, not doc status.** `certified ≠ current`: several PRD/canon pages self-declare DRAFT while `status:canonical`. Every coverage call cites a `file:line`/route/ticket. The engine's amazing-MCP primitives are code-real but **"live end-to-end is deploy-pending"** (`engine §3`, commit `91b3b115`) — "done" here means real+wired at HEAD, not behaviorally verified in prod.
2. **Serving service is a first-class column (PO-D4).** A capability parked on the wrong service is a gap, not coverage. Legend: `wiki-api` = the Go `/v1` tier · `engine` = AGPL engine primitive fronted by `/v1` (SoR for writes+collab, **never the AI path**) · `knowledge` = orvex-studio-knowledge (search/related/RAG) · `ai` = orvex-studio-ai (ask/chat/inline/image/bake) · `identity` = orvex-studio-identity (authn/JWKS) · `audit` = orvex-studio-audit (event store+query, R16/R25) · `billing` = orvex-studio-billing (entitlements) · `contracts` = orvex-studio-contracts (the spine). "wiki-api-fronted" means `/v1` composes/transports to a satellite.
3. **Disposition is argued from first principles (PO-D2), never "the fork had it":** `adopt` (carry as-is) · `adopt-reshaped` (carry, changed shape/home) · `rethink` (open design call, recommendation given) · `drop`.
4. **Two OBE ticket clusters.** The early ENG-13xx–19xx Done batch shipped the capability; the later ENG-24xx–25xx "from-scratch" Todo batch re-proposes it. Coverage cites the Done batch; §(b) flags the re-proposals.

## (c) Open design questions this list surfaces

1. **Agent document reads from `knowledge` index vs engine round-trip (PO-D4's named open question).** Serve `/v1` reads from the knowledge read-replica index for the 10k-customer/cell scale win — against freshness/indexing-lag, lossless DfM fidelity (does the index hold full content?), the per-page ACL enforcement point, and provenance/trust metadata. Engine stays SoR for writes+collab; this is only about the read path. **Design it and bring the trade-off back** — do not adopt blindly.
2. **Comments/labels/attachments — sub-resources under `{loc}` or top-level resources?** Modeling them as `/v1/wiki/{loc}/comments|attachments` preserves the D-S11 `wiki`-only grammar while giving CLI/MCP a real route (recommended). Until built, the CLI methods must become honest local stubs (like `wiki space`), never silent 404s.
3. **Spaces don't fit the `{resource}=wiki` grammar.** A space is not a "wiki" page — does the grammar gain a second resource type (violating the D-S11 hard gate), a `/v1/spaces/*` sibling tree, or does space-admin stay a distinct surface? Blocks the P0 spaces row.
4. **Does `/v1` front `related`/`duplicates`, or do MCP/CLI go direct to `knowledge` (D-M8)?** Consistency (one `/v1` truth) vs one-less-hop. `search` is already fronted; ENG-2526 proposes fronting the rest.
5. **`problem+json` vs the bespoke rich envelope.** The CLI's `{error_code,…,next}` is *richer* than RFC-9457's five fields — recommendation: converge onto it + add a `type` URI and serve under `problem+json` (extension members allowed), don't dogmatically strip to the RFC.
6. **`Idempotency-Key` on `/v1` — is CAS enough?** No: CAS is concurrent-writer safety, not retry-safety. Recommendation: add it (engine plumbing already exists).
7. **Collapse version semantics onto one integer now?** Breaking change to the read shape → contracts major bump + deprecation window; sequence before wide MCP/CLI adoption.
8. **Is engine-delegated 401 acceptable for Phase-0 edge auth?** No security hole (engine still 401s) but no edge defense-in-depth, and `/v1/audit` is dead until the real `CallerVerifier`+JWKS is wired. Recommendation: acceptable for Phase-0 only, scheduled as a named launch M-gate.
9. **Cursor changes feed + webhooks depend on the outbox→Kafka relay (D-S13).** The cursor feed can ship over the engine primitive first; webhook push waits on the relay. Sequence accordingly.
10. **The Linear support-issue relay — keep-renamed, or drop?** The directive says Linear is dropped "entirely"; the code frames the relay as a narrow "file a support ticket" leg, not the excluded view/graph/sync surface. Rename to a service-neutral `/v1/support/issues` or drop — **human should rule**.
11. **Per-page export verb — re-add, or is the read-ladder enough?** The flat `export` verb was retired; the `/internal/pages/{id}/export` seam is knowledge-facing. Decide whether agents need a dedicated markdown-export verb.
12. **Upstream EE page-level access/permissions — intentional cut or re-add?** The fork deleted 1438 lines of server+client per-page-sharing; no replacement, no ticket. Explicit call needed.
13. **Do the AI-first grammar affordances ship as a required eval gate?** `served-openapi-diff` + `emitted-event-validation` are draft/unwired — the only mechanism that stops the 404/drift class recurring. Recommendation: move to blocking CI.

### One-paragraph verdict for the PRD redo

The `/v1` grammar is **live and broad** — read ladder, whole-doc `PUT`, `{loc}` `blocks:batch`, integer-CAS receipts, block registry, convert, import, `pages/bulk`, drift-verify, studio verbs, and public capabilities/instructions are all real at HEAD (ENG-1969 mounted the Dispatcher into `cmd/wikiapi`). But the tier is **not yet a trustworthy composition tier**: its **edge auth is inert** (both verifiers `notImplemented`), **`/v1/audit` is unusable**, it serves **no comments/labels/attachments and no spaces resource** (cascading into shipped-broken CLI 404s), its **flagship search 502s in prod**, its **error path is four divergent mappers**, its **read version can't feed a write CAS**, and its **`gen/` is hand-authored off a stale tag** — the root cause of the drift. The governance value-prop (spec-gate, ratify transport) and the lifecycle verb are scaffolded-not-wired. Under PO-D4 the serving homes are clean — search/related→`knowledge`, ask/chat/bake→`ai`, authn→`identity`, audit-events→`audit`, quota-read→`billing`+engine, and the engine stays SoR-for-writes but never the AI path — with **one genuinely-open design question the PO named: serving agent document reads from the knowledge index instead of the engine.** The ENG-2511–2543 "from-scratch" Todo cluster is **largely OBE** re-proposals of the already-Done ENG-1366–1969 batch and must be reality-probed, not scheduled; ENG-1483's Linear relay is misaligned and pending removal; ENG-2082 (prod 502) is mis-parked in Backlog.
