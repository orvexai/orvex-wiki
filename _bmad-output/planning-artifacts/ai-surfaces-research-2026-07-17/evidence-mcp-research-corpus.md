## Prior-art anchor — 10 already-settled conclusions a fresh-goggles redesign must build on, not re-derive

1. **The tool-surface architecture is decided: Architecture B, domain-namespaced intent verbs, hero-13 ceiling.** Ruled R-WDS-1 by the PO on 2026-07-17, chosen explicitly over an ask-first-only architecture and a code-mode/execute() architecture (both fully specced as alternatives and rejected — not overlooked). Re-opening this is a real regression risk, not a fresh contribution, unless there's new evidence the ruling session didn't have.

2. **The MCP's product role is bidirectional-lite, and this resolves a genuine multi-week canon conflict.** Four contradictory prior framings (Vision: bidirectional; Librarian brief: write-only save door; addendum: universal fallback; Architecture Spine: read-only retrieval) all had "canonical" status simultaneously and code implemented all four at once. R-WDS-1 is the single ruling that ends this — any redo re-litigating "what should the MCP even do" without citing this history will re-create exactly the ambiguity that was just closed.

3. **Naming/envelope/error-taxonomy conventions are a governed design-system authority, not implementation detail.** `save_X(id?)` upsert consolidation is a binding naming rule; the envelope is plain-text-authoritative with `structuredContent` as an optional typed mirror, never a JSON dump; the error taxonomy (`QUOTA_EXCEEDED`, `VERSION_MISMATCH`, `AI_UNCONFIGURED`, `ABORTED_UNCERTAIN_STATE`, `READ_ONLY`, `NOT_AVAILABLE_YET`) is fixed with actionable `recovery_hints` on every response. This is a same-day-ratified spec, not a draft convention.

4. **Writes default ON with a hard READ_ONLY ceiling override, not a dark-by-default door (R-WDS-3).** This deliberately reverses the shipped `WRITES_ENABLED=false` default and composes the strongest single elements from Supabase/Slack/GitHub's write-safety patterns (dry-run + space-allowlist + confirm-token + CAS receipts + a hard ceiling flag), assessed as the strongest autonomous-write posture in the entire field survey.

5. **The three-repo E2E is the first Definition-of-Done anchor — and it went GREEN for the first time in any era today, 2026-07-17T12:48Z.** identity→MCP→wiki-api/ai→knowledge, with a real headless-minted bearer, 6/6 legs. This single fact changes the risk profile of any redo substantially: "never proven end-to-end" was true as of this morning and is no longer true as of this afternoon.

6. **Cited semantic `ask` is a genuinely uncontested competitive lead, confirmed twice three weeks apart.** The June 2026-06-25 50-server survey found no incumbent wiki/KB MCP ships a cited semantic ask; the 2026-07-17 benchmark re-confirmed it as the field's "biggest unclaimed differentiator." A redo does not need to re-research this claim — it needs to protect and extend it.

7. **A per-tenant Tenant Governance Profile (memory-write mode / sharing grants / publish path, all human-gated by default) is a new, deliberately cross-cutting design-system concept** introduced specifically to resolve the pattern that recurred across seams 2, 8b, and 8c: the PO's answer was consistently "configurable per tenant, human-gated default" rather than a single global rule. Any future governance/permissions design should extend this concept rather than inventing a parallel one.

8. **Progressive disclosure is being deliberately migrated off a fragile hand-rolled mechanism onto spec-native primitives.** `tool-visibility.ts`'s SDK-internals reach-through is explicitly named "the most fragile code in the repo" and is being replaced by `tools/list_changed` + reliance on the platform's default-on Tool Search Tool (~85% overhead reduction, independently validated by Anthropic's own measurements). Any new tool-discovery design should assume this migration, not the old mechanism.

9. **Knowledge routing is direct (MCP→knowledge, bypassing wiki-api) for search/related/RAG — this is settled (D-M8/PO-ruling-9/ENG-1403) but the family-root canon document currently still says the opposite and needs a correction ticket.** This is flagged explicitly as a live, unresolved documentation bug — worth carrying forward as a known gap rather than re-discovering it as new.

10. **The golden-tape KPI (≤2 calls / ≤~1k response tokens per common task, split from a separate ≈4.5–5.5k one-time connect-schema cost) is the project's own numeric efficiency constitution, and it has just been rebuilt as an enforced CI gate today (12/12, `kpi-golden-tape` required check) after being deleted by the 07-16 rebuild.** No competitor in the field survey publishes a numeric, CI-enforced ceiling like this — it is a genuine differentiator, but was unenforced for three weeks and is only real again as of today. A redo should treat this ceiling as the standing efficiency bar to measure against, not a number to re-derive.
