### Product Brief: Orvex Studio (`rgBOQh31p3`) — canonical, the umbrella brief
- North star: "your AI finally knows you." Three-layer context system: **Memory** (curated facts, RAM), **wiki** (kept content, HDD), **beads** (volatile agent narration, owned by `orvex-studio-workgraph`, ADR-0025). RAG/Turbopuffer is a retrieval *mechanism*, never a fourth store.
- Knowledge loop: intake (MCP save / beads monitoring / external chat-history import) → Librarian curation (propose-and-confirm) → the three stores → Prompt Composer consumption (marketplace skill + tweaks + Memory + wiki via RAG; now also task-first wizard-driven, ruled 2026-07-13) → outbound sync to native vendor memories (free tier; Claude has an adapter, ChatGPT is platform-blocked).
- **Delivery Doctrine** (the umbrella's closest analogue to a "delivery plan"): (1) contracts-first — resolve `orvex-studio-contracts` ownership before any service starts against prose; (2) build the two shared chokepoints (`orvex-studio-lib` auth verifier + the Go↔TS bridge) before anything else; (3) isolated builds against pinned contract fixtures, services tier-agnostic (consumer/business/enterprise differ only by UI + entitlements); (4) continuous integration on a standing dev cell, "Done" = an observed pass on the six-surface protocol, never a reported one; (5) per-service AND per-seam testing (api/mcp/cli/ai/rag/knowledge-sync = the acceptance floor); (6) prod cutover last, only after all six surfaces are green on dev with real data; (7) launch readiness is a separate gate (Complexity + Observability from the Readiness Assessment).
- State as of 2026-07-13 (self-reported): design canon mature (31/32 family ADRs canonical); satellite code real on one dev cell; identity furthest along (M9 closed); wiki-api Phase-0/1 facade live; knowledge wired to Turbopuffer; **mcp at ~19/73 tools**; production still runs vanilla Docmost with Orvex modules OFF; "no surface has a green, human-verified, real-data end-to-end pass"; contracts repo ~90% unbuilt/unowned; shared auth verifier is a deny-by-default stub.
- Pricing canon (locked 2026-07-13): free-forever individuals (200 pages/1 GiB/2,000 files/25 members cap), consumer AI **£7/mo**, Teams £70 framing, GBP-only, hidden Enterprise; trial = 7-day, card-required, downgrades to free at end (this specific point CORRECTED again 2026-07-17 — the earlier "free month, no card" framing is retired).
- Audience ruling: "for everyone non-technical" is the story; **teachers are the launch wedge** (supersedes the Vision doc's estate-agents/SMB-marketing Phase-1 paragraph — re-ratification of the Vision is owed but not yet done).
- Research phase (immediate next step, no timebox): 18 prioritized open questions in three tiers (P0 blocks the doctrine — contracts ownership, prove the dev cell E2E, build lib, decide the Go↔TS bridge; P1 ownership rulings; P2 pin-before-cutover items).

## 3. Bindings on the AI-surfaces redo (MCP / wiki-api / CLI)

Concrete commitments in this space that constrain or gate the api/mcp/cli PRD-architecture redo:

1. **MCP's memory role is fixed as read-only retrieval, not delivery.** AD-5b (Architecture Spine) and the Fold-in Map both state plainly: MCP exposes memory as a **knowledge-retrieval tool call under I-4** (ACL ∩ caller's delegated token-scope); MCP is explicitly **"not a compose adapter."** Any MCP verb design for memory must stay within read/retrieval semantics — the write/compose/delivery path is a *separate, new, currently-unowned* browser-extension component (F1/AD-5a), not MCP.
2. **No layering shortcuts — CLI must ride wiki-api's `/v1` grammar.** The six-surface acceptance protocol (Addendum §8) defines the cli acceptance test as "byte-faithful DfM round-trip through the `/v1` grammar" — i.e., CLI → wiki-api, never CLI → engine directly. This matches (and is the concrete evidentiary source for) the standing "no fallbacks / no layering shortcuts" memory rule.
3. **A named MCP tool-parity target of 73 exists in canon**, distinct from and dated earlier than the "19/19 live-green" claim reported outside this space (see Staleness flags below) — the redo needs to reconcile which number (19 intent-verbs vs. 73 tool-parity-with-wiki-api) is the actual target surface.
4. **Cross-service interaction is contracts-only (I-1).** Every new MCP verb, wiki-api endpoint, or CLI command that touches Memory must be pinned in `orvex-studio-contracts` (AD-11 names card schema, DeliveryAdapter port, and "MCP memory additions" explicitly as contracts-pinned artifacts) before build — "no service starts its isolated build against prose" (Delivery Doctrine §1).
5. **Retrieval is always principal-delegated (I-4).** Any MCP tool or wiki-api endpoint that reads memory/knowledge must carry the caller's delegated principal through to `knowledge`'s ACL∩scope enforcement — this is a hard tenancy/authz invariant for the redo's contract design, not an implementation detail.
6. **Consent/tenancy tokens are minted by `identity`, not composed client-side (AD-8).** Any delivery-adjacent MCP/API surface dealing with private memories needs a short-TTL, single-target, single-use token model — the trust boundary is the mint step, never the calling client.
7. **Quotas and versioning are cross-cutting acceptance requirements**, not features to skip: `ifVersion`-based optimistic concurrency with 409-on-stale (api surface), and 402 `QUOTA_EXCEEDED` on both REST and collab paths, are named in the six-surface protocol as baseline acceptance criteria any redo must preserve.
8. **Governance discipline (draft→canonical, ratify tokens) governs the PRDs/architecture themselves.** The Memory PRD/Architecture explicitly show the fold-in workflow (PRD → Architecture → fold-in map → service PRDs → delivery-phase update) as the template sequencing the AI-surfaces redo should also follow, and name an unresolved **cross-canon conflict** (AD-1 vs. the `ai` architecture doc `Cb0XBkNezd`) as a still-open example of exactly the kind of contradiction the redo must not introduce or leave unflagged.
9. **Tier-agnostic services / entitlements-not-forks.** The Product Brief's delivery doctrine states services must stay tier-agnostic — consumer/business/enterprise differ only via UI + billing entitlements, never via service forks. Any per-tier MCP/API/CLI behavior (e.g. free-tier direct-write vs. paid-tier curated confirm) must be modeled as an entitlement check, not a separate code path.
10. **Free-tier direct-write lane (D11, 2026-07-17) is a brand-new machine-write mode** that any MCP/API save/write verb design must account for: on Free, machine writes may land `unreviewed`-flagged and provenance-stamped without going through the staging confirm gate, gated by a single global user-level toggle — this is a live, very recent change to the write-path contract that postdates (and is not yet reconciled with) AD-4's "confirm gate is the sole mutation path" architecture text.
11. **Beachhead accessibility bar.** FR-D2/FR-O4 set a concrete, testable non-technical-user bar ("≤3 taps, ≤2 minutes, no config file, no JSON, no command line" to connect an assistant) that any redo of MCP/CLI onboarding or auth flows should be measured against, even though CLI itself is a technical-user surface by nature — the bar applies most directly to whatever the eventual delivery-client/extension does, but signals the family's general anti-friction bar.
