## 2. Deeper summaries — load-bearing pages

### Systems Architecture (`pTeVSp9P5f`) — canonical, the family map hub
- Frames the family as two thin product shells (wiki = engine+wiki-api; Studio = studio-ui+studio-api) over one shared platform of satellites (identity/ai/knowledge/mcp/console/billing/workflows/staging/workgraph/audit), driven by a single CLI.
- Two binding rules restated: Principle 1 "primitives stay in, brains move out" (engine keeps only write-chokepoint/ACL/lifecycle/export/session-consume); Principle 2 "events are the integration surface" (every mutation → same-txn outbox row → relay → Kafka).
- Full "Our Systems" table gives per-service build-state badges: only `orvex-wiki` (live, vanilla), `orvex-wiki-api` (partial), `orvex-studio-mcp` (partial), `orvex-studio-identity` (partial), `orvex-studio-workflows` (partial) are non-scaffold; audit is `planned`; everything else `scaffold`/`pre-build`.
- Explicit honesty flag: only mcp, workflows, and platform substrate carry real running code as of the page's last sync — this is now stale (see Staleness §).
- `orvex-studio-contracts` and `orvex-studio-lib` are drawn as build-time seams (dashed), never runtime hops.

### ADR-0007 — CloudEvent envelope (`pYkl7HpOgr`) — canonical
- Single structured-mode JSON envelope, required set `[specversion, id, source, type, orvexcell, orvextenant]`.
- `orvexcell`/`orvextenant` are the mandatory extensions; `partitionkey` = tenant id for per-tenant total ordering (topics are single-partition).
- `version` (monotonic per-document) lives inside `data`, never on the envelope — same counter as the engine's `ifVersion` CAS.
- Explicitly rejects mirroring the token's `cell`/`cell_epoch` wire-names onto the envelope (CloudEvents attribute-name constraints forbid the underscore) — that's exactly the gap ADR-0030 later closes on the Principal instead.

### Event-Driven Architecture (`z5K777kNkD`) — canonical, featured cross-cutting page
- The full narrative: transactional outbox (same-txn insert) → relay (sole publisher) → `studio-spine` Kafka → Knative Trigger consumers. At-least-once; consumers must dedupe on CloudEvent `id`.
- Cell fence (`orvexcell`) is fail-closed on mismatch — an event from the wrong region is dropped/dead-lettered, never processed.
- Thin vs rich payload profile split; rich-payload consumers must re-verify/re-resolve rather than trust the envelope.
- Two sanctioned raw in-process Kafka consumer exceptions to the default Knative-Trigger rule: MCP's revocation subscriber (ADR-0013) and orvex-studio-audit's ingest consumer (ADR-0037, AD-2) — both always-on services where scale-to-zero buys nothing.
- Source-adapter pattern: a new content publisher registers event types + a content resolver + a fail-closed ACL primitive + purge events, and knowledge indexes it through the same pipeline with zero pipeline changes. Workgraph (ADR-0027) is the first satellite to use this.
- Event catalog table given (ai.*, wiki.*, billing.*, studio.{skill,memory,chat,staging,workgraph}.*, audit.<category>.<verb>).
- Build-state-as-of-last-sync: spine provisioned+Ready but idle — the engine's outbox/relay is NOT BUILT, zero events emitted; this is now stale per the R21/three-repo-E2E evidence (see Staleness).

### ADR-0011 — Tenant-to-cell registry (`qZFmNWGY47`) — canonical
- The platform's ONE sanctioned global singleton (allow-list: `{registry, Temporal, edge, Stripe front-door, Clerk, CDN}`).
- identity is sole writer; two-store split by sensitivity (PII-free routing core, replicable/KV-mirrored vs. an in-jurisdiction sensitive shard).
- Discovery contract: discover-once-then-pin; `421 Misdirected Request` → exactly one re-discovery + retry; a tenant-move bumps `cell_epoch`.
- Cell claim is carrier-not-owner: mirrored into Clerk org/user metadata + a Keycloak mapper; lives ONLY in short-TTL session/access tokens, never in long-lived bearer/MCP tokens.

### ADR-0035 — Go↔TS contract/client bridge (`QbEBPuKcGR`) — canonical
- Per-repo codegen is the family rule (generalizing mcp's already-proven pattern): `openapi-typescript`+`openapi-fetch` for REST, `json-schema-to-typescript`+ajv for CloudEvents/JSON-Schema.
- Generated code lives PER-REPO, checked in — never a published `@orvex/openapi-clients` package (keeps contracts repo non-building, keeps the diff visible in the consuming PR).
- Codegen source is always a local checkout of authored artifacts (`ORVEX_STUDIO_CONTRACTS_DIR`), never a live/served descriptor — draft artifacts may be generated against for dev, but a GA cutover must pin an `x-status: pinned` artifact at a released tag.
- Regen is a reviewed PR with a CI determinism gate; additive bump = automated lane, breaking = ADR lane (slots under ADR-0008).
- POC's hand-authored zod `@orvex/shared` vocabulary is replaced tag-by-tag as BFF contracts land, never lifted wholesale.

### ADR-0037 — orvex-studio-audit founding (`Z6WrJjZDAO`) — canonical, ratified 2026-07-16
- Context: 5 services (wiki, wiki-api, cli, console, identity, +ai) each built a diverging audit answer; console's `ENG-1968` KafkaRecorder is the only real durable one and had already been WRONGLY assumed elsewhere (identity's own docs) to be the family-wide sink.
- Decision: new component `orvex-studio-audit`, consumer-only off the already-live `studio-spine` Broker (reuse, no new transport); `wiki-api`'s `/v1/audit` forward target becomes this new service (resolves OQ-A13/OQ-CLI3); console's `BrowseAudit` becomes a client of audit's read API (its own producer path is unaffected).
- Store engine & retention RESOLVED by companion PO design: same-stack tiered store (Postgres hot, Rook-Ceph WORM Parquet/NDJSON in Compliance-mode Object Lock, Glacier DR client-side-encrypted offsite); per-record hash-chain + hourly OpenBao-signed checkpoints (CloudTrail integrity pattern, NOT a ledger DB — immudb/QLDB/Trillian rejected); uniform 7-year retention floor (raised from 6, HIPAA-anchored); legal hold overrides; per-subject crypto-shred for GDPR erasure; seal runs as a Temporal workflow.
- Companion page `orvex-studio-audit` (`WoKjHEJlZA`, also canonical) gives the full architecture: hexagonal ports/adapters, raw in-process Kafka consumer (2nd ADR-0013-style exception), `/v1/audit/query`+`/export` API, `/internal/v1/audit/seal` step-API, always-on ingest Deployment + KPA-scaled query API.

### ADR-0033 — Work-claim arbiter (`yNFx3YyNap`) — canonical
- Ratifies Linear-status-as-claim (`In Progress` flip via linearis, read-back verified) as the delivery-engine work-claim mechanism — NOT a lock, but structural safety via four invariants (G1 single orchestrator, G2 single authoritative Linear writer w/ P1 auto-close guard, G3 provably-disjoint project partitions w/ preflight assertion, G4 intra-engine `claimedIds` guard).

## 3. Bindings on the AI-surfaces redo (MCP / wiki-api / CLI)

**Contracts & layering**
- ADR-0035 fixes the codegen mechanism for every TS satellite (mcp, api, ui): per-repo codegen from a local `orvex-studio-contracts` checkout, checked-in output, pinned tag + determinism CI gate. Any new MCP tool surface or CLI verb that crosses a contract boundary must land as an authored OpenAPI/CloudEvent artifact in contracts FIRST, tag-pinned, before client generation — never generated from the AGPL engine's served descriptor (D-CON-8/agpl-guard boundary), and never off a `draft` artifact for a GA cutover.
- ADR-0008 (change-authority, referenced throughout, not itself in this slice) governs which contract changes are additive/automated vs. reshaping/ADR-gated — binding for how the MCP/API/CLI redo evolves its own contract surface.
- orvex-studio-contracts page (`XZdd14uEFs`): the seam repo owns OpenAPI (incl. `wiki-api.yaml`, the MCP/CLI codegen source), CloudEvents catalog, SSE wire contract, frozen ~58-code error vocabulary + CLI exit-code buckets, golden fixtures — MCP and CLI must consume these, never invent parallel vocab.
- ADR-0021 (referenced, not in this slice) — wiki-api edge caller-token pass-through — is the pattern ADR-0029 item 4 extends to MCP's own egress: MCP forwards the caller's identity-minted bearer verbatim to wiki-api/ai, never mints or holds a service credential.

**Auth / tenancy**
- ADR-0029: MCP's edge verifier must be `IdentityJwtVerifier` (multi-issuer JWKS + deny-list + TTL-bounded introspection, fail-closed past the bound); session-identity rebind keys on `tenant:subject`, never `sid` (frozen Principal has no `sid` field, by design).
- ADR-0030: `cell`/`cell_epoch` are now legitimate OPTIONAL fields on the frozen `Principal` — any AI-surface redo reading routing claims off the Principal must read `principal.cell`/`principal.cell_epoch`, never invent a side-channel.
- ADR-0011: cell claim is carrier-not-owner and lives ONLY in short-TTL tokens — a redesigned MCP/CLI auth flow must never assume a long-lived bearer or MCP token carries `cell`.
- Clerk page: only identity talks to Clerk directly; MCP/API/CLI verify via `orvex-studio-lib/auth`, never call Clerk itself; tokens are frozen-at-mint (never re-rendered from live session/active-org) — a CI test proves this; any new token-minting surface in the redo must respect that invariant.

**Verb grammar / routing**
