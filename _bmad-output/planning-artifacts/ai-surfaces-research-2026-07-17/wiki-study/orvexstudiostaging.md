## 2. Deeper summaries — load-bearing pages

### PRD: orvex-studio-staging (mOzNDhn322) — canonical

- **Core bet:** agents get exactly one write verb into customer knowledge — `propose` — never direct wiki writes. Proposals group into per-session ChangeSets; the Librarian (AI reviewer, per-tenant prompt + autonomy dial) triages, routes (find-before-create), detects conflicts, beautifies, and applies under human review by default.
- **Autonomy Dial** (FR-STG10): `recommend` (default) → `auto-apply-low-risk` → `full-auto`; destructive ops (`replace-document`, `delete-document`) always human-gated below `full-auto`; all three positions ship in v1 (PO decision 2026-07-10, nothing deferred).
- **Conflict handling** (FR-STG9): base-version stamped at submit, re-checked against the authoritative wiki-api/ydoc read (never the engine's ≤45s-stale content column); never a silent three-way prose merge — conflicts always go to a human.
- **Apply engine** (FR-STG13): sanctioned wiki write path only (ProseMirror JSON/DfM, CAS `--if-version`), ChangeSet-level atomicity is a workflow-state gate not cross-store ACID; a paused mid-apply ChangeSet forces explicit human resume/abort.
- **Beautification-at-triage** (FR-STG14): the Librarian re-authors content BEFORE review so the rendered diff IS the final content; apply performs mechanical writes only, keeping LLM latency out of the apply SLO.
- **Platform surfaces** (FR-STG23-25): new `staging_*` MCP tools (`staging_propose`, `staging_changeset_submit/status`, `staging_list_mine`, `staging_withdraw`, `staging_review_queue`, `staging_decide`, `staging_apply`), new `orvex-cli staging` namespace. **Hard-cut migration of ALL agent wiki-write surfaces** — MCP `save_page`/`edit` re-point to emit Proposals; CLI wiki write verbs get 403'd at wiki-api ingress for agent-class credentials. Sequencing invariant: the hard cut is the LAST release step, gated on staging working end-to-end against non-501 `orvex-studio-ai` and `orvex-studio-knowledge`.
- **Events** (FR-STG26): `studio.staging.*` CloudEvents are telemetry only; wiki reindexing rides the engine's own `wiki.*` events (single mutation source of truth) — staging events never trigger a second reindex. Engine `wiki.*` outbox becoming live is itself a gating prerequisite.
- **Trust Tiers + learning loop** (FR-STG11, 15-17): per-producer acceptance history gates auto-apply eligibility; every accept/reject/edit is a Feedback Event that tunes future routing (exemplars + tier recalibration + Librarian-proposed prompt-pack self-revisions, never self-applied).
- **Supersedes** OPS "The Librarian"/Card Contract v1 and `orvex-studio-api`'s FR-SA12-16 Curator — Proposal is a strict superset of Card v1 (verbatim field-authority transfer).
- **NFR-STG5** names `orvex-wiki-api` as the enforcement chokepoint: the ONLY write ingress to the wiki; engine direct-write routes network-fenced; wiki-api rejects agent-class credentials with 403 on write verbs.
- Service: new platform service `orvex-studio-staging`, Go, six-tier family shape, own Postgres — `orvex-studio-api`'s Curator becomes a client of it.

### Architecture: orvex-studio-staging (GOacy9kdJz) — canonical, 14 architecture decisions (AD-1..AD-14)

- **AD-1**: one service — staging = Librarian, never a second deployable/contract surface. Binaries: `cmd/api`, `cmd/triage`, `cmd/relay`, `cmd/sweep`.
- **AD-2**: state machines are the only mutation path (CAS `UPDATE...WHERE state=$expected`, transactional with outbox).
- **AD-3**: conflict keying on `(document_id, section_anchor)` via stable engine block IDs (never heading text); base-version = wiki-api's `ifVersion` counter verbatim.
- **AD-4**: apply engine calls `orvex-wiki-api` exclusively (never the engine directly); per-(tenant,page) lease serializes concurrent ChangeSets; agent-class tokens 403 at wiki-api ingress.
- **AD-5**: durable orchestration lives centrally in `orvex-studio-workflows` (Temporal) — no satellite Temporal worker; staging exposes idempotent `/internal/*` step-APIs only.
- **AD-6**: all LLM calls go through `orvex-studio-ai`; duplicate/routing candidates from `orvex-studio-knowledge`; `divert-to-workgraph` targets `orvex-studio-workgraph` via typed client, never the wiki path.
- **AD-7**: the Autonomy Dial is a server-side domain verdict computed by the `policy` package from (dial position)×(Operation risk class)×(Trust Tier)×(routing confidence) — surfaces (MCP/CLI/UI) never compute it themselves.
- **AD-8**: tenancy/credentials — every request verified via `orvex-studio-lib/pkg/auth`; Trust Tiers key on the *verified* principal, never a body-supplied id; RLS ships in baseline migration.
- **AD-9**: `studio.staging.*` events are telemetry only; `wiki.*` (engine outbox) is the truth for reindex — never dual-triggered.
- **AD-10**: Proposal bodies/snapshots in S3 under `{tenant_id}/` keys; queue rows carry pointer + sha256 hash only.
- **AD-11**: quota verdicts are a domain fn against billing-owned entitlements; frozen 402 `QUOTA_EXCEEDED`, never 429/destructive.
- **AD-12**: supersession is verbatim (Card v1 → Proposal), the cut is loud (no dual-write, no shim).
- **AD-13**: beautify happens at triage, stored as an immutable sha256-addressed artifact; apply writes that exact artifact by hash, never re-authors.
- **AD-14**: learning is captured/versioned/admin-gated; a pack revision activates only through admin approval.
- Dependency-direction diagram: MCP/CLI/workflows → API → domain → {store, event, apply}; domain → {ai, knowledge, workgraph, billing}; apply → wiki-api → engine → `wiki.*` → spine → knowledge. Forbidden edges: staging → engine DB/HTTP directly, staging → any sibling's Postgres, ai/knowledge → staging store.

## 3. Bindings on the AI-surfaces redo (MCP / wiki-api / CLI)

These are hard constraints from this space that the MCP/API/CLI PRD-architecture redo must respect or actively design against:

- **The staging area is meant to become the sole agent write path into the wiki.** Per canonical PRD (FR-STG25, NFR-STG5) and Architecture (AD-4, AD-12): current MCP `save_page`/`edit` and current `orvex-cli`/`docmost-cli` wiki write verbs are slated for a **hard cut** — agent-class credentials rejected 403 at `orvex-wiki-api` ingress, redirected to `orvex-cli staging`/`staging_*` MCP tools. **This is not yet live** — it is sequenced LAST, gated on staging's own apply path being non-501 and on `orvex-studio-ai`/`orvex-studio-knowledge` being non-501. As of 2026-07-15 staging's apply path (`internal/apply`) is itself a `doc.go` stub — the gate has not opened. The redo must NOT assume this cut has happened, but MUST design the new API/MCP/CLI surfaces so they are the correct target of that eventual re-pointing (new `staging_*` tool names, new `orvex-cli staging` namespace already specified in canon — FR-STG23/24).
- **`orvex-wiki-api` is named the ONLY write ingress to the wiki** (NFR-STG5) — a chokepoint responsibility the wiki-api PRD/architecture redo directly inherits: it must enforce credential-class-based 403 on write verbs, become the authoritative CAS/version-read path (not the engine's debounce-lagged ≤45s-stale content column), and support per-page block ops + `ifVersion` CAS that staging's apply engine depends on verbatim.
- **New MCP tool surface staging expects to exist**: `staging_propose`, `staging_changeset_submit`, `staging_changeset_status`, `staging_list_mine`, `staging_withdraw`, `staging_review_queue`, `staging_decide` (bulk), `staging_apply` — registered as a hidden `staging` category via `list_tools`, before `installToolVisibility`. Machine-to-machine auth must be per-agent, never a shared service credential. This is a concrete naming/shape precedent the MCP redo should reconcile against (does the "amazing MCP" 19-tool surface already include or plan for these staging verbs, or are they additive later?).
- **New CLI namespace expected**: `orvex-cli staging <verb>` (`propose`, `changeset submit/status/list`, `review`, `decide`, `apply`) — parity with the MCP surface, for CI and human power users. The CLI redo must reserve/plan this namespace.
- **Content payload contract**: DfM or ProseMirror JSON only — lossy formats (raw markdown/html) rejected at the door across every write surface (staging AND presumably the underlying wiki-api contract). Schema-validation happens once at intake; unregistered ProseMirror nodes are silently stripped by the engine today — a known landmine for any write-path design.
- **Idempotency-Key required on all `/internal/*` and public write verbs** (cell-lint R11, AD-5, AD-11) — a pattern the wiki-api/MCP/CLI redo should adopt consistently, not just for staging.
- **Quota/429 doctrine**: quota exhaustion is ALWAYS a 402 `QUOTA_EXCEEDED` from the frozen `Error`/`ErrorCode` vocabulary — never 429, never a silent/destructive drop. Binding precedent for any admission-control design in the API/CLI/MCP redo.
- **Tenancy**: polymorphic `{user|org}` tenant everywhere (ADR-0001/0012) — an org-only assumption is treated as a bug across the family; applies equally to any new API/MCP/CLI surface.
- **Events**: any new `studio.<service>.<resource>.<past-tense>` CloudEvent types must be authored in `orvex-studio-contracts` FIRST (additive lane) before being emitted; wiki reindexing must ride the engine's own `wiki.*` events, never a duplicate trigger from a satellite's own event stream — a pattern directly relevant if the API/MCP redo introduces its own eventing.
- **Observability**: PII/content (Proposal/page content, end-user text) must NEVER appear in spans/logs/metrics labels; OTel via `pkg/obs`; troubleshooting surface is `orvex-studio-console`, not Grafana — same constraint likely applies to the redone MCP/API/CLI telemetry.
- **No-fallbacks doctrine, concretely broken today** (D-STG2, config silent defaults) — the redo's implementers should treat this as a warning: silent env-var fallbacks are exactly the anti-pattern the family forbids and this space shows a live violation of it.
