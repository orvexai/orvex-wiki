## 2. Deeper summaries — load-bearing pages

### Architecture: orvex-studio-ui (`DmJsnB5Z9Y`, canonical, ratified 2026-07-06)
- Thin Vite+React+TS SPA; canon end-state = Cloudflare Workers Static Assets served by the Studio Worker (global, region-agnostic); Alpha interim = nginx Deployment (OQ-UI4 open).
- A-SHELL: one upstream only, `orvex-studio-api` (BFF) over `/api`; two true-external egresses (Clerk Frontend API, Stripe hosted Checkout/Portal) — no direct satellite/engine/model-provider calls, ever.
- A-STREAM: AI arrives as SSE proxied by the BFF; wire frames/DTOs pinned in `orvex-studio-contracts` — the SPA is a conformant consumer, not the contract author.
- A-DRAFT: every AI output renders as a proposal requiring explicit human accept before showing as committed (mirrors BFF authority model).
- A-DS: Design System token fidelity + WCAG 2.1 AA (both themes) is a hard ship-gate (CI unlanded at audit time).
- A-STATELESS: no business logic/store/secrets; the token attached to `/api` is the **identity-exchanged, cell-bearing session token** (not the raw Clerk JWT) — the Cloudflare Worker routes by the `cell` claim at the edge, before the BFF.
- A-GLOBAL-URL: single global URL (prod orvex.ai / dev orvex.dev / crew crew-{name}.orvex.dev); SPA is region-agnostic; Worker resolves tenant→cell and proxies to grey-cloud DNS-only per-cell origins.
- A-AUTH-UI: Clerk session UI rendered client-side, but authorization always server-enforced; principal is **polymorphic {user|org}** (D-S17 — personal user = user-keyed, NO Clerk org; only Teams are Clerk orgs); entitlement/AI-cap/wiki-quota verdicts are server truth including engine's F-QUOTA 402.
- 3 ADR triggers named (serving topology, Clerk/Stripe external dep, session-token auth-flow) but **cannot be numbered** — Studio Decision Records registry is still TBD (open decision, CS §9).
- Explicitly says (§3) surfaces are design targets not claimed-built, repo is single-commit scaffold — **this framing is stale**, superseded by the Amendment draft and SDD (see Staleness).

## 3. Bindings on the AI-surfaces redo (MCP / wiki-api / CLI PRD-architecture)

These are commitments from `orvexstudioui` that constrain or inform the concurrent redo of the AI-facing surfaces (orvex-wiki-api, orvex-studio-mcp, orvex-cli):

- **Layering discipline (A-SHELL) is the strongest transferable pattern**: the Studio SPA talks to exactly ONE upstream (its own BFF) and treats every other service as invisible; direct-to-satellite calls are architecture violations enforced by CSP + code review. This is the same "no layering shortcuts" doctrine the family memory (`new-vision-no-legacy-no-shortcuts`) states for wiki-api /v1 vs direct-to-engine — the UI space is independent confirmation that this is a family-wide, not local, rule.
- **Contract pinning discipline**: SSE/DTO wire shapes are pinned in a shared contracts repo (`orvex-studio-contracts`) and consumers (TS satellites, per ADR-0035) get **generated clients**, never hand-authored types or Go stubs for TS satellites. If MCP/CLI/wiki-api generate clients for each other or for UI-facing consumers, the same generated-client-not-hand-rolled discipline should apply, and CS §7 "front types match the pinned contract exactly, no carve-outs" is the enforcement bar.
- **Entitlement/quota as a single SSE channel, server truth only**: the UI never computes entitlement or quota — it renders server verdicts, including the wiki engine's F-QUOTA 402 on writes. Any MCP/API tool exposing writes into the wiki must originate the same 402/QUOTA_EXCEEDED-style server truth; the UI pattern (single push channel + refetch-on-gesture fallback + bounded TTL) is a candidate pattern for how MCP-driven or CLI-driven long sessions should learn about quota/entitlement changes without polling storms.
- **Draft-vs-committed (A-DRAFT) as an AI-authorship invariant**: "AI proposes, human accepts" is implemented as one shared rendering layer across every AI surface in the UI. This is directly relevant to the MCP/API redo's own design-verb semantics (per memory `R21 MCP streaming folded into 19-tool surface`): any MCP/API write-capable verb that originates from AI should have an explicit accept/commit step surfaced back to the UI (or CLI) layer — the UI cannot itself decide something is committed; the API/MCP layer must emit that distinction as part of its response contract (proposal vs applied), not leave it to client-side heuristics.
- **Tenancy/session token model**: the SPA attaches the identity-exchanged, cell-bearing session token (not a raw Clerk JWT) to every `/api` call; the Cloudflare Worker reads the `cell` claim at the edge before the BFF. Any MCP/CLI auth flow that ultimately serves the same Studio users should mint/attach the same cell-bearing token shape — a raw IdP JWT without the cell claim breaks routing. This is a concrete token-shape constraint the MCP/CLI redo should verify against, not assume.
- **Governance / ADR mechanics**: the UI space is blocked on a **Studio Decision Records parent + numbering registry** (CS §9) that still does not exist — three named ADR triggers (serving topology, external-dependency, auth-flow change) sit unfileable. If the MCP/API/CLI redo also needs ADRs (e.g., for its own auth-flow changes, new external dependencies, or contract-breaking reshapes), it will hit the **same missing registry** — this is a shared, currently-open blocker across at least two Studio service spaces, worth escalating once rather than rediscovering per-space.
- **"Looks good AND works" / delivered bar**: for surface-owning services (like UI) delivered requires 7 evidenced gates ending in a human delight-check; CI-green/ArgoCD-Healthy/closed-ticket are explicitly non-evidence. The AI-surfaces redo (MCP/API/CLI) are not surface-owning in the same visual sense, but the analogous discipline — "19/19 tools green on dev" is a weak proxy exactly like ArgoCD-Healthy — is the same anti-fake-done doctrine already captured in family memory (`evidence-signals-are-not-observation`, `certified-is-not-current`). This UI space is a second, independently-authored instance of that exact epistemic trap (D-UI1 Hollow-Healthy), reinforcing it as cross-service canon, not a one-off lesson.
- **No UI-authored contract**: the UI explicitly authors zero OpenAPI/events/store and states this is deliberate, not a gap — a template for how the MCP/CLI PRDs should describe their own "what we do NOT author" boundaries relative to wiki-api, avoiding scope creep into each other's contract ownership.
- **Reviewer≠author + never-self-ratify discipline**: every Wave-4 draft artifact (PRD-delta, Contract Summary, SDD, Test Plan, Build Prompt) explicitly states "never self-ratify," human-ratify downstream — the same discipline the MCP/API/CLI redo's own PRD/architecture packs must carry if produced via the same Definition-Factory process.
