## 3. Bindings on the AI-surfaces redo (MCP / wiki-api / CLI PRD-architecture)

Constraints and commitments load-bearing on the redo of orvex-wiki-api, orvex-studio-mcp (intent-verb MCP), and orvex-cli:

**Contracts / verbs / layering**
- **Domain-pure routing is locked (D-S16/D-S18, A-APIBOUNDARY):** ALL wiki-shaped programmatic access — page CRUD, history/diff/version, bulk import/export, binary attachments, comments — must route through **wiki-api**, never the engine directly. The engine's raw API is internal-only; only the React client is exempted. The engine ships **no** `/docs`/Swagger endpoint — OpenAPI lives on wiki-api.
- **The engine exposes exactly one atomic write primitive**: `apply-ops` (typed PM-JSON ops + `ifVersion` CAS, atomic-or-409). wiki-api composes DfM→ops, block-patch grammar, heading-resolution, section-edit, and opaque-reattach ON TOP of this single primitive — it is NOT itself a second write path.
- **wiki-api owns:** verb grammar (get/search/list/save/edit; `ask` delegates to ai), block-patch composition, Markdown↔DfM convert/fidelity (`check_dfm_fidelity`, `/v1/convert/*`), drift-verify + wiki-first spec-gate (moved here from mcp per D-S8 — engine keeps only the `verified_against`/`verified_at`/`spec_confirmed` stamp fields), oversized-page content-handle (must be Redis-backed multi-node, not in-memory).
- **mcp is a stateless agent front-door with NO database**, re-pointed from in-process NestJS injection to REST clients (engine / wiki-api / knowledge / ai). `ask` delegates to ai, not wiki-api. `related_pages`/`search_semantic`/`search_hybrid` → knowledge. `generate_image`/`insert_excalidraw` → ai. The **live/current MCP tool count context** (per user memory, 2026-07-16/17) is 19 tools live-green on dev — the 2026-07-05 study's "73-tool fork parity target, WS-16 gap" framing is stale (see Staleness flags).
- **DfM import boundary (A-SEAMS, binding, corrects an earlier PRD error):** the AGPL TypeScript `@orvex/dfm` package is imported **ONLY** by the AGPL engine.
