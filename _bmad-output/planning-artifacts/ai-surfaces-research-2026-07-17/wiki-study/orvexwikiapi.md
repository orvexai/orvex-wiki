## 2. Deeper summaries — load-bearing pages

### Architecture: orvex-wiki-api (pG9s9OjqZA) — canonical
- Stateless Go service; dual-surface Phase-0 strangler facade: (a) byte-compatible `/api/orvex/*` reverse proxy (path-identical, so MCP's deployed client repoints via base-URL change only), (b) draft `/v1` verb grammar alongside; consumers migrate a→b via one deliberate client regeneration.
- A-RUNTIME: Go chosen specifically to avoid AGPL contamination from reusing `@docmost/editor-ext`; A-SERIALIZER: clean-room Go DfM package `orvex-studio-lib/pkg/dfm`, embedded in-process by wiki-api AND docmost-cli/orvex-cli, NEVER ported from the AGPL TS; CI parity test keeps engine's AGPL TS serializer and pkg/dfm honest.
- A-VERBS: search/get/save/edit/list over `{resource_type, locator}`, "wiki" only; NO ask verb (ai owns cited-ask, D-A12); write chokepoint = CAS `ifVersion`→409, full receipt `{url,id,version,persisted:true}`; engine keeps exactly one atomic `apply-ops` write primitive — wiki-api NEVER decomposes a batch into multiple engine calls.
- A-GOVERNANCE: living-wiki drift verification + wiki-first spec-gate move INTO wiki-api from the AGPL fork (D-S8); engine keeps only `orvex_page_meta` stamp fields; Linear-free (D-S11).
- A-TOKEN: pass-through only, no elevated credential; Phase-0 interim = K4 engine api-key pass-through (engine doesn't yet accept identity-minted tokens — named blocking dependency).
- A-STATE: no shaped-read body caching (would be a cached ACL-allow-decision by proxy); Redis holds only principal-independent artifacts (descriptor, block-schema catalog, authoring guide).
- Review-tightening addendum (unratified draft): HIGH finding — the "Redis→Kafka bridge" wording is wrong, real mechanism is engine transactional-outbox→relay→studio-spine Kafka→Trigger; host form UNPINNED (flat vs api.wiki, no cell segment in either candidate; shipped route is flat `wiki-api.orvex.ai/.dev`); /healthz must echo both CELL_ID and CLUSTER_NAME; observability front-end renamed control→console; docmost-cli should be orvex-cli as the family CLI.
- OD-7 (audit-sink ownership) RESOLVED via ADR-0037 (2026-07-16): shared service orvex-studio-audit, not the engine — authorized but unscheduled.

## 3. Bindings on the AI-surfaces redo (MCP / wiki-api / CLI PRD-architecture redo)

- **Verb surface is fixed and already partially serving in prod**: `search/get/save/edit/list` over `{resource_type, locator}` (wiki-only). NO `ask` verb ever — cited-ask is exclusively orvex-studio-ai's; any redo that adds an ask/RAG verb to wiki-api or MCP-against-wiki-api violates D-A12 and is an ADR-trigger.
- **Write chokepoint invariant**: atomic multi-op batches MUST map to exactly one engine `apply-ops` call; never decompose a batch into N engine calls (breaks the atomicity guarantee silently — flagged repeatedly as the single easiest invariant to violate "by being helpful"). CAS `ifVersion` required on every write → 409 on stale; full receipt `{url,id,version,persisted:true}` always, never the legacy `{block_id,status}`.
- **Contract authority**: MCP and CLI MUST codegen from the orvex-studio-contracts tag, never from wiki-api's served `/v1/openapi.json` (that's a conformance/drift-gate input only). Today (as of the mirror) that tag is NOT yet trustworthy for codegen — it's a flat grammar diverging from the served RESTful surface (MR-A1). Any redo PRD/architecture must either (a) wait on MR-A1's resolution or (b) explicitly own resolving it as part of the redo.
- **AGPL boundary is absolute and CI-guarded (NFR-A7)**: no closed repo (wiki-api, MCP, CLI) may import `@docmost/*` or any AGPL package in-process; all engine reuse is network-boundary only. The DfM serializer is the one clean-room Go package (`orvex-studio-lib/pkg/dfm`), shared by wiki-api and the CLI — MCP has no serializer role.
- **Token/auth model**: edge-verified pass-through only, via `orvex-studio-lib/auth`, no elevated service credential ever minted or exchanged. Phase-0 interim is K4 engine api-key pass-through pending an engine-side identity-token acceptance enabler — this is a named cross-repo blocking dependency for exiting Phase 0, relevant to any surface (MCP/CLI) that assumes identity-token auth already works end-to-end against wiki-api.
- **Quota propagation**: `QUOTA_EXCEEDED` (402) must propagate verbatim from engine through wiki-api to any consumer (MCP/CLI); never cached, never partial-applied; batch that trips the cap fails whole; error carries an upgrade/entitlement deep-link. wiki-api makes no entitlement decisions — billing owns values, engine enforces, wiki-api only relays. Any redo surface (MCP tool results, CLI error handling) must preserve this relay-only posture.
- **Doc-governance (drift + spec-gate)** is owned by wiki-api, exposed via CLI/MCP, NOT the engine directly. RATIFY/CONFIRM tokens are engine-minted; wiki-api (and by extension MCP/CLI) only ever transports them verbatim — never mints, never decides gate state locally.
- **Standalone/degradation posture** (D-S3/NFR-A5) must be preserved by any redo: every upstream (engine/knowledge/ai/identity) is a configured endpoint that MAY be absent; `get/save/edit/list` + drift/spec-gate (SHA/stamp mode) keep working with only engine present; `search/related` and ai-origin writes degrade to typed errors, never silent failure or fabricated success.
- **No second `resource_type`** beyond "wiki" is in scope (D-A5) — any redo introducing multi-resource grammar needs a fresh decision, not an assumption this is already generalized.
- **Layering discipline**: new wiki behavior composes in wiki-api over the engine's primitives via strangler-extraction — never via deep engine edits (INV-STRANGLER). This is the "wiki-api /v1, not direct-to-engine" rule already in this session's own memory index (new-vision-no-legacy-no-shortcuts) and is independently reaffirmed here as PRD/Architecture canon.
- **Observability/tenancy**: cell-lint's 14 rules apply to wiki-api and by extension anything MCP/CLI assume about its deployment (per-cell host, `/healthz` echoing CELL_ID+CLUSTER_NAME, cell-fail-closed on consumed events, Idempotency-Key on `/internal/*`). Most rules were still "not-yet" as of the Wave-3 SDD — a redo should not assume full cell-contract compliance is already in place.
- **Events**: wiki-api is a pure event CONSUMER (wiki.page.*, billing.entitlement.changed via engine outbox→relay→Kafka→Trigger), produces nothing. Any redo that has MCP/CLI expect wiki-api-originated events is wrong.
- **MCP-specific seam note**: MCP is named as the primary consumer of the WIKI verbs but per MR-A6 currently "vendors-but-bypasses" wiki-api (calls the engine directly rather than repointing) — this is exactly the kind of drift the AI-surfaces redo needs to resolve/decide explicitly, cross-pack with wiki-api.
