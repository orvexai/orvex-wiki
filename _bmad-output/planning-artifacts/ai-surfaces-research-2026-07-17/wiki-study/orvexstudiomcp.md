## 3. Bindings on the AI-surfaces redo (MCP/API/CLI PRD-architecture)

These are the commitments from this space that constrain or must inform the concurrent redo of orvex-wiki-api, the MCP, and orvex-cli:

- **Contracts and layering.** MCP consumes `orvex-studio-contracts` (`openapi/wiki-api.yaml`, `ai.yaml`, identity token-claims/scope/cell-claims schemas) via local-checkout TS codegen (ADR-0035) ‚Äî never a live descriptor, never the engine's AGPL 322-path descriptor (canon P3). This is the layering discipline the wiki-api/CLI redo must also respect: no direct-to-engine hops, everything through the `/v1` verb grammar once frozen.
- **Verb namespacing is now canon-ratified (R-WDS-1).** Tools are `namespace_verb` (`wiki_get`, `wiki_save`, `knowledge_search`, `ai_ask`, `memory_recall`, `staging_*`, `workgraph_*`). `save_page`+`edit` are MERGED into one `wiki_save(id?)` CAS upsert verb ‚Äî any wiki-api-facing redo should confirm its verb grammar matches this merged-upsert shape, not the old split save/edit pair.
- **Three (soon more) upstream seams, never MCP-direct-to-engine.** wiki verbs‚Üíwiki-api; ask/chat/inline/generate‚Üíai; search/related/neighborhood‚Üíknowledge DIRECT (D-M8/PO-ruling-9 ‚Äî this is the one path that bypasses wiki-api by design, MCP holds a `KNOWLEDGE_BASE_URL`); studio_*‚Üístudio-api (contract home now pinned per R-SEAM-1). Any wiki-api redo must keep the search/RAG path routed around itself to knowledge, not assume all reads flow through it.
- **Token pass-through, no minting, no elevation (FR-M11/M12).** MCP never calls `POST /v1/tokens`; caller's identity-minted scoped token rides verbatim on every upstream call. This is a hard invariant any redo of the auth/token surface (wiki-api, CLI) must preserve ‚Äî no service credential amplification anywhere in the chain.
- **RATIFY/CONFIRM tokens are server-minted and transport-only everywhere.** MCP MUST NOT promote draft‚Üícanonical on any path, ever ‚Äî this governance chokepoint lives in wiki-api/engine (D-A8), and any redo of the publish/ratify flow must keep MCP (and by extension CLI, if it shares the pattern) as a pure transporter of human-minted tokens.
- **Writes ON by default + hard READ_ONLY ceiling (R-WDS-3), with a Tenant Governance Profile** (memory-write mode staged|direct, sharing grants console-only|agent-enabled, publish path ratify-required|scoped-key) sitting inside that ceiling, all defaulting to human/Librarian-gated. Any redo of write-governance across MCP/API/CLI should adopt this same tenant-policy-object shape rather than inventing a parallel one.
- **CAS everywhere.** Every write is `ifVersion`-guarded ‚Üí `409 VERSION_MISMATCH`; receipts are `{url, id, version, persisted:true}` (verified read-after-write), never a re-rendered body. `402 QUOTA_EXCEEDED` (billing-owned, F-QUOTA) is a frozen error shape (ADR-0003) surfaced verbatim, never retried/masked/429'd.
- **DfM-only agent contract.** Raw ProseMirror JSON never crosses the MCP boundary either direction; opaque nodes (diagrams, embeds, mentions) travel only as typed handles. This DfM ownership lives in wiki-api (`lib/pkg/dfm`) ‚Äî any wiki-api redo must keep serializing DfM server-side, not push conversion to MCP or CLI.
- **Streaming is a requirement ON the existing verb surface, not new tools (R21/ADR-0002).** ask-class gets a dormant SSE relay (buffered-only live today, since ai's `/v1/ask` isn't streamed yet); edit-class gets local stage-progress. No `chat`/`inline`/`generate` tools were added to accommodate this. Any redo touching ai's `/v1/ask` surface should know the MCP has a pre-built, tested-but-dormant relay waiting on a pinned streamed-verdict grammar.
- **Statelessness is absolute (D-M4/FR-M17).** No DB, no worker, only bounded auth caches (JWKS/introspection TTL/deny-list). No tenant-content-derived cache ever (no page bytes, no search results cached) ‚Äî this is the same "no cache as ACL-decision-by-proxy" reasoning wiki-api applies (D-A10); the redo should keep this symmetric across services.
- **Cell contract (JGAUQRsw2g) applies uniformly.** `CELL_ID`‚âÝ`CLUSTER_NAME`, `/healthz` echoing both, fail-closed on `orvexcell` mismatch, cell-scoped Kafka topics, discover-once-pin routing (ADR-0020) for a flat public host. Any redo's deploy/runtime surface must carry the same fields ‚Äî MCP's `src/health`+`src/routing` are the reference implementation (landed, per SDD ¬ß2e rules 4 verified, ADR-0020 implemented).
- **CS ¬ß5 dependency-category discipline.** Own code = Row 1, never mocked. Sibling services (wiki-api/ai/knowledge/identity/studio-api) = Row 3, remote-but-owned ‚Äî test against contracts golden fixtures, never hand-rolled guesses. No Row-4 (true-external: Clerk/Keycloak/Stripe/Turbopuffer/LiteLLM) dependency should ever appear directly in MCP, wiki-api, or CLI ‚Äî those are reached only via identity/knowledge/ai/billing's own contracts.
- **Golden-tape KPI as a release gate (NFR-M6), human-ratified, CI may not silently raise it.** ‚â§2 calls/‚â§~1k tokens per common task.
