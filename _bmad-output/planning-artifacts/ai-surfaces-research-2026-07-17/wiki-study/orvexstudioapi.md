## 3. Bindings on the AI-surfaces redo (MCP / wiki-api / CLI)

These are commitments this space places on the concurrent orvex-wiki-api / orvex-studio-mcp / orvex-cli re-baseline:

- **Wiki as agent-composition tier, not Studio-specific.** studio-api's DocmostPort is explicitly the model for "wiki as a programmatic client surface" — OQ-SA6 frames the choice as engine-direct vs re-pointing to **wiki-api `8jAiCBifDW`**, contingent on wiki-api covering: space/page CRUD + CAS(`LostUpdateError`) + archive/restore/purge-gated + `getPageHistory`/`listPages` + magic-OTP transport + `recentPagesForUser`. Whatever wiki-api's `/v1` PRD/architecture lands, this SDD-level coverage check (PRD §2.3 reference) is the acceptance bar a downstream consumer will audit against.
- **Quota propagation contract.** The engine's F-QUOTA chokepoint returns a typed over-quota **402 `QUOTA_EXCEEDED`** (engine PRD `EPsdD7uK8e`); wiki-api MUST propagate this verbatim (never cached, never partial-written-around) — binding for any wiki-api write endpoint the CLI/MCP/API surfaces call. Never a 429 substitute for quota (429 is now a distinct D12 rate-limit axis, not interchangeable).
- **Tri-axis denial vocabulary (D12, 2026-07-17).** 402 QUOTA_EXCEEDED (period quota) / 403 ENTITLEMENT_REQUIRED (tier lock) / 429 + Retry-After (burst/abuse, distinct Redis token-bucket axis, per-principal AND per-credential/agent). "Reads never 429" (FR-R50) is a quota-axis promise only — burst protection may 429 any caller including MCP tools. Any MCP/API/CLI redo that touches rate limiting or error taxonomy should adopt this exact three-way split, not invent a new one; first consumers named are MCP tools (ENG-2329), the direct-write lane (ENG-2848), telemetry ingest (ENG-2847).
- **Two machine-write lanes (D11, 2026-07-17) — directly shapes MCP/CLI write semantics.** Machines (agents via MCP/API) get BOTH a DIRECT-WRITE lane into Memory and wiki (provenance-stamped agent id+timestamp, `unreviewed` flag, deterministic zero-model-call fail-closed screen, quota-bound) AND the curated Librarian lane (propose→AI-assist→human confirm, costs AI, `403 ENTITLEMENT_REQUIRED` without entitlement). The confirm gate is the sole *curated* mutation path, no longer the sole path overall. Direct-write availability is globally configurable per user; for Free users it may be the ONLY machine-write option. This directly bears on how MCP write-verbs (`submit_to_curator`, wiki write tools, etc.) must be specified — any redo needs both lanes represented, not a single propose-only model.
- **Act-as-user delegation is the identity pattern for every downstream/agent call.** studio-api's A-AUTH and FR-SA26 establish: delegated tokens are short-lived, audience-scoped, never persisted/cached/logged, never ride workflow inputs; approved wiki writes apply as the user, never as an anonymous service actor. Any MCP tool or CLI agent-identity design should mirror this rather than invent its own token model.
- **Cell/tenancy contract (JGAUQRsw2g, 14 rules) applies uniformly.** Same-cell cluster-local Service DNS only for peer calls (no public `orvex.(ai|dev)` host literals — cell-lint R1/R9); every route behind cell-guard returning 421 on mismatch (R3); `orvexcell` CloudEvent extension stamped on every published event, consumers fail closed on mismatch (R6); Idempotency-Key mandatory on every `/internal/*` and webhook forward (R11). Any redo of wiki-api/MCP's own deployment topology inherits this same rule set as the family cell contract, not something bespoke.
- **Host/region doctrine (D7, 2026-07-17).** Canonical service-host pattern is `api.studio.orvex.<env>` region-neutral, with a same-origin `/api` browser altitude behind global edge steering — the two are "two altitudes of one seam," not competing schemes. Region must never leak into hostnames or API contracts. This is the pattern any wiki-api / MCP public host naming should follow.
- **Go↔TS contract-codegen seam is UNRESOLVED and blocks typed clients.** ADR-0035 governs but the "how does a TS satellite consume a Go-authored contract pipeline" question is open (published npm TS security-ceiling port vs. per-request identity introspection; TS-client+golden-fixture emission from orvex-studio-contracts undecided). Since MCP and CLI are also TS/Node-shaped consumers in this family, they inherit the identical unresolved seam — the wiki-api/MCP redo should either wait on or independently resolve this, not silently assume Go tooling.
- **CloudEvent taxonomy discipline (ADR-0007/0008/0010).** Envelope = ADR-0007; type taxonomy = `studio.<svc>.<resource>.<past-tense>` (ADR-0010, e.g. `studio.api.memory.updated`); change-authority = ADR-0008 (additive-only lane, breaking/reshaping needs an ADR + human ratify). Any wiki/MCP-emitted event types should conform to this same pinned taxonomy and land in orvex-studio-contracts first (design-it-twice, CS §3.7) before code references them — exactly the discipline api's own MR-4 violates by NOT having done this yet (a cautionary, not yet exemplary, case).
- **"Memory" is a live 3-way naming collision (MR-2) directly touching MCP tool naming.** api `/v1/memory` (user-managed FormSpec) vs workgraph "memory" (multi-agent work-graph, explicitly NOT the product) vs knowledge (Memory-corpus retrieval/isolation grade still open) — any MCP tool or CLI verb using the word "memory" must disambiguate against this collision; PO+workgraph/knowledge authority, not this pack's to decide.
- **Curator↔Librarian ownership cut (MR-1) affects any MCP capture/submit tool surface.** Direction is settled (staging's Librarian supersedes api's Curator) but sequencing/ownership of the cut is not — `submit_to_curator`-style MCP verbs may need to repoint to staging mid-flight; do not hardcode an assumption either way in a fresh MCP verb design.
- **AD-1..AD-4/AD-12 Memory ownership contract (Memory gap-closure fold-in) is binding on any MCP `search_memory`/`add_memory` design:** api is the sole system-of-record for the Memory *card*; ai owns extraction compute/ephemeral recall only (renamed off "memory"); knowledge owns the read-only retrieval projection; the confirm gate (or now, the direct-write lane per D11) is the sole mutation path; bi-temporal Reconcile orders strictly by `valid_from` (world-time) never ingest-time.
- **Observability/troubleshooting surface is orvex-studio-console, NOT Grafana** (obs contract FR-C18) — any MCP/CLI tooling that surfaces service health/telemetry to a human should point at console, not raw LGTM.
