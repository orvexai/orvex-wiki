## 3. Bindings on the AI-surfaces (MCP / wiki-api / CLI) redo

- **Layering is settled and non-negotiable (D-S16, reinforced by ADR-0013's 2026-07-17 clarification and the family Index roster):** wiki verbs → `orvex-wiki-api` only; AI calls (ask/chat/inline/generate) → `orvex-studio-ai` only; **search/related/RAG/semantic-neighbor calls from MCP go DIRECTLY to `orvex-studio-knowledge`** — the wiki-api intermediary for search is explicitly retired (D-M8/PO ruling 9/ENG-1403, Done) and any lingering "MCP queries knowledge via wiki-api" text is stale. This must be reflected precisely in the redone MCP/API architecture — get the three lanes (wiki-api / ai / knowledge-direct) right per verb.
- **wiki-api is the sole wiki write ingress, target-state.** ADR-0021 (still draft/pending ratify at last sync) fixes the caller-token pass-through contract: wiki-api never mints or holds a service credential for its own engine/knowledge egress; it forwards the CALLER's verified token byte-for-byte and reuses `pkg/auth.RequireScope` directly (no bespoke scope algorithm). `IDENTITY_VERIFY_ENABLED=false` is a *pass-through* mode (upstream still enforces), not an unauthenticated hole — flipping the flag only adds a stricter pre-check, never changes what a successful request forwards. Any redo of wiki-api's edge auth must preserve this exact semantic.
- **Agent write surfaces are being hard-cut to route through Staging (FR-STG25), not deprecated-in-place.** This includes the CLI's own `wiki` write verbs under agent-class credentials (403 + migration error once staging proves end-to-end). The AI-surfaces redo must not silently re-legitimize a direct-write path for MCP/CLI once this cut lands — a redo landing BEFORE the cut still needs to design its write verbs so the cut is a clean swap-in, not a rearchitecture.
- **CLI is a from-scratch rewrite, not a rename** (D-S21 amended): `docmost-cli`'s behavior is the SPEC/reference only — no history seed, no blind copy. The new `orvex-cli` uses `orvex-studio-lib/pkg/dfm` (the clean-room Go serializer) instead of any hand-mirrored converter.
- **Contract-change authority is layered (ADR-0008), not the older "fully automated" or "always ADR" readings.** Mechanical/additive OpenAPI/CloudEvent changes ship through an automated CI-policed lane; anything breaking or seam-reshaping (renamed/removed field, changed `required` set, altered ADR-0007 envelope, retired endpoint, coordinated multi-repo cutover) is a mandatory ADR + human doc-ratify. The redo's contract changes must be pre-classified against this rubric before dispatch.
