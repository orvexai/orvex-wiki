## 3. Bindings on the AI-surfaces redo

Workgraph is a **sibling satellite, not a wiki-api/MCP/CLI-redo dependency** — it does not sit in the direct build path of orvex-wiki-api, orvex-studio-mcp, or orvex-cli. Its bindings are precedent and family-canon, not code-level:

- **Frozen family envelope family binds every satellite alike**: `Error{errorCode,message,details?}` (ADR-0003, `additionalProperties:false`), `QuotaExceeded` 402 superset, `Page{items,nextCursor,total?}`, and the generic `Banner{kind:elision|staleness|degraded,scope,detail}` honesty primitive. Any MCP/wiki-api/CLI redo should reuse this exact shape family rather than inventing a parallel one — workgraph's pages are the most fully fleshed-out worked example currently in canon.
- **ADR-0008 change-lane discipline** (additive lane vs ADR+human-ratify lane) is applied consistently across workgraph's contracts and is the same lane doctrine any MCP tool-surface change must follow.
- **The anti-footgun doctrine is family-wide, not workgraph-local** — "no error message ever contains a copy-pasteable destructive command" is cited from a real incident and should bind MCP tool descriptions / wiki-api error copy identically.
- **Token-budget doctrine (A-N1 abandonment guard)** — lazy schema discovery + `brief=true` + result-compaction-with-preview — is exactly the class of problem the MCP redo (per the user's own memory notes: 19-tool surface, streaming folded into design verbs) must also solve; workgraph's DS-P3/DS-P8 patterns and its `mcp-tool-surface.md` are a directly comparable worked precedent.
- **`orvex-cli workgraph` establishes the CLI's namespace-first multi-service pattern** (`wiki`/`search`/`ai`/`auth`/`admin`/`workgraph`) and the frozen 0–9 exit-code + errorCode vocabulary shared across all namespaces (FR-CLI8) — any CLI-surface redo work must not renumber or fork this scheme; workgraph's CLI-conventions page explicitly cites and extends it, never reinvents it.
- **ADR-0033 (claim arbiter, still OPEN) / ADR-0034 (credential lanes)** are family-wide governance ADRs workgraph found itself unnamed in; if the MCP/CLI redo introduces any multi-agent claim/dispatch semantics, these two ADRs are the binding precedent and the same "workgraph is missing from ADR-0034" gap-pattern (an unnamed consumer of a credential lane) is worth checking for the AI-surfaces redo's own build-lane agents.
- **`studio.memory.*` vs `studio.workgraph.*` naming reservation (ADR-0010/ADR-0025)** is a hard namespace collision boundary — any MCP tool naming for the AI-surfaces redo must respect the existing `studio_memory_*`/`studio.memory.*` reservation and must not invent new "memory" terminology that collides.
- **No new binding on wiki-api /v1 or the amazing-MCP 19-tool surface directly** — workgraph is architecturally isolated (cluster-internal only, no public HTTPRoute, own Postgres) and shares no runtime dependency with wiki-api. The one *cross-service* touch point is the promotion edge (workgraph→staging, not workgraph→wiki-api), which is itself unbuilt (see Staleness §4).
