## 2. Deeper summaries — load-bearing pages

### Architecture: orvex-cli (`pf10XC2Qjz`, canonical)
- Single Go/cobra binary `orvex` (+`orvex-full` for chromium/tree-sitter features); wiki is ONE namespace among peers (ai, auth, search, admin) plus client-local tooling (daemon, cache, config, doctor, audit, instructions).
- Built from scratch (D-S21 AMENDED) against the microservice architecture; `docmost-cli` is the untouched behavioural reference/oracle only, never imported code.
- The one structural redesign: **per-service endpoint registry** on `Profile` + **one service-aware transport** (chokepoint for base-URL, envelope, retry/backoff, correlation-ID, Idempotency-Key) replacing docmost-cli's single `InstanceURL` and ~90 raw-http callsites/triplicated HTTP layer.
- Routing is domain-pure (D-S16): wiki verbs → orvex-wiki-api; search/knowledge/SSE → knowledge; ask/chat → ai; auth → identity. Engine's raw API never contacted.
- Host convention (post F-A fix): canon flat service hosts (`wiki-api.orvex.{tld}`, `auth.orvex.{tld}`, `events.orvex.{tld}` SSE-only); ONLY wiki tenant host is cell-segmented (`{tenant}.wiki.{cell}.orvex.{tld}`). `ai` host and public knowledge query host and wiki-api host FORM remain unpinned (OQ-CLI2/2b).
- Per-service typed clients codegen from the orvex-studio-contracts tag, never a served descriptor.
- `pkg/dfm` (orvex-studio-lib) embedded in-process — bidirectional DfM↔ProseMirror, offline-capable, byte-lossless opaque round-trips (fixes docmost-cli's markdown→PM offline gap and the baked-Excalidraw data-loss bug by construction).
- Daemon + SQLite cache: one per-host daemon, single SQLite cache over Unix-socket IPC, kept fresh by one SSE connection per workspace; cursors qualified by (cell/events-host, space) to avoid cross-cell aliasing on tenant-move.
- 421 cell-mismatch handling: typed retriable-after-re-resolve condition (re-resolve tenant→cell, re-point host, cold-start cursor, retry once, else typed `CELL_MISMATCH`).
- Idempotency-Key minted on replayable writes (cell contract R11); CAS `ifVersion`→409 on wiki updates.
- Correlation/request ID originated at CLI ingress, propagated on every outbound call, surfaced as `request_id` in error envelope.
- Zero-trust: CLI holds no service credential beyond forwarded token, makes no authz decisions, propagates 401/403/404 verbatim.
- Not a dependency: orvex-studio-billing (Stripe-webhook-only; entitlement state reaches CLI only via knowledge FR-K20 projection).
- Owner: jezer.bacquian@orvex.ai. Last reviewed 2026-07-05. Note appended 2026-07-16: OQ-CLI3 (audit sink) resolved by ADR-0037 — sink is orvex-studio-audit, reached transitively via wiki-api forward, not the engine.

## 3. Bindings on the AI-surfaces redo (MCP / wiki-api / CLI PRD-architecture)

- **Contracts-first, never served-descriptor.** Every satellite client (CLI included) MUST codegen from the `orvex-studio-contracts` tag, never from a live/served OpenAPI descriptor. This is a repeated, hard-frozen rule (NFR-CLI7, the F-A/D-A7 anti-pattern) — any redo of MCP/API/CLI PRDs must reaffirm it, not relitigate it.
- **Domain-pure routing (D-S10/D-S16), engine raw API is internal-only.** All wiki-shaped access — including history/diff/version, import/export, binary attachments, drift, audit, comments — routes through orvex-wiki-api, never the engine directly. The React client is the SOLE platform-wide exception. This closes OQ-A7 and is binding on any redo: nothing new should reintroduce an engine-direct call for an AI-facing surface.
- **The 19-code frozen errorCode vocabulary + 0-9 exit codes are a shared client artifact**, reused (not wiki-only) across ai/search/auth namespaces — extend, never renumber. Any MCP/API redo touching error surfaces should reconcile against this vocabulary, not invent a parallel one (open co-ownership question flagged, not yet resolved by contracts).
- **Idempotency-Key is REQUIRED on all replayable writes** (cell contract R11) but is currently a stub in the CLI (NOT MET). Any redo of write-verb design across MCP/API/CLI must close this gap consistently, not just in one surface.
- **421 cell-mismatch handling + cursor cell-qualification** are load-bearing day-1 cell-contract obligations for any direct-to-cell client (CLI is direct-to-cell; MCP/API presumably share this posture — verify in their own audits). Cursors must be keyed by (cell/events-host, space), never space-alone, to avoid cross-tenant-move aliasing.
- **Host-routing form is STILL UNPINNED (MR-CLI1/OQ-CLI2/2b)**: flat service hosts vs per-cell form; `ai` host, public knowledge-query host, and wiki-api host FORM (`wiki-api.orvex.{tld}` vs `api.wiki.orvex.{tld}`) are all open. Any MCP/API/CLI redo that assumes a specific host form is getting ahead of a still-open contracts decision — flag, don't invent.
- **The Go codegen bridge for non-TS consumers is UNDECIDED (MR-CLI2).** ADR-0035 covers ONLY the three TS satellites (api/mcp/ui) and explicitly states "no Go stubs anywhere." Any redo assuming a Go client-generation mechanism for the CLI needs a NEW decision/ADR — do not extend ADR-0035's scope by inference.
- **SSE/events wire contract is far richer than any FR bullet list suggests** — 11 concrete behaviours (cold-start, cursor scheme, unprefixed names, thin content_updated, permission.changed semantics, frame/queue caps, backoff ladder, control frames, admin surfaces, exactly-once dedup, status_changed mapping) must be pinned VERBATIM in contracts EVENTS.md. This binds any streaming design in the MCP/API redo too (relevant given R21: streaming folded into existing tools, not separate ones — the underlying wire contract this note describes is still what any streaming verb rides on).
- **`content_pm` (raw ProseMirror) parity is the single biggest cross-surface parity risk** (OQ-CLI1) — the knowledge projection currently has no PM-JSON analogue. Any surface (CLI, presumably MCP too) that needs `--prosemirror`/round-trip/hash-skip fidelity is blocked on this decision.
- **pkg/dfm (clean-room, in-process, AGPL-provenance-gated) is the shared DfM↔ProseMirror conversion path** for both wiki-api and CLI (and by extension anything else that authors content offline). No surface may call an engine `/convert` endpoint going forward — mark as retired for consumers.
- **No fallbacks, hard cuts, fail-closed on unpinned config** — the CLI's posture (typed config error on an unpinned host, never a fabricated host) is the template; the family memory note "no-fallbacks-hard-cuts" applies identically to MCP/API redesign choices.
- **Linear is fully absent by design** (D-S11) from the CLI, with only `LINEAR_NOT_CONNECTED` as a single residual error code slated to drop. Consistent with family memory (linear-no-epics) — no surface in the redo should reintroduce Linear coupling.
- **Audit sink is now pinned**: ADR-0037 (ratified 2026-07-16) resolves the previously-open audit-sink question to the shared `orvex-studio-audit` service, reached transitively via wiki-api's forward target (not the engine). This closes G-audit for the CLI's audit dual-write design and should inform any MCP/API audit-surface design too — align to the same sink, don't invent a second one. (Per-service re-point authorized but not yet scheduled in code.)
- **CLI is a client, not a cluster workload** — no ArgoCD Application/Helm chart/cell pod; "deployed" = signed released binary. Relevant if the MCP/API redo is drafting a uniform deployment/observability template across surfaces — the CLI needs its own template (doctor probes, correlation-ID, no OTel/LGTM).
- **Governance/ADR discipline**: costly-to-reverse seam decisions (host form, from-scratch rewrite, pkg/dfm+AGPL boundary, contracts-tag codegen, hard-delete tombstones) are flagged as ADR triggers (CS §9) but filing is gated on the Act-1 ADR registry / Decision-Records parent landing (still TBD per this space, OQ-CLI9) — the redo work should check whether that registry has since landed and file these formally if so.
