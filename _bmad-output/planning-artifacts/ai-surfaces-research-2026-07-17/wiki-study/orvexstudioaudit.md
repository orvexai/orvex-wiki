## 2. Deeper summaries (load-bearing pages)

### PRD (`vhC5XXCYkC`) — canonical

- **Problem**: 6 uncoordinated audit surfaces + no owner (engine `NoopAuditService` no-op, wiki-api's prod-unreachable `/v1/audit` forward, cli dual-write, console's *private* `KafkaRecorder`/`console.audit.record` pipeline that others wrongly treat as the family sink, identity's Loki stopgap, ai's ad-hoc counter) **plus** billing's siloed entitlement table as a 7th record. ADR-0037 already chartered the fix.
- **Delivery model**: single-phase, PO-decided 2026-07-16 — every FR ships in release 1 (no staged rollout); a build-order still applies inside the phase (§8: foundations → core trail → compliance depth → cutover/hardening).
- **Goals**: G1 one durable tamper-evident trail; G2 meet strictest of GDPR+SOC2+ISO27001 (HIPAA design-compatible, not committed); G3 never on the critical path; G4 give console/identity/cli/wiki-api one sanctioned surface.
- **Success metrics are numeric and auditable**: relay-completeness/delivery-completeness ≥99.999% (SM1a/b), capture lag p50≤5s/p99≤60s/p99.9≤5min (SM2), 100% tamper-detection + 100% WORM tamper-prevention (SM3a/b), 100% of 7 producers onboarded (SM4), signed+verifiable exports (SM5a; public verification SM5b deferred), DSAR honored within GDPR statutory window (SM6). Guardrails: 0ms added latency, no availability coupling, minimized PII, cost SLO with 2x-median alarm.
- **7 producers named explicitly**: engine, wiki-api, cli, console, identity, ai, billing. **MCP is not named as a producer anywhere in this PRD.**
- **Scope carve-outs**: not ops-logs/observability; not the entitlements/AI-spend system-of-record (stays billing); not console's producer (only its reader re-points, ruling R16); not a live-tail/streaming UI; not the owner of the CloudEvent taxonomy (that's contracts' ADR-0010/MR-C3, a hard precondition it consumes not authors).
- **FR groups (38 FRs)**: ingestion/delivery (FR1–6, FR32 partial), tamper-evident storage (FR7–10, FR36), retention/legal hold (FR11–13, FR37), query/console-browse (FR14–16), evidence export (FR17–19), review/monitoring/completeness (FR20–21, FR31, FR38), privacy/erasure (FR22–24), integrity ops (FR32–35), multi-tenancy/residency/access (FR25–27), producer onboarding (FR28–30).
- **7 PO decisions resolved 2026-07-16 (OQ1–OQ7)**: 7-year uniform retention floor (6y regulatory + 1y margin); per-tenant floor-raising deferred past release 1; HIPAA-claim confirmation deferred as a go-to-market gate (not a build blocker); public/independent verification (SM5b) deferred; sealing job = Temporal; crypto-shred key granularity = per-subject; DPIA/RoPA are DPO deliverables, not in-service features.
- **Dependencies list is a direct blocker map**: ADR-0010/MR-C3 taxonomy blocks FR28; Ceph Object-Lock Compliance-mode confirmation blocks FR9; `orvex-studio-lib pkg/auth` blocks FR30 for wiki-api specifically; identity org-JWT recipe (M9) needed for FR27/FR24; identity's erasure-veto endpoint is new cross-team work for FR37; per-producer contract tests (all 7 producers) needed for FR21's completeness residual; engine/ai/billing producer onboarding needed for FR30/SM4 (none currently emit).

### Architecture Spine (`iWvaOcrzRD`) — canonical

- **Paradigm**: hexagonal, pipeline `accept → dedup → hash-chain → persist → seal → tier → expire`, pure domain core, everything external is an adapter behind a port. Source tree seeded (`internal/ingest`, `internal/core`, `internal/store`, `internal/retention`, `internal/query`, `internal/completeness`, `internal/privacy`).
- **Inherited invariants from family arch** (read-only, not re-derived): primitives-in/brains-out closed-Go-satellite rule; events-are-the-integration-surface (never point-to-point); CloudEvent envelope requires `orvexcell`/`orvextenant`/`partitionkey`; `audit.*` type family + `/v1/audit/*` governed by contracts (ADR-0008); workflow-free satellites (Temporal lives in `orvex-studio-workflows`, satellites expose idempotent step-APIs); per-cell + RLS + EU residency; P-never-shared-databases (no service reads the audit DB directly); data-plane pins (CNPG, Redis, S3/Rook-Ceph, Turbopuffer, OpenBao).
- **16 architecture decisions (AD-1…AD-16)** — the durable heart:
  - AD-1 durable-async, never critical path (one narrow local-outbox-insert-precondition exception for highest-stakes actions, never a downstream call to audit).
  - AD-2 raw in-process Kafka consumer, NOT a Knative Trigger (2nd documented exception after ADR-0013).
  - AD-3 exactly-once storage via `INSERT...ON CONFLICT DO NOTHING`; dedup collisions detected not dropped; projections mint deterministic event_id from source event id, active-emission mints UUIDv7.
  - AD-4 append-only, INSERT-only role, `BEFORE UPDATE OR DELETE` trigger; only removal paths are retention-drop and crypto-shred, each a signed chained authorized-removal event.
  - AD-5 tamper-evidence = hash-chain + signed checkpoints + WORM (not a ledger DB); Merkle rejected (single writer, bounded verifiers); immudb/QLDB/Trillian evaluated and rejected.
  - AD-6 tiered same-stack storage, audit owns its private store (P-never-shared-databases enforced).
  - AD-7 7-year uniform retention; legal hold overrides; partition-drop only after seal+WORM-anchor.
  - AD-8 CloudEvents-native `audit.<category>.<verb>`, schema-per-type in contracts, NO runtime schema registry, projection registry is a static build-time artifact; taxonomy consumes ADR-0010/MR-C3, never invents a third name.
  - AD-9 producer authenticity: transport-authenticated identity + declared tenant/cell validated against authenticated scope at ingest; mismatch rejected/DLQ'd.
  - AD-10 per-cell + RLS + EU residency; sole cross-vendor copy (Glacier) stays in-region, ciphertext-only.
  - AD-11 sealing via family Temporal control plane (`orvex-studio-workflows` drives audit's idempotent seal step-API); seal liveness monitored out-of-band, independent of the seal itself running.
  - AD-12 `/v1/audit/*` is the ONLY path to audit data; every read audited; dedicated audit-reader scope ≠ admin; double-layer tenant scoping (API filter + RLS); Turbopuffer search projection is also gated by the same scope.
  - AD-13 completeness detection out-of-band: producers emit outbox-written/relayed counts as a periodic reconciliation *event on the spine* (never a point-to-point pull, so Principle 2 holds) so all 7 producers implement identically.
  - AD-14 GDPR erasure preserves immutability: minimization + identity-as-tokenization-map + per-subject crypto-shred + retention-expiry; legal hold vetoes erasure via a defined `retention.legal_hold_applied/released` event (not a point-to-point call).
  - AD-15 numeric service-level invariants: capture-lag SLO, completeness SLIs ≥99.999%, DR RTO≤4h/RPO bound by lag SLO, chain-revalidating DR restores.
  - AD-16 active review & monitoring is orthogonal to completeness: anomaly alerting, breach-of-self detection, itself-audited review cadence, break-glass alarmed/time-boxed/recorded.
- **Consistency conventions table**: event naming (`audit.<category>.<verb>`, 7 closed categories: auth/data-access/admin/config/consent/retention/export); `event_id` = producer UUIDv7 dedup key; dual timestamps (`occurredAt` producer, `recordedAt` server-authoritative); ordering by trace-context+broker-offset never wall-clock; integrity fields (`recordHash`/`prevRecordHash`/`chainSeq`/`checkpointRef`/`keyEpoch`); provenance fields (`source`, `actor{id,type,sessionId}`, `orvextenant`/`orvexcell`); CloudEvents 1.0 structured mode + contracts error codes; INSERT-only state mutation; family single auth-verifier (`orvex-studio-lib/pkg/auth`) + org-JWT tenant GUC; OpenBao+External Secrets, no secret in image, offsite is ciphertext-only.
- **Deferred list**: per-tenant retention floor-raising; public/independent verification; dedicated `audit-events.<cell>` topic vs. filtering domain topics (open ADR-0008-gated decision); cold-query engine choice (reload-to-CNPG default vs. DuckDB); ClickHouse hot-store revisit (behind a measured >~10k events/sec trigger); DPIA/RoPA (DPO deliverables).

### PRD Technical Addendum (`bEZ31uqofJ`) — **draft, unratified**

- Explicitly scoped as PRD-local pointer/summary; "the authoritative, fully-cited detail is the research page ... and ADR-0037." Not itself the source of truth.
- Storage tiering concretized: hot Postgres ~90d online; warm+cold WORM Parquet/NDJSON on Rook-Ceph Object Lock Compliance mode (~90d→6-7y); offsite DR AWS Glacier Deep Archive Frankfurt, client-side encrypted, never a read/critical path, vendor-swappable. *PO decision 2026-07-16: Rook-Ceph primary + AWS Glacier offsite, not the reverse.*
- Tamper-evidence = CloudTrail log-file-integrity pattern (not a ledger DB); hourly seal signs via OpenBao Transit, written to Postgres AND WORM bucket.
- Ingestion = raw in-process Kafka consumer group, `enable.auto.commit=false`, static membership + cooperative-sticky, commit-after-persist, tenant-keyed multi-partition, per-topic DLQ, separate replay consumer group.
- Event contract: CloudEvents-native extension of ADR-0007 envelope, no runtime Schema Registry, 7-category taxonomy, passive-projection + active-emission lanes, OCSF only at export boundary; **hard precondition: MR-C3/ADR-0010 must reconcile `console.audit.record` vs `studio.admin.action.recorded` first**.
- Query/export: structured filter only (never free-text SQL), mandatory tenant+time, keyset pagination, AIP-151 async LRO shape, BagIt export bundle with SHA-256 manifests + detached Ed25519 signature, double-layered tenant scoping.
- Privacy: opaque `actor_id` + identity-tokenization + per-subject crypto-shred (per-subject DEK wrapped by OpenBao KEK) + retention-expiry as primary discharge; anchored to EDPB Guidelines 02/2025 + Recital 26; certification alone is NOT an Art.17(3) shield.
- Completeness: heartbeats 60s cadence/~180s alarm, two-stage reconciliation (outbox-written↔relayed↔persisted) at 5-min/hourly/daily windows, grace = p99.9 lag budget; named residual = a producer that never writes its outbox row (needs a per-producer contract test, not covered by this signal).
- Sizing: ~1M events/day baseline → ~100M/day upper bound; ~1KB/event; bounded worst case ~200GB/day, ~18TB/90-day hot pre-compression; revisit trigger for ClickHouse = sustained >~10k events/sec over rolling 7 days AND (hot p99 over SLO OR first-class OLAP need).
- Alternatives rejected list mirrors the research (SIEM-as-system-of-record, synchronous-block, fold into console/knowledge, immudb/QLDB/Trillian, ClickHouse-as-primary [deferred not rejected outright], MongoDB [raised by PO, set aside], OpenSearch/Elastic, object-store-as-primary, runtime Schema Registry, RFC3161-as-baseline, OCSF/ECS-as-storage-schema, Governance-mode Object Lock).
- r3 integrity-ops additions: authorized-removal accounting (FR36), signing-key epoch lifecycle (FR33), trusted time (FR32), producer authenticity (FR34), cross-service hold propagation (FR37), export-as-sealed-bundle semantics (FR17 — a bundle's signature attests data-state at export time; later erasure of source records does not invalidate a prior export).

### Research (`zPssXupzLZ`) — canonical, 545 lines, primary input to PRD+architecture

- **Verdict**: build `orvex-studio-audit` as the 18th closed Go microservice; not discretionary — named owner of a documented gap, 3 services blocked/degraded until it exists.
- **Current-state finding, load-bearing**: console (`orvex-studio-console`) is the ONLY service that has actually shipped a durable Kafka audit pipeline (`KafkaRecorder`, ENG-1968, `acks=all`, proven in CI) — but it's console-*private* (`console.audit.record` type, console's own store), and `identity` (ENG-2421) already wrongly conflates it as *the* family sink in its own shipped documentation — cited as the accidental-convergence failure mode the ADR exists to catch before a second service copies the mistake.
- **Engine has zero Kafka egress today**: `Outbox.enqueue()`/`Relay.publishPending()` in `orvex-studio-api` both unconditionally throw "not implemented," zero production call sites (`yXUWpQpRjx`) — a foundational gap not unique to audit; it blocks Principle-2 events-as-integration-surface generally.
- **Retention derivation**: the ONLY six-year number across all 4 regimes is HIPAA §164.316(b)(2)(i), and even that black-letter governs required *documentation*, not explicitly the raw audit event stream — the 6y (later 7y in the PRD/ADR) floor is a **deliberate strictest-of-four interpretive choice**, not settled law; OQ3/primary-HHS confirmation deferred as a go-to-market gate.
- **Effort**: 4-phase, ~3-3.5 calendar-quarter build, 2-3 Go engineers; explicitly **not yet backed by a story-count**, pre-epic, ±0.5q uncertainty band; the two hardest/most-novel pieces are completeness-detection and GDPR-erasure reconciliation.
- **Verification discipline**: 4 refuted/corrected claims from an adversarial pass are recorded and never repeated as fact (S3/Ceph Object Lock retrofit IS now supported since Nov 2023/Squid-Tentacle; pg_partman is PostgreSQL-License not BSD; RFC 8493 BagIt is Informational not standards-track; EDPB Guidelines 5/2019 does NOT cover crypto-shredding — the correct authority is EDPB 02/2025 on blockchain).
- **Storage decision matrix**: hot=Postgres/CNPG; warm/cold WORM=Rook-Ceph RGW Object Lock Compliance mode (Parquet/NDJSON); offsite DR=AWS Glacier Deep Archive Frankfurt client-side-encrypted (the one deliberate cross-vendor exception, never a runtime dependency); optional search=Turbopuffer. ClickHouse/MongoDB/OpenSearch/object-store-as-primary/DuckDB-as-default all evaluated and rejected or deferred with explicit measured revisit triggers.
- **Delivery lag SLA is concretely budgeted**: p50≤5s/p99≤60s/p99.9≤5min, with a budget decomposition (relay poll≤2s + Kafka sub-second + consumer batch dwell≤5s + DB commit sub-second) and alarm thresholds WARN-60s/CRITICAL-5min/potential-loss-page-15min.
- **Completeness/gap-detection (§8.3) called out as the single most novel piece**: heartbeats every 60s through the REAL outbox→relay→Kafka path (not a side channel); two-stage reconciliation (Stage A outbox-written vs relayed localizes relay-side gaps; Stage B relayed vs persisted localizes delivery-side gaps); rolling-5min + hourly-authoritative + daily-deep windows; durable `audit_reconciliation` counter rows chained into WORM checkpoints; named limitation = cannot catch a producer that never wrote its outbox row at all (a per-producer contract-test concern, explicitly not claimed as solved).
- **GDPR-erasure-vs-immutability resolution, in priority order**: (1) data-minimization — opaque refs not raw PII; (2) identity-service-as-tokenization-map — erase the identity link, not the record; (3) field-level crypto-shred for unavoidable fields (IP is personal data per *Breyer* C-582/14); (4) retention-expiry as primary discharge; (5) legal hold = the Art.17(3)(e) basis for declining erasure, scoped to a documented matter.
- **Phased build plan** (P0 contracts/foundations ~0.5q → P1 ingestion/hot-store/tamper-evidence ~1q → P2 WORM/retention/legal-hold/query-export ~1q → P3 completeness/cutover/hardening ~0.5-1q).
- **Full compliance requirements matrix** (§7) cross-references GDPR/SOC2/ISO27001/HIPAA per dimension (retention, immutability, access control, review/monitoring, evidence export, clock integrity, RoPA, breach documentation) against the concrete design response — this is the canonical mapping table for any future compliance-facing claim.

### ADR-0037 (`orvexstudioarch/Z6WrJjZDAO`) — canonical, ACCEPTED, companion/founding doc

- Founding decision: charters component #18; documents the 6-uncoordinated-surfaces problem (wiki engine, wiki-api, cli, console, identity, ai) as a textbook CS §9 ADR trigger (family-wide seam, costly to re-decide once N services each build their own answer).
- Resolves OQ-A13 (wiki-api) and OQ-CLI3 (cli): **"shared service, not the engine."** wiki-api's `/v1/audit` forward target becomes `orvex-studio-audit`; the engine's own local `OrvexAuditService` MAY continue as an engine-internal export-audit record (not forced to change) but is no longer the family-wide destination.
- cli's dual-write code is explicitly UNCHANGED — only wiki-api's downstream target changes; cli does not build a second sink or its own hash-chaining.
- console's `BrowseAudit` reader re-points to the new component (ruling R16) — console's *producer* (`KafkaRecorder`) is unaffected and keeps publishing its own admin-action events; the reader re-point is named as genuinely new work, not a relabel, and its old `Store.Browse`/consumer-group code must be explicitly retired, "or this ADR's 'one sink' claim silently degrades back into two."
- **Same-day self-update (2026-07-16)**: "Store engine & retention — RESOLVED by the PO design" section absorbs the PRD's decisions verbatim, including a **floor increase from 6y (this ADR's original number) to 7y** (per the PRD/addendum), and pins the **raw in-process consumer group** as a correction to the ADR's own earlier Decision-section wording ("Knative Trigger/consumer group") — readers must use the "Consumer refinement" correction, not the original Decision §1 sentence.
- Named unresolved follow-on risk: "The engine's own produce-to-spine question... is left to the pack's Architecture, and un-ruled it risks becoming a SEVENTH quiet local mechanism if a future engine story assumes an answer nobody actually gave" — i.e., whether the engine ever gains its own Kafka egress vs. relying purely on wiki-api's HTTP forward is still open.

## 3. Bindings on the AI-surfaces redo (MCP / wiki-api / CLI PRD-architecture)

These are commitments already ratified in `orvexstudioaudit` + ADR-0037 that constrain the concurrent redo of orvex-wiki-api, orvex-studio-mcp, and orvex-cli.

**A. Layering / no-shortcuts (hard constraints)**
- Every service integrates via the transactional-outbox → relay → CloudEvent spine — **never point-to-point calls to `orvex-studio-audit`** (Principle 2, AD-13's "never a point-to-point pull" applies generally). No AI-surface design may propose a direct HTTP call from MCP/CLI to the audit service, or a direct DB read of its store (P-never-shared-databases, AD-6/AD-12 — audit's store is private, all access via `/v1/audit/*`).
- `orvex-studio-lib pkg/auth` is the **single family auth-verifier** and is a named, explicit blocker on wiki-api's own audit re-point (FR30 dependency) — any MCP/API/CLI redo that assumes a different or parallel auth-verification path conflicts with this.
- No new message-bus technology, no second Kafka-class client library (CS §7 one-adapter rule, reaffirmed by ADR-0037 point 1) — the AI-surfaces redo must reuse the existing `pkg/events`/`pkg/kafkautil` surface for anything event-emitting, not invent a parallel client.

**B. Producer obligations that apply to wiki-api and (transitively) CLI**
- **wiki-api is one of the 7 named sanctioned producers** (FR30); its `/v1/audit` forward re-points from the engine to `orvex-studio-audit` once `pkg/auth` lands. **CLI's dual-write code stays unchanged** — it continues through wiki-api's transport; CLI must NOT build its own sink or its own hash-chaining (ADR-0037 explicit, echoing OQ-CLI3's resolution).
- Any mutation-emitting endpoint (wiki-api /v1, and by extension anything CLI or MCP drives through it) must have a **real, working transactional outbox row per mutation** — the research flags that `Outbox.enqueue()`/`Relay.publishPending()` in the engine still throw "not implemented" with zero production call sites as of the research date; this is a **live blocking dependency**, not resolved history, and the AI-surfaces redo inherits it as a precondition for FR1/FR21.
- **Deny/failure/consent/export actions must be actively emitted** (AD-8's active-emission lane), not just passively projected from domain state changes — a naive "mirror every CloudEvent" design silently drops exactly the events auditors most want. Any wiki-api /v1 endpoint, MCP tool, or CLI command whose failure/denial path has no corresponding domain-state mutation (e.g. a permission-denied response, a rejected export request) needs an explicit audit-event emission path designed in, not inferred from success-path events.
- **Producer authenticity (AD-9)**: every emitted event must carry transport-authenticated provenance and a declared tenant/cell validated against that authenticated scope at ingest — a mismatch is rejected/DLQ'd. This is a concrete field-level contract any AI-surface event-producing code must satisfy.
- **Per-producer contract test (FR21 residual, explicitly a cross-team ask on "all seven producers")**: wiki-api (and any backing service CLI/MCP route through) needs a CI-enforced test proving every mutation path emits its outbox row — this is a Definition-of-Done item the AI-surfaces redo's test-architecture should adopt, not an audit-service-internal concern.
- **Reconciliation heartbeat/counter events (AD-13)**: any onboarded producer emits `studio.audit.heartbeat` every 60s and periodic outbox-written/relayed counts as standard event types on the spine — a concrete new emission obligation if wiki-api (or any newly-onboarded surface) is brought fully onto the sanctioned path.
- **Never block on audit (AD-1, CM1/CM2)**: 0ms added latency, no availability coupling — no AI-surface verb, MCP tool call, or CLI command may synchronously wait on audit persistence, with a single narrow allow-list exception (a local outbox-insert-precondition only, never a downstream reachability coupling) reserved for an explicitly enumerated highest-stakes-action class. If the AI-surfaces redo introduces any "must be logged before allowed" semantics for destructive/irreversible MCP tools or CLI commands, it must use this narrow local-precondition pattern, not a call to the audit service.

**C. MCP-specific gap — an explicit open decision the redo must resolve**
- **`orvex-studio-mcp` is NOT named among the 7 sanctioned producers** (engine, wiki-api, cli, console, identity, ai, billing) anywhere in the PRD, addendum, research, or ADR-0037. Given the family architecture note that TypeScript (and by extension MCP) is "reserved for the engine, BFFs, and the MCP gateway" and the "no legacy shortcuts — use wiki-api /v1, not direct-to-engine" governing rule (memory: new-vision-no-legacy-no-shortcuts), the working assumption is that **MCP inherits audit coverage transitively by calling wiki-api's audited `/v1` surface** rather than becoming its own producer. This is NOT stated explicitly anywhere in the audit canon — it is an inference the AI-surfaces PRD/architecture redo needs to state and either confirm or escalate. If any MCP tool makes a decision that never reaches wiki-api (e.g. an MCP-layer authorization denial, a client-side redaction, a tool-call rejected before any wiki-api call), **that decision path has no defined audit-event coverage today** and is a genuine gap the redo should surface (candidate for an escalation analogous to OQ-A13/OQ-CLI3, not something to quietly resolve either way).
- If MCP is later onboarded as an 8th producer, it inherits every producer obligation in section B above (outbox+relay, active-emission for denies, heartbeat, contract test, no-critical-path-coupling) — the redo should design for that possibility rather than assume MCP is exempt indefinitely.

**D. Read-side / query bindings — relevant if any AI-surface exposes audit-browse or compliance-query capability**
- All audit reads (including any future MCP "show me the audit trail" tool or CLI `audit query` command) MUST go through `/v1/audit/*` with the dedicated **audit-reader scope, distinct from admin/operator**, mandatory tenant+time scoping, RLS double-layered with the API filter, and — critically — **every read is itself an audited action** (AD-12, FR27). No AI surface may treat audit data as a general-purpose queryable knowledge source outside this contract.
- Large/async audit queries must follow the **AIP-151 long-running-operation shape** (submit → poll → fetch) with per-tenant concurrency caps (FR16) — any MCP tool or CLI command wrapping a bulk audit pull must use this pattern, not a synchronous large-payload call.
- The org-JWT tenant-scoping recipe (M9, Clerk org-keyed) that audit's `audit-reader` role reuses is the **same primitive** the AI-surfaces already use/should use for their own tenant-scoped calls — reinforces the family-wide identity contract rather than a parallel one.

**E. Governance / change-control bindings**
- The `audit.<category>.<verb>` event taxonomy is governed by **`orvex-studio-contracts` (ADR-0008 change-authority)**, gated by **ADR-0010/MR-C3** (not yet landed as of this study) reconciling `console.audit.record` vs. a would-be `studio.admin.action.recorded`. Any new event type an AI-surface redo introduces that touches audit semantics must go through this contracts change-authority — **never invent a third naming convention** (stated as a hard rule in the PRD, addendum, research, and ADR-0037 alike).
- **NFR9 — harder-than-typical change-gate**: "an audit-store bug is a compliance-evidence bug." Any AI-surface code that emits, forwards, or touches audit-relevant events inherits a stricter review bar by association, even though it lives outside the audit-service repo itself.
- **Audit ≠ ops logs** is a hard boundary (Loki/Mimir/Tempo stay separate from the `audit.*` compliance trail) — the AI-surfaces' own observability/tracing work must not conflate the two; this is consistent with the standing memory note that message-tracing-across-outbox is a known observability gap on this program.
- **Quotas / cost**: audit tracks GB/tenant/month as an SLO with a per-cell cost ceiling alarmed at 2x trailing-90-day median (CM4) — if any AI-surface redo introduces new high-volume audit-relevant event types (e.g. a chatty MCP tool-invocation log), it should be sized against this ceiling rather than assumed free.

## 4. Staleness flags

Overall: the `orvexstudioaudit` space and ADR-0037 are **same-day-fresh** relative to the mirror sync (both ratified/updated 2026-07-16; mirror synced 2026-07-17) and internally self-consistent — no contradictions found against each other. The following are genuine flags, not corpus contradictions:

1. **Addendum page (`bEZ31uqofJ`) is still `status: draft`** while the PRD, Architecture Spine, and Research it summarizes are all `canonical`. It is explicitly self-described as "PRD-local pointer + summary," so its content is fully covered by the canonical Research page — but any downstream author citing the Addendum directly as authoritative should be aware it has not itself been ratified. Low risk (content is a subset/summary of ratified material) but worth a ratify pass for hygiene.

2. **wiki-api `/v1/audit` prod-reachability claim needs re-verification against the current 2026-07-16/17 live state.** The PRD/ADR/research all state, as of their authoring, that wiki-api's `/v1/audit` forward is "genuinely unreachable in production today" pending an `orvex-studio-lib/pkg/auth` primitive (a 401-on-every-request gap). The session context states wiki-api's **general /v1 cutover is done** as of 2026-07-16/17 — this is a *different* claim (overall API surface cutover, not specifically the `/v1/audit` auth-gating fix) and the two are not necessarily in conflict, but the AI-surfaces study should explicitly re-check whether `pkg/auth` has landed and whether `/v1/audit` is now reachable, since FR30 (SM4's producer-onboarding target) is gated on exactly this.

3. **Engine's audit egress (`NoopAuditService` binding, zero Kafka egress) is stated as current fact in the research/ADR as of 2026-07-16** — this is a build-blocking dependency for FR30/SM4 (engine onboarding as a producer). Nothing in the reviewed corpus indicates this has changed; the AI-surfaces study should not assume it has been fixed without independent verification, since it sits outside the audit space's own control (owned by the engine team).

4. **ADR-0037 contains a self-correcting internal inconsistency, already resolved in-page but easy to mis-cite.** Its original "Decision" §1 text says the service "runs its own Knative Trigger/consumer group against the same `studio-spine` Broker." A later same-day "Consumer refinement" section explicitly overrides this: the actual, correct design is a **raw in-process consumer group, NOT a Knative Trigger** (2nd exception after ADR-0013). Anyone citing ADR-0037's Decision section directly (rather than the addendum/research/architecture, which all correctly state "raw consumer") risks propagating the superseded sentence.

5. **Retention floor number changed within the same document set** — the research report (`zPssXupzLZ`) argues for and adopts a **6-year** uniform floor throughout its body; the PRD (`vhC5XXCYkC`) and the Addendum ship a **7-year** floor (6y regulatory + 1y safety margin), and ADR-0037's "Store engine & retention — RESOLVED" section notes this explicitly as "up from the six-year floor this ADR originally resolved." This is not a contradiction (the PRD is the later, canonical, PO-decided number and self-documents the change) but any reader skimming only the Research page's headline recommendation would land on the superseded 6y figure — flag it for anyone quoting retention duration from the Research page alone.

6. **MCP's audit-producer status is undefined by omission, not by an explicit "N/A" ruling** (see §3.C above) — this is not a contradiction between pages, it is a **silent gap**: the audit canon enumerates exactly 7 producers and never mentions MCP, and no page in this space or ADR-0037 states whether MCP is out of scope by design or simply not yet considered. Given this program's own precedent (OQ-A13/OQ-CLI3 were exactly this kind of silent-gap-turned-accidental-conflation before ADR-0037 caught it), this is the single most actionable staleness/gap flag for the AI-surfaces redo to close explicitly rather than let calcify.

7. **No contradiction found with the stated 2026-07-16/17 live-state facts** (MCP live-green 19 tools on dev; streaming folded into existing tools per R21; audit-service PRD canonical) — the audit corpus doesn't reference MCP tool counts or streaming at all, so there is nothing here to reconcile against those facts beyond the producer-list gap noted in point 6.
