## Item-by-item verdicts

**1. Fixes the real divergence points for the level below (build stories) and misses none.**
**PASS, with one carried-not-newly-introduced gap.** AD-EXT-1..8 map cleanly onto B1–B6 and close the obvious divergence points: which providers plug in and how (AD-EXT-1), how tier flips without a rebuild (AD-EXT-2), where consent/firewall actually bind (AD-EXT-3), what a "delivery" transaction even is (AD-EXT-4), the MV3 trust surface (AD-EXT-5), auth/scope binding (AD-EXT-6), read-consistency (AD-EXT-7), and the contracts seam (AD-EXT-8). See Finding 2.

**2. Every AD-EXT Rule is enforceable and actually PREVENTS its stated divergence.**
**PASS.** AD-EXT-1 (registry as the single add/remove point, testable via a lint/structure check), AD-EXT-2 (tier-as-data with a fail-safe default, testable), AD-EXT-3 (token presenter with mint enforced server-side — the Shape 2 contract is the enforcement point), AD-EXT-4 (canary state machine, directly unit-testable through the `DeliveryAdapter` interface per CS §5), AD-EXT-5 (bundle-network-audit + host_permissions review + AGPL denylist in CI — concretely enforceable, not aspirational), AD-EXT-6 (fail-closed session/scope, unit-testable), AD-EXT-8 (drift gate + denylist in CI) are all backed by a concrete mechanism, not merely stated as prose. AD-EXT-7 is the one Rule whose enforcement lives partly outside this component (in `api`/`knowledge` behavior) — reasonable, since the extension is a client with no SoR, but worth naming as the one AD-EXT that is asserted rather than independently enforceable *by this component's own tests*.

**3. Nothing under Deferred/open-items could let two units diverge.**
**PASS.** OQ-E1 (Claude native-memory route) is explicitly walled off from FR-D1 by the "MUST NOT be conflated" language. OQ-E2 (Google ToS re-diff) and the counsel/store-policy preconditions are legal gates, not implementation choices, so they can't cause code-level divergence. OQ-E3 (Grok grok.com-only for v1) is scoped into the structural seed itself (`grok.ts # DEGRADE (grok.com only, OQ-E3)`), so it can't silently drift. The numeric `[ASSUMPTION]` targets (p95 400ms, 60s clipboard clear, 24h remediation SLA) are stated as concrete working numbers even though flagged provisional, which is the right shape — a placeholder value everyone builds against beats an unstated one.

**7. No new AD-EXT weakens or contradicts an inherited invariant (I-1/I-4/AD-5a/6/8/11).**
**PASS.** AD-EXT-3 keeps enforcement at the mint boundary exactly as AD-8 requires (client only ever "presents," never "decides"). AD-EXT-4's read-back step is explicitly scoped to the extension's own just-written text, consistent with AD-6 clause 3 (no Output scraping) — this is called out directly, not left to inference. AD-EXT-7 implements AD-11's RYW promise rather than loosening it. AD-EXT-8's telemetry and compose calls both ride the generated-TS-client/OpenAPI channel, satisfying I-1's "OpenAPI or CloudEvent" seam requirement without inventing a third channel. **8. Every dimension this altitude owns is decided/deferred/open — especially the operational/environmental envelope.**
**CONCERN.** Three of the four named sub-dimensions are handled well: the four provider surfaces are fully decided (per-provider tier, selector strategy, write technique, SPA resilience — §5/§6); CI is decided (§11's cell-lint note: self-hosted runners, lint/tsc/tests/contract-drift/bundle-network-audit gates, store bundle as the reproducible-build artifact); store distribution is explicitly OPEN and named as such (§12: Chrome-first, Firefox AMO/Edge phased, the latency-vs-takedown risk split). The gap is **backend-environment targeting**: nowhere does the document decide, defer, or even flag as open how the shipped extension (and any dev/unpacked build used during B1–B6 implementation and store-review QA) resolves *which* `orvex-studio-api`/`identity`/`knowledge` environment it talks to (dev-cell vs. prod, per this family's own documented dev/prod topology split). The only text touching this is one clause — "config via env / build injection only, no embedded secrets (CS §10)" — which names the *mechanism* (build-time env injection) but not the *topology* (what environments exist, which one a WXT dev build points at by default, how a build story tests delivery against a live provider DOM without either hitting prod or having no backend at all). Given six build milestones that all depend on a working backend connection from day one of B1, and given this is exactly the kind of silent-dimension gap the checklist calls out by name, this is a real gap, not a nit. Canary *operations* (§6/§11) are decided at the design/routing level (Shape 3, OTLP+Loki, paging alert) but the *ownership* of the 24h remediation SLA — which team authors and ships the signed selector-pack fix — is never named; a minor, lower-severity instance of the same "operational envelope" gap.

## Overall verdict: **CONCERNS**

The eight AD-EXTs are individually sound, mutually consistent, enforceable, and faithful to both the parent spine and the PRD — no invariant is weakened, no verdict caveat is laundered away, and FR coverage is complete. The document's stated goal ("zero architecture decisions left for the build agent") holds for everything provider-, port-, canary-, token-, and tier-shaped. It does not fully hold for the operational/environmental envelope: the backend-environment-targeting dimension is left effectively silent (not decided, not explicitly deferred, not flagged open), which is precisely the failure mode item 8 exists to catch. This is closable with a short addition (name the environments, name which one a dev/unpacked build defaults to, name who owns it if still open) rather than a structural rework — hence CONCERNS rather than FAIL.

## Findings

1. **[MEDIUM] Backend-environment targeting (dev-cell vs. prod) is not decided, deferred, or flagged open anywhere in the document**, despite six build milestones (B1 onward) depending on the extension reaching a working `orvex-studio-api`/`identity`/`knowledge` backend from day one, and the family's own documented dev/prod topology split. The only relevant text — "config via env / build injection only, no embedded secrets (CS §10)" (Consistency Conventions, "State & cross-cutting") — names the injection mechanism but not the environment topology itself.
   *Fix:* Add a short subsection (or extend §11/§12) naming: which environments the extension can target (dev-cell, prod), how a WXT build variant selects one, what a B1–B6 implementer's local/unpacked dev build points at by default, and — if genuinely undecided — explicitly list it under §12 Deferred/open rather than leaving it unaddressed.

2. **[LOW, inherited] AD-EXT-7's RYW Rule carries forward AD-11's own unresolved "until projection catches up (or the confirm ack is withheld until it does)" branch** without picking one. This is inherited verbatim from the read-only parent invariant, not introduced here, so it doesn't breach item 7 or item 5 — but it does leave one live two-way choice for whichever B4 story implements the RYW read, at a document that otherwise closes every other build-facing choice.
   *Fix:* Either pick one branch explicitly for this component (with a one-line rationale), or name it explicitly in §12 as a deferred choice inherited from AD-11, rather than letting it read as settled inside an AD-EXT Rule statement.

3. **[LOW] Canary-remediation ownership is undecided.** §6/§11 decide the remediation *channel* and *SLA target* (signed remote-JSON, ≤24h assumption) and the *routing* of the paging alert (OTLP metric + Loki log via `api`), but never name who is paged / who owns shipping the selector-pack fix within that SLA — a minor instance of the same operational-envelope gap as Finding 1.
   *Fix:* One line naming the owning team/on-call rotation for canary-trip remediation, or an explicit open-item entry in §12 if genuinely TBD.
