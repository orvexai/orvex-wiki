# ADR-0034: Credential lanes for agent execution — deny-by-default per-lane scoping, mint-scoped-ephemeral, allow-list dispatch

| | |
| --- | --- |
| **Status** | Proposed |
| **Date** | 2026-07-13 |
| **Deciders** | Daniel (PO) |
| **Supersedes / relates** | Refines ADR-0006 §2 and **partially reverses its 2026-07-07 amendment** — the amendment allowed a raw provider key so the run never blocks; §3 narrows that to a scoped `orvex-studio-ai` broker-token (with a scoped-raw-key interim fallback until that broker ships), preserving never-block while removing the standing master-key exception. Also supersedes the hard refuse-gate restatement of orchestrator-prompt §2.3 (the allow-list model here replaces it). Inherits the vended-ephemeral **agent**-credential principle from **Houston ADR-0009** (not Studio ADR-0009); reuses the OIDC→OpenBao mint pattern; parallels the deny-by-default scope gate of ADR-0029 and the caller-token pass-through of ADR-0021 |

## Context

The autonomous delivery loop (orchestrator prompt `gkkUDzn277` §6A) fans out **build** and **verify/review** sub-agents across the family repos. Each agent runs code, reads canon, runs tests against real backing services, commits to a work branch, and reports a verdict — and to do that it touches credentials: the Linear token, GitHub (git push *and* the REST/GraphQL merge token — two distinct credentials, `gkkUDzn277` §6A.5 / 2026-07-09 lesson), the cluster kubeconfig, the `docmost-cli` wiki token, and — behind all of them — OpenBao secret paths. The question this ADR settles is the one `gkkUDzn277` §2.3 left open and ADR-0006 answered only for the orchestrator's own seat: **how does a *sub-agent* obtain credentials, and what is it structurally prevented from ever holding?**

**What is already decided (the substrate this ADR builds on):**

- **ADR-0006 §2 — seat credential model.** The orchestrator process (operator laptop) holds the powerful credentials: cluster-admin kubeconfig (build-only validation / read, **never** `kubectl apply` — §3.17), `DOCMOST_API_TOKEN`, `gh` + git-over-SSH, and kubectl-reachable in-cluster OpenBao. Sub-agents hold **none of them**; LLM auth is the harness's own.
- **ADR-0006 amendment (2026-07-07).** The original *refuse-dispatch-on-raw-key* hard gate was **withdrawn**: build/review agents **MAY** be handed a provider API key when a task genuinely needs one, because the PO's overriding priority is that the run **never blocks** — "a hard credential gate that halts dispatch is a worse failure mode than handing a scoped key to an agent that needs it; keys are still scoped and passed only on need." That last clause is load-bearing here.
- **CS §10 (zero-trust baseline).** Least privilege; secrets via env only; secrets reach workloads via the GitOps/ExternalSecrets path, never the repo; **a provider key found outside its owning service is a build failure** (§3.2 / §6 LLM confinement — LiteLLM lives only in `orvex-studio-ai`).
- **The OIDC→OpenBao mint precedent (in production today).** Consumer CI fetches the private `orvex-studio-lib` module with **no stored secret**: GitHub Actions OIDC → OpenBao `jwt-github` auth → a short-lived (`TTL ≤ 15m`), narrowly-scoped (`contents:read`) `clone-only` token, minted on demand and never exported. This is the family's proven shape for "credential a workload holds": **minted, scoped, ephemeral — never a long-lived master key at rest in the workload.**
- **ADR-0029 §5 / ADR-0021.** The family already runs a **deny-by-default scope gate** (`requireScope()` before every upstream) and a **caller-token pass-through** (forward the already-verified principal, never re-mint a service credential). Same instincts, one layer out.

**The tension.** ADR-0006's amendment optimized for *never block*, and in doing so left the *loosest* posture as the only written rule: an agent may receive a raw provider key. Read literally against CS §10, that is a standing exception to zero-trust — a long-lived master key in a sub-agent's dispatch env (or worse, echoed into a committed test fixture — CS §5 "hand-authored responses are a defect", and fixtures are committed) has a family-wide blast radius. Meanwhile the amendment's *own rationale* already points at the resolution: keys should be **scoped and passed only on need.** Nobody actually decided "raw master key"; the amendment decided "don't halt the run."

The two goals only *look* opposed because ADR-0006's mechanism was a **deny-list refuse-gate** — detect a forbidden key, then refuse. A refuse-gate can only ever *block*, so withdrawing it was the only way to stop it blocking, and that threw out the confinement with the block. The mechanism, not the intent, was the problem. This is a security-boundary change that reasonable engineers differ on and that is costly to reverse once every dispatch path depends on it (CS §9 ADR trigger; SE-Arch `8sYi523i4t` §4i), so it is filed as a family-tier ADR.

## Decision

**Confine each agent role to a named, deny-by-default credential *lane*. A dispatch env starts empty and is populated by an *allow-list* of exactly the lane's declared credentials — each minted short-lived and narrowly scoped on the OIDC→OpenBao pattern. Forbidden credentials are never *candidates* for injection, so nothing is ever "refused" and no dispatch is ever blocked.** This replaces ADR-0006's withdrawn raw-key *deny-list refuse-gate* with a *credential allow-list*, which honors CS §10 least-privilege and the ADR-0006 amendment's never-block mandate **simultaneously**.

### 1. The lanes

**Control lane — orchestrator process only, never enters any dispatch env.** Cluster-admin kubeconfig (build-only/read; never `apply`/`sync` — ADR-0006 §2, §3.17), the `gh` REST/GraphQL **merge** token, the `docmost-cli` **write** token, kubectl-reachable in-cluster **OpenBao** auth, and the live **Linear write** token. These are the operator-held credentials of ADR-0006 §2; this ADR does not widen them and forbids their appearance in any child env.

**Build lane — build sub-agent.** Receives exactly: (a) a **work-branch-scoped git push** credential — an SSH deploy key or short-lived push token confined to `feat/*`/`salvage/*` work branches, **never** the `gh` REST/merge token (the two are distinct — the 2026-07-09 lesson — and merge stays orchestrator-serialized, §6A.1.6, so no agent ever needs merge authority); (b) a **read-only** worktree checkout (`isolation: 'worktree'`); (c) **read-only wiki grounding** — a `docmost-cli` read-scoped token or pre-fetched canon context, never the write token; (d) **Linear reads via the repo-local `.cache/linear/`** (cache-first, `gkkUDzn277` §3.11) — **no live Linear token**; (e) **injected per-service connection URLs** for the store-integration tests it must run (§1.3/§3.15), sourced by the orchestrator's OpenBao read and passed as ordinary env — **never the kubeconfig, never OpenBao auth itself.** LLM auth is the harness's own by default (item 3).

**Verify/verify lane — review sub-agent (reviewer ≠ implementer, opus).** Receives a **read-only** diff + worktree, the ability to run the exact CI gates locally (`make ci-local`/`lint-backend` at the pinned version — the §6A.5 deterministic-gate lesson), read-only wiki grounding, and Linear read-cache. **No push credential, no merge token, no write token of any kind.** A reviewer that cannot mutate anything cannot launder a bad verdict into a merge.

### 2. Enforcement is structural (allow-list), not detective (deny-list)

The dispatch env is constructed by **enumerating the target lane's declared credentials and injecting only those.** There is no scan-for-forbidden-key step because a forbidden credential is never a candidate — it is structurally absent, the way ADR-0029's `requireScope()` is deny-by-default rather than deny-listing bad scopes. Consequences:

- **No dispatch is ever refused or blocked at dispatch time** — the ADR-0006 amendment's mandate holds by construction, not by exception. "Handed a key when needed" becomes "the lane *declares* the key it needs and gets exactly that," which is the amendment's own "scoped, passed only on need" made mechanical. (A lane that under-declares a credential still surfaces later as a per-ticket task failure, not a run halt — see Negatives; the guarantee is never-block-the-run, not never-stall-a-ticket.)
- **The control-lane credentials cannot leak downward by omission-error**, because a child env is built up from empty, not filtered down from the parent. This is the single most important property: cluster-admin, the merge token, and OpenBao auth are absent because they were never added, not because a filter happened to catch them.

### 3. Model access is an AI-brokered scoped token, never a raw provider key

The default remains **harness-mediated** LLM auth (ADR-0006 §2). When a task genuinely needs model capability beyond the harness, the build lane declares a **scoped, metered, short-TTL token brokered through `orvex-studio-ai`'s contract API** (CS §6 LLM confinement — no provider/LiteLLM master key exists in any repo but `orvex-studio-ai`). This is the amendment's "hand a key when needed" preserved in full — **narrowed to a scoped derivative rather than a raw master/gateway key.** Nothing about delivery blocks; the blast radius of the thing handed shrinks from "the family's provider account" to "one metered, expiring, ai-scoped token."

**Interim rule — until the `orvex-studio-ai` broker ships.** The broker-token scope vocabulary is an open follow-up (below), so the broker token is not yet issuable. In the interim a **scoped, short-TTL raw provider key remains a legal, lane-declared credential for the narrow model-capability case** — minted and metered like any other lane credential, never a standing master key at rest. This is deliberate: the ADR must not forbid the only available fallback while its replacement is unshipped, or it re-creates exactly the block the 2026-07-07 amendment withdrew. Once the broker ships, the scoped broker-token supersedes the raw key and the raw-key lane is retired.

### 4. Credentials are minted scoped-and-ephemeral, on the OIDC→OpenBao pattern

Every lane credential is **short-lived and narrowly scoped**, modeled on the in-production `clone-only` mint (OIDC → OpenBao → `TTL ≤ 15m`, single capability). On the single-operator **laptop seat** (ADR-0006 §1) the sub-agents are in-process children of the orchestrator, so "mint" is concretely the **orchestrator deriving a scoped, expiring credential from its own OpenBao read and injecting it** into the lane env — not a full per-agent OIDC identity exchange. A true per-agent OIDC-vended identity (Houston-style) is **deferred to the in-cluster / multi-operator seat decision** already open as an ADR-0006 §5 follow-up; adopting it now would be over-engineering for a seat where the orchestrator is the parent process and the trust root.

## Consequences

**Positive**

- **Reconciles ADR-0006's amendment with CS §10 without re-litigating either.** Never-block survives (allow-list can't block); least-privilege returns (deny-by-default lanes). The apparent conflict was a mechanism artifact — a deny-list can only block — and swapping to an allow-list dissolves it.
- **Structural, not vigilant, confinement.** cluster-admin, the merge token, `DOCMOST` write, OpenBao auth, and provider master keys are absent from build/verify envs because they are never added — no scan to forget, no filter to misconfigure. Matches the family's existing deny-by-default posture (ADR-0029 §5) and pass-through-don't-re-mint posture (ADR-0021).
- **Bounded blast radius per lane.** A compromised or leaky build agent holds at most a work-branch push key, a read token, and expiring per-service test URLs — it cannot merge, cannot `apply` to the cluster, cannot read arbitrary OpenBao paths, cannot mint against the family provider account.
- **Merge authority stays where ADR-0006/§6A already put it** — the orchestrator's serialized per-repo lane — so the tightest credential (the `gh` REST token whose loss dried both engines on 2026-07-09) is held in exactly one place and preflighted once.
- **Reuses a proven, in-production shape** (OIDC→OpenBao scoped-ephemeral mint) rather than inventing a new one.

**Negative / accepted cost**

- **The orchestrator must derive-and-scope credentials per dispatch** instead of the current "hold everything, inject a fixed env." That is real work in the dispatch path — a per-lane credential manifest and a derive/inject step — where today the seat mostly passes connection URLs. Accepted: it is the cost of least-privilege, and it is bounded (a handful of lanes, not per-ticket bespoke).
- **On the laptop seat "mint" is orchestrator-derived, not OIDC-vended**, so the sub-agent's ultimate trust root is still the orchestrator process holding the control lane. This does **not** protect against a compromised *orchestrator* — only against a compromised or leaky *sub-agent*. Full per-agent identity waits on the in-cluster seat (ADR-0006 §5). Honest scope: this ADR shrinks the *sub-agent* blast radius, not the *seat* blast radius.
- **A lane that under-declares will surface as a task failure** ("test needs a connection URL the lane didn't carry") rather than silently over-privileging. This is the intended failure direction (fail toward least-privilege, escalate the gap per §3.17 / §6A.3 — a missing credential is a clean escalation, never a fake-pass), but it means lane manifests need maintenance as build needs evolve.

**Follow-ups (open, not decided here)**

- Per-agent OIDC-vended identity for the in-cluster / multi-operator seat (rides ADR-0006 §5).
- The concrete lane-manifest format and the `orvex-studio-ai` broker-token scope vocabulary (an `orvex-studio-ai` PRD / contracts item).
- Whether the review lane's read-only `docmost` grounding should itself be a scoped read token vs. pre-fetched context (a `docmost-cli` capability question).

## Alternatives considered

- **Keep the ADR-0006 amendment as-is (an agent may receive a raw provider key when needed).** Rejected: it leaves the loosest posture as the only written rule and stands as a permanent exception to CS §10 §3.2/§6 — a long-lived master key in a dispatch env or a committed fixture has family-wide blast radius. This ADR *keeps the amendment's intent* (never block; hand what's needed) while narrowing "what's needed" to a scoped derivative — which the amendment's own rationale already asked for.
- **Reinstate ADR-0006's original refuse-dispatch hard gate.** Rejected: the PO explicitly withdrew it to stop it blocking the run, and re-imposing a run-halting gate contradicts both that ratified amendment and the §6A never-block mandate. The allow-list gets the confinement *without* the block, so the refuse-gate is strictly dominated.
- **Full per-agent OIDC identity vending now (Houston-style central refresher + metered gateway lane).** Rejected for the current seat / deferred: over-engineered for a single-operator laptop where sub-agents are in-process children and the orchestrator is the trust root (ADR-0006 §1/§5). Revisit at the in-cluster/multi-operator seat decision, where it becomes the right shape.
- **One shared "agent" credential set for both build and verify.** Rejected: it would hand the review lane a push/write capability it must never have (reviewer ≠ implementer, `gkkUDzn277` §6A.1.4) — the separation of the two lanes is the point.
