## Context

The Delivery Orchestrator (`gkkUDzn277`) runs Act 3 as an autonomous loop: compute the ready frontier → **claim** an issue → build → adversarially review → deterministic Done-gate → merge → advance. The *claim* — the mechanism by which exactly one build agent takes ownership of one issue so no two agents build the same work — is flagged in §2.2 as an **OPEN DECISION with a known race window**, and the prompt's own default names a stopgap it forbids from silently calcifying: *"Linear-status-as-claim under a single orchestrator … the Act-1 run must either ratify it or replace it."* ADR-0006 §5 ratified the operator-workstation seat but deferred this arbiter question outright. This ADR closes it.

**What the claim actually is today** (grounded in the landed engine, `tools/act3/delivery-engine.js` v9). A claim is: (1) an explicit Linear state flip of the issue to **In Progress** via `linearis` — *"you are the single claimer"* — followed by (2) a `linear-sync.sh issue <id>` refresh that read-backs the flip and rewrites the local cache. There is no compare-and-swap: the flip is an unconditional write, and read-after-write is the claim-verify. The frontier that feeds candidate selection is *derived*, not authoritative — it is computed locally from `.cache/linear/initiative.json` (`state == Todo|Backlog`, all `blockedBy` edges resolved to Done/Canceled/Duplicate), at zero API cost, under the cache-first model (§3.11) where every engine write refreshes its own ticket so the cache stays current.

**Why this is safe today — the mutual exclusion is structural, not lock-based.** Two independent layers already make same-issue double-claim *impossible* in the current architecture, which is why no CAS has been needed:

- **Intra-engine:** an in-memory `claimedIds` Set (v9 rolling-saturation loop) guarantees a single engine never re-dispatches an issue it already claimed, even off a briefly-stale cache read between flip and refresh.
- **Cross-engine:** to reach the §3.28 ceiling of 32 agents, delivery runs **two engines over static, disjoint *project* partitions** (`args.partitionProjects`). An issue lives in exactly one project and each repo in exactly one partition, so two engines can never select the same issue. §3.18 states this explicitly: *"this static partition IS the claim arbiter required … before a second concurrent claimer; the orchestrator session remains the single orchestrator."* Blockers are still judged across the *whole* initiative cache, so cross-partition dependencies resolve correctly.

So the "race window" §2.2 warns about is not an open contention race in the shipped system — it is the *absence of a lock* that would matter **only if** the structural invariants (single orchestrator, single Linear writer, provably-disjoint partitions) were broken. The real, present residual risk is a different failure mode: **stale claim** — an issue stuck In Progress after an agent or engine dies mid-build, invisible to the Todo/Backlog frontier and silently clogging capacity (§3.31). A secondary, well-understood hazard is **spurious claim-release**: Linear's GitHub integration auto-flips a linked ticket to Done seconds after its branch merges (pattern P1, confirmed victims ENG-1405/ENG-1395), which would surrender a claim the gate has not yet earned.

**The forces:**

- **Correctness vs. cost.** A real arbiter (a CAS store the engine consults before building) buys mutual exclusion under *concurrent contention for the same issue* — a state the current architecture makes structurally unreachable. Building one now is a synchronizer for a race that cannot occur.
- **Single-source-of-truth (§3.5).** Linear is canonical for delivery state. Any second claim store is a *second* delivery-state source that must be kept in lockstep with Linear — a standing dual-write hazard (§3.12) and exactly the kind of split-brain the doctrine works hardest to prevent.
- **Quota (§3.32).** Linear's 2,500 req/hr workspace budget already bounds the engine. Linear-status-as-claim adds zero coordination calls beyond the flip that would happen anyway; it rides the writes-only steady state.
- **The §2.2 mandate.** The stopgap must be *ratified or replaced* — it may not stay an undocumented default. Ratifying it requires converting the implicit invariants into an explicit, enforceable contract and a named trip-wire, or it calcifies exactly as forbidden.

## Decision

**Ratify Linear-status-as-claim as the delivery work-claim arbiter, bound by an explicit guardrail contract, with a pre-designed trip-wire to a Temporal deterministic-ID CAS in orvex-workflows the moment any invariant it rests on can no longer hold.** No standalone arbiter service and no external CAS store is introduced now.

### 1. The claim record is the Linear `In Progress` state; the arbiter is the invariant set, not a lock

The authoritative claim on an issue is its Linear workflow state being **In Progress**, written by the engine's explicit `linearis` flip and verified by the mandatory refresh read-back. This is ratified — not merely tolerated — because mutual exclusion is provided *structurally* by the four invariants below, which the engine already embodies and which this ADR now makes binding contract:

- **G1 — Single orchestrator.** Exactly one orchestrator session drives delivery (ADR-0006 §5). Additional *engines* are permitted; additional *independent orchestrators* are not, until the trip-wire (§4) fires.
- **G2 — Single authoritative Linear writer.** The delivery engine(s) are the SOLE *authoritative* writer of in-scope issue *state*. No human drag and no second automation may authoritatively flip delivery state during an autonomous run. One known non-authoritative writer is tolerated by design — Linear's GitHub integration auto-closing a linked ticket to Done on merge — and that write is detected-and-reverted per §3 (pattern P1), never left standing. External drift is *detected* (not prevented) by the §3.11 45-minute bulk-resync bound; a human edit mid-run is an out-of-contract event the operator owns.
- **G3 — Provably-disjoint project partitions.** When two engines run, their `partitionProjects` sets MUST be disjoint at the *project* granularity, and every in-scope issue's project MUST fall in exactly one partition. This is promoted from a convention to a **preflight assertion**: before a second engine launches, the engine computes the intersection of the two partition project-sets over the live initiative and REFUSES to start if it is non-empty or if any in-scope project is unpartitioned. Partition-at-project-granularity is the cross-engine arbiter; issue-level parallelism across engines is prohibited under this ADR (it would break the disjointness proof — see §4).
- **G4 — Intra-engine dispatch guard.** The in-memory `claimedIds` Set remains mandatory: an engine never dispatches an issue it has already claimed this run, independent of cache freshness.

### 2. Claim liveness — harden stale-claim reclamation (the one genuine present gap)

Stale claim is the real residual risk, and it is a *liveness* problem (work stranded), not a *safety* problem (no double-build). Two mechanisms, one existing and one added:

- **Startup reclaim (existing, kept).** On launch, before the first frontier computation, the engine resets every In Progress / In Review issue in its partition with no open PR back to Todo (routing open-PR issues to review+merge), so a prior engine death never leaves work stranded behind a dead claim (§3.20c / §3.31).
- **Mid-run claim lease (added).** Startup reclaim only fires at engine *restart*; an agent that dies while its engine keeps running strands its claim until the next restart. The claim therefore carries a **lease keyed on a concrete, engine-owned liveness signal** — a live dispatched-agent handle table (the in-process PID/child-handle the engine holds for each agent it launched this run), **not** the issue's `updatedAt` (which a slow-but-alive build leaves unmoved). The rolling loop may reclaim an In Progress issue in its own partition **only when all three hold: (i) the engine holds no live dispatched-agent handle for that issue, (ii) there is no open PR, and (iii) a hard wall-clock staleness window has elapsed** — reset to Todo and re-picked. Because reclaim keys on the engine's own liveness table rather than a timestamp, a slow-but-alive agent is never reclaimed (its handle is live), so the lease stays a pure *liveness* fix and cannot manufacture a double-build. This closes the mid-run hole with no external store: the engine-owned handle table and the Linear state the claim already uses are the only inputs.

### 3. Spurious claim-release stays guarded

The P1 anti-auto-close guard remains mandatory and binding: immediately after any merge (and after pushing an identifier-named branch/PR), the engine re-reads the ticket's LIVE status, and if it is Done but this engine's Done-gate did not perform that transition, reverts it to In Progress with the audit comment and continues. Done is gate-owned; an engine-unauthored Done (Linear's GitHub-integration auto-flip) is never left standing. This is a property of the claim contract, not incidental engine code.

### 4. Trip-wire — the exact conditions that force migration to a real arbiter, and the named successor

Linear-status-as-claim is ratified **only while all of G1–G4 hold**. Any one of the following breaks an invariant and is the **mandatory trigger** to migrate to a real arbiter *before* the change ships — this is what keeps the decision from calcifying:

- **T1** — More than one *independent* orchestrator operator (not merely >1 partitioned engine under one orchestrator). Two humans launching engines against a shared initiative breaks G1/G2.
- **T2** — Any sanctioned *concurrent* non-engine writer to Linear delivery state (a second automation, a bot, a routinely-hand-driven board during runs). Breaks G2.
- **T3** — A parallelism model that cannot keep partitions provably disjoint at project granularity — most concretely, *issue-level* cross-engine claiming, or dynamic re-partitioning mid-run. Breaks G3.

**The named successor is a Temporal deterministic-workflow-ID CAS hosted in orvex-workflows — not a new service, not Dolt, not a bespoke arbiter.** ADR-0023 (`pjacNq8SQY`) already establishes orvex-workflows as the family's single durable Temporal control plane, and its decision 4 already uses exactly this primitive for dedup: `client.StartWorkflowOptions.ID` server-side ID-reuse rejection is a mutual-exclusion CAS. A future `ClaimWorkflow` keyed on a deterministic `claim-<ENG-N>` id gives real, quota-independent, concurrent-safe mutual exclusion with **zero new infrastructure and zero new failure domain** when a trip-wire actually fires. Recording the successor now is the difference between a bounded stopgap and an undocumented default.

## Consequences

**Positive**

- Ratifies the mechanism that already delivered work with no double-build in the shipped v9 engine — zero migration, zero new infra, no second delivery-state source to keep in lockstep with Linear (honors §3.5 single-source-of-truth and avoids the §3.12 dual-write hazard).
- Adds no Linear-quota cost (§3.32): the claim rides the In-Progress flip the engine already performs; the frontier stays a free local recompute.
- Converts three implicit invariants into an enforceable contract — the G3 partition-disjointness **preflight assertion** turns a code convention into a launch-time guard, removing the single way today's cross-engine safety could be silently violated.
- Closes the one genuine present gap (mid-run stale claim) with an in-band lease rather than an external lock, so liveness improves without introducing a coordination store.
- Satisfies §2.2's ratify-or-replace mandate honestly: the stopgap becomes a *bounded, invariant-gated* architecture with an explicit trip-wire and a pre-designed, zero-new-infra successor — the opposite of silent calcification.

**Negative / accepted risk**

- **The arbiter is a set of invariants, not a lock.** Its correctness is only as strong as G1–G4 hold. A violation (a second human engine, an issue in two partitions, a mid-run manual board edit) reintroduces the double-claim race with no runtime backstop beyond detection. Mitigated by the G3 preflight assertion and the §3.11 drift bound, but a *deliberate* invariant breach that skips the trip-wire is unprotected — accepted, because that breach is exactly the T1–T3 condition this ADR forbids shipping without the successor.
- **Linear is on the claim's critical path.** A Linear outage or rate-limit stalls claiming (the flip cannot be written). Accepted: Linear is already canonical for delivery state and already gates every write; the claim adds no new dependency the engine did not already have.
- **The mid-run lease adds a small correctness surface** (a wall-clock staleness window + the engine-owned live-handle check). Reclaim is gated on the engine holding **no live dispatched-agent handle** for the issue (plus no open PR and an elapsed hard window), so a genuinely-slow-but-alive agent — whose handle is still live — is never reclaimed. The safety of the lease rests on that liveness signal alone; the intra-engine `claimedIds` guard does **not** backstop it, because re-picking a reclaimed issue necessarily removes it from `claimedIds`. A too-eager wall-clock window (with the liveness check still honored) is a liveness annoyance, never a double-build (the reclaimed issue re-enters the same single-claimer frontier).
- **Deferring the real arbiter is a bet that the team stays single-operator.** If the team grows faster than expected, the T1 migration lands on the critical path. Mitigated by pre-designing the successor here so the migration is a scoped `ClaimWorkflow` add in an existing service, not a new architecture.

## Alternatives considered

- **Build a real claim arbiter now (Temporal `ClaimWorkflow` CAS, or a purpose-built arbiter service).** Rejected *for now*, not on merit — it is the correct successor (§4). Building it today is a synchronizer for a race the current architecture makes structurally unreachable (G3/G4), and it introduces a second delivery-state source of truth that must be reconciled with Linear on every claim/release — a standing §3.12 dual-write split-brain hazard and a new failure domain, for zero present benefit while the operator count is one. Premature generality that fights §3.5.
- **Adopt Houston's Dolt CAS as the arbiter.** Rejected. Dolt CAS is Houston machinery with no other foothold in the Studio family; adopting it would add a brand-new engine and operational surface where ADR-0023's already-committed Temporal control plane offers the identical CAS primitive (`StartWorkflowOptions.ID` reuse-rejection) with no new infrastructure. If a real arbiter is ever needed, Temporal is the strictly cheaper home.
- **Keep Linear-status-as-claim but leave it undocumented (the literal status quo).** Rejected — this is precisely the "silently calcify into the architecture" outcome §2.2 forbids. The mechanism is sound; leaving its invariants implicit and its trip-wire unstated is the actual defect.
- **Guard the race with an optimistic CAS on Linear's own state** (read state, conditionally flip only if still Todo). Rejected as insufficient and misleading: Linear's API offers no conditional-write primitive on workflow state, so a "CAS" would be a read-then-write with a TOCTOU gap — weaker than the structural disjointness G3 already provides, while implying a safety guarantee it cannot keep. The partition invariant is a stronger arbiter than a simulated CAS on a non-transactional field.
