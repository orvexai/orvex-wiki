# ADR-0035: Go↔TypeScript contract/client bridge — per-repo codegen from the pinned seams

- **Status:** Proposed · Date: 2026-07-13 · Deciders: Daniel (PO)
- **Ticket:** ENG-2035

## Context

`orvex-studio-contracts` is the family's single seam repo. It ships **authored artifacts** — OpenAPI documents (`openapi/*.yaml`), JSON Schema (`events/schemas/*.json`, `identity/*.schema.json`, `schemas/*.json`), the CloudEvent catalog (`events/catalog.yaml` + `events/schemas/_envelope.json`, ADR-0007 `pYkl7HpOgr`), and the frozen error/exit-code vocabulary. It is explicitly a **non-building** repo — the Makefile states "nothing here runs as a service, so there is no build … the gates are the whole job." The two TypeScript packages it does ship (`@orvex/dfm`, `@orvex/audit-contract`) are **clean-room contract LOGIC**, not generated clients, and they are the *only* runtime TS in the repo.

The seam is Go-centric on the producer side (engine primitives, lib, the outbox relay) and TypeScript-centric on the consumer side. The three consuming satellites — `orvex-studio-api` (BFF), `orvex-studio-mcp` (agent gateway), `orvex-studio-ui` (SPA) — are all TS and must consume the *same* pinned seams the Go legs produce. Today they do it three different ways, and two of them are placeholders:

- **`orvex-studio-mcp` — the proven pattern.** It already generates its clients with **`openapi-typescript` (types) + `openapi-fetch` (runtime)** via `scripts/gen-client.ts` / `scripts/gen-discovery-client.ts`. The codegen SOURCE is a **local checkout** of the contracts repo (`ORVEX_STUDIO_CONTRACTS_DIR`, defaulting to a sibling clone) — never a live fetch, and pointedly **never** wiki-api's served `/v1/openapi.json`, which is a conformance artifact and not a codegen source (ENG-1406 R2, D-A7). Output lands **per-repo, checked in** (`src/backstage/generated.d.ts`, `src/routing/generated.d.ts`); the pin is recorded in `package.json` (`contractsRelease: "v0.1.2"`, `orvexContractsPin`); and a CI determinism gate (`test/ci/eng-1406-r2-regen.test.ts`) asserts the checked-in output matches a fresh regen against the pinned source. Regen is "a deliberate, reviewed PR — never a background refresh."
- **`orvex-studio-api` — waiting on this decision.** Its typed surfaces are hand-authored with explicit standing TODOs: `src/types/domain.ts` ("land once orvex-studio-contracts pins a TS codegen path") and `src/cards/card.ts` ("hand-authored typed surface until orvex-studio-contracts pins …"). It has no codegen wired.
- **The POC — the vocabulary being replaced.** `orvex-prompt-studio-poc` ships `@orvex/shared`: a **hand-authored, zod-based DTO vocabulary** (`skill-dto`, `skill-card-dto`, `memory/card`, `me-stats`, `feed`, `blocks`, …) plus a large `copy/*` UI-string bundle, consumed across the POC's apps. This is the shared shape vocabulary the studio product currently runs on, and it predates any authored BFF contract.

There is **no CloudEvent type generation anywhere** — the Go legs read the event schemas directly, but the TS consumers (mcp's revocation consumer, studio-api's `/internal/events` sink, ui via the BFF) have no generated envelope/payload types. ADR-0029 (`WZWmazrlS0`) item 6 already records that the TS Kafka *binding* is unshipped; the *types*, however, are independent of the driver and can land now.

This is a **family-wide seam-consumption contract** (six-plus repos pin the contracts tag), so the mechanism must be decided once, not per repo. It is a CS §9 ADR trigger: reasonable engineers differ on centralized-package vs per-repo codegen, and it is costly to reverse once satellites cut over. Per ADR-0008 (`JLMuwgc9fV`), the codegen *mechanism* is a reshaping decision (ADR lane); routine *regen against an additive contract bump* is mechanical (automated lane).

## Decision

Adopt a **per-repo codegen bridge** driven by a **contracts-owned, centralized generator recipe**, sourced from a **local pinned checkout** of the authored artifacts. Generalize the proven mcp pattern into the family rule; do **not** introduce a published client package.

### 1. Toolchain — `openapi-typescript` + `openapi-fetch` for REST; `json-schema-to-typescript` for events

- **REST seams (OpenAPI):** generate **types only** with `openapi-typescript`; call them at runtime through `openapi-fetch`. This is already the mcp toolchain and is the family standard for every TS consumer of an `openapi/*.yaml` document. `openapi-typescript` emits an idiomatic types-only `.d.ts` with zero runtime dependency; `openapi-fetch` is a ~2 KB typed `fetch` wrapper. No Java, no heavyweight generated client classes.
- **CloudEvent / JSON-Schema seams:** generate **types** with **`json-schema-to-typescript`** off `events/schemas/*.json` + `_envelope.json` (and the standalone `schemas/*.json` shapes). Runtime *validation* at a trust boundary is done with **ajv against the shipped JSON Schema** — the schemas are already the SoT and are already meta-validated in contracts CI (`scripts/validate_schemas.py`), so the wire validator is the authored schema itself, not a hand-copied mirror.

### 2. Generated code lives **per-repo, checked in** — not in a contracts `gen/` tree, not in a published package

Each TS satellite generates exactly the tags it consumes and **commits** the output (mcp's `generated.d.ts` model, generalized). The contracts repo stays a non-building seam repo: it ships **no** generated TS and **no** npm-published client artifact.

- **What is centralized is the RECIPE, not the output.** The contracts repo ships one shared codegen script + a **codegen manifest** (`codegen.manifest.yaml`) that pins, for the whole family: the generator versions, the deterministic flag set (stable ordering, no timestamps), and — per consuming satellite — which authored artifact(s) are that satellite's legal sources. Each satellite invokes that shared recipe; config is centralized, output is distributed. This removes the only real downside of per-repo generation (config drift) while keeping every other advantage.
- **Why per-repo over a central `@orvex/openapi-clients` package:** (a) it keeps the seam repo non-building — no npm publish, no registry, no license-surface or publish infra to grow (any npm publish would force the deliberately non-building seam repo to grow a package-license + registry-publish surface it lacks); (b) it matches ADR-0006 (`79YR5C7caM`) — persistence is git + GitHub, no new infra, no auto-elastic anything; (c) each satellite keeps its own tsconfig target/module system rather than inheriting one package's; (d) the generated-type diff surfaces **in the consuming PR** — a breaking OpenAPI change shows up as a red diff in `generated.d.ts`, a **downstream** visibility of a break already classified at the contracts repo per ADR-0008 — a second confirmation, not the CS §9 review surface itself (that gate fires on the contracts-repo schema change).

### 3. Codegen source eligibility follows `x-status`; never the AGPL served descriptor

- The codegen source is **always a local checkout of the authored YAML/JSON** (`ORVEX_STUDIO_CONTRACTS_DIR`), never a live/served descriptor (D-A7). Generating TS types from the Apache-2.0 authored OpenAPI is clean-room; generating from the AGPL engine's served descriptor is forbidden (D-CON-8 / the agpl-guard boundary).
- **Draft artifacts may be generated against for in-flight development** (mcp already codegens off `wiki-api.yaml`, still `x-status: draft`), but a satellite may not **ship a GA cutover** whose frozen dependency is a draft-sourced client: the pin recorded for a launch must reference a **`x-status: pinned`** artifact at a released tag (SEAMS rule: "nothing downstream may pin a `draft` artifact"). This keeps the M0-frozen six-row seam set honest while unblocking day-to-day builds.

### 4. Version pinning = the contract tag + a recorded per-artifact pin, regen is a reviewed PR

Generalize mcp's pin model as the family rule:

- Each consumer records `contractsRelease` (the git tag, e.g. `v0.1.3`) and an `orvexContractsPin` map (per-artifact commit SHA) in its `package.json`.
- Regeneration is a **deliberate, reviewed PR** — never a background refresh. A **CI determinism gate** (mcp's `eng-1406-r2-regen` generalized) re-runs codegen against the pinned source and fails if the checked-in output drifts.
- **This slots under ADR-0008's two lanes:** a regen against an *additive* bump (new optional field, new path) is the **mechanical/automated lane** — bump the pin, regen, green diff, merge, no ADR. A generated-type diff that **deletes or renames** a field surfaces the drift-gate's MAJOR classification downstream — but that classification fires at the **contracts repo per ADR-0008**, *before* any consumer regens; the consumer's regen diff is a second, downstream confirmation of a break already classified upstream, not where the ADR-gate itself fires.

### 5. The POC `@orvex/shared` vocabulary is replaced tag-by-tag, not lifted wholesale

- **DTO shapes migrate to generated-off-contract.** As the BFF surface (`openapi/studio-api.yaml`) and its product sub-domains (`skills`/`memory`/`marketplace`, `noop-501` today per SEAMS) are authored and pinned, each POC DTO with a served counterpart becomes a **generated type off `studio-api.yaml`**. The generated type is the type SoT.
- **The zod runtime role is preserved at the trust boundary, not thrown away.** `openapi-typescript` emits types only; the POC leaned on zod for runtime *parsing*. Each replaced DTO keeps a **thin hand-authored zod validator at the BFF/UI ingress** that parses into the generated type (or, for the JSON-Schema-backed shapes, an ajv validator per §1). A schema→zod generation step may be added later if hand-authoring proves costly (hey-api is the noted fallback, §Alternatives).
- **UI-only content does not move.** The `copy/*` string bundles and UI-only helpers in `@orvex/shared` are **not** a cross-service seam and stay in the UI repo — they never belong in contracts. `@orvex/shared` cleaves: DTO shapes → generated-off-BFF-contract; copy/UI helpers → stay put. No legacy shim, no baked POC shapes as canon before the BFF contract exists (no-fallback rule).

### 6. CloudEvent types unblocked independently of the Kafka driver

TS CloudEvent types (envelope + per-type payloads) are generated per §1 and land now **as draft-sourced types for in-flight dev per §3 — a GA cutover still waits on the catalog types reaching `x-status: pinned`**; they do **not** wait on the unshipped `orvex-studio-lib` TS Kafka binding (ADR-0029 item 6). The consumer *wiring* stays blocked on that shipment; the *types* do not.

## Consequences

**Positive**

- One toolchain, one recipe, one pin model across every TS satellite — mcp's already-proven, already-CI-gated pattern becomes the family rule instead of three divergent approaches. `orvex-studio-api` unblocks its two standing codegen TODOs immediately.
- The contracts repo stays a non-building seam repo: no npm publish, no registry, no build infra, no license-posture collision. Consistent with ADR-0006's "git + GitHub is the substrate, no new infra."
- The generated-type diff lands in the consuming PR and becomes the visible seam-change review surface, wiring ADR-0008's lane classifier (additive → automated; breaking → ADR) to something a reviewer can actually see.
- CloudEvent consumers get typed envelopes/payloads now, decoupled from the Kafka-driver blocker.
- The POC vocabulary is retired the right way — replaced as real BFF contracts are authored, with runtime validation preserved at the boundary — rather than fossilized into canon.

**Negative / accepted risk**

- **N regeneration sites.** Every satellite re-runs codegen and commits output. Mitigated by the contracts-owned shared recipe + manifest (one config) and the per-repo determinism gate (drift can't merge silently) — but it is genuinely more moving parts than a single `npm i @tag`.
- **No single-artifact "install the clients" story.** A newcomer wires the shared recipe + a local checkout rather than adding one dependency. Accepted as the cost of keeping the seam repo non-building and the diff reviewable.
- **Runtime validation is hand-authored (zod at the boundary).** `openapi-typescript` gives compile-time types only; the runtime guard is written by hand (or ajv-off-schema). If this proves costly at the BFF ingress, §Alternatives' hey-api path is the escape hatch.
- **Local-checkout coupling.** Codegen needs a contracts checkout on the box (`ORVEX_STUDIO_CONTRACTS_DIR`); there is no registry fallback by design (D-A7). Fine for the ADR-0006 laptop seat and CI-with-a-checkout; a hard requirement to document.

## Alternatives considered

- **Central published `@orvex/openapi-clients` package in a contracts `gen/` tree.** The tidy option — one generator config, `npm i @orvex/openapi-clients@v0.1.3`, semver == the contract tag, zero per-repo drift. **Rejected:** it forces the deliberately non-building seam repo to grow build + npm-publish + registry infra and a single tsconfig/license posture onto every consumer, and it hides the seam-change diff from the consuming PR (the CS §9 review surface). Its drift-avoidance upside is recovered by shipping the shared *recipe* instead of the *output* (§2).
- **`openapi-generator` (typescript-fetch / typescript-axios).** Rejected: a heavyweight Java toolchain producing verbose client classes, strictly worse TS-idiomatic output than `openapi-typescript` + `openapi-fetch`, which mcp has already proven end-to-end.
- **`orval` or `@hey-api/openapi-ts`.** Genuinely attractive — both can emit runtime **zod** validators alongside the client, which would close the §5 hand-authored-zod gap for free. Rejected as the default because they pull heavier, opinionated runtimes (orval couples to react-query; hey-api ships its own client + plugin model) that the BFF and mcp do not use, and they introduce a second generator idiom next to the one mcp already runs. **Noted as the explicit fallback** if hand-authored zod-at-the-boundary proves too costly at scale — adopting hey-api's schema→zod plugin would be an additive follow-up, not a re-decision of the type/client seam.
- **Lift POC `@orvex/shared` wholesale into contracts as the shared DTO package.** Rejected: it is product-UI vocabulary (including `copy/*` string bundles), not a frozen cross-service seam; importing it into the seam repo violates the repo's scope and would bake POC shapes as canon *before* the BFF contract is authored — a legacy shim of exactly the kind the no-fallback rule forbids.
- **Keep per-repo hand-authored types (status quo for api/ui).** Rejected: it is the drift-prone state the two standing TODOs in `orvex-studio-api` are explicitly waiting to exit, and it gives no mechanical link between an OpenAPI change and the consumer's types (no determinism gate, no ADR-0008 classifier surface).
