## 🎯 Story
As the outbox→Kafka relay and every content consumer, I want the normative publisher-obligations section authored in the catalog header — content events MUST carry a monotonic per-document `version` that IS the document's persisted write-commit counter (the same one FR-C1's `ifVersion` reads; the relay carries, never mints), tenant as a CloudEvents extension attribute, each type's ordering key, publish-only-registered-types, and the batch fan-out rule (the engine's `pageIds[]` batch emissions fanned to one CloudEvent per pageId, each with its own version + per-page ordering key; the SSE gateway MAY re-batch for legacy byte-compatibility) — so that the event CAS and the FR-C4 keyword-upsert CAS live in one comparable version space.

**Definition of Done:** ONE named test `TestVersionProvenanceAndFanout` — a CI-layer test asserting, through `validate.yaml`, that the obligations header pins `version` as the write-commit counter (not a relay/outbox sequence), that every content type declares an ordering key + tenant extension, and that no batch (`pageIds[]`) event type is emitted on the spine (only per-page). *Final H1–H17 elaboration + exact contract tag/versions are pinned at pack certification (ENG-2091); this story is dispatch-blocked until that tag exists.*
