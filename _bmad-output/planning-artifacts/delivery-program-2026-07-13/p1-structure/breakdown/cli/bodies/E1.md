## 🎯 Epic — Client Foundations, Service Registry & Transport Chokepoint

orvex-cli is rewritten **from scratch** in an initially-empty repo (`github.com/orvexai/orvex-cli`) as deliberate commits against the microservice architecture — never a fork or file-copy of `docmost-cli` (D-S21 AMENDED). This epic lays the three structures the reference could never fix in place: a **per-service endpoint registry** on the profile (replacing the single `profile.InstanceURL`), **one service-aware transport** every callsite funnels through (envelope decode, retry/backoff/`Retry-After`, correlation-ID origination, `Idempotency-Key` on replayable writes), and **per-service typed clients codegenned from the orvex-studio-contracts tag** (never a served descriptor). It also lands the direct-to-cell reliability posture the cell contract (JGAUQRsw2g) binds on a client that bypasses the Studio edge Worker: the 421 cell-mismatch re-resolve path (F-B). Everything downstream (wiki verbs, search/ai, daemon) queues behind this seam. [Source: PRD R4AOVBLST7 §Overview/F1/F4; Architecture pf10XC2Qjz §Components/§Reliability]

Why it exists: the audit (PACF13d3I3) found three structural facts — no service-routing seam, a triplicated HTTP layer with no chokepoint, and codegen bootstrapped from the LIVE engine served descriptor (the exact anti-pattern the contracts doctrine forbids). This epic turns each into a deliberate one-place structure so base-URL, envelope, error, and observability changes never drift across ~90 callsites.
