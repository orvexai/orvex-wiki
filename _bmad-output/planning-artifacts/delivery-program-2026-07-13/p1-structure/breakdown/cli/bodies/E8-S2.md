## 🎯 Story

As a **docmost-cli user**, I want the keychain service rename (`docmost-cli → orvex`) and token-dir move to perform a one-time read-through migration (read old → write new → delete old) across both the OS keyring and the encrypted fallback file, so that my stored tokens are never silently orphaned (closes defect D5).

**Definition of Done:** ONE named test `TestKeychainReadThroughMigrationNoOrphan` — an auth-layer test asserting, through the credential store, that on first run a token stored under the `docmost-cli` service is read, written under `orvex`, and only then deleted from the old service — for both the OS keyring and the encrypted fallback file — with no window where the token is lost. *Final H1–H17 elaboration + exact contract tag/versions are pinned at pack certification (ENG-2105); this story is dispatch-blocked until that tag exists.*

## ✅ Acceptance Criteria

- [ ] **AC1** — Given a token under keychain service `docmost-cli`, When first run, Then it is read, written under `orvex`, then deleted from `docmost-cli` (read-through order). *Assertion: new-write precedes old-delete; token retrievable under `orvex` after.* [Source: FR-CLI19]
- [ ] **AC2** — Given the encrypted fallback file, When migrating, Then the same read→write→delete order applies. *Assertion: fallback file migrated with no loss window.* [Source: FR-CLI19]
- [ ] **AC3 (crash-safety)** — Given a crash between write-new and delete-old, When re-run, Then the token is still present under `orvex` (idempotent, never orphaned). *Assertion: interrupted migration leaves a recoverable credential, not a vanished one.* [Source: FR-CLI19]
- [ ] **AC4 (negative)** — Given no old credential, When first run, Then migration is a no-op with no spurious deletion. *Assertion: absent old cred → no error, no delete.* [Source: FR-CLI19]

## 🔨 Tasks
