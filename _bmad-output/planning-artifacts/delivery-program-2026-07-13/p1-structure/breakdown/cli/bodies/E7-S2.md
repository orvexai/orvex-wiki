## 🎯 Story

As a **platform security owner**, I want each service edge to verify the forwarded token via the Go `lib/auth` binding, an api-key pass-through interim until identity ships end-to-end, and strict zero-trust propagation (no elevated CLI credential, 401/403/404 verbatim, no existence oracles), so that the CLI never makes an authorization decision and no ACL logic leaks client-side.

**Definition of Done:** ONE named test `TestEdgeVerifyAndZeroTrustPropagation` — an auth/transport-layer test asserting, through the transport, that a forwarded token is verified at the edge via `lib/auth` (stubbed edge), that the api-key pass-through interim path forwards raw keys when identity is not live, and that upstream 401/403/404 propagate verbatim with no existence oracle and no elevated credential injected. *Final H1–H17 elaboration + exact contract tag/versions are pinned at pack certification (ENG-2105); this story is dispatch-blocked until that tag exists.*

## ✅ Acceptance Criteria

- [ ] **AC1** — Given a request to any edge, When issued, Then only the caller's forwarded token is sent (no CLI-owned service credential). *Assertion: no elevated credential in any outbound call.* [Source: NFR-CLI4]
- [ ] **AC2** — Given identity is not yet live, When authenticating, Then api-key pass-through is used and the engine enforces its own keys (FR-A16 interim). *Assertion: raw api-key forwarded; interim path clearly gated on the identity enabler.* [Source: FR-CLI17]
- [ ] **AC3** — Given an upstream 401/403/404, When received, Then it propagates verbatim with no existence oracle. *Assertion: status/errorCode unchanged; no leak distinguishing "forbidden" vs "not found" beyond upstream.* [Source: NFR-CLI4]
- [ ] **AC4** — Given a token, When an edge verifies it, Then verification uses the Go `lib/auth` binding (the CLI makes no authz decision). *Assertion: edge-verify path invokes lib/auth; CLI does not evaluate ACL.* [Source: FR-CLI17/NFR-CLI4]
