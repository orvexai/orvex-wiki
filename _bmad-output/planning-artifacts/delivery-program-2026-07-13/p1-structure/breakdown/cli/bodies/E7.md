## 🎯 Epic — Multi-Service Auth via Identity

`orvex auth login` mints an **identity-minted, space-scoped token** against the identity host; each service edge verifies it via the Go `lib/auth` binding. `whoami`/`status` read the caller principal/role from token **claims** — replacing the reference's `POST /api/users/me` probe. `Profile` holds a credential **per service** (or one bearer forwarded to all edges). Until identity ships end-to-end, **api-key pass-through is retained** and the engine enforces its own keys (FR-A16 interim) — this gates the auth cutover on the identity/token-enabler workstream. Throughout, orvex-cli is a **zero-trust client**: it holds no service credential of its own beyond the caller's forwarded token, makes no authorization decision, and propagates 401/403/404 verbatim with no existence oracles (NFR-CLI4). Long-lived bearers carry no `cell` claim (cell contract §2 token rule) and are routed by their stable tenant/org claim — which is why the E1-S5 421 path exists. [Source: PRD R4AOVBLST7 §F7 FR-CLI17 + NFR-CLI4; Architecture pf10XC2Qjz §Components Auth/§Boundaries]

Why it exists: identity is the dual-IdP spine; the CLI must forward-only and verify-at-edge so no ACL logic leaks client-side, while an honest interim keeps agents working before identity is fully live.
