## 🎯 Epic — Daemon, SQLite Cache & SSE Freshness

orvex-cli runs a **per-host background daemon** owning one SQLite cache (WAL, single-writer goroutine, flock lifecycle) served over Unix-socket length-prefixed-JSON IPC, kept fresh by **one SSE connection per workspace** to the knowledge events host. Reads resolve from the cache; writes target wiki-api. Cache-sync is a **cursor-delta projection download** from the knowledge FR-K20 projection (not a full crawl — a ≥5× throughput win, NFR-CLI2/NFR-K2), ACL-narrowed, where **permission-loss tombstones hard-DELETE** cache + FTS + on-disk mirror so revoked content never lingers (closes defect D13). The SSE consumer preserves the reference's richer-than-FR-K15 wire contract verbatim: `/events/head` cold-start bootstrap, a **numerically-monotonic, epoch-ms-carrying cursor** (defect D6 / NFR-CLI5), unprefixed event names, thin `page.content_updated` frames triggering refetch, `permission.changed` prune semantics, 1 MiB frame cap + 512-frame queue reconnect-not-drop, <45s heartbeat, control frames (`stream.gap/backpressure/terminated`), `applied_events` exactly-once dedup, and the `page.status_changed` silent-drop fix (defect D3). Cursors are qualified by **(cell/events-host, space)** with events-host cold-start on tenant-move (F-C); a ~6h belt-and-braces reconcile runs regardless. [Source: PRD R4AOVBLST7 §F6 FR-CLI14/15/16 + NFR-CLI2/3/5; Architecture pf10XC2Qjz §Data flow/§Reliability]

Why it exists: freshness that silently breaks is worse than a missing cache; this epic makes the daemon non-load-bearing for correctness (cold/stale = typed staleness error, never wrong or over-exposing data) while sustaining the projection throughput target.
