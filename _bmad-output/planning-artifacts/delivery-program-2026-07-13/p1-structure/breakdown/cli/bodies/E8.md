## 🎯 Epic — Back-Compat Migration & Release Chain

Cutover from `docmost-cli` must be **non-destructive across one deprecation window** and the release chain rebuilt fresh under the new org. This epic ships: the `docmost-cli` shim (one-time stderr deprecation warning, then `exec orvex`); `DOCMOST_* → ORVEX_*` env aliases with read-old-if-new-unset + warn-once (`DOCMOST_URL→ORVEX_URL` + per-service, `DOCMOST_API_TOKEN→ORVEX_TOKEN`; `DOCMOST_SCHEMAS_*` dropped — contracts-sourced); config-dir `~/.config/docmost-cli → ~/.config/orvex` with first-run auto-migration; project-file `.docmost-cli.yaml → .orvex.yaml`; and hidden namespace aliases. Critically, the **keychain rename + token-dir move perform a one-time read-through migration** (read old → write new → delete old) across both the OS keyring and the encrypted fallback file, so stored tokens never silently vanish (closes defect D5). The release chain is **built fresh under `orvexai`**: two-variant goreleaser, brew formula `orvex` (+ `docmost-cli` alias), cosign keyless + SLSA provenance + reproducible builds for both variants, and a CI assertion that **module-owner == release-owner** (fixes the `orvex-ai`-vs-`orvexai` mismatch, defect D4). [Source: PRD R4AOVBLST7 §F8 FR-CLI18/19/20 + NFR-CLI6; Architecture pf10XC2Qjz §Components Release chain]

Why it exists: existing agents/scripts and stored credentials must survive the repoint (G2), and a working, provenance-clean release chain under the correct org is ship-blocking (G5/NFR-CLI6).
