## 🗺️ Feature area

Beyond the wiki verbs, the MCP fronts the **Studio product surface** and the **memory-retrieval surface**. Under Decision 4 / C21 the two search-flavored `studio_*` tools (`marketplace_search`, `skill_get`) re-home onto **orvex-studio-knowledge** (the family's one multi-corpus search service, reached MCP→knowledge-direct per D-M8), leaving **six product tools** (`memory_get/save`, `library_list/save`, `librarian_session`, `comment_post`) fronting the Studio backend. Per the memory gap-closure fold-in (AD-5b, PRD `g9vWbSYplh`), the MCP also exposes the **memory-retrieval surface**: memory is *read* as a `knowledge` tool call under the caller's delegated principal (governed by I-4, ACL ∩ scope) — explicitly **not** a write/compose adapter (that's the browser-extension client port AD-5a) and **not** stateful sync (AD-5c).

It exists to keep every non-wiki agent capability on the same trust posture as the wiki path: caller-token pass-through, the same edge verifier + deny-by-default scope gate, fail closed on absent/invalid/insufficient — no service credential, no placeholder Studio JWT (FR-M19). The only open item is the `studio_*` **contract home** (OQ-M1), cutover-blocking before R2.
