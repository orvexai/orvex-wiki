## 🎯 Story

As the **event spine**, I want identity to emit its lifecycle events via a **transactional outbox + relay publishing DIRECT to Kafka** as CloudEvents (stamping `orvexcell`), so that emission is exactly-once-ish and the retired Redis→Kafka bridge is never rebuilt.

**Definition of Done:** `TestOutboxRelayExactlyOnce` (integration layer) — through a lifecycle change, an outbox row is written in the same DB transaction as the change, the relay publishes it straight to Kafka as a CloudEvent with `orvexcell` stamped, and a consumer receives it exactly once (no dual-write, no Redis Streams stage).
*Final H1–H17 elaboration + exact contract tag/versions are pinned at pack certification (ENG-2101); this story is dispatch-blocked until that tag exists.*

## ✅ Acceptance Criteria

- [ ] **AC1** — Given a lifecycle change (token revoked / org converted / provisioning done), When it commits, Then an outbox row is written in the SAME transaction. *assert: change + outbox row atomic (rollback drops both)* [Source: FR-I15]
- [ ] **AC2** — Given outbox rows, When the relay runs, Then it publishes straight to Kafka as CloudEvents (no Redis→Kafka bridge). *assert: relay → Kafka; no Redis Streams stage exists* [Source: FR-I15, D-S13]
- [ ] **AC3** — Given a published event, When inspected, Then the CloudEvents envelope stamps `orvexcell` and consumers fail closed on mismatch. *assert: event has `orvexcell`; mismatched-cell consumer rejects* [Source: FR-I15, cell-contract rule 6]
- [ ] **AC4** — Given the push-revocation broadcast, When a token is revoked, Then the broadcast rides this same spine. *assert: `token.revoked` reaches the E1-S4 deny-list consumer* [Source: FR-I15, FR-I3]
- [ ] **AC5** (error-path) — Given a relay crash mid-publish, When it restarts, Then rows publish exactly once (idempotent relay, no duplicate side effects). *assert: crash+restart → no duplicate consumer effect* [Source: FR-I15]
