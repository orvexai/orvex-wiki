## 🎯 Feature area — CLI & agent client seam (Orvex CLI)

Identity owns the auth surface the reborn **Orvex CLI** (`orvex auth …`, D-S10) and agent clients depend on, closing CLI gaps G7/G-verify/G-mint/G-health (FR-I20). A stable **`GET /v1/whoami`** verifies a token and returns the IdP-agnostic principal (replacing the engine's `POST /api/users/me` and the `login` probe of `POST /api/workspace/info`); a **device-authorization / token-grant** flow lets `orvex auth login` mint an identity-minted, space-scoped (C5) token against `auth.orvex.ai` rather than storing a user-supplied bearer; and a `GET /healthz` alias gives `orvex doctor` a per-service reachability target. An **interim** path lets the CLI forward an engine api-key through until the engine accepts identity-minted tokens (WS-6/WS-8) — identity's verify path must not break that transition (FR-A16).

Why it exists: PRD §4 F7 (FR-I20); Architecture §3 API surface. Served on the same FR-I1/FR-I2 verification path.
