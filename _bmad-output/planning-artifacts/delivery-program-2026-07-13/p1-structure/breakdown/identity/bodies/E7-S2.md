## 🎯 Story

As a **CLI user**, I want `orvex auth login` to mint an identity-minted, space-scoped token via a device-authorization / token-grant flow against `auth.orvex.ai`, so that the CLI stores a real identity-issued token rather than only a user-supplied bearer.

**Definition of Done:** `TestDeviceAuthGrant` (integration layer) — through the device-authorization flow, `orvex auth login` completes an interactive grant and receives an identity-minted, server-side-scoped (C5) token; scope cannot exceed the authorizing user's; the grant reuses the FR-I5/FR-I6 mint path.

## ✅ Acceptance Criteria

- [ ] **AC1** — Given `orvex auth login`, When it starts, Then a device-authorization (or equivalent first-party) grant is issued (user_code + verification URL). *assert: device-auth start → pending grant* [Source: FR-I20]
- [ ] **AC2** — Given the user authorizes, When the CLI polls, Then it receives an identity-minted token. *assert: authorized → token issued* [Source: FR-I20]
- [ ] **AC3** — Given the minted token, When issued, Then scope is server-side-derived (C5) and cannot exceed the authorizing user's. *assert: `minted.scope` server-derived; `<= user`* [Source: FR-I20, FR-I6]
