## 🎯 Feature area — Global org→cell registry & routing (the only global component)

Identity owns the platform's `tenant → cell` **routing registry** — the ONLY global component (A-CELL-REGISTRY) — serving BOTH products: the wiki's `{tenant}.{cell}.wiki.orvex.ai` hostname scheme (D-S5) and Studio's org→cell resolution behind the single URL orvex.ai via the edge geo-router. Per SE-Arch F5 it is **TWO schema-isolated stores** in a dedicated global CNPG (identity sole writer): a **PII-free routing core** (`tenant_id uuidv7, cell_id, residency_pin, cell_epoch, status`) that is cross-jurisdiction replicable + KV-mirrored, and an **in-jurisdiction sensitive shard** (`stripe_customer`, human `hostname_claims`, contacts). At tenant creation identity assigns a cell by geo-proximity + residency choice and reserves globally-unique `tenant_id`/hostname. The read path is edge-cached; a deliberate tenant-move is a typed step-API (cell-contract rule 10, the non-retrofittable part). It must **not become a routing SPOF** (OQ-I7 gates the second cell).

Why it exists: PRD §4 FR-I19; Architecture A-CELL-REGISTRY; SE-Arch F5. Routing/registry authority only — **not** an entitlement authority (billing SoR per-cell-replicable, D-P3).
