## 🗺️ Feature area

The thin-fork foundation is what makes orvex-wiki *the only AGPL artifact Orvex owns* survivable across upstream Docmost releases. The engine is a fold-in onto **vanilla Docmost v0.95.0** (D-S1) that keeps only irreducible primitives and pushes everything else across a network boundary. This epic builds the machinery that keeps that promise mechanically enforceable: the additive `apps/server/src/orvex/*` module tree behind a single `OrvexRootModule` import, the **frozen 13-row inline-edit allow-list** (A-THIN), the **FR-30 divergence gate** budgeted on *weighted hot-file conflict-hunks* (not raw file count), the routine **overlay-rebase-per-tag** import runbook (A-IMPORT), the **15-item correctness-hardening allow-list class** (A-HARDENING / D-S18), and the boot-migration hygiene that stops the renamed-migration duplicate-run trap and keeps `db.d.ts` stock-verbatim.

Why it exists: without the gate and the module boundary, every upstream release re-opens the conflict surface and the fork rots (G1 — thin, upstream-mergeable engine). Every other epic lands code *inside* this boundary, so this is the first-built substrate.
