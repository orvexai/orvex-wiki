## 🎯 Story

As a **downstream consumer (knowledge, billing, the event service) and the GDPR-deletion owner** I want **the engine to emit durable `workspace.deleted` / `user.deleted` events through the transactional outbox on tenant/user removal — the distributed-cleanup contract every external consumer deletes its own copy off** so that **the cross-service FK cascades the monolith relied on are replaced by the only cleanup channel once the brains are external, making the events part of the GDPR-deletion guarantee, not a nicety** (FR-37).

**Definition of Done:** ONE named test `TestWorkspaceDeletedEmitsCleanupOutboxEvent` (integration layer — asserts through the workspace/user removal path that a `workspace.deleted` (and `user.deleted`) event row commits in the SAME tx as the removal (FR-17), carries an explicit schema version (FR-36), and is drained by the relay as a CloudEvent an external consumer can delete off). *Final H1–H17 elaboration + exact contract tag/versions are pinned at pack certification (ENG-2103); this story is dispatch-blocked until that tag exists.*

## ✅ Acceptance Criteria

- [ ] **AC1** — Given a tenant (workspace) removal, When it commits, Then a durable `workspace.deleted` event row commits in the SAME tx via the outbox (FR-17). *Assert: removal ↔ `workspace.deleted` row atomic (both or neither).* [Source: FR-37]
- [ ] **AC2** — Given a user removal, When it commits, Then a durable `user.deleted` event row commits in the same tx. *Assert: `user.deleted` row committed atomically with the removal.* [Source: FR-37]
- [ ] **AC3** — Given the cleanup events, When published, Then each carries an explicit schema version (FR-36) and is a typed contracts CloudEvent round-tripping golden fixtures (via the E5-S2 envelope). *Assert: event ∈ catalog; schema version present; fixtures pass.* [Source: FR-37, FR-36]
- [ ] **AC4 (negative)** — Given the monolith's cross-service FK cascades, When the fold-in lands, Then engine-side cleanup is the emitted event, not an FK cascade into external stores. *Assert: no cross-service FK cascade remains; cleanup is event-driven and the consumer-deletion contract is documented (knowledge vectors, event-service stream state, billing ledger).* [Source: FR-37]
- [ ] **AC5 (error-path)** — Given a removal whose tx aborts, When it rolls back, Then no orphan cleanup event exists (no false deletion broadcast). *Assert: zero orphan `workspace.deleted`/`user.deleted` rows.* [Source: FR-37]
