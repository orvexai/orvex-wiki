## 🎯 Story

**Audit: already implemented** (evidence in §4) — this story is VERIFY + harden: prove the behaviour via the named DoD test + adversarial review; do not rebuild.

As a **wiki user authenticating via identity** I want **the engine to consume an identity-minted exchange token (RS256/JWKS or introspection) and mint its session cookie via `createSessionAndToken`, with native email/password login removed fully (no break-glass)** so that **identity owns verification/mint, the symmetric `APP_SECRET` `jwt.verify` on every request is replaced, and this acceptance unblocks wiki-api Phase-0 and knowledge** (FR-W6, FR-15, A-AUTH, D-S3).

**Definition of Done:** ONE named test `TestExchangeTokenMintsSessionRs256` (integration layer — asserts through the auth entrypoint that a valid identity-minted RS256/JWKS exchange token mints a session, an unsigned/HS256/APP_SECRET token is rejected, and no native-login route exists). *Final H1–H17 elaboration + exact contract tag/versions are pinned at pack certification (ENG-2103); this story is dispatch-blocked until that tag exists.*

## ✅ Acceptance Criteria

- [ ] **AC1** — Given an identity-minted RS256/JWKS exchange token, When presented, Then the engine consumes it and mints a session via `createSessionAndToken`. *Assert: session cookie minted; verified against JWKS.* [Source: FR-W6, FR-15]
- [ ] **AC2** — Given every request, When authenticated, Then RS256/JWKS against identity replaces the symmetric `APP_SECRET` `jwt.verify`. *Assert: no `APP_SECRET` verify path on the request hot path.* [Source: FR-W6]
- [ ] **AC3** — Given native email/password login, When the fold-in lands, Then it is removed fully — no break-glass, hosted or standalone (a standalone bundles identity). *Assert: no native-login route; no break-glass path.* [Source: FR-W6, D-S3]
- [ ] **AC4 (negative)** — Given an unsigned or HS256/APP_SECRET token, When presented, Then it is rejected. *Assert: 401; no unsigned-claims path; no test-mode bypass.* [Source: FR-W6]
- [ ] **AC5** — Given SSO-enforcement + session-revocation, When configured, Then they stay engine-side. *Assert: enforcement + revocation honoured.* [Source: FR-W6]
