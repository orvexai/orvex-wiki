## 🎯 Story

As a **cell operator / abuse-response owner** I want **the engine to enforce tenant suspension as a live, push-invalidated check — blocking a suspended tenant from all but read/export/auth surfaces within a bounded propagation window (≤ a few seconds cell-wide), NOT read from the cached-entitlements TTL (FR-16)** so that **a suspended tenant is locked out near-instantly rather than at cache expiry, wiring the existing-but-unused tenant status** (FR-22, NFR-7).

## ✅ Acceptance Criteria

- [ ] **AC1** — Given a suspended tenant, When it issues a write/mutating request (page create, member add, invite), Then it is blocked (all-but-read/export/auth surfaces). *Assert: write/member-add/invite rejected for a suspended tenant.* [Source: FR-22]
- [ ] **AC2** — Given a suspended tenant, When it issues a read, export, or auth request, Then it still succeeds and no data is deleted. *Assert: read/export/auth succeed under suspension; never a delete.* [Source: FR-22]
- [ ] **AC3** — Given a suspend signal, When it fires, Then the block is live / push-invalidated — NOT served from the cached-entitlements blob TTL (FR-16). *Assert: suspend takes effect from the push signal, independent of entitlement-cache TTL.* [Source: FR-22, FR-16]
- [ ] **AC4** — Given a suspend, When propagated across the cell's replicas, Then it binds within a bounded window (target ≤ a few seconds, NFR-7). *Assert: propagation ≤ target window across replicas.* [Source: FR-22]
- [ ] **AC5 (error-path)** — Given the suspension signal source is briefly unavailable, When a mutating request arrives, Then suspension fails closed: a tenant already known-suspended stays blocked and unknown state never silently un-suspends. *Assert: no fail-open un-suspension on signal loss.* [Source: FR-22]
