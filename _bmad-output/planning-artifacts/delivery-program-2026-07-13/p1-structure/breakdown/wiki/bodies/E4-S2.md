## 🎯 Story

As a **platform owner** I want **a pre-flight O(1) Redis fast-counter quota check at page-create, attachment-upload, and member-add that fires on BOTH the REST API and the Hocuspocus collab persistence path** so that **a single tenant cannot create unbounded pages or store unbounded bytes, and the Yjs store cannot silently bypass a REST-only check** (FR-W10, FR-W12, A-QUOTA).

**Definition of Done:** ONE named test `TestQuotaChokepointBlocksCollabPath` (integration layer — asserts through the collab persistence extension that a page/attachment write over the Yjs path is rejected when the `quota:*:{tenant}` counter is at cap, proving enforcement is not REST-only). *Final H1–H17 elaboration + exact contract tag/versions are pinned at pack certification (ENG-2103); this story is dispatch-blocked until that tag exists.*

## ✅ Acceptance Criteria

- [ ] **AC1** — Given a tenant under cap, When a page is created, Then the O(1) `quota:pages:{tenant}` INCR-on-write check passes pre-flight (never a `COUNT(*)`). *Assert: no per-request COUNT/SUM; counter increments.* [Source: FR-W12]
- [ ] **AC2** — Given a tenant at the page cap, When a page is created via REST, Then it is rejected. *Assert: write blocked (verdict handled in E4-S3).* [Source: FR-W10]
- [ ] **AC3** — Given a tenant at cap, When a page/attachment write arrives via the collab `persistence.extension.ts` path, Then it is also rejected. *Assert: `TestQuotaChokepointBlocksCollabPath` blocks the ydoc-store write.* [Source: FR-W10]
- [ ] **AC4** — Given an attachment upload, When it runs, Then aggregate storage bytes, per-file byte cap, and file count are all checked. *Assert: each of `quota:bytes/files:{tenant}` enforced.* [Source: FR-W10, FR-W12]
- [ ] **AC5 (negative)** — Given reads/exports/deletes, When performed at or over cap, Then they always succeed. *Assert: no quota block on read/export/delete.* [Source: FR-W13]
