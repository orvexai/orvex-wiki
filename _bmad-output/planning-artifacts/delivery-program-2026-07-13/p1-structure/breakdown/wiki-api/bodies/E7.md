## 🗺️ Epic — Operations: cache eviction, observability, health, latency & degradation

The operability spine. wiki-api is **stateless** with an event-evicted Redis cache restricted to **principal-independent artifacts only** (descriptor, block-schema catalog, authoring guide) — shaped-read body caching is **dropped** (D-A10): a `{tenant, scope-hash, resource, version}` key has no principal, so a hit would serve ACL-gated bytes without consulting the engine — a cached allow decision by proxy, violating FR-A17. The engine is the cache [Source: FR-A18, Arch A-STATE]. The evict Trigger consumes `wiki.page.*` off the studio-spine Kafka topics fed by the engine's **transactional-outbox relay** (no Redis→Kafka bridge — the canon-corrected mechanism, review A1), and every CloudEvent carries the `orvexcell` extension so the consumer **fails closed on a cell mismatch** (review A6).

Why it exists: agents run on wiki-api's hot path, so latency, health, observability and graceful degradation are product requirements, not afterthoughts. Health must distinguish liveness (engine-independent — the proxy stays in rotation, returns 502 on upstream failure) from a dependency round-trip health surface for the console — a degraded upstream must never read green-by-omission (review B2). `/healthz` echoes **both CELL_ID and CLUSTER_NAME** per the day-1 cell contract (review A3). Latency budgets (NFR-A1/A1a/A1b/A2a) get benchmarks; a hung non-SSE upstream body is bounded (review B4). Redaction must survive the relocation line (NFR-A3) and standalone posture degrades cleanly (NFR-A5) [Source: FR-A18, FR-A19, NFR-A1, NFR-A3, NFR-A5].
