## 🗺️ Epic — Block-patch write chokepoint, CAS, human-gating & quota

The write half of the grammar and the load-bearing invariant of the whole service: **one write chokepoint** with atomic multi-op batches, CAS, and human-gating transport. The primitive-vs-composition line is explicit because atomicity **cannot relocate** — post-thinning the engine keeps exactly one atomic `apply-ops` primitive (typed PM-JSON ops + `ifVersion`, applied atomically against the ydoc it holds), and wiki-api owns everything above it: DfM→ops translation, heading/anchor resolution, opaque reattach from the `ifVersion` base. wiki-api **never decomposes a batch into multiple engine calls** — that would silently regress the shipped atomicity guarantee to partial-batch writes [Source: FR-A6, Arch A-VERBS].

Why it exists: every write returns the verified read-after-write receipt `{url,id,version,persisted:true}` (never a re-rendered body), even block-authoring writes (closing contract-gap G4); CAS `ifVersion → 409 VERSION_MISMATCH` makes the widened read-base→transform→write race window safe (stale ⇒ 409, never partial). Human-gated actions are transport-only (R3, D-A8) — wiki-api never mints RATIFY/CONFIRM tokens. The engine's F-QUOTA 402 propagates verbatim and a batch that trips the cap fails whole (FR-A22). ai's inline-edit writes land through this same chokepoint carrying the caller's delegated token, no privilege (D-A12) [Source: FR-A7, FR-A8, FR-A9, FR-A22].
