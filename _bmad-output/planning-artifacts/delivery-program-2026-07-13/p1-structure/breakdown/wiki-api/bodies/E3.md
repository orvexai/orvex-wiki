## 🗺️ Epic — Verb grammar, read ladder & knowledge search fronting

The resource-agnostic verb grammar `search / get / save / edit / list` over `{resource_type, locator}` terminates here (grammar artifact authored in contracts). There is **no `ask` verb** — the cited-ask loop is orvex-studio-ai's (D-A12); agents ask ai directly. Wiki-only in practice (`resource_type:"wiki"`), with the parameter kept vestigial-but-future-proof (D-A5). This epic delivers the read half of the grammar: locator resolution (composing the engine's resolvers, never re-implementing them), the read ladder (`info → outline+token_estimate → blocks/:id?format=dfm` + ranged reads), structural nav, and the `search/related/duplicates` fronting that routes to knowledge's query API [Source: FR-A4, FR-A5, FR-A11].

Why it exists: agents need one stable read surface that stays context-light (≤ ~1k tokens / ≤ 2 calls per common task, G5) regardless of which upstream serves it. The knowledge fronting is wiki-api's verb-grammar surface, **not** a supersession of knowledge FR-K15/K20/K21 — the CLI's direct knowledge paths stand ratified, and this fronting is optional-for-MCP unless OQ-A7 resolves otherwise [Source: FR-A11]. With knowledge absent, `search/related` degrade to typed errors while `get/list` keep working (NFR-A5).
