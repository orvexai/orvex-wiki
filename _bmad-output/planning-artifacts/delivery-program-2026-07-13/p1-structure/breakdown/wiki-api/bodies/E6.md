## 🗺️ Epic — Doc-governance re-home: drift verify, spec-gate & audit-write (D-S8)

The 2026-07-05 program fold: the living-wiki **drift verifier** (`verifyPage`/`getDrift`/force-new-token) and the **wiki-first spec-gate** (`{satisfied, token, reason}`) move out of the AGPL fork into wiki-api, exposed via CLI/MCP. Both are **composition over engine primitives**: the engine keeps only the `orvex_page_meta` stamp fields (`verified_against`/`verified_at`/`spec_confirmed`), which wiki-api writes through the engine's lifecycle-stamp primitive — never a second write path, no new column on `pages` [Source: FR-A20, FR-A21, Arch A-GOVERNANCE].

Why it exists: doc-governance is the living-wiki's drift + spec-gate machinery; keeping it in the fork keeps the agent contract in AGPL code. Re-homing is a **clean-room reimplementation** gated by parity, not a data migration. Two corrections land on re-home: the meaningless `behindBy=1` drift signal becomes a real drift metric, and the spec-gate work-item trigger is **de-Linearized (D-S11)** — the `contentHasLinearIssue` ProseMirror-node scan is deleted, keeping only the text-only story-id confirmation path (must not silently drop confirmation on pages that used the Linear path). CONFIRM/RATIFY tokens stay engine-minted; wiki-api transports verbatim (D-A8). The audit-**write** endpoint composes here but the audit **sink stays engine-side** (WORM, purge-exempt) [Source: FR-A20, FR-A21, FR-A23].
