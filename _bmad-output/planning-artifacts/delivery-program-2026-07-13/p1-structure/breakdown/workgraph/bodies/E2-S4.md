### 🎯 Story
As a **coordinating agent** I want **a `batch` verb that executes N mutations in one transaction** so that **an agent loop makes one round-trip instead of N (the write-amplification killer)**.

**Definition of Done:** one named test `TestBatchAtomicWithPerResourceEvents` (domain+event layer) asserts all N mutations commit or roll back together and emit N per-resource events with deterministic ids derived from `(Idempotency-Key, resource_id)`. *Final H1–H17 elaboration + exact contract tag/versions are pinned at pack certification (ENG-2095); this story is dispatch-blocked until that tag exists.*

### ✅ Acceptance Criteria
- [ ] **AC1** — Given N mutations, When submitted as one `batch`, Then they apply in a single transaction. *assert: all-or-nothing commit.* [Source: FR-MEM6, AD-2]
- [ ] **AC2** — Given a batch commit, When events emit, Then N per-resource events fire inside the one transaction with deterministic ids so replay dedups. *assert: N events; duplicate replay → single apply (transactional atomicity ≠ event coarseness).* [Source: AD-7]
- [ ] **AC3** — Given a batch where mutation k fails, When processed, Then the entire batch rolls back (no partial state). *assert: failure at k → zero mutations persisted, zero events emitted.* [Source: FR-MEM6, no-fallbacks]
- [ ] **AC4 (negative)** — Given a batch exceeding the tenant item cap, When submitted, Then it returns the frozen 402 without partial application. *assert: 402; nothing persisted.* [Source: AD-13]
- [ ] **AC5** — Given a `blocks` edge in the batch that forms a cycle, When processed, Then the whole batch is rejected. *assert: cycle in batch → full rollback.* [Source: AD-4]
