## 🧭 Epic — Surfaces & adapters: MCP, CLI, KG interop, Claude memory-tool

The four-way front door. Every adapter is a thin protocol translator onto the same published
workgraph API verbs and the AD-9 authz chokepoint — none reads or writes the store directly.
This epic delivers the MCP `workgraph_*` section on the existing studio-mcp server (the ~15
tools of FR-MEM14, with context-budget levers from day one: lazy schema discovery, `brief=true`
minimal returns, result compaction above threshold) plus the `studio_memory_get/save`
disambiguation pointers (the split ruling — user-memory tools stay product-side, only their
descriptions repoint agent-state usage here), the CLI `orvex-cli workgraph` section with verb
parity and `--json`, the KG interop adapter (entities/relations/observations ↔ facts/edges/items,
read + write, pinned to `@modelcontextprotocol/server-memory` 2026.7.4, golden-fixtured), and the
Claude memory-tool backend serving `memory_20250818` `/memories` file ops from the workgraph store.

Why it exists: MCP-first with CLI parity is the delivery posture — chat platforms have no hook
system. AD-10 forbids three write paths with three semantics and adapter-side authz gaps; FR-MEM22/21
carry the user/agent memory split (the naming firewall: `memory` names the user product, never this
service). A ChatGPT-native memory bridge is platform-blocked — an external constraint, not a deferral.
