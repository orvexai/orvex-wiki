## 🎯 Story

As a **user who just confirmed a Memory card**, I want the **Prompt Composer to read my just-confirmed cards from the `api` system-of-record until the retrieval projection catches up**, so that "confirmed = visible" never breaks — my enrichment weaves the fact I just approved even if the `knowledge` projection still lags behind the confirm.

**Definition of Done:** one named test **`TestComposerReadYourWritesOnConfirm`** (integration layer — confirms a Memory card via the sole confirm gate, then runs Composer enrichment for that same user *before* the `memory.*` projection has landed, and asserts the just-confirmed card is woven from the `api` SoR read path — while a monotonic-version replay of two out-of-order `memory.*` events over the same card-id projects last-writer-wins, through the enrichment + SoR read interfaces). *Final H1–H17 elaboration + exact contract tag/versions are pinned at pack certification (ENG-2098); this story is dispatch-blocked until that tag exists.*

## ✅ Acceptance Criteria
- [ ] **AC1** — Given the confirming user has a just-confirmed card not yet in the projection, When the Composer enriches for that user, Then the card is read from the `api` Memory SoR and woven (read-your-writes). *Assertion: just-confirmed card appears in enrichment despite projection lag.* [Source: AD-11]
- [ ] **AC2** — Given the confirm ritual, When a card is confirmed, Then either the confirm ack is withheld until the projection reflects it, or the Composer SoR read path covers the gap — exactly one strategy, applied deterministically. *Assertion: no window exists where a confirmed card is invisible to its own confirming user.* [Source: AD-11]
- [ ] **AC3** — Given `api` publishes `memory.*` CloudEvents, When emitted, Then they are partitioned by `card-id` for per-card ordering and carry a monotonic card version. *Assertion: events for one card share a partition key = card-id; each carries a strictly increasing version.* [Source: AD-11, FR-SA31]
- [ ] **AC4** — Given two out-of-order `memory.*` events for one card, When projected, Then last-writer-wins by monotonic version — a `superseded` never overtakes a lower-version `created`. *Assertion: idempotent LWW; stale-version replay is a no-op.* [Source: AD-11]
- [ ] **AC5** (negative) — Given a *different* user than the confirmer, When the Composer enriches, Then the RYW SoR read path does not apply — that user reads only the projection (no cross-user SoR peek). *Assertion: RYW is scoped to the confirming user only.* [Source: AD-11, AD-7]
- [ ] **AC6** (error-path) — Given the projection is unreachable, When a confirmed card is read via the SoR path, Then enrichment still weaves it and no user-facing error surfaces. *Assertion: SoR read path is independent of projection availability.* [Source: AD-11, FR-SA11]
