## 🎯 Story

As the data tier, I want Postgres (CNPG, per-cell) with tenant RLS and a repository layer that sets the tenant GUC per request for `ai_chats`, `ai_chat_messages`, `ai_drafts`, `ai_memories`, `ai_prompts`, `tenant_ai_config` (no cap value stored), so that every read/write is tenant-isolated by construction (FR-AI23, NFR-AI3).

**Definition of Done:** one named test `TestTenantRLSRepositoryIsolation` (integration) asserts that a repository call under tenant A's GUC cannot read/write tenant B's rows, and that a cross-tenant probe is denied at the RLS layer. *Final H1–H17 elaboration + exact contract tag/versions are pinned at pack certification (ENG-2097); this story is dispatch-blocked until that tag exists.*

## ✅ Acceptance Criteria

- [ ] **AC1** — Given a request, When the repository runs, Then it sets the tenant GUC and all queries are RLS-scoped to that tenant. *Assert: GUC set; RLS predicate applied.* [Source: FR-AI23, NFR-AI3]
- [ ] **AC2** — Given tenant A's session, When it queries tenant B's rows, Then it returns nothing (RLS deny). *Assert: cross-tenant read ⇒ empty.* [Source: NFR-AI3]
- [ ] **AC3** — Given `tenant_ai_config`, When persisted, Then no cap value column exists (billing owns the entitlement). *Assert: no cap column.* [Source: FR-AI23]
- [ ] **AC4** (negative) — Given a missing tenant GUC, When a query runs, Then it fails closed (no unscoped access). *Assert: absent GUC ⇒ no rows / error, never all-tenants.* [Source: NFR-AI3]
