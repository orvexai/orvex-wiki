## 🗺️ Feature area

orvex-studio-ai owns the conversational surface of the whole family: the streaming chat turn, the session/message store with fork-at-message branching, and per-turn context assembly. This is the `cmd/server` always-warm streaming path — the reason Knative never fronts it (A-STREAM, NFR-AI2). Cutover is byte-compatible: the React client keeps calling `/api/ai/*` and never regenerates (G3, D-AI4), so the SSE data-stream frame sequence is reproduced from recorded tapes, not redesigned.

**Why it exists:** chat is the primary AI feature and the migration risk — the AI-SDK data-stream protocol has many frame types and an unforgiving client (PRD §4 F1; Arch A-STREAM). Conversational state (branching, TTL, reconciliation) is ai's own Postgres, not the fork's.
