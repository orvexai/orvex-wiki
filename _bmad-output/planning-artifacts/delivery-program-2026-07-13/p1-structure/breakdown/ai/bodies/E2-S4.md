## 🎯 Story

As a user whose tool run failed, I want a `tools-retry` surface that re-runs failed tool executions against a persisted turn, and a SearXNG web-search tool that is SSRF-guarded with fixed egress, so that transient tool failures recover and web search can't be abused (FR-AI4).

**Definition of Done:** one named test `TestToolsRetryAndSearxngGuard` (integration) persists a failed tool execution, re-runs it via `POST /api/ai/tools/retry`, asserts success, and asserts a SearXNG call to a non-allowlisted/internal target is blocked. *Final H1–H17 elaboration + exact contract tag/versions are pinned at pack certification (ENG-2097); this story is dispatch-blocked until that tag exists.*

## ✅ Acceptance Criteria

- [ ] **AC1** — Given a failed tool execution, When persisted, Then a retryable record is stored against the turn. *Assert: retryable record exists.* [Source: FR-AI4]
- [ ] **AC2** — Given a retryable record, When `POST /api/ai/tools/retry` is called, Then the tool re-runs against the persisted turn and updates the frame. *Assert: retry succeeds; turn updated.* [Source: FR-AI4]
- [ ] **AC3** — Given a web-search tool call, When it targets an internal/metadata IP or non-allowlisted host, Then SearXNG egress is blocked (SSRF guard, fixed egress). *Assert: internal target ⇒ blocked.* [Source: FR-AI4]
- [ ] **AC4** (negative) — Given a retry against another tenant's turn, When attempted, Then it is denied. *Assert: `status==404`.* [Source: FR-AI4, NFR-AI3]
