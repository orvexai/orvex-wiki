## 🎯 Story

As a chat/inline turn, I want a data-driven tool registry with a bounded in-request multi-step loop (`stepCountIs(MAX_TOOL_STEPS)`) and tool-output validation before anything reaches the stream or a write path, so that the model can act safely without runaway loops (FR-AI4).

**Definition of Done:** one named test `TestToolLoopBoundedAndValidated` (integration) runs a tool-calling turn and asserts the loop halts at `MAX_TOOL_STEPS` and that an invalid tool output produces a typed tool-error frame rather than reaching the stream. *Final H1–H17 elaboration + exact contract tag/versions are pinned at pack certification (ENG-2097); this story is dispatch-blocked until that tag exists.*

## ✅ Acceptance Criteria

- [ ] **AC1** — Given a tool-calling turn, When the loop runs, Then it is bounded at `MAX_TOOL_STEPS` and terminates. *Assert: step count ≤ MAX_TOOL_STEPS; no infinite loop.* [Source: FR-AI4]
- [ ] **AC2** — Given a registered tool, When invoked, Then its output is validated against the tool's schema before streaming or writing. *Assert: invalid output ⇒ typed tool-error frame, not streamed.* [Source: FR-AI4]
- [ ] **AC3** — Given block-emission + page-edit + web-search tools, When the registry loads, Then it is data-driven (declarative), with no Linear tool set. *Assert: registry contains no Linear tool (D-S11).* [Source: FR-AI4]
- [ ] **AC4** (escalation/negative) — Given a long/expensive loop, When it exceeds the interactive bound, Then it escalates to the durable path rather than running unbounded in-request. *Assert: escalation signal emitted at bound.* [Source: FR-AI4, FR-AI21]
