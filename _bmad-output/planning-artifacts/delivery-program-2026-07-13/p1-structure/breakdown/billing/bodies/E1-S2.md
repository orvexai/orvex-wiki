## 🎯 Story

As a **billing engineer wiring Stripe**, I want **one Product per purchasable plan with two recurring Prices referenced exclusively by versioned `lookup_key`s**, so that **test/live mode differences never leak into code and a price change grandfathers existing subscribers Stripe-side**.

**Definition of Done:** `TestLookupKeyResolvesInConfiguredStripeMode` — a store/integration test asserting every catalog `lookup_key` (`personal.monthly.v1`, `personal.annual.v1`) resolves to a live Price in the configured Stripe mode at startup, and that a mis-wired key fails the pod (not a customer). *Final H1–H17 elaboration + exact contract tag/versions are pinned at pack certification (ENG-2100); this story is dispatch-blocked until that tag exists.*

## ✅ Acceptance Criteria

- [ ] **AC1** — Given the Stripe object model, When Products/Prices are provisioned, Then there is one Product per purchasable plan (`Orvex AI Studio — Personal`) with two recurring Prices (monthly, annual), each carrying `currency_options` and `tax_behavior: inclusive`. *assert provisioned Prices have `tax_behavior == "inclusive"` and `currency_options` present* [Source: FR-B2, A-PRICE]
- [ ] **AC2** — Given code needs a Price, When it references one, Then it uses the versioned `lookup_key` only — never a raw `price_...` id. *static/CI assert: no `price_[A-Za-z0-9]+` literal outside Stripe SDK objects* [Source: FR-B2, D-B8]
- [ ] **AC3** — Given startup, When the catalog loads, Then every `lookup_key` is asserted to resolve in the configured Stripe mode; a mis-wired mode fails the pod. *assert startup returns non-nil error when a key does not resolve* [Source: FR-B2, A-PRICE]
- [ ] **AC4 (grandfathering)** — Given a price change, When it ships, Then it is a new Price + `.v2` lookup key attached to a new plan-version; existing subscriptions keep their old Price. *assert a `.v2` key maps to a new plan-version and the old key still resolves* [Source: FR-B2]
- [ ] **AC5 (negative)** — Given a catalog `lookup_key` that resolves in test but not live mode, When the live pod starts, Then the pod fails readiness with a mode-mismatch error. *assert readiness false + mode-mismatch error* [Source: FR-B2]
