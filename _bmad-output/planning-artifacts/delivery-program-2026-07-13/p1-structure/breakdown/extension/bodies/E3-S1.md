## 🎯 Story

As a **client developer** I want **a provider-agnostic delivery-adapter framework — a provider registry plus a single injection port that writes a composed prompt into the user's own compose surface** so that **per-provider DOM fragility is isolated behind one interface and adding/repairing a provider never touches core delivery logic (FR-D1, FR-D5)**.

**Definition of Done:** one named test `TestAdapterFrameworkInjectsIntoOwnComposeSurfaceOnly` (integration — a registered fake provider adapter receives a composed prompt and places it into the designated compose node via the port; the framework refuses to inject anywhere but the sanctioned compose selector and never auto-submits). *Exact adapter interface + payload schema are pinned at ENG-2690; this story is dispatch-blocked until that contract exists.*

## ✅ Acceptance Criteria

- [ ] **AC1** — Given a registered adapter and a composed prompt, When delivery runs, Then the prompt is placed into that provider's compose surface. *Assert: compose node value == prompt; no network scrape.* [Source: g9vWbSYplh FR-D1]
- [ ] **AC2** — Given any adapter, When it injects, Then it writes only to the user's own compose field and never auto-presses send. *Assert: no submit event fired; only compose node mutated.* [Source: g9vWbSYplh FR-D5; NFR-2 (user-initiated)]
- [ ] **AC3** — Given an unregistered/unsupported provider, When delivery is attempted, Then the framework returns a typed "unsupported" result, not a best-effort guess. *Assert: unsupported → typed result, no DOM write.* [Source: g9vWbSYplh FR-D3]
- [ ] **AC4 (error-path)** — Given the compose node cannot be resolved, When injection runs, Then it fails typed and signals the E5 fallback path, never a silent no-op. *Assert: resolve-fail → `INJECTION_TARGET_MISSING`.* [Source: g9vWbSYplh FR-D7]
