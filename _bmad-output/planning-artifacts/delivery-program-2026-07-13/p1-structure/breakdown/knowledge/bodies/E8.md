## 📖 Epic — Rebuild, backfill & disaster recovery

The index is a projection — it must be rebuildable from the engine at any time; index loss is an availability event, never data loss (NFR-K4). This epic builds the **idempotent rebuild step-APIs** the central Temporal workflows in orvex-studio-workflows drive (D-WF-1 — knowledge runs no in-service Temporal worker): `rebuild/namespace`, `walk?cursor`, `index-batch`, `delta-repull`, `alias-swap`, `retire` (A8). Rebuild is fresh-namespace → checkpointed walk → walk watermark → delta re-pull of events landed during the walk → atomic AliasSwap → retire old, with **projection epochs** so a rewrite forces delta clients to full-resync. The **event-archive sink** (S3 + Postgres append table, D-S12) backs long-horizon replay beyond Kafka retention.

Why it exists: DR RTO — single tenant re-indexed < 15 min, full cell < 24 h (the sizing input for FR-K24), event-loss = 0 across crash-inject + replay. No KEDA (A-W6/C19): the central worker is a fixed-replica Deployment; knowledge's step-APIs are KPA-autoscaled HTTP surfaces. Live Trigger upserts continue against the old namespace during the walk so no events are lost at swap.
