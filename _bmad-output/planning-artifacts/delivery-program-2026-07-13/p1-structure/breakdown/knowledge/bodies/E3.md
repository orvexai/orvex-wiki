## 📖 Epic — Query & retrieval API, ranking, and read-path cache

The always-warm query surface every consumer hits: the wiki UI search box, MCP `search`/`related`, the interim in-fork AI and later orvex-studio-ai retrieval, and the Orvex CLI read verbs. This epic delivers the query endpoints (keyword|semantic|hybrid, RAG top-k, related, duplicates, attachment search — FR-K11), the **permanent ranking home** (D-S14: RRF fusion + freshness + optional nil-safe rerank — knowledge owns ranking forever, ai only consumes ranked results), the Redis read-path cache with event-driven eviction (A11), and the latency + fairness budgets (FR-K14, NFR-K10/K11/K13).

Why it exists: G1 — one retrieval substrate, one ranking path, one ACL choke point. Ranking was **struck as a non-goal** (D-S14, U13 closed) — it lives here permanently. The Redis cache is what makes it "blindingly fast" while staying safe: candidate sets are pre-ACL and always pass the live engine allow + token-scope post-filter (A1), so cache keys need no principal component (staleness, never exposure).
