## 📖 Epic — Orvex CLI projection cache-sync

The Orvex CLI keeps a local SQLite cache filled today by an O(pages) engine crawl. This epic replaces the fill mechanism with a **cursor/delta projection-download API** (FR-K20: "everything in space X changed since cursor C"), returning bodies + metadata from the projection, ACL-narrowed per FR-K12, epoch-bearing, at ≥ 20 pages/s (NFR-K2) — the ≥ 5× G3 target. It carries the full projection field set the CLI needs (FR-K20a: lifecycle metadata, verification stamps, presentational fields, byte-stable markdown, and the **content_pm parity ruling**), page-level ACL **hard-DELETE** tombstones (FR-K20b, a NEW capability), and preserves the CLI read verbs + parity claim (FR-K21/K22).

Why it exists: the CLI's offline reads, `page list --status`, supersession resolution, and verify/staleness all regress without these fields. The **content_pm parity risk (OQ6)** is the single biggest FR-K22 parity risk — contracts must either include raw PM-JSON in the download or explicitly accept a markdown-only degrade (pinned by the Definition Pack). Page-level tombstones must be hard-DELETE or a lost-access page lingers as "trashed" in the cache (a data-exposure defect).
