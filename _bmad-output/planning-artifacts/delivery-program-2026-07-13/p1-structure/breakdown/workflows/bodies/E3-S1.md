## 🎯 Story

**As a** workflows platform engineer
**I want** two named retry-option constructors in `internal/workflows` — AUTHORITATIVE (`MaximumAttempts:0`, StartToClose 2m, backoff 1s→1m ×2.0) and BEST-EFFORT (StartToClose 30s, 3 attempts) — applied per activity, correcting the as-built 10-attempt cap
**so that** a callee outage of any length stalls-and-retries a lifecycle workflow instead of silently failing it (closes the G2 loss window).

**Definition of Done:** one named test `TestAuthoritativeRetryUnbounded` — workflow-test-env layer: an AUTHORITATIVE activity failing for longer than the old ~5-min budget still retries (never exhausts); a BEST-EFFORT activity exhausts at 3 and the workflow continues

Note: *final H1–H17 elaboration + exact contract tag/versions are pinned at pack certification (ENG-2107); this story is dispatch-blocked until that tag exists.*

## ✅ Acceptance Criteria

- [ ] **AC1** — Given an AUTHORITATIVE activity, When the callee is down for 30 minutes, Then the activity keeps retrying and the workflow does not fail — *RetryPolicy.MaximumAttempts==0; workflow still Running after simulated long outage* [Source: NFR-W5 / A-W4]
- [ ] **AC2** — Given a non-retryable 4xx from an AUTHORITATIVE step, When returned, Then the step fails fast (negative) — *4xx marked non-retryable → activity fails without retry loop* [Source: NFR-W5]
- [ ] **AC3** — Given a BEST-EFFORT activity, When it exhausts 3 attempts, Then exhaustion is logged and the workflow continues — *MaximumAttempts==3; workflow continues past exhaustion* [Source: NFR-W5]
- [ ] **AC4** — Given the shipped 10-attempt cap, When code is scanned, Then no AUTHORITATIVE activity uses a bounded cap (negative) — *no AUTHORITATIVE option sets MaximumAttempts>0* [Source: A-W4 / §5]
