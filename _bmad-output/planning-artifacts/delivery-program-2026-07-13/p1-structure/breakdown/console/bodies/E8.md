## 🗺️ Feature area

A **new operator/staff-plane pane** answering *"how are our users behaving in the wiki and Studio, and what is our go-to-market doing"* — activation, feature adoption, engagement, retention, plan mix, **plus session-replay recordings and GA4-style web & marketing analytics** — for internal **product + growth/marketing + sales** staff (Clerk admin-org, role-scoped; inside the §3 no-tenant-facing-dashboards non-goal). The store is DECIDED: **PostHog Cloud EU** (Frankfurt, a managed external data processor). Console's share is **fronts + renders only** — it *"owns no analytics state and makes no analytics decision"* (D-CT2): the same A-PANE/A-EDGE posture it uses over LGTM, with an open embed-vs-query mechanism (OQ-UBA-2, leaning embed for the native replay player + web-analytics dashboard). **Explicitly NOT console's share** (other services): the server-side CloudEvent→PostHog **bridge** (a separate spine consumer, OQ-UBA-1b — knowledge or a dedicated service; *console is never the bridge*), the client-side **posthog-js** capture in the frontends (engine React client + orvex-studio-ui, OQ-UBA-10), and the `product.*` CloudEvent authoring in contracts (OQ-UBA-1a). This epic covers the console-resident pieces: the guardrailed analytics BFF + `pkg/posthog` port (no computation in console, FR-CT-UBA-9/3/10), analytics authz + PostHog-credential confinement + read-audit (FR-CT-UBA-6), the five product-insight surfaces (FR-CT-UBA-1/2), aggregate-first + k-anonymity render (FR-CT-UBA-4), honest ILLUSTRATIVE pre-launch panels (FR-CT-UBA-5), the session-replay and web-analytics embed panes (FR-CT-UBA-11/12), and the privacy/residency/erasure-fronting + cost SLI (FR-CT-UBA-7/8/8a + NFRs).

It exists because *"reading behaviour means querying PostHog, not mining logs"* (CS §10 telemetry split) — a different data plane from the LGTM ops pane, fronted the same disciplined way.
