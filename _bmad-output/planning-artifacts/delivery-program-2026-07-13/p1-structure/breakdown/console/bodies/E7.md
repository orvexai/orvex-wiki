## 🗺️ Feature area

The last phase and the sharpest-edged one: **break-glass**, identity's IdP-independent offline-sealed emergency admin path surfacing in console as a *"restricted, heavily-audited route class — separate from the normal Clerk-verified plane"* (FR-CT15, A-BREAKGLASS). This epic builds break-glass endpoints in **their own router group with their own middleware stack**: verification delegates ONLY to identity's offline-sealed credential path (FR-I12/OQ-I5) — no Clerk, no Keycloak, no admin-org lookup, no cached-introspection dependency — so the end-to-end path (operator → console → identity offline verify → action) works **with both IdPs down** (NFR-CT3). A CI test proves the break-glass handlers import nothing from the Clerk-verified middleware chain (dependency direction enforced, not assumed). Plus per-use rate limits, out-of-band alerting on every invocation, and an action allow-list **structurally bounded** by what break-glass can authenticate downstream: with both IdPs down no other satellite can verify anything console presents, so allowed actions are identity-local + console-local recovery only — the workflows FR-W5 grace-cancel is explicitly out of scope (an accepted gap per the assumptions index) unless OQ-I5 grows offline emergency tokens.

It exists because with both IdPs down there must still be a recovery path — and it must be un-bypassable, alarmed, and honest about its own coverage gap.
