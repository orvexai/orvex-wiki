# PRD: Orvex Studio Extension — Cross-AI Delivery

> **tl;dr** — The Orvex Studio Extension is the family's **17th component**: a browser extension that delivers a memory-enriched prompt **into the AI assistant a non-technical user already uses** — ChatGPT, Claude, Gemini, Grok — by writing it into that assistant's own compose box, **without copy/paste**, so the user reviews and sends it themselves. It is the delivery arm of the F1 flagship in `g9vWbSYplh`; that umbrella PRD gates *all* of F1 on a viability spike, which is now decided (`V6hlDjecfh`): the extension mechanism is **viable** — GO for ChatGPT and Gemini, **DEGRADE to copy/paste for Claude and Grok** (their ToS "automated means" clauses) — **contested, not resolved: a co-equal analysis in `V6hlDjecfh` read the persistent-extension mechanism as NO-GO for Claude and Grok, pending the counsel review (FR-PA4); this doc ships DEGRADE as the working posture, not a closed question** — with a fail-loud breakage canary as the durability spine. This PRD folds F1's FR-D1..D7 down into the extension's own home, and adds what only the extension owns: one-tap connection, per-use consent + the personal↔employer firewall at the point of injection, breakage detection, extension-trust minimization, store distribution, and the build approach (**fork** `supermemory` + pattern-donate from `mem0`, per `IgOjzk034v`). It deliberately does **not** restate the memory model, the Librarian, or retrieval — those stay in canon. Three things gate the build and are surfaced here, never assumed away: a **legal-counsel review** covering both the "automated means" interpretation and Chrome Web Store third-party-ToS policy, **store-policy viability** with a named fallback distribution posture if the store rejects the build, and a **live-DOM prototype** per provider (including Gemini's closed-shadow-DOM check) before the canary is trusted.

## 1. Context

**The gap this closes.** The mid-2026 competitive assessment (`FC23qWA8n3`) found Orvex's biggest hole is *delivery*: its memory advantage is unreachable inside the assistants people actually use — above all ChatGPT, which has no memory-write API. Rivals who reach ChatGPT for a non-technical user (MemoryPlugin, Supermemory) do it with a **browser extension** — a channel Orvex had specified nowhere. `g9vWbSYplh` §5 makes F1 the Tier-1 flagship and gates it on a legal + technical viability spike, because "if it degrades to copy/paste it delivers nothing past the status quo, so viability is a go/no-go for the flagship — not a detail." The spike is done. This PRD is the component that carries F1 to build.

**Why a browser extension, and why now (the resolved decision).** The viability verdict `V6hlDjecfh` ruled the mechanism question (`g9vWbSYplh` OQ1): a **Chrome MV3 browser extension** is the ship vehicle — the only option with auto-update and real breakage telemetry, both load-bearing for fail-loud durability. The bookmarklet is a ToS-hygiene comparator only (no auto-update; CSP already breaks it on X); a native helper is rejected on trust and ToS grounds. The extension acts **only inside the user's own already-authenticated tab, on the user's action** — it is not scraping, not a headless bot, not automated sending.

**The competitive white space (why now).** Three current facts frame the opening (landscape research, accessed 2026-07-15):
- **The extension is the proven non-technical channel into ChatGPT.** Multi-model sidebar extensions sustain paid subscriptions at scale — Sider (~4–6M users), Monica (10M+), Merlin (5M+) at $8–20/mo — but *memory-first* extensions have not cracked that distribution: MemoryPlugin, the closest analog, is reported at only ~3,800 users **(2026-07-15 landscape-research digest — a research finding, not an independently verified count; `IgOjzk034v` §3 lists MemoryPlugin's activity as "unknown," closed-SaaS, no repo)**. The channel works; the memory wedge is unclaimed.
- **A one-sided switcher-poaching race has left a neutrality vacuum.** Anthropic **reportedly** shipped Claude memory-import (pulling from ChatGPT/Gemini/Grok) in early 2026 — **reported, not primary-source-verified** (`V6hlDjecfh` §5: corroborated only by secondary coverage, not fetched against Anthropic's own primary page text; see OQ-E1 and FR-D6, which carry the same hedge). Google followed on 2026-03-27 with Gemini "Import Memory" + "Import Chat History." OpenAI has built **zero** reciprocal import either direction. Neither #2 nor #3 is neutral — they poach #1's users. A **neutral, cross-AI** memory layer is exactly what none of them will build.
- **Native memory is now table-stakes but silent by default.** OpenAI's "Dreaming" (2026-06-04) and Google's "Memories"/"Personal Intelligence" build and apply memory with **no per-use confirmation** — privacy researchers (MIT Tech Review, CDT) are already flagging invisible cross-context consolidation. Orvex's **human-confirm curation + per-use consent + the personal↔employer firewall** is the deliberate opposite.

No incumbent occupies Orvex's position: neutral cross-AI memory (portable, not lock-in), human-confirm curation, the personal↔employer firewall, and per-use consent — enforced right at the moment of delivery. The firewall in particular is **genuinely unclaimed**: no AI-memory product does it, and an estimated 93% of enterprise ChatGPT use running through ungoverned personal accounts **(2026-07-15 landscape-research digest — a research finding cited as such, not an independently verified statistic)** plus the EU AI Act's high-risk provisions landing 2026-08-02 are a real compliance tailwind. `[ASSUMPTION — Anthropic's import tier-gating is reported inconsistently across sources (paid-only vs. all-tiers); this PRD cites no specific tier number pending primary-source verification.]`

**Build reality (`IgOjzk034v`).** This is not greenfield: two MIT open-source extensions are genuine prior art. `supermemoryai/supermemory` (apps/browser-extension, MIT, actively maintained) is the recommended **fork base** for its WXT/MV3 scaffold and per-provider layout; the archived MIT `mem0ai/mem0-chrome-extension` is the **pattern donor** for the correct composer-write technique and the Gemini/Claude/Grok selectors supermemory lacks. But the two hardest, most Orvex-specific pieces — the fail-loud breakage canary and the pre-send **review pause** (both prior-art extensions auto-inject silently, architecturally opposite to Orvex's consent model) — have **zero OSS prior art** and are net-new.

## 2. Goals & success metrics

**Goals.** Make Orvex memory (a) *reachable* inside the assistant a non-technical user already uses, (b) *deliverable in one guided step with no technical setup*, and (c) *trustworthy* — consented, firewalled, and honest about breakage and ToS.

**Success metrics.** Targets marked `[ASSUMPTION]` are provisional (inherited from `g9vWbSYplh`), each naming its measurement method; owner: PM + analytics; revisit at architecture.

- **Delivery reach** — ≥ 50% `[ASSUMPTION]` of weekly-active users invoke Orvex *inside ChatGPT or Gemini* (not only Claude / copy-paste). *Method:* per-assistant delivery telemetry (FR-D1).
- **Connect activation** — ≥ 40% `[ASSUMPTION]` of new extension installs complete a first assistant connection *and* produce one memory-enriched output within 7 days. *Method:* connect + first-enriched-output funnel.
- **One-step connect** — a non-technical user connects an assistant in **≤ 3 taps and ≤ 2 minutes, no config file / command line** (FR-CN1). *Method:* first-run usability test (hard gate, from FR-O4).
- **Breakage honesty** — **100%** of injection breakages are detected and surfaced (fail-loud), **0** silent mis-injections. *Method:* canary match-failure telemetry classified NOTFOUND/NOTWRITABLE/AMBIGUOUS/VERIFY (FR-BC5) vs. user-reported failures. Detection is measured separately from **remediation** (canary-trip-to-fix), which is bounded by the FR-BC7/NFR-8 SLA — a detected-but-unfixed breakage is not counted as "honest" past the SLA window.
- **Trust** (hard, not provisional) — **zero** personal↔employer leaks; **100%** of private-memory injections consented; **0** background network calls to any non-Orvex endpoint. *Method:* firewall/consent audit log + a shipped-bundle network audit.

**Counter-metrics.**
- Connection support volume ("it's not working in ChatGPT") rising — a durability or UX regression.
- False-positive breakage alerts (canary cries wolf on an A/B UI variant) eroding trust in the fail-loud signal.
- Users abandoning at the permission-install prompt (extension-trust cost too high).
- Injected-but-wrong: text landing in the composer but not registering with the app's editor model (the supermemory-style missing-dispatch failure).

## 3. Scope

**In scope.** The browser extension that: writes an Orvex-composed, memory-enriched prompt into the compose surface of ChatGPT / Claude / Gemini / Grok web UIs (per-assistant GO/DEGRADE per `V6hlDjecfh`); connects an assistant in one guided step; enforces per-use consent + the personal↔employer firewall at injection; detects its own breakage and fails loud; degrades to copy/paste; and is distributed and updated through the extension stores. The build approach (fork/adopt) is in scope as a decision; its mechanics live in the addendum.

**Out of scope.** The memory model, stores, Librarian ritual, retrieval/eval, and proposal quality (all in canon — `g9vWbSYplh`, `iiCcKhGptV`, knowledge/ai service PRDs). Server-side scraping or reading conversation content out of the provider (explicitly forbidden — FR-D5). Outbound Memory *sync* for vendors with native memory APIs (knowledge-owned, FR-D6). A mobile app or non-browser surface. Mid-conversation background auto-recall inside third-party clients beyond what their surfaces permit.

**Boundary with the umbrella.** `g9vWbSYplh` remains the cross-cutting F1 owner (inherited invariants, cross-cutting NFRs, the memory model). This PRD owns the *extension component* only; where the two overlap (FR-D1..D7), this doc is the component's authoritative home and inherits the umbrella's intent verbatim.

## 4. Users & context

The user is a **non-technical professional running their day through AI** — the beachhead being regulated/high-stakes roles (a teacher like **Laura Pendleton**; doctors; lawyers). Two consequences bind the whole component: delivery and setup must need **zero technical skill** (no MCP, no JSON, no local server), and **consent/firewall are load-bearing**, not nice-to-haves.

**UJ-1 — Laura connects ChatGPT and gets an enriched draft (happy path, GO provider).** Laura, a secondary-school teacher, installs the Orvex extension from the Chrome Web Store. On first run it shows one card: "Connect an assistant." She clicks *ChatGPT*; the extension confirms she's signed in to chatgpt.com in her own tab and marks it Connected (≤ 3 taps, ≤ 2 min — no config). Later, composing a parents'-evening summary in ChatGPT, she clicks the Orvex button; the extension shows the memories it will add (her class list, the term's topics) and asks her to confirm — private items flagged, per-use consent. She confirms; the enriched prompt appears **in ChatGPT's own compose box**, she reads it, edits a line, and hits send herself. No copy/paste.

**UJ-2 — Laura on Claude (DEGRADE provider).** Laura clicks Orvex inside claude.ai. Because Claude is a DEGRADE provider (ToS), the extension does **not** silently inject; it hands her the enriched prompt via a **copy-ready panel** ("Claude requires copy/paste — copied, paste into Claude") and says so plainly. The "without copy/paste" promise is honestly withdrawn for Claude; she still gets the enriched prompt, one click away.

**UJ-3 — a provider changes its UI (breakage).** Google ships a Gemini redesign; the compose selector Orvex relied on is gone. The next time Laura clicks Orvex in Gemini, the extension's **pre-inject probe fails**: instead of writing into the wrong place or nothing, it **fails loud** — tells Laura "Gemini changed; use copy/paste for now," puts the prompt on her clipboard, and **alerts the Orvex team** so a fix ships before most users notice.

## 5. Features & functional requirements

Delivery FRs (FR-D*) are folded from `g9vWbSYplh` §5 F1 and are the component's authoritative home; the rest are extension-owned.

### F-A — Delivery (folded F1, FR-D1..D7)

- **FR-D1** The extension MUST deliver a composed prompt enriched with relevant Memory into the **ChatGPT and Gemini web UIs without copy/paste**, by writing into the user's own compose surface. *(GO per `V6hlDjecfh`.)*
- **FR-D2** A non-technical user MUST connect an assistant in **one guided step** — no MCP config, no local server, no JSON — with clear per-assistant connection status. *Acceptance:* FR-CN1.
- **FR-D3** Delivery MUST cover the beachhead assistants (ChatGPT, Claude, Gemini, Grok), each with honest per-assistant capability messaging (GO = inject; DEGRADE = copy/paste), and honest "not yet supported" for others.
- **FR-D4** Delivery MUST honor the personal↔employer firewall and **per-use consent** for private memories — never silent injection. *See F-D.*
- **FR-D5** Delivery MUST NOT scrape or breach provider ToS: it injects only into the user's *own* session/compose surface, and **degrades to copy/paste** when a provider blocks or disfavors injection. *(DEGRADE for Claude, Grok per `V6hlDjecfh`; "without copy/paste" is explicitly withdrawn for those two.)*
- **FR-D6** Outbound Memory *sync* for vendors with native memory APIs (Claude adapter) is **knowledge-owned**, not this component; the extension MUST NOT duplicate it. **This PRD MUST NOT conflate FR-D6 with FR-D1 delivery**: memory-sync reaching an independent GO is a separate mechanism with a separate ruling — it is *not* validated as equivalent to enriched-prompt injection, and it does not upgrade or substitute for Claude's FR-D1/FR-PA1 injection DEGRADE. Cross-reference only. `[ASSUMPTION — the Claude native-memory path may become a superior DEGRADE *route* (not an FR-D1 equivalent); confirm with knowledge, OQ-E1.]`
- **FR-D7** Delivery MUST **detect its own breakage** and fail loud — notify the user, fall back to copy/paste, and alert the team — rather than silently injecting into the wrong place or nothing. *Expanded in F-E.*

### F-B — Per-assistant mechanism & ToS posture (adopts `V6hlDjecfh` verbatim)

- **FR-PA1** The extension MUST adopt the viability verdict's per-assistant ruling as its shipped behavior: **ChatGPT — inject (GO)**, **Gemini — inject (GO, additionally conditioned on the closed-shadow-DOM gate, FR-PA6)**, **Claude — copy/paste (DEGRADE)**, **Grok — copy/paste (DEGRADE)**, each conditioned on the breakage canary (F-E). **Claude and Grok's DEGRADE is contested, not resolved**: a co-equal analysis in `V6hlDjecfh` read the persistent-extension mechanism as NO-GO for both providers; this PRD ships DEGRADE as the working posture pending the counsel review (FR-PA4), not as a closed question. For Claude specifically, this injection DEGRADE MUST NOT be conflated with the independent native-memory-sync GO (FR-D6): the two are separate mechanisms under separate rulings, and memory-sync reaching GO does not upgrade or substitute for FR-D1 delivery. The ruling page `V6hlDjecfh` is the cited source of record.
- **FR-PA2** The mechanism MUST be the **Chrome MV3 browser extension** (OQ1 resolved); bookmarklet and native-helper mechanisms are rejected and MUST NOT be shipped as the primary vehicle.
- **FR-PA3** A provider's tier MUST be **changeable without a code rewrite** — a per-provider config/manifest promotes Claude or Grok from DEGRADE to GO (or demotes a GO) when a legal opinion, provider permission, or ToS change warrants, so the verdict's "route to counsel" outcome is honored by data, not a rebuild. This tier config ships over the same versioned remote-JSON data channel as the FR-BC7 selector packs (data, not code — FR-TS4 unaffected).
- **FR-PA4** Because two providers' GO/DEGRADE rulings rest on a **legal interpretation** of "automated means" clauses (a co-equal analysis read them as NO-GO), the component MUST NOT ship injection for any provider until a **legal-counsel review** has confirmed its tier. That review MUST cover both (a) each provider's "automated means"/ToS interpretation, and (b) the Chrome Web Store Developer Program Policy's prohibition on facilitating another service's ToS violations (FR-PA5) — a provider-ToS "win" does not guarantee a store listing survives. This is an explicit build precondition, tracked in §7 and §8.
- **FR-PA5** Chrome Web Store (CWS) policy viability is a **hard build precondition**, co-equal with FR-PA4's counsel review — not merely a fix-latency assumption. CWS policy prohibits extensions that facilitate violations of another service's terms, and a reviewer can apply that same reasoning to reject or later take down the injecting build, **including for the GO providers, not only the DEGRADE ones**. The component MUST (a) have its store listing frame injection as **user-initiated own-session assistance** — the user reviews and sends, the extension does not automate sending or scrape — and (b) name a **fallback distribution posture** for use if CWS rejects or removes the listing: self-host/enterprise sideload first, Firefox AMO as a second channel. See §7 (build preconditions) and §8 (store-distribution posture).
- **FR-PA6** Every GO provider's live-DOM prototype MUST **rule closed-shadow-DOM in or out** as a categorical go/no-go, distinct from the generic selector-validation caveat: if a provider's composer root is a shadow root with `mode:'closed'`, DOM injection is **categorically impossible** — not a selector-fragility issue the canary can absorb — and that provider MUST fall to DEGRADE (copy/paste) regardless of ToS posture, routed through the FR-PA3 re-tier-by-config path (not a code change). This gate is most salient for **Gemini** (Angular/Material). *Calibration (name the gate, do not pre-judge it):* Gemini's composer is Quill/Angular, and Quill renders into a **light-DOM `contenteditable`** (addendum §A), which *lowers* the closed-shadow-DOM prior versus a Material-web-component assumption; neither spike did live-DOM inspection, so a closed root stays a hypothesis to rule in/out with the prototype, not a prediction that Gemini is blocked. See OQ-E2 and addendum §C.

### F-C — Connection & onboarding

- **FR-CN1** A non-technical user MUST complete a first assistant connection in **≤ 3 taps and ≤ 2 minutes with no config file or command line**, verified by first-run usability testing. *(Folds FR-O4.)*
- **FR-CN2** The extension MUST show **per-assistant connection status** (Connected / Not signed in / Not yet supported / Needs copy/paste) truthfully — a status never claims a capability the canary can't currently deliver.
- **FR-CN3** Connection MUST require **only that the user is signed in to the provider in their own browser** — no Orvex-held provider credentials, no OAuth to the provider, no separate account step *for the provider connection*. (This governs the provider-connection step only; the extension's own authentication to Orvex is a separate, required obligation — FR-AU1.)
- **FR-CN4** Every connection surface MUST have a **designed empty/again state** that teaches the next action (not a blank panel). `[ASSUMPTION — aligns with FR-O3; UI detail owned by ui service.]`

### F-D — Consent & firewall at the point of delivery

- **FR-CF1** Before any injection or copy, the extension MUST **show the memories it will include** and obtain **per-use consent** for private memories — never silent injection (FR-D4). Consent is per-use by default; automatic runtime recall is default-off (NFR-2). Where delivery is the clipboard fallback (FR-BC4), the same consent gate applies and MUST disclose the clipboard exposure (FR-CF3) before copying private/employer content — the fallback is not a neutral, unconsented path.
- **FR-CF2** The **personal↔employer firewall** MUST be enforced at delivery: employer-scoped memories are never mixed into a personal session (or vice-versa) without explicit, scoped consent. A firewall breach is a hard failure, not a warning. The scope enforced here is the one bound by FR-AU2 (F-H) — the firewall's subject — and the identity match is guarded by FR-CF7.
- **FR-CF3** The extension MUST **disclose the deletion boundary** at consent time: content already injected into a third-party session, **or placed on the system clipboard** (FR-BC4), is outside Orvex's later deletion reach (FR-L5) and is readable by other apps, other extensions, and OS clipboard sync/history on the device — stated plainly, not assumed away.
- **FR-CF4** Injected/retrieved memory MUST be treated as **untrusted data, not instructions** (defense against stored prompt-injection carried in a memory) — the extension composes, it does not execute memory content. *(Inherits FR-S6, `g9vWbSYplh`.)*
- **FR-CF5** The extension MUST support **consent revocation and connection removal**: a user can disconnect an assistant or revoke a standing consent at any time, taking effect on the next delivery attempt — a revoked consent MUST fail closed, never fall back to a cached prior consent.
- **FR-CF6** On uninstall or explicit disconnect, the extension MUST **clear locally stored connection state, cached memory bundles, and the consent/injection audit log** (NFR-6). The audit log MUST NOT retain memory content beyond what breakage/consent evidence requires, and MUST be bounded (a stated retention window and size cap) and protected at rest.
- **FR-CF7** Before injecting scoped (personal or employer) memory, the extension MUST verify the target provider tab's signed-in identity matches the memory's scope wherever the provider surface exposes that signal (e.g., an account/workspace switcher); where the provider exposes no reliable signal, the extension MUST require explicit user confirmation of the target account before injecting scoped memory. This is the firewall's outward-facing half — FR-CF2 governs *which* memory is selected, FR-CF7 governs *which identity* receives it. A detected account mismatch, an account switch mid-session, or an undetectable target MUST fail closed to copy/paste, never silent injection.

### F-E — Breakage detection & resilience (FR-D7, made concrete)

The canary **is** the delivery transaction, not a separate check bolted onto a write.

- **FR-BC1** Every delivery MUST run as a single canary transaction: **resolve** the target compose node → **verify it is writable** (not merely present/attached) → **insert** via a native-setter + dispatched `InputEvent` (**never** a naive `.value =` / `.textContent =` assignment, which does not register with the app's editor model) → **read back** the result within a **~250 ms** budget → **branch** to success or one of four failure classes (FR-BC5). Nothing reaches the surface the user can see until resolve **and** verify-writable have both passed — an abort at either stage touches nothing visible.
- **FR-BC2** The read-back step MUST re-read the target node and assert the injected text is present at the intended node **and** registered with the app's editor model (guards the silent-no-op failure where a write never reaches ProseMirror/React). Injection is not "done" until read-back passes within the FR-BC1 budget.
- **FR-BC3** On a failure at any FR-BC1 stage, the extension MUST **fail loud**: abort **before** the write reaches the visible conversation/compose surface the user can see, surface a visible user notification, and **never** write into the wrong place or nothing.
- **FR-BC4** On abort, the extension MUST offer the **deterministic copy/paste fallback** (FR-D5) with the prompt already on the clipboard / in a copy-ready panel. Because the clipboard is readable by other apps, other extensions, and OS clipboard sync/history, this fallback MUST: (a) be covered by the FR-CF1 consent gate and disclose the clipboard exposure at consent time, not only at the moment of abort; (b) **minimize exposure** by auto-clearing the clipboard payload after a short timeout (`[ASSUMPTION — target 60s; PO/architecture to confirm]`) or on the next copy, whichever comes first; (c) visibly flag when the copied content includes employer-scoped memory (FR-CF3).
- **FR-BC5** The extension MUST emit breakage telemetry classified into **four distinct failure classes**, each alerted separately (provider + selector-version + class), so a provider UI change pages the team before users are broadly affected:
  - **NOTFOUND** — the compose node could not be resolved.
  - **NOTWRITABLE** — the node resolved but failed the writability check (disabled, read-only, wrong node type).
  - **AMBIGUOUS** — more than one candidate node matched and could not be disambiguated.
  - **VERIFY** — insert completed but read-back did not confirm the text landed/registered within budget.
- **FR-BC6** The canary MUST tolerate **per-user/per-region A/B UI variants and locale** without false-positive alerts (a multi-candidate selector manifest); selectors MUST NOT key on **localized ARIA labels or visible text** — an EU-beachhead French/German/other-locale DOM renders different `aria-label`/placeholder strings, which would false-trip the canary — prefer locale-invariant `data-testid`/`role`/structural anchors. Every trip MUST be logged — no silent degrade.
- **FR-BC7** Selector/DOM contracts MUST ship as **versioned, remote-fetched JSON data** per provider (signed/validated by the extension) — data, not code, so MV3's no-remote-code rule (FR-TS4) is unaffected — patchable within a bounded remediation SLA of `[ASSUMPTION — target ≤ 24h from canary trip to shipped selector-data fix; PO/architecture to confirm]` measured canary-trip-to-fix (NFR-8), hours not store-review days. A **code-level** fix (e.g., a new insertion technique) still ships as a reviewed store update and is bound by ordinary store-review latency instead.
- **FR-BC8** Before insertion (FR-BC1), the extension MUST detect existing non-empty content in the compose node and MUST NOT overwrite an in-progress user draft — insert at cursor / append, or prompt the user to choose (replace / append / cancel), never silently clobber user-authored text.

### F-F — Trust, security & permission minimization

- **FR-TS1** The extension MUST request the **minimum host permissions** required (only the beachhead provider origins it actively supports) and MUST justify each permission in the store listing — no broad `<all_urls>`.
- **FR-TS2** The shipped bundle MUST make **no background network call to any non-Orvex endpoint** and MUST NOT phone home to any third party; the memory backend is Orvex's own (replacing any `api.supermemory.ai` / `api.mem0.ai` calls inherited from forked code). *Verified by a bundle network audit (a success metric).*
- **FR-TS3** The extension MUST NOT read or transmit conversation content, credentials, or page data beyond confirming its own just-written text landed (FR-BC2). Read scope is minimal and disclosed.
- **FR-TS4** MV3 constraints MUST be honored: **no remote code** (every logic change ships as a reviewed store update), isolated-world content scripts, and a documented update cadence.
- **FR-TS5** The extension MUST address the **extension-trust cost** (canon flags real 900k / 8M-user malicious-extension exfiltration incidents) with atomic, testable obligations: (a) publish a **per-permission justification** in the store listing for every requested host/API permission; (b) ship a **reproducible build** — documented, independently re-runnable steps from tagged source to store bundle; (c) publish a **plain-language data-flow statement** (what leaves the browser, to which Orvex endpoint, verified by the FR-TS2 bundle network audit). The open-source-licensing question is a separate, unresolved PO decision tracked only in §8; this FR does not depend on it.

### F-G — Build approach (adopts `IgOjzk034v`)

- **FR-BA1** The component SHOULD be built by **forking `supermemoryai/supermemory` (apps/browser-extension, MIT)** for its WXT/MV3 scaffold and per-provider layout, **not** greenfield — pruning its non-F1 scope (bookmark/capture, broad permissions) on fork.
- **FR-BA2** The **composer-write technique and the Gemini/Claude/Grok selectors** SHOULD be pattern-donated from `mem0ai/mem0-chrome-extension` (MIT, archived) — including the input-event dispatch that supermemory's own writers omit — mined as patterns, never depended on as a live dependency, with its phone-home stripped (FR-TS2).
- **FR-BA3** The breakage canary (F-E), the pre-send **review pause** (F-D), the GO/DEGRADE split (F-B), the copy/paste degrade UI, and Orvex memory-enrichment wiring are **build-fresh** — no OSS prior art exists; the pack MUST NOT credit the fork with delivering these.
- **FR-BA4** License hygiene MUST be preserved: both sources are MIT (clean for a store-distributed, Orvex-backend extension); borrowed files retain their copyright/license notice; copyleft (GPL/AGPL) sources are excluded.

### F-H — Orvex authentication & scope binding

The firewall (FR-CF2) has no subject without this section: nothing else in this PRD states how the extension knows *who* and *which scope* it is delivering for.

- **FR-AU1** The extension MUST authenticate to Orvex independently of any provider connection: it MUST hold an Orvex session token in extension storage, refresh it before expiry, and provide an explicit sign-out that clears the token and all cached scope/memory state. Losing the Orvex session MUST NOT silently fall back to a cached or default scope.
- **FR-AU2** Every delivery (injection or copy/paste fallback) MUST bind to an **explicit, currently-active personal-vs-employer scope** resolved from the live Orvex session at the moment of delivery — never inferred from the provider tab, never defaulted. If the Orvex session is expired or absent at compose time, the extension MUST **fail closed**: no delivery, no unscoped or last-known-scope bundle, a visible re-auth prompt instead. This is the subject FR-CF2's firewall enforces against.

## 6. Cross-cutting NFRs

- **NFR-1 Latency** — the end-to-end **click → visible-in-composer** action MUST meet a stated, testable budget: p95 ≤ `[ASSUMPTION — 400ms; PO/architecture to confirm at architecture]`, decomposed as warm memory retrieval (inherits the knowledge p95 target, ~50 ms) + the FR-BC1 canary transaction (resolve → verify-writable → insert → read-back, budgeted at ~250 ms) + render. *Method:* client-side click-to-visible timing instrumentation, p95 over a rolling window.
- **NFR-2 Privacy & consent** — automatic runtime recall is **default-off**; per-use opt-in for private memories; delivery is a **user-initiated** act (compose-and-send), not silent background recall — this is what keeps FR-D1 consistent with privacy.
- **NFR-3 Neutrality** — no lock-in; the extension works across all supported assistants; memory stays portable (locked decision, inherited).
- **NFR-4 Graceful degradation** — a memory/AI/extension outage MUST NOT block the user's own use of the assistant; delivery degrades to copy/paste; never white-screen the provider's page.
- **NFR-5 Non-technical accessibility** — every extension feature usable with **no technical setup and no prompt engineering**.
- **NFR-6 Auditability & reversibility** — consent decisions and injections are logged (locally/audit) and reviewable, bounded and cleared per FR-CF6 (retention window, size cap, protection at rest, wipe-on-uninstall); the firewall/consent audit log is the trust evidence.
- **NFR-7 Localization** — extension surfaces MUST support i18n; English first, EU-beachhead languages phased. `[ASSUMPTION — locale priority set with GTM.]`
- **NFR-8 Store compliance & durability** — the extension MUST satisfy Chrome Web Store (and Firefox AMO, if targeted) MV3 review policy, including CWS's third-party-ToS-facilitation policy (FR-PA5) — **store rejection or takedown is a named channel risk, not only review latency.** Selector/DOM-contract churn is handled by the versioned remote-JSON selector-data channel (FR-BC7) within its bounded remediation SLA (target ≤ 24h canary-trip-to-fix, `[ASSUMPTION]`); a code-level fix still needs a store update and is bound by ordinary review latency instead.
- **NFR-9 Security** — least-privilege permissions, no third-party network egress, no remote code (FR-TS*), and a supply-chain-clean build.
- **NFR-10 Accessibility** — the extension's own surfaces (connect card, consent/memory-preview panel, copy-ready panel, breakage notifications) MUST be keyboard-operable and screen-reader-announced (WCAG 2.1 AA as the target level); injection MUST NOT silently steal focus from the user's cursor without an announced, predictable focus transition.
