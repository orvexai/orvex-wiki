# Prior Art Deep-Dive: Google Docs Suggesting Mode

## What it is
Google Docs "Suggesting" mode is an editor permission level (View / Comment / **Suggest** / Edit) in which a user with comment-level access can propose insertions, deletions, and formatting changes that render as colored, attributed, non-destructive overlays on the live document rather than being applied directly. Anyone with edit access can accept or reject each suggestion individually (or in bulk via Tools ▸ Review suggested edits ▸ Accept all / Reject all), at which point the change is merged into or dropped from the canonical document. It launched June 25, 2014 on the Rapid Release track, replacing Docs' earlier separate "track changes"-style workflow. [Suggest edits in Google Docs (Help Center)](https://support.google.com/docs/answer/6033474), [Google Workspace Updates: Suggested edits in Google Docs (launch post, 2014-06-25)](https://workspaceupdates.googleblog.com/2014/06/suggested-edits-in-google-docs.html)

## Behind it & traction
Owned and operated by Google as a built-in feature of Google Docs / Google Workspace; no separate company, no standalone GitHub repo (it is closed-source, part of the Docs web client). No public usage/traction numbers (DAU of suggesting mode specifically) were found in this research — Google does not publish feature-level adoption metrics for Docs suggesting mode. unknown (no sourced figure).

## Architecture & data model
Suggestions are modeled as first-class objects inside the Docs document JSON, not as a separate diff/patch store:
- Text runs carry `suggestedInsertionIds` and `suggestedDeletionIds` arrays that tag a run of text as a pending insertion or deletion, keyed by suggestion ID.
- Formatting-only changes are modeled via `suggestedTextStyleChanges` / `textStyleSuggestionState`, which record exactly which style fields were suggested vs. applied.
- A `suggestionsViewMode` parameter controls how a fetched document renders: `SUGGESTIONS_INLINE` (show pending changes inline, needed to get correct character indices for further edits), `PREVIEW_SUGGESTIONS_ACCEPTED`, or `PREVIEW_WITHOUT_SUGGESTIONS` (as if all suggestions were rejected).
- Each suggestion thread is backed by a comment thread (the "head post" of a suggestion is system-generated and not directly editable).
[Work with suggestions | Google Docs API (Google for Developers)](https://developers.google.com/workspace/docs/api/how-tos/suggestions)

## API & integration surface (MCP? SDK? CLI? chat-platform plugins?)
- **REST API** (`documents.batchUpdate`, `documents.get`) with official SDKs (Java, Node/googleapis, Go, Python, etc.) — no MCP server, no CLI, no chat-platform plugin from Google itself.
- **Read**: fully supported today — fetch a document with `suggestionsViewMode=SUGGESTIONS_INLINE` and enumerate pending suggestions and their IDs/authors.
- **Write (create/accept/reject) is a Developer Preview feature, not GA**: per the official how-to, you set `writeMode: SUGGEST` on the `WriteControl` object of a `batchUpdate` request to make edits land as suggestions instead of direct writes, and use dedicated `AcceptSuggestionRequest` / `RejectSuggestionRequest` request types to resolve them programmatically — but this is explicitly gated behind the [Google Workspace Developer Preview Program](https://developers.google.com/workspace/preview), meaning it can change or be pulled before GA and requires preview enrollment. [Work with suggestions | Google Docs API](https://developers.google.com/workspace/docs/api/how-tos/suggestions), [documents.batchUpdate reference](https://developers.google.com/workspace/docs/api/reference/rest/v1/documents/batchUpdate)
- Suggest-mode writes explicitly cannot: add/delete tabs, headers, or footers; create/delete named ranges; or change document/header/footer format settings — a documented scope limitation on what "suggest" can express vs. a direct edit. [Work with suggestions | Google Docs API](https://developers.google.com/workspace/docs/api/how-tos/suggestions)
- No third-party/community MCP wrapper for Docs suggestions was identified in this search; general "Google Docs MCP servers" exist in the wild for basic doc CRUD but none surfaced specifically implementing suggestion accept/reject. unknown.

## Staging & review workflow
- **Propose → review → apply/reject**: a Suggest-permission user's edits are captured as pending ops attached to the live doc (no separate staging document/branch); an Edit-permission user reviews inline and accepts or rejects each one, which either merges the op into the canonical content or discards it.
- **Change granularity**: per-run text insertion/deletion and per-field style change — i.e., sub-paragraph granularity, not whole-document or whole-section. Each is independently addressable and resolvable.
- **Batch review at scale**: the UI-level primitives are only "Accept all" and "Reject all" (all-or-nothing) plus one-by-one accept/reject navigation; there is no documented native UI or API primitive for reviewing/accepting a filtered subset (e.g., "accept all suggestions from author X" or "accept all in section Y") — user community threads confirm people repeatedly ask for partial/selective bulk accept and are told to do it manually one at a time or via Apps Script hacks. [Accepting multiple changes, but not all changes (Docs Editors Community)](https://support.google.com/docs/thread/95333134/accepting-multiple-changes-but-not-all-changes), [Need help accepting one single suggestion at a time (Docs Editors Community)](https://support.google.com/docs/thread/10289073/need-help-accepting-one-single-suggestion-not-a-comment-at-a-time-not-accept-all-or-reject-all)
- **Conflict handling vs. live content**: suggestions are stored as offsets/annotations against the current document state; because indices shift as suggestions are added/resolved, the API documentation explicitly warns that you must fetch with `SUGGESTIONS_INLINE` to get correct indices before issuing further batchUpdate ops — i.e., conflict-safety is handled by requiring the client to always re-fetch current state rather than by the server performing three-way merges. No optimistic-locking/CRDT mechanism beyond `WriteControl`'s revision-check for batchUpdate calls is documented. [documents.batchUpdate reference / WriteControl](https://developers.google.com/workspace/docs/api/reference/rest/v1/documents/batchUpdate)
- **Reviewer UX + automation**: no native auto-approve rules and no native AI reviewer are documented as built into Docs suggesting mode itself; any such automation would have to be built externally (Apps Script or the Developer Preview API) by a third party. unknown / not offered by Google.

## Scale & operational evidence
No published evidence was found of Google Docs suggesting mode being stress-tested or documented at "100 changes in one session" scale, nor any public engineering post-mortem, benchmark, or capacity discussion of the suggestion data model at high suggestion counts. Community threads only describe UX friction (no partial bulk-accept) at modest scale, not systemic/performance failures. This is a genuine evidence gap: unknown whether the underlying data model degrades or how the UI performs with e.g. hundreds of concurrent pending suggestions in one doc.

## Pricing & positioning
Suggesting mode is a standard feature of Google Docs, included at no additional cost in every Google Workspace tier and in the consumer/free Google Docs product — it is not a paywalled or usage-metered capability. The programmatic write path (`writeMode: SUGGEST`, Accept/RejectSuggestionRequest) is gated by enrollment in the free Google Workspace Developer Preview Program, not by a separate price. [Suggest edits in Google Docs (Help Center)](https://support.google.com/docs/answer/6033474), [Work with suggestions | Google Docs API](https://developers.google.com/workspace/docs/api/how-tos/suggestions)

## STEAL - 3-5 concrete ideas for Orvex
1. **Model staged ops as annotations on the live document, not a shadow copy.** Docs' `suggestedInsertionIds`/`suggestedDeletionIds` pattern (tag runs in-place rather than diffing two full document trees) is a strong pattern for Orvex's staging store: attach proposal IDs directly to the target wiki content addresses (doc/section/field) so the librarian sees exactly what changed in context, and accept/reject is a local id-scoped merge/discard rather than a full-document three-way merge.
2. **Force a "current-state re-fetch" discipline before any staged op resolves.** Docs' documented requirement to re-fetch with `SUGGESTIONS_INLINE` for correct indices before further writes is a cheap, effective conflict-avoidance rule Orvex's librarian agent should adopt explicitly: never resolve a staged op against a cached snapshot of the wiki page — always resolve against current canonical state, and surface a hard conflict (not a silent merge) if the underlying content moved since the agent proposed the change. Fits the "no fallbacks — hard cuts" house rule.
3. **Distinguish content suggestions from style/format suggestions as separate op types.** Docs' split between text suggestions and `suggestedTextStyleChanges` maps well onto Orvex's five op kinds (add/section/edit/replace/delete) — keep "beautification" (the librarian's stylistic pass) as its own reversible op type distinct from substantive content changes, so a customer can bulk-accept style cleanups without also blindly accepting content edits.
4. **Ship selective bulk-review from day one — Docs' biggest, most-repeated user complaint is the gap Orvex must not repeat.** Years of Docs Editors Community threads beg for "accept all suggestions from this author / in this range" and never got it (only all-or-nothing + one-by-one). Since the librarian must triage up to 100 proposed changes in a single chat session, build filterable bulk accept/reject (by agent, by document, by proposal type, by confidence) as a core primitive, not an afterthought.
5. **Treat "who proposed this and why" as first-class, addressable metadata, not just a comment thread.** Docs attaches a system-generated, non-editable comment thread as the suggestion's provenance record. Orvex's staged ops should similarly carry immutable provenance (which agent, which conversation/session, rationale) that survives into the applied wiki history, so a librarian's bulk-accept decisions remain auditable after merge — Docs loses most of that thread context once a suggestion is accepted and the thread resolves.

## AVOID / where Orvex differs
- **Docs has no reviewer automation or AI-assisted triage at all** — every accept/reject is a manual human click, even at 2026 Developer Preview state (the preview API only lets *code* create/accept/reject suggestions the way a human would, it does not add any policy/rule engine). Orvex's librarian agent (an AI reviewer with a tunable prompt) is a genuine capability gap Docs does not fill — do not assume "Docs already solved this," build it.
- **No selective/filtered bulk accept-reject exists in Docs even after 12 years** — this is a proven, long-standing product gap, not a minor omission; Orvex should not copy Docs' all-or-nothing bulk model.
- **Docs' write/accept/reject API is still Developer Preview, not GA, and access-gated** — it is not safe or wise for Orvex to model its own staging engine as "we'll integrate with Docs's suggestion API," nor to assume that pattern is battle-tested at scale; it is unproven even by Google's own admission.
- **No documented scale evidence for high suggestion counts** — do not cite Docs suggesting mode as proof that annotation-in-place staging survives 100-proposal sessions; Orvex needs its own load testing, since Google has not published anything to lean on here.
- **Docs suggestions live only inside the single target document** — there is no cross-document or cross-workspace staging queue in Docs; Orvex's staging store spans many customer documents and needs its own queue/inbox abstraction that Docs' single-document model doesn't provide a pattern for.
