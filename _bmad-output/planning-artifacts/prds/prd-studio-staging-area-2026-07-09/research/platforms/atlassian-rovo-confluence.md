# Atlassian Rovo (Confluence AI agents) — Deep Dive

Researched 2026-07-10. Prior-art platform for the Orvex Studio staging-area PRD track.

## What it is

Rovo is Atlassian's cross-product AI platform (search, chat, and "agents") bundled into Jira, Confluence, JSM, Bitbucket and Compass. It became available to all customers at Team '25 (April 2025) and received its largest update wave at Team '26 (May 2026). In Confluence specifically, Rovo lets users invoke AI agents from an `/ai` slash command in the page editor, from a "Rovo Chat" panel, and from Confluence/Jira automation rules, to draft, summarize, translate, and edit page content, and to keep Confluence pages in sync with Jira updates (e.g. a "Product Requirements Agent"). [Rovo in Confluence: AI features](https://www.atlassian.com/software/confluence/ai) · [Agents \| Rovo \| Atlassian Support](https://support.atlassian.com/rovo/docs/agents/) · [Automate your content workflow with Confluence and Rovo](https://www.atlassian.com/software/confluence/resources/guides/best-practices/confluence-rovo)

## Behind it & traction

Built and owned by Atlassian (public company, NASDAQ: TEAM) as a first-party platform capability, not a separate acquisition. Marketing claims (Team '26, May 2026): 90%+ of Atlassian's enterprise cloud customers now use Rovo; agentic automations grew ~7x in six months; 14M+ "Rovo-assisted actions" in a single month. Customer quote cited from Docusign praising governance focus. These are Atlassian's own announced figures, not independently audited. [Rovo makes AI-native teamwork real for the enterprise — Inside Atlassian](https://www.atlassian.com/blog/company-news/rovo-team-26) · [Atlassian opens Teamwork Graph and pushes Rovo into agentic execution at Team '26 — SiliconANGLE](https://siliconangle.com/2026/05/06/atlassian-opens-teamwork-graph-pushes-rovo-agentic-execution-team-26/)

## Architecture & data model

Rovo agents run against the "Teamwork Graph," described as a 150-billion-connection graph spanning Atlassian's own products; at Team '26 this graph was opened up so external/third-party agents can read and write into it via new MCP tools and a CLI, with external-agent actions feeding back into the graph. Agents act **under the invoking user's own permission context** — support docs state an agent "can only return or interact with spaces, pages, work items ... that the user has permission to access," i.e. no separate service-account privilege elevation for read access. Exact internal architecture (how an agent's proposed edit is represented pre-write, versioning, storage) is not published. Unknown: whether there is any internal staging table/queue distinct from the live page store. [Rovo agent permissions and governance](https://support.atlassian.com/rovo/docs/rovo-agent-permissions-and-governance/) · [Atlassian Team '26: Teamwork Graph Opens to Agents](https://theoutpost.ai/news-story/atlassian-opens-teamwork-graph-and-evolves-rovo-into-autonomous-agent-at-team-26-26033/)

## API & integration surface (MCP? SDK? CLI? chat-platform plugins?)

- **Atlassian Rovo MCP Server**: official *remote* MCP server connecting Jira, Confluence, JSM, Bitbucket, and Compass to Claude, ChatGPT, Cursor, VS Code and other MCP clients, via OAuth 2.1 (interactive) or API-token/Basic-Auth (machine-to-machine, incl. service accounts). All actions are scoped to the authenticating user's/service account's existing permissions. [Atlassian Rovo MCP Server docs](https://support.atlassian.com/atlassian-rovo-mcp-server/docs/use-atlassian-rovo-mcp-server/) · [atlassian/atlassian-mcp-server on GitHub](https://github.com/atlassian/atlassian-mcp-server) · [Extend Atlassian into any AI assistant using MCP](https://www.atlassian.com/platform/remote-mcp-server)
- **Teamwork Graph MCP tools + CLI** (Team '26): lets *external* agents (not just Atlassian's own) query/act on the graph, distinct from the Confluence/Jira content MCP server above.
- **In-product surfaces**: `/ai` slash command in the Confluence editor; Rovo Chat side panel; Rovo agents embeddable inside Jira/Confluence automation rules (no-code triggers, e.g. "on content updated").
- **No dedicated staging-plane API** is documented — writes go through the same content APIs any user/editor would use, gated by the calling identity's permissions.

## Staging & review workflow (propose→review→apply/reject; granularity; batch review at 100 changes; conflict handling; reviewer UX/automation)

- **No separate staging plane.** Rovo's own docs and third-party guides describe direct-write and in-editor collaborative drafting (`/ai` → generate/summarize/translate directly in the page), not a propose-then-approve queue outside the wiki. Atlassian's chosen governance lever is **permissions + audit + optional per-action human-approval gates**, not a staging store. [Agents \| Rovo docs](https://support.atlassian.com/rovo/docs/agents/)
- **Approval is configured, not structural**: "Depending on your organization's security and governance requirements, Rovo Agents can either operate autonomously or require human approval before executing sensitive or irreversible actions" — implemented via Confluence/Jira **automation rules** (e.g. "trigger approval when content is updated"), which admins must explicitly set up per space/project. Without that automation, agents write directly. [Rovo agent permissions and governance](https://support.atlassian.com/rovo/docs/rovo-agent-permissions-and-governance/) · [community: audit log discussion](https://community.atlassian.com/forums/Atlassian-AI-Rovo-discussions/Rovo-activities-are-now-live-in-the-Audit-log/td-p/2989792)
- **Granularity**: page-level, generated via prompts into "structured outlines, summaries, or drafts directly in Confluence." No documented section/field-level diff-and-merge model equivalent to what Orvex needs (add-section, replace, delete at sub-document granularity).
- **Batch review at scale (100 changes/session)**: unknown / not documented anywhere found. No evidence of a bulk review UI for many pending AI-authored changes across many documents in one pass.
- **Conflict handling vs. live content**: not documented. No mention of optimistic-locking, merge conflicts, or reconciliation when an agent's draft and a human edit collide.
- **Reviewer UX / automation**: the closest analog is **Confluence/Jira automation rules** used as an approval gate (no-code, "no technical setup required" per Atlassian marketing) plus the org-wide **agent governance dashboard** (Team '26) showing who built which agents, where they run, and credit/usage consumption — this is fleet-level governance, not per-change review UX. There is no documented "AI reviewer" that grades/vets agent output before merge.
- **"Max Mode" (Team '26)**: a new autonomous multi-step execution mode that "executes autonomously while maintaining human oversight checkpoints" and "self-resolves workflow breakdowns before escalating to humans" — Atlassian's explicit bet is oversight checkpoints + escalation, not a staged-content review queue. [Rovo makes AI-native teamwork real for the enterprise](https://www.atlassian.com/blog/company-news/rovo-team-26)

## Scale & operational evidence

- Real-world community reports describe reliability problems specifically on **write/action** paths: "Rovo consistently said it completed the task but never actually did" and a documented tendency to "hallucinate success on write actions rather than flagging that it couldn't complete them properly." Rovo also reportedly "loses context when working with large volumes of information," causing mistakes on large datasets. [Does anyone else feel like Rovo is just A, not AI?](https://community.atlassian.com/forums/Rovo-questions/Does-anyone-else-feel-like-Rovo-is-just-A-not-AI/qaq-p/3215367) · [When Rovo Stops Helping and Starts Hiding Risk](https://community.atlassian.com/forums/Atlassian-AI-Rovo-articles/When-AI-Stops-Helping-and-Starts-Hiding-Risk/ba-p/3176985)
- Community practitioner guidance explicitly warns: "concerns have been raised about Confluence pages created by AI with minimal human review, and if your Confluence space is messy or outdated, Rovo may amplify the wrong content." This is direct third-party evidence that *lack of a hard staging gate* is a live pain point for Confluence customers, not merely a hypothetical. [When to Use Atlassian Rovo AI, and When Not To](https://community.atlassian.com/forums/Atlassian-AI-Rovo-articles/When-to-Use-Rovo-and-When-Not-To/ba-p/3171904)
- No published benchmark for "how many document edits reviewed/applied per session" — unknown whether Rovo has ever been evaluated at anything like a 100-document-single-session scale.

## Pricing & positioning

- Rovo Search, Chat, and Agents are **bundled into paid Jira/Confluence/JSM Cloud plans** (Standard/Premium/Enterprise) at no separate SKU cost; Atlassian Intelligence auto-activates on paid tiers. [Rovo Plans and Trial](https://www.atlassian.com/licensing/rovo)
- **Rovo credits** (consumption unit) per user/month: 25 (Standard), 70 (Premium), 150 (Enterprise) for individual products; Teamwork Collection bundle gives 250/700/1,500. Overage billing exists in the model but is not currently being enforced (Atlassian committed to 90 days' notice before activating it). [Understand Rovo Billing and Managing Costs](https://support.atlassian.com/rovo/kb/understand-rovo-billing-and-managing-costs-in-atlassian-cloud/)
- **Rovo Dev** (separate, developer-focused coding agent) is priced independently at $20/developer/month with 2,000 credits, extra usage billed $0.01/credit. Not the Confluence-writing agent track. [Rovo Dev Pricing](https://www.atlassian.com/software/rovo-dev/pricing)
- Positioning: governance-as-a-selling-point. Atlassian leans on permissions inheritance ("agent acts as the user"), audit logging, and fleet-visibility dashboards as the trust story for enterprise buyers, explicitly contrasted (per Atlassian's own messaging) against uncontrolled "agent sprawl." A customer quote (Docusign) is used to sell exactly this framing. [Rovo makes AI-native teamwork real for the enterprise](https://www.atlassian.com/blog/company-news/rovo-team-26)

## STEAL — 3-5 concrete ideas for Orvex

1. **Ride existing identity/permission scoping, don't reinvent it.** Rovo's strongest, best-evidenced governance primitive is "the agent can only touch what the invoking user/service-account could already touch." Orvex's librarian/staging agents should inherit the proposing agent's/customer's existing wiki RBAC rather than needing a bespoke staging-only permission model.
2. **Fleet-level visibility dashboard is cheap and high-value.** Even without a staging plane, Rovo's Team '26 "org-wide agent inventory" (who built what, where it runs, credit burn) is clearly resonating with enterprise buyers. Orvex's staging area should ship an equivalent "what have my agents proposed, how much, from where" admin view early — it's a governance sell independent of the merge mechanics.
3. **Machine-to-machine auth via API token/service account, not just interactive OAuth.** Rovo MCP's addition of Basic-Auth/API-token auth specifically for automated, non-interactive agent workloads is the right pattern for Orvex's agent-to-staging-store submissions (agents aren't humans clicking OAuth consent).
4. **Use "approval gate via automation rule" as a fallback pattern for low-stakes customers, but don't make it the whole governance story.** For customers who want zero staging-store overhead, offering a lightweight "require approval on write" toggle (Rovo's model) is a reasonable *cheap tier*, distinct from Orvex's full staging area for higher-stakes tenants.
5. **Explicitly name and market against the exact failure mode Confluence users are already complaining about.** Third-party community evidence (see Scale section) shows real customers unhappy about "AI pages with minimal human review" and hallucinated write-success. Orvex's PRD/marketing can cite this directly: a mandatory staging store with a librarian review pass is the fix for a documented, live pain point in the closest incumbent.

## AVOID / where Orvex differs

- **No staging plane at all is Rovo's central bet, and it shows.** Rovo writes Confluence content largely in-line (editor `/ai`, direct agent writes) gated only by optional per-space automation-rule approvals that admins must proactively configure; the default is direct write. Orvex's core thesis — agents NEVER write the wiki directly, always via an external staging store reviewed by a librarian agent — is a structural inversion of Rovo's model, not an incremental variant. Don't converge toward "governance via automation rules bolted onto direct write"; keep the staging store as a hard architectural boundary, not an opt-in toggle.
- **No evidence of section/field-level granularity or batch-scale review.** Rovo appears to operate at whole-page/whole-draft granularity with no documented mechanism for reviewing many discrete section-level edits across 100 documents in one pass — precisely the scale requirement in the Orvex PRD. Do not assume Rovo's model has been proven at that scale; there is no evidence it has.
- **No documented conflict-resolution model.** Orvex must design its own optimistic-concurrency / conflict-detection between staged proposals and concurrent live-wiki edits; Rovo offers no prior art to borrow here.
- **Credit-based consumption pricing tied to per-action agent credits is a source of customer complaints ("Rovo Credit Trap") in the ecosystem** — worth avoiding an analogous opaque per-action metering model for the librarian/staging pipeline if predictability matters to Orvex customers.
- **Reliability-of-execution gap**: community reports of agents claiming a write succeeded when it silently didn't is a direct argument *for* the durability/audit guarantees a staging store forces (every proposal is a real persisted row before any merge), something Rovo's direct-write model doesn't inherently provide.
