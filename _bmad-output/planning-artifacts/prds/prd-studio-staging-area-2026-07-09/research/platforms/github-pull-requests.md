# Prior Art Deep-Dive: GitHub Pull Requests + Suggested Changes + Merge Queue

Researched 2026-07-10 for the Orvex Studio AGENT STAGING AREA track (propose → review → merge model).

## What it is

GitHub's native code-review and integration pipeline: a **pull request (PR)** isolates a proposed diff on a branch away from the protected base branch; **suggested changes** let a reviewer author an inline, one-click-applyable edit on top of the proposer's diff (the "beautify" seam); **CODEOWNERS** auto-routes review requests to the right people/teams by file path; and **merge queue** batches many approved PRs, re-validates them together against a moving base branch, and lands them serially without breaking main. Together these form the reference propose→review→apply model for the staging-area track. [GitHub Docs: Reviewing proposed changes](https://docs.github.com/en/pull-requests/collaborating-with-pull-requests/reviewing-changes-in-pull-requests/reviewing-proposed-changes-in-a-pull-request), [GitHub Docs: Managing a merge queue](https://docs.github.com/en/repositories/configuring-branches-and-merges-in-your-repository/configuring-pull-request-merges/managing-a-merge-queue), [GitHub Docs: About code owners](https://docs.github.com/en/repositories/managing-your-repositorys-settings-and-features/customizing-your-repository/about-code-owners)

## Behind it & traction

Built and operated by GitHub (Microsoft-owned). Octoverse 2025 reports 180M+ developers on GitHub (36M+ new in the past year, "more than one new developer per second") and 518.7M pull requests merged in the year (+29% YoY), ~43.2M merged PRs/month on average (+23% YoY) — this is the largest live corpus of propose/review/merge workflow usage in the industry. [GitHub Blog: Octoverse — a new developer joins GitHub every second](https://github.blog/news-insights/octoverse/octoverse-a-new-developer-joins-github-every-second-as-ai-leads-typescript-to-1/), [GitHub Blog: What 986 million code pushes say about the developer workflow in 2025](https://github.blog/news-insights/octoverse/what-986-million-code-pushes-say-about-the-developer-workflow-in-2025/)

Merge queue reached General Availability in 2023 and remains a maintained, actively evolving feature — GitHub previewed a native "stacked PRs" workflow (`gh-stack` CLI extension) in private preview starting April 13, 2026, explicitly to address large-merge / branch-capacity pain that plain merge queue doesn't solve. [GitHub Blog: GitHub merge queue is generally available](https://github.blog/news-insights/product-news/github-merge-queue-is-generally-available/), [InfoQ: GitHub Targets Large Merge Problem with Stacked PRs](https://www.infoq.com/news/2026/04/github-stacked-prs/)

## Architecture & data model

- **PR** = a diff between a head ref (branch/fork) and a base ref, plus a thread of review comments, suggestion blocks, and status-check results; it is metadata layered over git, not a separate store.
- **Suggested changes** are review comments carrying a fenced ```suggestion block scoped to specific diff lines; applying one creates a real commit on the compare branch (single-suggestion or batched into one commit for multiple suggestions). [GitHub Docs: Incorporating feedback in your pull request](https://docs.github.com/en/pull-requests/collaborating-with-pull-requests/reviewing-changes-in-pull-requests/incorporating-feedback-in-your-pull-request)
- **CODEOWNERS** is a plain-text file (`.github/`, root, or `docs/`, checked in that priority order) mapping path globs to usernames/teams; branch protection can require their approval before merge. [GitHub Docs: About code owners](https://docs.github.com/en/repositories/managing-your-repositorys-settings-and-features/customizing-your-repository/about-code-owners)
- **Merge queue** materializes temporary branches per queue entry, each containing the base branch + all preceding queued PRs + the candidate PR (a "merge_group"), and reruns required status checks against that speculative merged state before allowing the real merge — this is the core conflict-vs-moving-target defense. Batch size (min/max PRs per group) and build concurrency (1–100 parallel merge_group builds) are admin-configurable. [GitHub Docs: Managing a merge queue](https://docs.github.com/en/repositories/configuring-branches-and-merges-in-your-repository/configuring-pull-request-merges/managing-a-merge-queue)

## API & integration surface (MCP? SDK? CLI? chat-platform plugins?)

- **REST API**: full CRUD on PRs, reviews, and review comments (`/rest/pulls/reviews`, `/rest/pulls/comments`), review-request management (`/rest/pulls/review-requests`). [GitHub Docs: REST API endpoints for pull request reviews](https://docs.github.com/en/rest/pulls/reviews), [REST API endpoints for review requests](https://docs.github.com/en/rest/pulls/review-requests)
- **GraphQL API**: exposes mutations not present in REST and vice versa — GitHub explicitly documents REST and GraphQL as a non-overlapping Venn diagram, so integrators sometimes need both. [GitHub Docs: Migrating from REST to GraphQL](https://docs.github.com/en/graphql/guides/migrating-from-rest-to-graphql)
- No dedicated "apply-suggestion" REST endpoint was found in the docs surveyed; applying a suggestion appears to be a UI/GraphQL-mutation-backed action rather than a documented public REST verb (unconfirmed — flagged, not verified further in this pass).
- **CLI**: `gh pr` subcommands cover create/review/merge; a new `gh-stack` extension (private preview, waitlisted, April 2026) adds native stacked-PR support. [InfoQ: GitHub Targets Large Merge Problem with Stacked PRs](https://www.infoq.com/news/2026/04/github-stacked-prs/)
- **Chat-platform plugins**: none native to core PR review; third parties (Slack/GitHub app notifications) exist but are integration add-ons, not part of the reviewed core surface — unknown depth, out of scope for this pass.
- **AI reviewer**: GitHub Copilot code review can be configured to auto-run on every PR, but it always posts a "Comment" review — never "Approve" or "Request changes" — so it cannot satisfy CODEOWNERS or required-reviewer gates; it is additive, not a substitute reviewer. As of June 1, 2026 Copilot code review runs are billed as AI credits and also consume GitHub Actions minutes on private repos. [GitHub Docs: Configuring automatic code review by Copilot](https://docs.github.com/en/copilot/how-tos/copilot-on-github/set-up-copilot/configure-automatic-review), [GitHub Changelog: Copilot code review will start consuming Actions minutes](https://github.blog/changelog/2026-04-27-github-copilot-code-review-will-start-consuming-github-actions-minutes-on-june-1-2026/)

## Staging & review workflow

- **Propose → review → apply/reject**: branch = staging area; PR = the proposal envelope; suggested-changes = the inline "beautify" seam a reviewer uses to fix up the proposal without a separate round-trip; merge = apply; close-without-merge = reject. This is the closest existing analogue to Orvex's "agent proposes, librarian reviews/beautifies/merges" flow.
- **Change granularity**: PRs are diff/commit-grained (file + hunk level), not document/section/field-aware — GitHub has no native concept of "this PR proposes one wiki document" vs "one section." Granularity finer than a file is only line ranges within a diff.
- **Batch review at scale (100 changes in one session)**: no native single mechanism reviews 100 independent content changes as one unit. The two real levers are (a) "apply suggested changes as a batch" — bundles multiple suggestions into one commit, but that's multiple suggestions *within one PR/session*, not 100 separate proposals — and (b) merge queue, which batches multiple already-approved PRs into shared validation groups (default max batch ~5, admin-configurable 1–100) purely for CI re-validation, not for human review batching. [GitHub Docs: Incorporating feedback](https://docs.github.com/en/pull-requests/collaborating-with-pull-requests/reviewing-changes-in-pull-requests/incorporating-feedback-in-your-pull-request), [GitHub Docs: Managing a merge queue](https://docs.github.com/en/repositories/configuring-branches-and-merges-in-your-repository/configuring-pull-request-merges/managing-a-merge-queue)
- **Conflict handling vs. live content**: merge queue's core trick — build a temporary "merge_group" branch (base + all prior queued PRs + candidate) and rerun required checks on that speculative state; if checks fail or there's a base-branch conflict, only the offending PR is evicted and the rest of the queue is regrouped and retested automatically. This directly maps to Orvex's problem of many agent proposals racing against a live-editable wiki.
- **Reviewer UX + automation**: CODEOWNERS gives deterministic path-based routing plus an optional hard gate ("require review from code owners"); best-practice guidance layers load-balancing on top so no single owner is bottlenecked. Auto-approve rules aren't a first-class GitHub primitive — they're typically bolted on via bots/Actions (e.g., auto-merge on green CI + approval), not a core PR feature. AI review (Copilot) is comment-only by design, preserving a human (or human-equivalent) approval gate — GitHub deliberately did not let AI self-approve.

## Scale & operational evidence

- 518.7M PRs merged across GitHub in the trailing year per Octoverse 2025, ~43.2M/month average — evidence the underlying data model holds up at internet scale, though this is aggregate platform volume, not evidence for any single repo handling 100 simultaneous proposals in one session. [GitHub Blog: Octoverse](https://github.blog/news-insights/octoverse/octoverse-a-new-developer-joins-github-every-second-as-ai-leads-typescript-to-1/)
- Merge queue GA'd in 2023 and is still being actively extended in 2026 (stacked PRs work) — signals the batching model needed follow-up investment rather than being a solved, static feature. [InfoQ: GitHub Targets Large Merge Problem with Stacked PRs](https://www.infoq.com/news/2026/04/github-stacked-prs/)
- One third-party operational report (Solomon Neas, independent blog, not GitHub-official) describes April 2026 GitHub availability incidents including "merge queue corruption" under heavy agentic-commit load — third-party claim, not corroborated by an official GitHub status postmortem in this pass; treat as anecdotal evidence that merge queue has real failure modes under high automated-agent write volume, which is directly relevant to Orvex's agent-driven staging use case. [Solomon Neas: GitHub Availability in April 2026](https://solomonneas.dev/blog/github-availability-agentic-load-report)
- Independent research cited by third parties (not GitHub) found PRs of 200–400 lines had 40% fewer defects and were approved 3x faster than larger PRs, and that stacks beyond 3–4 PRs incur more coordination overhead than benefit — marketing/community claim from the stacked-PR coverage, not a GitHub-published statistic; useful directionally for "keep individual proposed changes small." [InfoQ: GitHub Targets Large Merge Problem with Stacked PRs](https://www.infoq.com/news/2026/04/github-stacked-prs/)

## Pricing & positioning

- **Free**: unlimited public/private repos, unlimited collaborators on public repos, 2,000 Actions minutes/month (private repos). PR/suggested-changes/CODEOWNERS are available; merge queue requires org ownership and (for private repos) GitHub Enterprise Cloud.
- **Team**: $4/user/month — unlocks protected branches and required reviewers, 3,000 Actions minutes/month.
- **Enterprise**: from $21/user/month — SSO, SCIM, audit logs, IP allow lists, Advanced Security bundled, 50,000 Actions minutes/month; this is the tier required for merge queue on private repos.
- **Add-ons**: GitHub Advanced Security split into Secret Protection ($19/active committer/month) and Code Security ($30/active committer/month); Copilot ranges free tier up to $39/user/month (Pro+/Enterprise), and Copilot code review now separately consumes AI credits + Actions minutes as of June 2026.
- Positioning: PR + merge queue + CODEOWNERS is bundled into the core product (not sold as a discrete "staging area" product) — GitHub monetizes via seat tier + Advanced Security + Actions minutes + Copilot credits, not by pricing the review/merge pipeline itself. [Third-party pricing aggregation: eesel.ai GitHub pricing 2026](https://www.eesel.ai/blog/github-pricing) — marketing/third-party source, treat exact figures as approximate; not GitHub's own pricing page content, cross-check at [github.com/pricing](https://github.com/pricing) before quoting numbers externally.

## STEAL — concrete ideas for Orvex

1. **Speculative-merge validation (merge queue's core trick)**: before landing a librarian-approved staged change, materialize it against the *current* live wiki state (not the state at proposal time) and re-validate before applying — directly solves "100 agents propose against a wiki that's moving underneath them."
2. **Suggested-changes as the beautify seam**: give the librarian agent (or human) an inline "propose an edit to the agent's proposed edit" primitive that the original diff can absorb with one action, rather than forcing reject-and-resubmit — this is exactly Orvex's "beautify" step, already validated at huge scale by GitHub reviewers.
3. **CODEOWNERS-style path/topic routing**: map staged content (by target document, space, or tag) to a routing table so the librarian's prompt/policy — or the right human escalation — is selected deterministically by *what* is being changed, not left to a single global reviewer.
4. **Batch-apply as one commit**: GitHub's "apply suggested changes as a batch → single commit" pattern is the right shape for landing many approved staged edits atomically — Orvex should let the librarian apply N approved proposals as one wiki revision rather than N sequential writes.
5. **Comment-only AI reviewer gate**: deliberately keep the AI (librarian) from being able to self-satisfy the approval gate for high-stakes content — mirror Copilot's "Comment never Approve" design so a human/policy checkpoint stays load-bearing for anything above a configurable risk threshold.

## AVOID / where Orvex differs

- **No document/section/field granularity**: GitHub's unit is a file/line diff; Orvex's PRD explicitly needs document/section/field-level proposals against a rich-text/structured wiki (Docmost), which PRs don't model natively — don't try to force wiki content through git-diff semantics.
- **No native 100-proposals-in-one-session batch review UX**: GitHub has nothing that lets a reviewer triage 100 independent content proposals in one pass; merge queue batches for CI re-validation, not human review throughput. Orvex needs to design this UX itself, not borrow it.
- **No auto-approve rules as a first-class primitive**: GitHub leaves auto-merge/auto-approve to bots bolted on via Actions; if Orvex wants configurable auto-approve tiers (e.g., low-risk edits auto-merge), that has to be purpose-built, not adapted from a GitHub feature.
- **Reported reliability risk under automated/agentic load**: the one (third-party, uncorroborated) report of merge queue corruption under agentic commit volume is a warning to load-test Orvex's own speculative-validation path under many-agent concurrent write scenarios rather than assume the GitHub pattern is bulletproof at agent-driven volumes.
- **Pricing model doesn't transfer**: GitHub monetizes seats + CI minutes + security add-ons, not the review/merge pipeline itself — not directly informative for how Orvex should price the staging/librarian feature.
