# Prior art deep-dive: Wikipedia Pending Changes / Articles for Creation / FlaggedRevs

## What it is
Three related but distinct MediaWiki mechanisms for buffering untrusted edits against a stable public version:
- **FlaggedRevs** — the underlying extension. Tracks two pointers per page: the current (draft) revision and the latest "stable" revision meeting a wiki-configured review bar. Unregistered/anonymous readers see the stable version by default; logged-in users can see the pending draft. ([Help:Extension:FlaggedRevs](https://www.mediawiki.org/wiki/Help:Extension:FlaggedRevs))
- **Pending Changes** — English Wikipedia's lightweight deployment of FlaggedRevs, applied *selectively* to specific pages (via admin-set page protection) rather than site-wide. Edits by non-autoconfirmed/anonymous users on a protected page are saved but hidden from the public until a "reviewer" accepts them. ([Wikipedia:Pending changes](https://en.wikipedia.org/wiki/Wikipedia:Pending_changes), [Wikipedia:Reviewing pending changes](https://en.wikipedia.org/wiki/Wikipedia:Reviewing_pending_changes))
- **Articles for Creation (AfC)** — a separate, older, human-process (not code-enforced) workflow: unregistered/new editors can't create mainspace articles directly, so they write in the `Draft:` namespace and submit for review; an AfC reviewer accepts (moves to mainspace) or declines with feedback. ([Wikipedia:WikiProject Articles for creation](https://en.wikipedia.org/wiki/Wikipedia:WikiProject_Articles_for_creation), [Draft namespace](https://www.mediawiki.org/wiki/Draft_namespace))

## Behind it & traction
- Built and operated by the Wikimedia Foundation / MediaWiki community as open-source (GPL), not a commercial vendor — no company, no funding rounds, no pricing.
- FlaggedRevs adoption: **25+ Wikipedia language editions** (German, Russian, Polish, Arabic, Hungarian, etc.) plus assorted Wiktionaries, Wikibooks, Wikinews, Wikiquote/Wikisource. German Wikipedia runs it on ~99.99% of articles ("Sighted/Quality" model); English Wikipedia applies Pending Changes to under 0.05% of articles (targeted protection only). ([Flagged Revisions – Meta-Wiki](https://meta.wikimedia.org/wiki/Flagged_Revisions))
- **Moratorium on new deployments since April 2017** — the extension is considered legacy/hard-to-maintain, and WMF has not enabled it on new wikis since 2013–2014. This is a strong maintenance-burden signal from a decade-plus-old prior-art system. ([Flagged Revisions – Meta-Wiki](https://meta.wikimedia.org/wiki/Flagged_Revisions))
- A 2010–2011 trial to adopt full Pending Changes on English Wikipedia required "at least one contractor" and 8 WMF staff over 7 months — evidence this class of review-gate system is organizationally expensive to roll out even inside a mature engineering org. ([Flagged Revisions – Meta-Wiki](https://meta.wikimedia.org/wiki/Flagged_Revisions))

## Architecture & data model
- Two-revision-pointer model per page: **current/draft revision** (most recent edit, reviewed or not) and **stable revision** (latest revision that satisfies the configured review flags). Public/anonymous readers are served the stable revision; the draft exists in the same revision history stream, not a separate store. ([Help:Extension:FlaggedRevs](https://www.mediawiki.org/wiki/Help:Extension:FlaggedRevs))
- Flags are **hierarchical/configurable per wiki** — e.g., "sighted" (vandalism-checked) vs "quality" (content-reviewed) tiers on German Wikipedia; wikis define which flag level counts as "stable."
- **Autopromotion** (trust-tiering) is config-driven: account age, edit count, edits over time, namespace edit history, block/revert history, edit-summary usage — users who cross the threshold get their edits auto-flagged as reviewed without human review. This is the closest analog to an auto-approve/trust-tier rule engine.
- AfC/Draft namespace is architecturally simpler: it's just a MediaWiki namespace (`Draft:`) plus categories/templates for status tracking (unsubmitted, awaiting review, declined) — no separate flagging table, the "queue" is a maintained category + Special:NewPagesFeed listing.
- Precise DB schema (table names) is not documented in current public docs; the extension's own help page carries a stale/out-of-date-content warning as of the version fetched (2026-07). ([Help:Extension:FlaggedRevs](https://www.mediawiki.org/wiki/Help:Extension:FlaggedRevs))

## API & integration surface (MCP? SDK? CLI? chat-platform plugins?)
- No MCP, no vendor SDK (it predates that ecosystem entirely; MediaWiki is 2000s-era software).
- Standard **MediaWiki Action API** exists for reading/writing pages generally, but no dedicated public API surface for FlaggedRevs review actions was found in current docs (fetch of Help:Extension:FlaggedRevs did not surface hook/API details).
- **Page Curation / PageTriage extension** provides the actual reviewer tooling: a filterable list view (new pages feed) + an in-page curation toolbar for tagging, flagging deletion, marking reviewed/unreviewed. It integrates with **ORES** (WMF's ML scoring service) for damaging/goodfaith predictions surfaced as suggestions, and with **CopyPatrol** for copyright-violation detection. ([Page Curation](https://www.mediawiki.org/wiki/Page_Curation))
- **ORES** (Objective Revision Evaluation Service) is a standalone ML-as-a-service API scoring edits 0–1 for "damaging" and "goodfaith" likelihood; third-party tools (Huggle, Snuggle, Wiki Ed dashboards) consume it via API to prioritize/automate review. This is the closest thing to an "AI reviewer" in this ecosystem — but it produces a *risk score for human triage*, not an autonomous accept/reject decision. ([ORES – MediaWiki](https://www.mediawiki.org/wiki/ORES), [Diff: AI x-ray specs for Wikipedians](https://diff.wikimedia.org/2015/11/30/artificial-intelligence-x-ray-specs/))
- **AFCH (Articles for Creation Helper)** is a community-built browser userscript (not an official product), not an API/CLI/SDK — it's UI automation on top of the existing wiki edit API to speed up single-reviewer accept/decline actions with templating and notifications. ([Reviewing instructions](https://en.wikipedia.org/wiki/Wikipedia:WikiProject_Articles_for_creation/Reviewing_instructions))
- No chat-platform plugins found (no Slack/Teams/ChatGPT integration for review).

## Staging & review workflow
- **Propose → review → apply/reject**: Pending Changes — anonymous/new-user edit is saved as a pending revision; reviewer diffs it against the last stable revision; accept publishes it as stable, revert/undo discards it. AfC — draft is authored in Draft: namespace, explicitly submitted, reviewer accepts (moves to mainspace) or declines (leaves in draft with feedback) or rejects (rare, deletion-track). ([Reviewing pending changes](https://en.wikipedia.org/wiki/Wikipedia:Reviewing_pending_changes), [AfC reviewing instructions](https://en.wikipedia.org/wiki/Wikipedia:WikiProject_Articles_for_creation/Reviewing_instructions))
- **Change granularity**: Pending Changes reviews are **diff-level** (edit-by-edit against a page), not field/section level — MediaWiki has no sub-page granularity primitive. AfC review is **whole-document** (accept/decline the entire draft article).
- **Batch review at scale (~100 changes/session analog)**: Real-world evidence is *backlog-driven*, not single-session. AfC backlog stood at **3,268–5,000+ pending drafts** in 2025, driving month-long organized "Backlog Elimination Drives" (June 2025, December 2025) with a stated goal of cutting review latency from "3+ months" to under 1 month. There is no evidence of a UX for one reviewer processing 100 changes in one sitting — the model assumes many reviewers picking off a shared queue over weeks. ([June 2025 Backlog Drive](https://en.wikipedia.org/wiki/Wikipedia:WikiProject_Articles_for_creation/June_2025_Backlog_Drive), [December 2025 Backlog Drive](https://en.wikipedia.org/wiki/Wikipedia:WikiProject_Articles_for_creation/December_2025_Backlog_Drive))
- **Conflict handling vs. live/stacked edits**: When multiple pending edits stack on top of each other, the documented procedure is: pull up full page history, review each contributor's edits individually, manually undo the bad ones (each undo becomes its own new revision), and once satisfied, **accepting the single most-recent pending revision implicitly accepts all earlier pending revisions beneath it** (no per-edit "accepted by" record for the intermediate ones). This is a pragmatic collapse-the-stack approach, not true 3-way conflict resolution. ([Reviewing pending changes](https://en.wikipedia.org/wiki/Wikipedia:Reviewing_pending_changes))
- **Reviewer UX + automation**: Page Curation toolbar (list + in-context flyout panels) is the UX; **ORES scores are advisory/sorting signals only** — flagged as likely-damaging edits get surfaced higher/faster to human reviewers, but the design explicitly keeps "algorithmic predictions serving as suggestions only," i.e. no autonomous AI accept/reject in production. **Autopromotion** (trust-tier thresholds: account age, edit count, revert history, etc.) is the actual auto-approve mechanism — trusted users' edits bypass the queue entirely and post directly as stable. ([Page Curation](https://www.mediawiki.org/wiki/Page_Curation), [FlaggedRevs autopromotion](https://www.mediawiki.org/wiki/Help:Extension:FlaggedRevs))

## Scale & operational evidence
- English Wikipedia Pending Changes protection is applied to **<0.05% of articles** — i.e., a narrow, targeted anti-vandalism tool, not a blanket review gate. German Wikipedia's FlaggedRevs (broader "Sighted" model) covers ~99.99% of articles, showing the two very different scale postures the same underlying extension supports. ([Flagged Revisions – Meta-Wiki](https://meta.wikimedia.org/wiki/Flagged_Revisions))
- Some FlaggedRevs-enabled wikis show **unreviewed-edit backlog ages exceeding 1000 days on average**, and review delays of "several days" even for registered/unregistered edits on heavily flagged wikis — hard evidence that a pure human-review-queue model degrades badly without either enough reviewer capacity or aggressive autopromotion/automation. ([Flagged Revisions – Meta-Wiki](https://meta.wikimedia.org/wiki/Flagged_Revisions))
- AfC-specific backlog: **3,268 pending submissions** (per Category:Pending AfC submissions, 2025 snapshot) with prior "3+ months" wait times before organized drives; **5,000+** cited in another 2025 source — the numbers are noisy/inconsistent across pages but consistently in the thousands. ([Wikipedia:WikiProject Articles for creation](https://en.wikipedia.org/wiki/Wikipedia:WikiProject_Articles_for_creation))
- No official machine-readable throughput/latency dashboard was found; Wikipedia:Pending changes/Metrics is explicitly a *metrics-collection plan*, not a report of actual figures — it documents intended stats (Special:ValidationStatistics) rather than publishing them inline, and states "we need a better understanding of the ratio of reviewers to articles," implying WMF itself never fully closed the loop on measuring this system's health. ([Wikipedia:Pending changes/Metrics](https://en.wikipedia.org/wiki/Wikipedia:Pending_changes/Metrics))

## STEAL — 3-5 concrete ideas for Orvex
1. **Trust-tiered autopromotion as the throughput valve.** FlaggedRevs' autopromotion (account age, edit count, revert/block history → auto-bypass review) is directly portable: agents/customers with a track record of clean staged proposals should earn a "trusted producer" tier where the librarian auto-merges below some risk threshold, reserving human/heavy-LLM review for new or previously-reverted agents. This is the single biggest lever for surviving 100-document sessions without a reviewer bottleneck.
2. **Stable-vs-draft dual-pointer, not a separate staging DB.** Consider keeping proposed changes as revisions *of the same document* (draft pointer) rather than a fully separate staging store — simplifies diffing, history, and rollback, and avoids a merge-conflict class of bugs. (Tension with the PRD's explicit "staging store OUTSIDE the wiki" requirement — see AVOID below.)
3. **Collapse-the-stack accept semantics.** When an agent proposes multiple sequential edits to the same doc before review happens, let the librarian accept the *latest* proposal and treat it as implicitly superseding/absorbing earlier pending ones on that doc, rather than requiring N separate approve clicks — directly addresses the "100 documents in one session" requirement.
4. **Advisory ML risk scoring (ORES pattern), never autonomous accept.** Score each staged proposal for "risk" (e.g., deletes large sections, touches high-traffic pages, low-trust submitter) and use that purely to *sort/prioritize* the librarian's queue and flag proposals for closer read — keep the actual accept/reject decision as a distinct, auditable step (matches Orvex's "no fallbacks/silent rot" ethos: no silent auto-merge of risky content).
5. **Backlog-drive pattern as an operational fallback.** Even a well-designed queue can silently degrade to 1000-day backlog ages (their own hard data point) — build in an explicit alerting/dashboard for staging-queue age/depth from day one (not an afterthought "metrics plan" as WMF's own page admits happened to them), so Orvex customers get a visible SLA on staged-content latency rather than discovering silent rot months later.

## AVOID / where Orvex differs
- **No separate outside-the-wiki staging store in Wikipedia's model** — their "draft" is just another revision/namespace inside the same wiki. Orvex's PRD explicitly requires staging OUTSIDE the customer wiki; Wikipedia's architecture doesn't validate or invalidate that choice since it never faced a multi-tenant customer-owned-wiki constraint — don't over-index on their in-wiki-draft pattern as proof it's sufficient for Orvex's isolation requirements.
- **Diff/whole-page granularity only — no field/section-level review primitive.** MediaWiki review workflows operate at revision-diff or whole-document granularity; Orvex's PRD calls for add-document/add-section/edit/replace/delete as first-class proposal types, which is finer-grained than anything FlaggedRevs or AfC natively models. Don't assume their diff UX transfers directly to structured section-level proposals.
- **Reviewer UX is built for one human at a time, at low-to-moderate scale, with weeks-long backlogs tolerated as normal.** Their own backlog-drive cadence (organizing month-long volunteer pushes to clear thousands of items) is not a model for surviving "single chat session, 100 documents" — Orvex needs synchronous/near-real-time batch review UX that Wikipedia's system was never designed to provide.
- **No AI-authored content problem.** Wikipedia's trust model is calibrated for human vandals/newbies; it has no concept of *AI-agent-authored* proposals at scale, and ORES scoring targets vandalism/damage patterns learned from human edit behavior — likely a poor fit, untuned, for scoring LLM-generated content risk (hallucination, scope creep, tone) without retraining.
- **Deployment/maintenance cost is a red flag, not a template.** The extension has been under a new-wiki deployment moratorium since 2017 due to legacy code complexity and a 7-month/8-person WMF trial cost in 2010–11 — a strong signal that a from-scratch FlaggedRevs-style system is expensive to build and maintain; Orvex should treat this as a caution on architectural complexity, not inspiration to replicate the same code-level approach.
