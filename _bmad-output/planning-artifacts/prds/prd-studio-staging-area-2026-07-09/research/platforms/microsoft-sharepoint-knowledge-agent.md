# Prior art deep-dive: Microsoft 365 Copilot / SharePoint Knowledge Agent

Researched 2026-07-10. Focus: the STAGING track for Orvex Studio's agent-staging-area PRD.

## What it is

"Knowledge Agent" is Microsoft's branding for AI-in-SharePoint: a built-in agent (plus the ability to build custom `.agent` files) that grounds Microsoft 365 Copilot and declarative agents on a tenant's SharePoint sites, pages, and document libraries, respecting each user's existing SharePoint permissions ([Manage access to agents in SharePoint](https://learn.microsoft.com/en-us/sharepoint/manage-access-agents-in-sharepoint)). It is not a separate product — it's the grounding/retrieval layer of Microsoft 365 Copilot layered on top of SharePoint's existing search index, plus a growing set of SharePoint Advanced Management (SPAM) governance controls that let admins scope what that grounding layer is allowed to see, independent of who has read access ([Restricted SharePoint Search](https://learn.microsoft.com/en-us/sharepoint/restricted-sharepoint-search), [Restricted Content Discovery](https://learn.microsoft.com/en-us/sharepoint/restricted-content-discovery)). Metadata columns (managed properties/taxonomy tags) are also increasingly used to shape *what* Copilot retrieves and *how* it reasons over results — filtering, grouping, sorting by metadata, with a March-2026-era roadmap item extending this to attached libraries/folders ([Add SharePoint as a knowledge source](https://learn.microsoft.com/en-us/microsoft-copilot-studio/knowledge-add-sharepoint), search summary from [SharePoint Showcase blog title](https://techcommunity.microsoft.com/blog/spblog/sharepoint-showcase-how-metadata-and-the-knowledge-agent-elevate-microsoft-365-c/4464079) — full body unreachable via fetch, treat as marketing-title-only evidence).

## Behind it & traction

Microsoft/SharePoint engineering team, shipped as part of the Microsoft 365 Copilot suite; SharePoint Advanced Management is a licensed add-on tier for governance features. unknown — no independently-sourced adoption/usage numbers found (Microsoft doesn't publish SharePoint-agent-specific MAU or GA-customer-count figures in the pages fetched). Feature velocity is high and dated: Restricted SharePoint Search is already being retired (new enablement blocked from 2026-07-31) in favor of Restricted Content Discovery ([Restricted SharePoint Search](https://learn.microsoft.com/en-us/sharepoint/restricted-sharepoint-search)), and the "Site Assets > Copilots" agent-storage convention changed as recently as June 2026 per search-summarized third-party docs — evidence this is an actively-iterating, not-yet-settled surface, not a mature stable API.

## Architecture & data model

- **No separate staging store.** Custom agents are represented as `.agent` files stored directly in a SharePoint document library (`Site Assets > Copilots`), governed by ordinary SharePoint file permissions — whoever can create/access files on the site can create/access agents ([Manage access to agents in SharePoint](https://learn.microsoft.com/en-us/sharepoint/manage-access-agents-in-sharepoint)).
- **Grounding is retrieval-time, not write-time.** Content safety/discoverability is enforced as metadata flags on the *site* (or, per marketing copy, potentially file-level) that the search/indexing pipeline checks at query time — this is layered on top of, and independent from, the underlying ACL/permission model. A user can have read access to a file yet Copilot will still refuse to ground on it if the site carries a Restricted Content Discovery (or the retiring Restricted SharePoint Search) flag ([Restricted Content Discovery](https://learn.microsoft.com/en-us/sharepoint/restricted-content-discovery)).
- **Propagation is asynchronous and can be slow at scale**: flag changes propagate through the search index and are not instant — sites with >500,000 items can take "more than a week" to fully reflect a discoverability change ([Restricted Content Discovery](https://learn.microsoft.com/en-us/sharepoint/restricted-content-discovery)).
- **DLP/sensitivity labels are a second, orthogonal metadata layer**: Purview DLP policies keyed on sensitivity labels can exclude specific files from grounding even though the file still shows up as a citation (the citation appears, but content isn't used) — notably the `.agent` file itself can't yet carry a sensitivity label, so DLP targeting it must match on file extension instead ([Manage access to agents in SharePoint](https://learn.microsoft.com/en-us/sharepoint/manage-access-agents-in-sharepoint)).

## API & integration surface (MCP? SDK? CLI? chat-platform plugins?)

- PowerShell cmdlets for admin control: `Set-SPOSite -RestrictContentOrgWideSearch`, `Set-SPOTenant -DelegateRestrictedContentDiscoverabilityManagement`, `Start-SPORestrictedContentDiscoverabilityReport` ([Restricted Content Discovery](https://learn.microsoft.com/en-us/sharepoint/restricted-content-discovery)).
- Admin-center UI toggles (SharePoint admin center > Active sites > Settings) mirror the PowerShell surface.
- Agent authoring/consumption surfaces through Microsoft 365 Copilot Studio (declarative agents, knowledge-source attachment) and native SharePoint agent-builder UI; agents surface in Copilot Chat, Teams, and the SharePoint Agent Store ([Add knowledge sources to your declarative agent](https://learn.microsoft.com/en-us/microsoft-365/copilot/extensibility/agent-builder-add-knowledge), [Share and manage agents built with Microsoft 365 Copilot](https://learn.microsoft.com/en-us/microsoft-365/copilot/extensibility/agent-builder-share-manage-agents)).
- No MCP surface found in the fetched docs. unknown whether Microsoft exposes an MCP server for this specifically (Copilot Studio does have separate MCP-related features, but that's out of scope of what was verified here).
- Billing/metering: agents beyond the included Copilot license consume Azure pay-as-you-go credits, requiring an Azure subscription tied to the tenant ([search summary, Microsoft 365 Copilot pricing](https://www.microsoft.com/en-us/microsoft-365-copilot/pricing)) — unknown exact per-request cost structure, not independently verified beyond search-summarized third-party guides.

## Staging & review workflow

- **No propose→review→apply pipeline for *content*.** There is no analog of an agent drafting a wiki edit that a human approves before it lands — SharePoint agents only *read* (ground on) existing content; Microsoft's flow governs *discoverability of already-published* content, not intake of AI-proposed changes. This is architecturally the opposite of what Orvex needs for the staging area.
- **The one genuine "propose → approve" gate in this ecosystem is for *agents themselves*, not content.** Site owners can approve a custom Copilot agent (an `.agent` file) into a site's `Approved` subfolder, at which point it surfaces in the agent picker for all site visitors and in the org-wide Agent Store; until approved it's only usable by its creator/those with direct file access ([Manage agents in SharePoint](https://support.microsoft.com/en-us/sharepoint/copilot-in-sharepoint/manage-agents-in-sharepoint), search-summarized). If org-wide sharing is restricted, a tenant admin must approve+deploy before broader availability ([Get started with agents in SharePoint](https://support.microsoft.com/en-us/sharepoint/copilot-in-sharepoint/get-started-with-agents-in-sharepoint), search-summarized).
- **Change granularity**: none for content (read-only grounding); for agent governance, granularity is per-agent-file and per-site (coarse — a whole site's discoverability, not per-document or per-section).
- **Batch review at scale (100 changes in one session)**: no equivalent found. The closest "batch" mechanism is *bulk application of a discoverability flag across many sites* via PowerShell/admin center, explicitly designed for large governance sweeps, with documented throttling/guardrails and index-propagation delay as the primary scale constraint ([Restricted Content Discovery](https://learn.microsoft.com/en-us/sharepoint/restricted-content-discovery)). This is a bulk *lockdown* mechanism, not a bulk *content-change review* mechanism — it doesn't map to "librarian reviews 100 staged document edits."
- **Conflict handling vs live content**: unknown/not applicable — since there's no write-staging concept, there's no conflict-with-live-edits problem to solve. SharePoint's ordinary co-authoring conflict resolution (last-write-wins with version history) is the only relevant analog, and it predates and is unrelated to Copilot.
- **Reviewer UX + automation (auto-approve rules, AI reviewer)**: no AI-assisted or auto-approve reviewer found for content. The only "automation" is policy-based, not case-by-case: DLP rules and sensitivity-label-driven exclusion apply uniformly at query time rather than being reviewed per change. Site-owner agent approval is manual, per-agent, with no bulk/AI-assisted approval path documented. Note: site admins delegated to manage Restricted Content Discovery must supply a written justification for each change, which is a lightweight accountability/audit mechanism worth stealing even though it's for a lockdown flag, not a content edit ([Restricted Content Discovery](https://learn.microsoft.com/en-us/sharepoint/restricted-content-discovery)).

## Scale & operational evidence

- Bulk discoverability changes are guardrailed and throttled; sites over 500,000 items can take over a week to fully propagate a single flag change through the search index ([Restricted Content Discovery](https://learn.microsoft.com/en-us/sharepoint/restricted-content-discovery)) — hard evidence that Microsoft's own index-propagation model does not treat "flip a safety flag" as instantaneous or cheap at scale, a useful cautionary data point for any Orvex design that assumes fast re-indexing after a librarian merge.
- Restricted SharePoint Search — an earlier version of the same idea — is already being deprecated in favor of Restricted Content Discovery, with new enablement blocked from 2026-07-31 ([Restricted SharePoint Search](https://learn.microsoft.com/en-us/sharepoint/restricted-sharepoint-search)), indicating this whole area is still being iterated/replaced rather than a settled, load-bearing pattern to copy verbatim.
- unknown — no publicly documented figures for concurrent agent volume, number of `.agent` files per tenant, or review-queue throughput at enterprise scale.

## Pricing & positioning

- Microsoft 365 Copilot: Business tier $18/user/month rising to $21/user/month in July 2026 for orgs up to 300 users; Enterprise tier $30/user/month — always additive to a base Microsoft 365 license ([search-summarized, Microsoft 365 Copilot pricing](https://www.microsoft.com/en-us/microsoft-365-copilot/pricing)); treat exact figures as marketing/reseller-summarized, not independently verified against the primary pricing page content.
- Custom/declarative agents beyond what's bundled require an Azure subscription and are billed pay-as-you-go on Azure; publishing agents to external (non-employee) channels requires standalone Copilot Studio capacity at ~$200 per 25,000 "Copilot Credits"/month (third-party guide figure, unverified against Microsoft's own pricing page) ([search-summarized pricing guides](https://www.microsoft.com/en-us/microsoft-365-copilot/pricing)).
- Positioning: SharePoint Advanced Management (which carries the discoverability/content-safety controls) is licensed as a governance add-on aimed at "Copilot readiness" — i.e., Microsoft sells the safety/scoping layer as a *pre-requisite governance product*, not a feature of Copilot itself. This is a meaningful go-to-market signal: enterprises are expected to buy and configure a separate governance layer before they can trust Copilot on their content.

## STEAL — 3-5 concrete ideas for Orvex

1. **Decouple "AI-groundable" from "human-readable" as a first-class flag, independent of ACLs.** SharePoint's core insight — a document can be visible to a human but explicitly withheld from AI grounding via a metadata flag — maps directly onto the librarian's job: a customer should be able to mark a wiki page/section "human-only, do not let agents cite or stage-edit this" without touching sharing permissions. Orvex's staging area should carry an equivalent per-page/per-space flag, enforced at the point agents read *or* propose changes, not just at write-approval time.
2. **Require a written justification for governance-affecting changes, even automated/delegated ones.** The delegated Restricted-Content-Discovery justification prompt is a cheap, high-value audit primitive — apply the same pattern to librarian auto-merge/auto-approve rules: any rule that lets staged content bypass human review should force the rule's author (or the triggering event) to record a one-line justification, captured in the audit trail.
3. **Treat propagation/indexing lag as a designed-for constraint, not an edge case.** Microsoft's own numbers (a week+ for large sites) show that "safety flag changed" and "system fully reflects it everywhere" are not the same instant. Orvex's librarian-merge pipeline should explicitly model and surface this gap (e.g., "merged, re-indexing, agents may still see stale content for N minutes") rather than assuming atomicity, especially at the 100-documents-in-one-session batch scale the PRD targets.
4. **Site-owner-as-approval-gate is a good coarse-grained pattern for tenant-level defaults, but Orvex must go finer.** Approving an *agent* per site is roughly analogous to a customer approving a *librarian prompt/config* per space — reuse that as the top-level trust boundary — but Orvex additionally needs the per-change (document/section-level) review loop that Microsoft has no equivalent for, since Orvex agents are expected to *propose content*, not just *read* it.
5. **Sell/position the safety-scoping layer as a distinct, visible governance surface**, the way Microsoft bundles it into SharePoint Advanced Management rather than hiding it inside Copilot. For enterprise trust, Orvex's librarian review/staging safety controls (content-safety flags, auto-approve rule justification, audit log) should be a named, demoable capability in the PRD/pricing story, not an implementation detail buried in the librarian agent.

## AVOID / where Orvex differs

- **Microsoft has no propose→review→apply content pipeline at all** — Copilot/Knowledge Agent is strictly read/ground, never write. Orvex's entire staging-area premise (agents propose edits that a librarian reviews and merges) has no direct precedent here; don't expect to port a review-queue UX from this platform — it doesn't exist. Build it from scratch or borrow from Orvex's other prior-art comparisons instead.
- **Microsoft's "batch" mechanism is a bulk lockdown, not a bulk content-review.** Applying a discoverability flag to 1,000 sites is a fundamentally different operation from a librarian reviewing 100 distinct proposed document edits in one session with individual accept/reject/merge decisions — don't conflate SharePoint's bulk-governance-ops scale story with the PRD's "single session proposing updates to 100 documents" requirement; they solve different problems.
- **Granularity is coarse (site-level) for safety controls, and file-level (whole `.agent` file / whole document) elsewhere** — there's no section/field-level metadata-flag precedent to steal; Orvex's staging area explicitly needs section/field granularity, which this platform doesn't attempt.
- **No AI reviewer / auto-approve-with-rationale pattern for content** — everything here is deterministic policy (ACL, DLP label, discoverability flag) rather than an AI agent judging AI-proposed content, which is core to Orvex's librarian concept. Don't assume Microsoft has solved (or even attempted) the "AI reviews AI's staged work" problem.
- **Propagation delay at scale (week+ for huge sites) is a real operational cost Microsoft has visibly not fully solved** — Orvex should avoid inheriting a fully-synchronous-index assumption and should design the librarian merge step to be explicit about eventual consistency rather than copying an architecture that already shows strain at scale.
