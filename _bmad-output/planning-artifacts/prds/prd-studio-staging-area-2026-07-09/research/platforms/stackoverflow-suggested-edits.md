# Prior Art Deep-Dive: Stack Overflow Suggested Edits + Review Queues

## What it is
Stack Overflow's **Suggested Edits** feature lets users below the direct-edit reputation threshold (and anonymous/logged-out users) propose a change to a question or answer instead of writing it live. The proposal enters a **review queue** where users who hold the "Edit Suggestions" review privilege (2,000+ reputation) vote to approve or reject it; the edit is auto-applied once it clears, and never touches the live post until then. [Suggested Edits and Edit Review (Stack Overflow blog, 2011)](https://stackoverflow.blog/2011/02/05/suggested-edits-and-edit-review/)

In 2025 the same moderation-queue DNA was industrialized into the enterprise product: **Stack Internal** (formerly Stack Overflow for Teams / Stack Overflow for Teams Enterprise), whose "Knowledge Ingestion" pipeline pulls raw content from Confluence/Microsoft Teams, has AI structure and score it into atomic Q&A, and routes it through a "scalable human-in-the-loop" review step before publishing. [Introducing Stack Internal (Stack Overflow blog, 2025-11-18)](https://stackoverflow.blog/2025/11/18/introducing-stack-internal-powering-the-human-intelligence-layer-of-enterprise-ai/)

## Behind it & traction
- Public Stack Overflow: owned by Prosus (via Stack Overflow, part of Prosus's portfolio); the community site's own question volume has collapsed since ChatGPT's Nov-2022 launch — monthly questions fell from ~108,563 (Nov 2022) to ~25,566 (Dec 2024), a 76.5% drop, and posts in April 2025 were down 64% YoY per Stack Overflow's own Data Explorer. [Stack Overflow's decline (Eric Holscher, Jan 2026)](https://www.ericholscher.com/blog/2025/jan/21/stack-overflows-decline/), [Are LLMs making StackOverflow irrelevant? (Pragmatic Engineer)](https://blog.pragmaticengineer.com/stack-overflow-is-almost-dead/)
- The company's strategic pivot is explicit: reposition the same trust/review machinery as an enterprise "human intelligence layer" that grounds AI coding agents (GitHub Copilot, ChatGPT, Cursor) instead of competing for human question-askers. [Introducing Stack Internal](https://stackoverflow.blog/2025/11/18/introducing-stack-internal-powering-the-human-intelligence-layer-of-enterprise-ai/)
- Stack Overflow for Teams itself rebranded to "Stack Internal" in Nov 2025; users reported confusion and link-rot during the transition. [Stack Internal: What Happened to Stack Overflow Teams? (Dev Solutions)](https://devsolus.com/stack-internal-what-happened-to-stack-overflow-teams/), [A new look for what's next (Stack Overflow blog, May 2025)](https://stackoverflow.blog/2025/05/08/a-new-look-for-whats-next/)
- No customer counts, revenue, or quantified adoption numbers for Stack Internal were disclosed in the launch materials found. **unknown** beyond marketing language ("measurable productivity gains," "faster onboarding").

## Architecture & data model
- Public SO: a post (question/answer) has a canonical body; a **suggested edit** is a diff object (title/body/tags delta) linked to the target post, sitting outside the live content until approved — structurally the same "proposal object separate from canonical content" pattern Orvex needs for the staging area. [Suggested Edits and Edit Review](https://stackoverflow.blog/2011/02/05/suggested-edits-and-edit-review/)
- Full post history (including every edit, suggested or direct) is preserved in the `PostHistory` table of Stack Overflow's public data dump, giving a durable audit trail per revision. [Stack Overflow Data Dumps (Brent Ozar)](https://www.brentozar.com/archive/2018/06/new-stack-overflow-public-database-available-2018-06/)
- Stack Internal's ingestion pipeline restructures unstructured source content (Confluence pages, Teams threads) into **atomic questions and answers** before scoring — i.e., it normalizes to Q&A granularity rather than keeping arbitrary document diffs. [Introducing Stack Internal](https://stackoverflow.blog/2025/11/18/introducing-stack-internal-powering-the-human-intelligence-layer-of-enterprise-ai/)
- Exact queue data model (approval-count fields, reviewer-cap tables, starvation-avoidance bookkeeping) is **unknown** — not published; stackoverflow.com/meta.stackoverflow.com pages that likely document the precise N-approvals rule were unreachable (WebFetch blocked on both hosts) and not indexed by search in a form with the exact numbers.

## API & integration surface (MCP? SDK? CLI? chat-platform plugins?)
- Stack Internal ships a **first-party MCP server** that runs inside customer infrastructure and connects AI dev tools (GitHub Copilot, ChatGPT, Cursor) to the verified knowledge base, described explicitly as **bidirectional**: agents can query it for grounding *and* write new candidate knowledge back into it (e.g., a Copilot session that reaches a good solution can push it back via the MCP server). [Introducing Stack Internal](https://stackoverflow.blog/2025/11/18/introducing-stack-internal-powering-the-human-intelligence-layer-of-enterprise-ai/)
- A dedicated **Stack Internal MCP Server for Microsoft 365 Copilot** connects Copilot Studio to the org's Q&A, letting users search or create questions from within M365 apps. [MCP server / Copilot Studio integration](https://stackoverflow.blog/2025/11/18/introducing-stack-internal-powering-the-human-intelligence-layer-of-enterprise-ai/)
- Basic and Business tiers of Stack Overflow for Teams include a **read API**; Business adds broader "API access for connecting to other tools." [Stack Overflow for Teams pricing (eesel AI summary of stackoverflow.co/internal/pricing)](https://www.eesel.ai/blog/stack-overflow-for-teams-pricing)
- No dedicated CLI product surfaced. Public Stack Overflow itself exposes a classic read-only Stack Exchange REST API (well known, not central to this deep-dive). **unknown** whether Stack Internal has a general-purpose write SDK beyond the MCP server and Slack/Teams integrations.

## Staging & review workflow (Staging/review workflow (propose -> review -> apply/reject), change granularity (document/section/field), batch review at scale (100 changes in one session), conflict handling vs live content, reviewer UX + automation (auto-approve rules, AI reviewer))
- **Propose → review → apply/reject, public SO:** any user (including anonymous, after a post is 10 minutes old) can submit a suggested edit; changes must exceed 6 characters and include a rationale comment; it lands in a queue instead of touching the live post; a user with the review privilege (2,000+ rep) approves or rejects; on approval the diff is applied atomically to the canonical post and the submitter earns +2 reputation (capped at +1,000 lifetime per user). [Suggested Edits and Edit Review](https://stackoverflow.blog/2011/02/05/suggested-edits-and-edit-review/)
- **Change granularity:** field/segment-level — a suggested edit is a diff against title, body, and/or tags of a single post, shown to reviewers as a **diff view** (before/after), not a whole-document replace. [Suggested Edits and Edit Review](https://stackoverflow.blog/2011/02/05/suggested-edits-and-edit-review/)
- **Reviewer cap / queue backpressure:** "If the queue is currently full, we temporarily stop accepting edits" — a hard admission-control valve rather than unbounded queue growth, i.e. the system protects reviewer throughput by rejecting new proposals at the door once WIP is saturated. [Suggested Edits and Edit Review](https://stackoverflow.blog/2011/02/05/suggested-edits-and-edit-review/)
- **Abuse/quality control on submitters:** users whose suggested edits are consistently rejected can be temporarily banned (7 days cited) from suggesting further edits — a submitter-side reputation gate that protects reviewer time from repeat bad actors. [Edit a Question or Answer (Stack Overflow Internal Help Center, via search summary)](https://internal.stackoverflow.help/en/articles/8858605-edit-a-question-or-answer)
- **10,000+ reputation** unlocks a dedicated `/review` tab listing the entire pending queue (not just edits on posts you're viewing) — i.e., a power-reviewer surface for batch triage. [Suggested Edits and Edit Review](https://stackoverflow.blog/2011/02/05/suggested-edits-and-edit-review/)
- **Exact N-approvals-to-clear threshold and conflict-with-live-content handling:** **unknown with a precise number** — search and available fetches surfaced the general "vote to accept or reject" mechanic and confirmed a canonical meta post exists with "edge condition" rules, but the specific approval count (commonly cited elsewhere as 2 approvals to auto-apply / 3 rejections to auto-reject, but NOT independently verified here) could not be confirmed from a fetchable, citable 2025-2026 source — stackoverflow.com and meta.stackoverflow.com pages are not fetchable in this environment. Treat any specific number as unconfirmed.
- **Stack Internal (enterprise) workflow — batch/automation angle:** ingestion at scale is explicitly AI-assisted: content is auto-structured into atomic Q&A and **auto-scored for accuracy/relevance**, and only content clearing that AI score is queued for the "scalable human-in-the-loop" review before publish. This is the closest documented analog to an **AI-assisted auto-triage / auto-approve-adjacent** mechanism, though the article does not disclose an actual auto-approve threshold, a batch-review UI for reviewing many proposals in one sitting, or explicit conflict-resolution logic against already-published knowledge. [Introducing Stack Internal](https://stackoverflow.blog/2025/11/18/introducing-stack-internal-powering-the-human-intelligence-layer-of-enterprise-ai/)
- **100-changes-in-one-session batch review:** **unknown** — no evidence found (marketing or engineering) that either product has been evaluated at, or exposes UX explicitly built for, single-session review of ~100 pending changes.

## Scale & operational evidence
- Public Stack Overflow's own scale is undisputed historically (peak >200,000 questions/month in 2014) but is now in structural decline post-ChatGPT: 76.5% fewer monthly questions by Dec 2024 vs Nov 2022, and April 2025 posts down 64% YoY / >90% off the 2020 peak — meaning the suggested-edits queue mechanism was proven at very high historical volume but current live volume is much lower. [Stack Overflow's decline](https://www.ericholscher.com/blog/2025/jan/21/stack-overflows-decline/), [Pragmatic Engineer: Stack Overflow is almost dead](https://blog.pragmaticengineer.com/stack-overflow-is-almost-dead/), [ppc.land: Stack Overflow traffic collapses](https://ppc.land/stack-overflow-traffic-collapses-as-ai-tools-reshape-how-developers-code/)
- 84% of developers now report using AI coding tools, with 81.4% using OpenAI GPT models specifically — cited as the demand-side driver of the community site's decline and the rationale for Stack Overflow's enterprise pivot toward being an AI-grounding data layer rather than a human Q&A destination. [Pragmatic Engineer newsletter summary](https://newsletter.pragmaticengineer.com/p/are-llms-making-stackoverflow-irrelevant)
- No published operational metrics (queue depth, review latency, reviewer count, approval rate) for either the public suggested-edits queue or Stack Internal's ingestion review pipeline were found. **unknown**.

## STEAL - 3-5 concrete ideas for Orvex
1. **Proposal-object-separate-from-canonical, with a diff view.** SO never mutates the live post until approval; the suggested edit is its own row/object with a diff view for reviewers. This maps directly onto Orvex's staging store: agent proposals should be first-class objects (add/edit/replace/delete + rationale) that the librarian reviews as a diff against current wiki state, never in-place writes.
2. **Admission control on the queue, not just on individual items.** SO's "if the queue is full, stop accepting new edits" is a simple, brutally effective backpressure valve. For the 100-documents-in-one-session case, Orvex's staging area should cap in-flight proposals per customer/session and reject (with a clear retry signal) once the librarian's queue is saturated, rather than let 100 proposals silently pile up unreviewed.
3. **Submitter-side reputation gate to protect reviewer time.** SO temporarily bans users whose suggested edits are consistently rejected. Orvex could track a per-agent (or per-prompt-version) "proposal quality score" and throttle/auto-downgrade agents whose staged proposals get rejected by the librarian repeatedly — this is cheap, symmetric, and doesn't require a human in the loop to decide per-agent trust.
4. **AI-assisted pre-scoring ahead of human/AI-librarian review (Stack Internal's ingestion pattern).** Score every staged proposal for confidence/relevance before it reaches the librarian queue, and use that score to sort/triage (not necessarily auto-approve) — directly reusable for Orvex's librarian agent doing first-pass routing across a 100-item batch.
5. **Bidirectional MCP as the integration seam.** Stack Internal's MCP server both serves grounding context to agents and accepts write-back proposals from the same agents in the same protocol call surface. Orvex's staging area should expose the same MCP tool surface for "read wiki" and "propose wiki change" so agents don't need a separate write path/integration.

## AVOID / where Orvex differs
- **No published, verifiable N-approvals-to-clear number could be confirmed here** — do not cite "2 approvals" or similar as fact for SO; if a specific quorum number is needed for the PRD, source it directly from a fetchable meta.stackoverflow.com privileges page or treat it as a design choice Orvex sets independently.
- SO's model assumes a large pool of *volunteer human reviewers* with reputation-earned trust, built over 15+ years of community — Orvex's librarian is a single per-customer AI agent (with a tweakable prompt), not a crowd. The starvation-defense mechanics (reviewer caps, distributing load across many trusted humans) don't transfer directly; Orvex's bottleneck is a single agent's throughput/quality, not fairness across many reviewers.
- SO's queue optimizes for public-web trust/vandalism-resistance (anonymous strangers editing anyone's content). Orvex's staging area is single-tenant per customer — no cross-tenant adversarial submitters — so the heavier trust machinery (reputation thresholds, review privilege at 2,000 rep, /review tab at 10,000 rep) is over-built for the problem; a simpler per-agent trust score is likely sufficient.
- Stack Internal's ingestion pipeline is document-agnostic source ingestion (Confluence/Teams) transformed into atomic Q&A — Orvex's staging area instead needs to preserve the wiki's native document/section structure (per the PRD's "add document, add section, edit, replace, delete" granularity), not flatten everything into Q&A pairs.
- SO's public suggested-edits feature is in relative eclipse alongside the site's traffic collapse — it is not evidence of a moderation-queue pattern currently operating at fresh high volume; the load-bearing evidence for "proven at scale" is historical (pre-2023), not current.
