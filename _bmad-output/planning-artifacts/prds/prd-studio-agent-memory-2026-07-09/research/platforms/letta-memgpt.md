# Letta (formerly MemGPT) — Platform Deep Dive

## What it is
Letta is an open-source (Apache-2.0) platform for building "stateful agents" — AI agents with persistent, self-editing memory that survives across sessions and can grow beyond the LLM's fixed context window. It is the direct successor/rebrand of the MemGPT research project (the "OS-inspired" agent memory paper), now expanded into a full platform: agent runtime/server, REST API, SDKs (Python, TypeScript, Rust), a visual agent development environment (ADE), a desktop app, and — as of 2026 — a CLI-based coding agent product called "Letta Code." ([GitHub](https://github.com/letta-ai/letta), [Letta homepage](https://www.letta.com/), [Core concepts docs](https://docs.letta.com/core-concepts/))

## Architecture & data model
Letta's memory model is explicitly OS-inspired (the MemGPT lineage) and has settled into an implementation taxonomy of three tiers, confirmed by docs and a third-party academic survey:
- **Core memory** — small, always-in-context "blocks" pinned to the system prompt (persona + key user facts). Analogous to RAM; the agent reads/writes it directly via tool calls. Blocks are first-class objects that can be **attached/detached from multiple agents at once**, enabling shared memory across an agent fleet. ([Letta core-concepts docs](https://docs.letta.com/core-concepts/), [Letta core memory guide](https://docs.letta.com/guides/ade/core-memory/))
- **Archival memory** — a vector database for long-term storage; agents write observations and retrieve them later via search tool calls (cold storage). ([Third-party survey summary via WebSearch of docs.letta.com](https://docs.letta.com/core-concepts/))
- **Recall memory** — full conversation history persisted in a database; agents can search past messages instead of relying solely on the fixed context window; the docs confirm "all state — memories, user messages, reasoning, tool calls — are persisted in a database, so they are never lost," and old messages remain retrievable via the API even after context compaction/eviction. ([Letta core-concepts docs](https://docs.letta.com/core-concepts/))
- **Letta Filesystem** — newer addition: a filesystem-like tool API letting agents interact with data that doesn't fit in context, aligned with how modern models are trained on file/tool-use patterns. ([Letta Filesystem blog](https://www.letta.com/blog/letta-filesystem))
- **Agent File (.af)** — a portability format to serialize a full stateful agent (memory + config) so it can be moved/reproduced across servers (cloud ↔ local). ([Agent File blog](https://www.letta.com/blog/agent-file))
- The LLM itself acts as its own memory controller — it decides via tool calls when to promote/demote information between tiers, rather than an external heuristic pipeline making those calls. This self-management design is the core philosophical bet distinguishing Letta from RAG-bolt-on memory products. ([Letta core-concepts docs](https://docs.letta.com/core-concepts/))

## API & integration surface (MCP? SDK? CLI? chat-platform plugins?)
- **APIs/SDKs**: production REST API; official SDKs for Python, JavaScript/TypeScript, Rust. ([GitHub](https://github.com/letta-ai/letta))
- **MCP**: Letta agents can consume external MCP servers as tools — registering just the server URL, with execution happening on the MCP server (distinct from Letta's own sandboxed "server tools" or app-side "client tools"). Separately, third parties have built a "Letta MCP Server" that exposes Letta itself as an MCP tool provider to clients like Claude Desktop, GitHub Copilot, Cursor, Replit, Sourcegraph Cody — this is a community/third-party integration, not confirmed as an official first-party Letta product. ([Letta MCP tools docs](https://docs.letta.com/guides/core-concepts/tools/mcp-tools), [Letta MCP Server by SNYCFIRE-CORE, third-party, via Glama listing](https://glama.ai/mcp/servers/SNYCFIRE-CORE/letta-mcp-server))
- **CLI**: Letta Code — a terminal-native coding agent, positioned separately from the core memory-agent SDK/API line.
- **Chat-platform surface**: "Channels" (2026) is Letta's own agent-to-chat-platform bridge, replacing an earlier "LettaBot"; unknown which specific platforms (Slack et al.) are supported — not independently verified in this pass beyond a WebSearch summary sentence.
- **Deployment**: self-hostable via Docker (App Server + Postgres/pgvector for embeddings), or Letta Cloud (managed). Also has one-click deploy templates on Railway/RamNode. ([Self-hosting docs](https://docs.letta.com/guides/selfhosting/), [Railway deploy](https://railway.com/deploy/letta))

## Memory lifecycle & sharing
- **Capture**: conversation turns and explicit agent tool calls write into core/archival/recall tiers as they happen — capture is agent-driven (the LLM decides what's worth remembering), not an automatic background scrape.
- **Consolidate/decay/compact**: **Sleep-time compute** is Letta's answer — dedicated "sleep-time agents" share memory with a primary agent but run asynchronously in the background (not latency-sensitive), reprocessing conversation history/data sources to refine and rewrite memory blocks. Because it's off the hot path, Letta recommends pairing a fast/cheap model for real-time chat with a larger, slower model (e.g., GPT-4-class or Claude Sonnet) for the sleep-time consolidation pass — an explicit token-cost/quality tradeoff design. ([Sleep-time compute blog](https://www.letta.com/blog/sleep-time-compute/), [Sleep-time agents docs](https://docs.letta.com/guides/agents/architectures/sleeptime/))
- **Retrieve**: archival memory via vector search tool calls; recall memory via message-history search; core memory is always-resident so retrieval is implicit (already in context).
- **Multi-agent sharing / namespaces / permissions**: memory **blocks** are the sharing unit — a block can be attached to multiple agents simultaneously, giving a native primitive for shared team/fleet knowledge. Docs confirm this capability exists; unknown (not found in this pass) whether Letta has fine-grained per-block ACLs/permissions beyond attach/detach, or a formal multi-tenant namespace model beyond organization-level API keys. Enterprise tier adds RBAC and SAML/OIDC SSO, implying org-level (not necessarily block-level) permissioning. ([Core concepts docs](https://docs.letta.com/core-concepts/), [Pricing/Enterprise tier](https://docs.letta.com/guides/cloud/plans/))
- **Cross-session token-cost story**: the core pitch is that core-memory blocks keep the always-in-context footprint small and curated (agent actively prunes/edits them) while archival/recall push everything else to external, tool-queried storage — the claimed benefit is bounded per-turn token cost regardless of how much history accumulates. Sleep-time compute lets that curation happen on a cheaper/async cadence rather than inline. This is Letta's own architectural framing; no independent third-party benchmark of actual token-cost savings vs. naive full-context or RAG approaches was found in this pass (unknown magnitude). One relevant academic benchmark exists — "Benchmarking AI Agent Memory: Is a Filesystem All You Need?" — but its findings weren't extracted in depth here. ([Letta benchmarking blog](https://www.letta.com/blog/benchmarking-ai-agent-memory/))
- **Chat-platform integration**: "Channels" (2026, chat-platform bridge) and third-party MCP server exposing Letta to Claude Desktop / ChatGPT-style clients — see API section above; depth of native ChatGPT/Claude.ai integration (vs. developer SDK wiring) is unknown/unverified.

## Pricing & positioning
Letta runs two distinct pricing surfaces that are easy to conflate:
- **Letta Code (coding-agent product)**: Free tier ($0, limited agents/usage, BYOK); Pro ($20/mo, weekly+monthly "Letta Auto" quota, pay-as-you-go overage, up to 20 stateful agents); Enterprise (custom, volume rates, RBAC, SAML/OIDC SSO, dedicated support). Reported real-world spend: "casual" users ~$100+/mo, "power" users $200+/mo. ([Letta Code pricing docs](https://docs.letta.com/letta-code/pricing))
- **Letta Agent/API platform (the memory-agent product)**: usage-based — $20/mo base (API/Developer tier) + $0.10 per active agent/month + $0.00015 per second of tool execution + pay-as-you-go LLM token cost at provider rates; a credits model also appears in docs (Developer: 5,000 monthly credits, 2 agent templates, 1GB storage; Pro: 20,000 credits, unlimited agents, 20 templates, 10GB storage, overage billing); Enterprise adds elevated quotas, RBAC, SSO, private model deployment. ([Letta API plans docs](https://docs.letta.com/guides/cloud/plans/))
- Positioning: Letta sells itself as *infrastructure for stateful agents* — a memory/runtime layer other agent builders plug into — plus, increasingly in 2026, its own first-party coding-agent product (Letta Code) competing more directly with Claude Code / Cursor-style tools. The dual pricing surfaces suggest the company is straddling "memory infrastructure vendor" and "agent product company" simultaneously — worth watching for focus drift. Unknown how much revenue/usage comes from each line.

## STEAL — ideas for Orvex
1. **Tiered memory-as-context-budget, not memory-as-database.** Letta's core/archival/recall split is really a token-budget management pattern: keep a small, agent-curated "always resident" block, and push everything else behind a retrieval tool call. Orvex's cross-agent memory service should adopt the same shape — a tiny per-agent/per-customer "core" summary block injected into every session, with beads-style structured records as the queryable archival tier — rather than dumping raw history into context.
2. **Shared memory blocks as the multi-agent primitive.** Letta's "attach one block to many agents" is a clean, simple primitive for cross-agent handoff/coordination — directly relevant to the CROSS-AGENT MEMORY track. Consider a similar "shared context block" object (scoped per customer/team) that multiple agent sessions can attach to, rather than requiring every agent to pull/merge full history.
3. **Sleep-time / async consolidation as a cost lever.** Rather than paying premium-model tokens inline to curate memory during a live chat, run consolidation (dedupe, summarize, promote/demote) asynchronously with a cheaper cadence — directly applicable to both the librarian (staging-area track) and the memory service's "make future sessions cheaper" goal.
2b. Grouped correctly above — treat as one line.
4. **Portable agent-state export (Agent File analog).** A `.af`-style export/import format for an agent's full memory state would let Orvex support agent migration, debugging, and reproducibility across environments (dev/staging/prod, or customer-to-customer support escalation) — cheap to build, high leverage for support/ops.
5. **Self-editing memory via tool calls, with guardrails.** Letta lets the LLM itself decide what to write/promote/evict. For the AGENT STAGING AREA track specifically, Orvex already gates writes through a librarian — but the *memory* track (beads-style logging) could borrow the "agent proposes memory edits as structured tool calls" pattern rather than free-text logging, making the librarian/reviewer's job schema-driven instead of prose-parsing.

## AVOID / where Orvex differs
- **Letta's memory is agent-self-managed and un-gated** — the LLM directly writes/edits its own memory blocks with no separate human/reviewer approval step. This is the opposite of Orvex's AGENT STAGING AREA principle (agents never write the customer wiki directly; a librarian reviews first). Don't import Letta's "agent has direct write access to persistent state" trust model wholesale — Orvex's staged-write requirement is a deliberate, stronger safety posture that Letta does not have for its core memory.
- **Single-tenant-shaped memory blocks vs. Orvex's multi-tenant customer isolation.** Letta's attach/detach block sharing is convenient but its permission model (per docs found here) is coarse — org-level RBAC/SSO at Enterprise tier, not confirmed fine-grained per-block ACLs. Orvex is inherently multi-tenant (per-customer wikis, per-customer librarian prompts); any shared-memory primitive must be namespace-isolated by tenant from day one, which is a harder requirement than what Letta's docs evidence.
- **Product-focus drift risk.** Letta's 2026 pivot toward "Letta Code" (a coding-agent product) alongside its original memory-infrastructure pitch is a cautionary signal: a memory-infra vendor can end up competing with its own downstream customers' use cases. Orvex should keep MEMORY and STAGING as clearly-scoped infra services, not let them grow into competing end-user chat products.
- **Unverified scale claims.** Letta's Terminal-Bench "#1 model-agnostic" claim and "casual/power user spend" figures are marketing/docs copy, not independently audited — don't cite them as validated benchmarks in Orvex planning docs without independent verification.
- **Dolt vs. Letta's Postgres+pgvector.** Letta's storage layer is Postgres/pgvector, not Dolt. It offers no direct evidence for or against Orvex's "possibly Dolt" storage question — that decision remains open and should be researched against a version-control-native memory store separately, since Letta's stack doesn't address the git-like diff/branch requirements Dolt would bring.
