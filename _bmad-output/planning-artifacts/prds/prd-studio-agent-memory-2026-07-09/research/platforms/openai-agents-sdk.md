# OpenAI Agents SDK — Prior Art Deep Dive (2025–2026)

## What it is
The OpenAI Agents SDK is a lightweight, open-source (MIT-licensed) Python/TS framework for building multi-agent workflows around three primitives: **agents**, **tools**, and **handoffs**, plus a **Sessions** memory layer and **guardrails** for validation/HITL. It is the production successor to OpenAI's experimental "Swarm" framework. It is provider-agnostic (works with OpenAI's Responses/Chat Completions APIs and 100+ other LLMs via a compatible interface). It ships alongside **AgentKit** (announced at DevDay, Oct 6 2025), a broader platform layer that adds a visual **Agent Builder** canvas, evals, guardrails-as-a-service, and deployment tooling on top of the SDK.
Sources: https://github.com/openai/openai-agents-python , https://openai.github.io/openai-agents-python/ , https://openai.com/index/introducing-agentkit/

## Behind it & traction
Built and maintained directly by OpenAI. GitHub repo `openai/openai-agents-python`: **27.8k stars**, MIT license, 106 releases, latest v0.18.0 (2026-07-07) — actively maintained with frequent releases. A major "next evolution" update shipped 2026-04-15 adding native sandbox execution, a model-native harness, and configurable memory — described by OpenAI as consolidating primitives "teams previously assembled themselves." Third-party comparison pieces (2026) position it as "the easiest choice" for getting a simple-to-medium-complexity agent running quickly, with clean tool calling and strong docs, but weaker than Anthropic's SDK for reasoning-heavy/multi-agent orchestration and weaker than Google ADK for schema-validated structured output.
Sources: https://github.com/openai/openai-agents-python , https://openai.com/index/the-next-evolution-of-the-agents-sdk/ , https://blog.stackademic.com/openai-vs-anthropic-vs-google-agent-sdk-the-brutal-truth-981200082684

## Architecture & data model
- **Agent**: an LLM configured with instructions, tools, guardrails, and possible handoff targets.
- **Handoff**: a named primitive (`handoff()` function) that exposes a transfer to another agent as a callable tool to the model (e.g., an agent named "Refund Agent" becomes the tool `transfer_to_refund_agent`). Supports `tool_name_override`, `tool_description_override`, an `on_handoff` callback, a typed `input_type` for handoff arguments, and `is_enabled` (static or dynamic) to gate availability. Handoffs transfer control entirely to the receiving agent **within the same run** — they are not a separate memory/session boundary by default.
- **Input filters**: functions that receive `HandoffInputData` (full history, pre-handoff items, new items, run context) and can prune what the receiving agent sees — e.g., a built-in filter strips prior tool-call noise from history before handoff.
- **Sessions**: the memory/state object. Before each run, the runner fetches session history and prepends it to new input; after the run, generated items (messages, tool calls) are auto-appended back into the session. Multiple agents can share one `Session` instance, in which case all participants see identical history; different session IDs are fully isolated.
- **RunContextWrapper.context**: separate from conversation history — app-level state (user id, DB connections, etc.) threaded through a run but not sent to the model.
Sources: https://openai.github.io/openai-agents-python/handoffs/ , https://openai.github.io/openai-agents-python/sessions/ , https://openai.github.io/openai-agents-python/agents/

## API & integration surface (MCP? SDK? CLI? chat-platform plugins?)
- **SDK**: Python and equivalent TS/JS packages; this is the primary distribution mechanism (no standalone CLI product for end users; AgentKit's Agent Builder is a hosted visual canvas, not a local CLI).
- **MCP**: MCP servers are supported as a tool source for agents (listed alongside "functions" and "hosted tools" in the SDK's tool system), but the docs treat MCP as orthogonal to session/memory management — MCP is for capability/tool wiring, not for memory transport.
- **Sessions have pluggable storage backends**: `SQLiteSession` / `AsyncSQLiteSession` (local dev), `RedisSession` (shared/low-latency across workers), `SQLAlchemySession` (existing SQL infra), `MongoDBSession` (horizontally scalable), `DaprSession` (cloud-native, 30+ backend DBs via Dapr sidecar), `OpenAIConversationsSession` (OpenAI-hosted server-side storage), and `EncryptedSession` (transparent encryption wrapper over any backend). Custom backends implement the `SessionABC` async interface (`get_items`, `add_items`, `pop_item`, `clear_session`).
- **Chat-platform plugins**: not native to the SDK itself; distribution to ChatGPT/Claude/portals is out of scope for this layer — it's a backend framework, not a chat-surface integration.
Sources: https://openai.github.io/openai-agents-python/sessions/ , https://cookbook.openai.com/examples/agents_sdk/session_memory , https://github.com/openai/openai-agents-python

## Memory lifecycle & sharing
- **Capture**: automatic — every run's generated items (user input, assistant output, tool calls/results) are appended to the active session after each turn; no manual `response.output` tracking needed.
- **Consolidate/compact**: `OpenAIResponsesCompactionSession` wraps a session and calls the Responses API's `responses.compact` to rewrite/shrink stored history. Two modes: `"previous_response_id"` (chains off Responses API state) and `"input"` (rebuilds from current session items); `"auto"` picks the safest based on whether the underlying run is stateful (`store=True`). Compaction blocks run completion by default; can be decoupled via `should_trigger_compaction=lambda _: False` + manual `run_compaction()` for low-latency paths.
- **Retrieve**: `get_items(limit=None)` pulls history; a `limit` caps how much is pulled per run; `RunConfig.session_input_callback` lets developers custom-merge retrieved history with new turn input before it's sent to the model (reordering/filtering here does not mutate persisted storage).
- **Decay/summarize (April 2026 update)**: two explicit tiers — **short-term memory** is per-session, ephemeral, bounded, and auto-summarized by the SDK as the context window fills (summarization aggressiveness is configurable); **long-term memory** is persistent, explicitly scoped, and explicitly *written* — exposed to the agent as a callable tool, so the agent itself decides what's worth remembering and under what scope, rather than everything being auto-captured.
- **Multi-agent sharing / namespaces**: sharing = literally sharing one `Session` instance/ID across agents (all see full history); isolation = separate session IDs (even against the same backing DB). There is no finer-grained namespace/ACL model documented beyond session-ID-as-boundary — no per-field permissions, no cross-tenant scoping primitive described.
- **Cross-session token-cost story**: compaction is the primary lever (rewriting/shrinking history via the Responses API); nested handoffs (opt-in beta, `RunConfig.nest_handoff_history`) collapse prior transcript into a single assistant summary wrapped in a `<CONVERSATION HISTORY>` block to bound tokens passed to a newly-handed-off agent.
- **Chat-platform integration**: no first-class ChatGPT/Claude/portal memory bridge; `OpenAIConversationsSession` ties into OpenAI's own hosted Conversations API rather than any third-party chat surface. MCP is used for tool access, not documented as a memory-sharing channel.

## Scale & operational evidence
- Session storage backends explicitly designed for horizontal scale/production: Redis (distributed workers), MongoDB ("horizontally-scalable, multi-process"), Dapr (30+ backend DBs, cloud-native sidecar model), SQLAlchemy (bring-your-own production DB). This is evidence of *design intent* for scale, not published customer-scale numbers.
- April 2026 update added durability primitives: snapshotting and rehydration so that losing a sandbox container mid-run doesn't lose agent state — it restores into a fresh container from the last checkpoint. Sandbox execution can run in Blaxel, Cloudflare, Daytona, E2B, Modal, Runloop, or Vercel, or a custom sandbox.
- No published concrete numbers found (no "N requests/day," no named enterprise customer scale figures) — unknown beyond the 27.8k GitHub stars and 106-release cadence as an activity proxy.
- **Known production gap (documented via GitHub issue #636 and third-party analysis)**: the `needs_approval` human-in-the-loop mechanism has no built-in notification (no email/Slack alert when a run pauses for approval), no timeout/escalation (paused runs wait indefinitely), and no structured audit trail (who was asked, when, what they chose). Developers must build interception detection, notification delivery, context formatting, decision capture, timeout handling, audit logging, and resume/abort translation themselves — described by the community as producing "brittle, custom logic." Workaround pattern: `state.to_json()` / `RunState.from_json()` for manual async-approval persistence.

## Pricing & positioning
- The SDK itself is **free and open-source** (MIT license) — no separate SDK pricing tier.
- Cost is entirely pass-through API usage: Responses API / Chat Completions API / Realtime API / Batch API / Assistants API "are not priced separately" — tokens bill at the chosen model's standard input/output rates. Hosted tool add-ons: Web Search ($10 per 1K calls + ~8K tokens/search), File Search ($2.50 per 1K queries + $0.10/GB/day storage), Code Interpreter ($0.03/session).
- Positioning per third-party comparisons (2026): OpenAI Agents SDK = best for **execution-heavy pipelines** and fastest time-to-first-agent; Anthropic's SDK = better for reasoning-heavy/multi-agent orchestration; Google ADK = better where schema-validated structured output matters most.
Sources: https://developers.openai.com/api/docs/pricing , https://blog.stackademic.com/openai-vs-anthropic-vs-google-agent-sdk-the-brutal-truth-981200082684

## STEAL - 3-5 concrete ideas for Orvex
1. **Named handoff primitive as a first-class tool, not a side channel.** Modeling handoff as `transfer_to_<agent>` — a real callable tool the model chooses to invoke, with an optional typed `input_type` payload (reason/priority/etc.) — is a clean pattern Orvex's cross-agent memory/handoff feature could copy directly: agents "hand off work" by calling a tool, and the handoff payload (not raw history) is what the librarian/registry logs.
2. **Two-tier memory split (ephemeral session vs. explicit long-term write).** Their April 2026 model — auto-summarized short-term session memory vs. an agent-invoked, explicitly-scoped long-term memory tool — maps well onto Orvex's "agents continuously log what they're doing" design: don't auto-persist everything into cross-agent memory; give agents a deliberate "remember this" tool call, keeping the durable store curated rather than a firehose transcript dump.
3. **Input filters at the handoff boundary.** Their `HandoffInputData` filter pattern (e.g., stripping tool-call noise before the next agent sees history) is a reusable idea for Orvex's librarian: when staged content or memory context is handed to the next agent/session, apply a filter step to trim irrelevant tool-call chatter before it counts against context/token budget.
4. **Pluggable session-storage interface (`SessionABC`) as a portability lesson.** Shipping one clean async interface (`get_items`/`add_items`/`pop_item`/`clear_session`) with Redis/Mongo/SQLAlchemy/Dapr/encrypted backends out of the box is a strong precedent for Orvex's Dolt-vs-Postgres decision: design the memory-store contract as a thin interface first, so the storage engine choice becomes swappable rather than load-bearing in the API.
5. **Steal the failure mode as a spec, not just the feature: build the HITL approval loop's "boring parts" in from day one.** Their own community explicitly names the gap (#636: no notification, no timeout/escalation, no audit trail) — Orvex's librarian review gate should ship with notification hooks, a timeout/escalation policy, and a structured decision log (who/when/what/why) as core requirements, not bolt-ons, precisely because this is the most-cited production complaint against the closest prior-art pause-for-approval primitive.

## AVOID / where Orvex differs
- **No namespace/permission model beyond session-ID isolation.** OpenAI's sharing model is binary (same session = full shared history, different session = fully isolated) with no per-field ACLs or cross-tenant scoping documented. Orvex's multi-tenant, per-customer requirement needs a real namespace/permission layer — don't copy the "session ID is the only boundary" simplicity.
- **No chat-platform memory bridge.** There's no native ChatGPT/Claude/portal-side memory integration — `OpenAIConversationsSession` only ties into OpenAI's own Conversations API. Orvex explicitly needs cross-platform (ChatGPT-style portals, Claude, CLI) memory sharing, which this prior art doesn't solve or even attempt.
- **HITL approval is a primitive, not a workflow.** `needs_approval` is a low-level interrupt/resume mechanism; it deliberately punts notification, escalation, and audit to the developer. Orvex's librarian review gate is meant to be a full workflow (review/route/merge/beautify) — treat OpenAI's primitive as a floor, not a model to imitate structurally.
- **Compaction can block the run.** Default compaction behavior is synchronous and can stall streaming/latency-sensitive paths unless manually decoupled — a design wrinkle Orvex should avoid by defaulting to async/background consolidation for its memory-decay story rather than blocking agent turns.
- **This is a developer SDK, not a hosted multi-tenant service.** Orvex's cross-agent memory is meant to be a *hosted* service (MCP tools + CLI) with per-customer tenancy; the Agents SDK is self-hosted infrastructure a team assembles per-app. The traction/scale evidence found here is about framework adoption (stars, release cadence), not about operating a hosted memory service at scale — treat "scale & operational evidence" gaps in this doc as a reminder that OpenAI hasn't proven this pattern at hosted-multi-tenant scale either.
