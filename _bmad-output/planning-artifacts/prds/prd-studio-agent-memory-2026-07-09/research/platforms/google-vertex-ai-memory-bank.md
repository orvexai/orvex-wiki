# Google Vertex AI Memory Bank — Deep Dive

## What it is
Memory Bank is a managed, Gemini-powered long-term memory service that is part of Google's **Vertex AI Agent Engine** (rebranded in 2026 as the **Gemini Enterprise Agent Platform**). It sits alongside "Sessions" (short-term conversation state): Sessions hold the current conversation, Memory Bank persists facts/preferences *across* sessions so an agent "remembers" a user over days or weeks. It launched in **public preview on 2025-07-09** and Google states Sessions + Memory Bank reached **General Availability in 2026**, with production billing effective **2026-01-28**.
[Vertex AI Memory Bank in public preview — Google Cloud Blog](https://cloud.google.com/blog/products/ai-machine-learning/vertex-ai-memory-bank-in-public-preview) · [Agent Engine Memory Bank overview](https://cloud.google.com/agent-builder/agent-engine/memory-bank/overview)

## Behind it & traction
Built and operated by Google Cloud as part of Vertex AI / Agent Engine (not a third-party or open-source project). The extraction/consolidation approach is described as grounded in "a novel method from Google Research, accepted at ACL 2025" (paper referenced but not independently verified by us). No named enterprise customers, published case studies, or third-party production-scale write-ups were found in this pass — coverage is dominated by Google's own docs/blog and community how-to posts (Medium, DEV.to, oneuptime.com), not independent adoption evidence.

## Architecture & data model
Three-stage pipeline per Google's own description:
1. **Extraction** — Gemini models asynchronously analyze conversation history from Agent Engine Sessions and pull out facts/preferences/context ("memory generation"), exposed via a `GenerateMemories` operation.
2. **Storage & consolidation** — new facts are merged into existing memory using Gemini to resolve contradictions and deduplicate, organized under configurable **scopes**: user ID, app, and session. The docs also mention "memory profiles" and "revision tracking" for audit trails.
3. **Retrieval** — via `RetrieveMemories`, either simple retrieval (all facts in scope) or similarity-search retrieval (embedding-based relevance ranking against the current conversation).
Google's own FAQ states TTL (time-to-live) for memories is **not yet available** (roadmap item as of preview) and automatic deduplication is included. Memory Bank is described as built tightly around the Gemini model family — not model-agnostic ("not at the moment" for open-source model compatibility, per Google's forum FAQ).

## API & integration surface (MCP? SDK? CLI? chat-platform plugins?)
- **Native path**: Agent Development Kit (ADK) → Agent Engine Sessions → Memory Bank, "out of the box."
- **Direct API**: `GenerateMemories` / `RetrieveMemories` REST-style operations callable from any framework (Google publishes sample integrations for **LangGraph** and **CrewAI** in addition to ADK).
- Docs additionally reference support/guides for **LangChain, Agent2Agent (A2A), AG2, and LlamaIndex**.
- **Express Mode**: lets ADK developers sign up with just a Gmail account/API key to try Sessions + Memory Bank inside free-tier quotas before upgrading to a full GCP project.
- **MCP**: docs reference "Google MCP servers" and "external MCP servers" in codelab titles, but we could not confirm a first-class MCP tool surface specifically for Memory Bank (i.e., whether Memory Bank itself is exposed as MCP tools, vs. agents built with ADK/MCP separately calling it). Treat as **unknown/unconfirmed**.
- No evidence found of a standalone CLI for Memory Bank, or of direct ChatGPT/Claude-side plugins (expected, since this is a Google-agent-runtime-coupled service, not a portable memory API).

## Memory lifecycle & sharing
- **Capture**: Gemini extracts facts asynchronously at session/turn boundaries from Session transcripts (`GenerateMemories`).
- **Consolidate**: Gemini merges new facts into existing memory records per scope, resolving contradictions and deduplicating; a "memory profile" per scope is the unit of organization, with revision tracking for auditability.
- **Retrieve**: `RetrieveMemories` supports full-scope fetch or embedding-based similarity search against the live conversation.
- **Decay/compact/summarize**: TTL/expiry is explicitly **not yet supported** — Google's own FAQ says it's "on the roadmap," meaning memories are not known to auto-decay today. No compaction/summarization mechanism beyond consolidation was documented.
- **Multi-agent sharing / namespaces / permissions**: scoping is by **user ID / app / session**, which acts as the namespace boundary. Access control is IAM-based (service accounts, OAuth clients, IAM Conditions) for granular session/Memory Bank access — this is GCP-project-level IAM, not a fine-grained per-agent ACL model. A2A (Agent2Agent protocol) is positioned as the way separate agents/teams compose systems with "opaque boundaries," but we found no documented mechanism for two independent agents to directly share/read the *same* memory scope beyond both being granted IAM access to it — cross-agent memory sharing is implied but not architecturally detailed in what we fetched.
- **Cross-session token-cost story**: Google's stated motivation is explicitly to avoid stuffing full session transcripts into the context window (calls out cost, latency, and "lost in the middle"/context-rot degradation) — Memory Bank's summarized-fact retrieval is pitched as the token-cost mitigation itself, rather than there being a separate published token-savings benchmark.
- **Chat-platform integration**: no ChatGPT/Claude/generic-portal integration found — this is coupled to Google's own Agent Engine/ADK runtime, reachable from other frameworks (LangGraph, CrewAI, LangChain, LlamaIndex, AG2) only via direct API calls.

## Pricing & positioning
- **Preview period**: free (per Google's own FAQ, "Currently free during preview," with pricing to be published later).
- **GA pricing (effective 2026-01-28)**: Sessions and Memory Bank are billed separately at **$0.25 per 1,000 stored events or memories**. This is described as an operation/storage-adjacent charge rather than a fine-grained read/write split.
- One third-party estimate (aiagentmemory.org, not an official source) suggests a realistic production support-agent bill of **$500–$2,000+/month**, but gives no methodology — treat as a rough marketing-adjacent estimate, not verified evidence.
- Underlying compute for retrieval (e.g., similarity search) appears to bill through general Vertex AI/Agent Platform compute pricing (referenced elsewhere as "Agent Compute vCPU-h," ~$0.085/vCPU-h bucketed per read volume) rather than a Memory-Bank-specific rate — this figure came from a secondary aggregator source, not Google's own pricing page, and should be treated as **unconfirmed**.
- Positioning: Memory Bank is marketed as removing the need to hand-build extraction/consolidation/dedup pipelines ("handles the messy state management you really don't want to build yourself" — Google's own words), aimed squarely at teams already inside the ADK/Agent Engine ecosystem, not as a portable/standalone memory product.

## STEAL — 3-5 concrete ideas for Orvex
1. **Separate "consolidation" from "retrieval" as distinct async operations.** Memory Bank's `GenerateMemories` (background, batched fact extraction/merge) vs `RetrieveMemories` (sync, scoped/similarity-ranked fetch) is a clean split Orvex's cross-agent memory service should mirror — write-side consolidation should never block an agent's live turn.
2. **Explicit conflict-resolution/dedup as a first-class step, not an afterthought.** Google calls out "resolves contradictions and keeps memories up-to-date" as core value; Orvex's memory/coordination service (and the librarian for the staging area) should treat contradiction-resolution as a named pipeline stage with its own review surface, not something silently done by an LLM prompt.
3. **Scope model (user / app / session) as the namespace primitive.** A minimal three-tier scope is a good floor for Orvex's per-customer, per-agent, per-conversation memory namespacing — simpler to reason about than ad hoc tagging.
4. **Lead with the token-cost narrative explicitly in positioning.** Google's own framing — "stuffing full transcripts into context causes cost, latency, and context rot" — is a ready-made justification Orvex can reuse verbatim when pitching the memory/coordination service internally and to customers.
5. **"Express mode" low-friction trial** (sign up with just an email, try inside free-tier quotas, upgrade later) is a good adoption pattern for Orvex's memory/coordination service and staging area — let a team try cross-agent memory in a sandboxed low-commitment mode before wiring in full multi-tenant config.

## AVOID / where Orvex differs
- **Not model-agnostic**: Memory Bank is tightly coupled to the Gemini model family (per Google's own FAQ: "not at the moment" for open-source/other-model compatibility). Orvex's memory/coordination service must stay LLM-agnostic since Orvex agents run ChatGPT, Claude, and others — this is a structural difference, not a preview-stage gap to wait out.
- **No TTL/decay yet**: Google explicitly has no memory expiry as of this pass ("on the roadmap"). Orvex should not assume "decay" is a solved problem to copy — it's an open gap even in the reference platform, so Orvex's own decay/compaction design is genuinely novel ground, not something to defer to prior art.
- **No confirmed MCP-native memory surface**: unlike Orvex's MCP-first mandate, Memory Bank's MCP story is ambiguous/unconfirmed from available docs — Orvex should not assume MCP tool parity exists to crib from; Orvex's MCP tool surface for memory would be original design.
- **IAM-only permissions, not fine-grained per-agent ACLs**: Memory Bank's access control rides on GCP project IAM/OAuth, which is coarser than what a multi-tenant, per-customer-tweakable librarian/staging model likely needs. Orvex should not assume IAM-level scoping is sufficient for the staging area or cross-agent memory's permission model.
- **Runtime-coupled, not a portable service**: Memory Bank only really works well inside Agent Engine/ADK; cross-framework use is "direct API calls" bolted on. Orvex's memory/coordination service is explicitly meant to be hosted and reachable from many chat platforms/CLIs — a more platform-neutral design goal than Memory Bank ever attempts.
- **Thin, marketing-heavy evidence base**: there is no independent production-scale evidence, no named customers, and pricing details are largely reconstructed from secondary sources rather than a fully transparent Google pricing page in this pass — Orvex should not treat Memory Bank's operational maturity as proven at scale, only as a well-marketed preview/early-GA service.
