# CrewAI — Platform Deep Dive

## What it is
CrewAI is an open-source Python multi-agent orchestration framework (independent of LangChain, built from scratch) for composing "crews" of role-playing autonomous agents that collaborate on tasks via sequential or hierarchical processes. It fuses both halves of the Studio memory mandate in one product: a native 4-tier memory system (short-term, long-term, entity, procedural — now unified) AND agent-to-agent handoff via task delegation / hierarchical manager-worker orchestration. Commercialized as CrewAI AMP (Agent Management Platform, formerly "CrewAI Enterprise"), a hosted control plane for deploying, monitoring, and governing crews. [Pricing](https://crewai.com/pricing), [GitHub](https://github.com/crewaiinc/crewai)

## Architecture & data model
- Core unit is a **Crew**: a set of **Agents** (role, goal, backstory, tools, LLM) executing **Tasks** under a **Process** — `sequential` (fixed order) or `hierarchical` (a manager agent/LLM plans, delegates, and validates work dynamically). [docs.crewai.com/en/concepts/processes](https://docs.crewai.com/en/concepts/processes)
- Hierarchical process: requires either a `manager_llm` or a custom `manager_agent`; the manager agent is auto-created if not supplied. Manager receives the goal, decides which worker agents run, in what order, with what inputs, and reviews/validates outputs — this is CrewAI's handoff/delegation mechanism. Delegation is gated by `allow_delegation=True` on the manager and typically `False` on specialist agents. [ActiveWizards](https://activewizards.com/blog/hierarchical-ai-agents-a-guide-to-crewai-delegation/), [docs.crewai.com/en/concepts/processes](https://docs.crewai.com/en/concepts/processes)
- Known operational friction: community/GitHub issues report hierarchical delegation misrouting tasks to the wrong agent, and hierarchical mode adds latency/cost from extra manager LLM calls — cited as a reason to prefer sequential process for straightforward pipelines. [GitHub Issue #4783](https://github.com/crewAIInc/crewAI/issues/4783), [Towards Data Science](https://towardsdatascience.com/why-crewais-manager-worker-architecture-fails-and-how-to-fix-it/)
- **Memory data model (current, unified as of CrewAI v1.10+)**: a single `Memory` class replaced the previously separate short-term/long-term/entity/external memory types with one API. Default storage backend is **LanceDB** (local, `./.crewai/memory`); custom backends implement a `StorageBackend` protocol. Default embedder is OpenAI `text-embedding-3-large` (3072-dim), swappable (Ollama, Azure, Cohere, HF) for fully local operation. Default analysis LLM is `gpt-4o-mini`. [docs.crewai.com/concepts/memory](https://docs.crewai.com/concepts/memory)
- Legacy/pre-unification model (still referenced across most third-party writeups): Short-Term Memory (ChromaDB + RAG, resets each `kickoff()`), Long-Term Memory (SQLite3, persists across sessions — lessons/strategies/findings), Entity Memory (structured facts about people/companies/concepts encountered during execution), Procedural Memory (learned task-execution patterns). [CallSphere](https://callsphere.ai/blog/crewai-memory-short-term-long-term-entity-persistent-crews), [Crewship](https://www.crewship.dev/learn/crewai-memory), [GeeksforGeeks](https://www.geeksforgeeks.org/artificial-intelligence/memory-in-crewai/)

## API & integration surface (MCP? SDK? CLI? chat-platform plugins?)
- **MCP**: native support as a first-class tool source. Agents can consume tools hosted on MCP servers via an `mcps` field on the agent DSL (string refs for quick setup, structured config for full control) or the lower-level `MCPServerAdapter` class in `crewai-tools` for manual connection lifecycle management. Supports Stdio (local), SSE, and Streamable HTTP transports; failures on one server degrade gracefully (warns, keeps using working servers). [docs.crewai.com/en/mcp/overview](https://docs.crewai.com/en/mcp/overview), [PR #2496](https://github.com/crewAIInc/crewAI/pull/2496)
- **SDK**: Python-native framework (`pip install crewai`); this is the primary integration surface — no other language SDK found in research.
- **CLI**: `crewai` CLI exists for scaffolding/running crews (referenced across docs/tutorials; not independently verified in depth here — unknown exact command surface).
- **Chat-platform plugins (ChatGPT/Claude/portals)**: unknown / not found in research. No evidence of a native ChatGPT plugin, Claude Desktop extension, or turnkey chat-portal embedding. CrewAI AMP exposes crews via its own hosted UI/API rather than third-party chat platforms.
- **CrewAI AMP** adds: visual crew editor, AI copilot, observability, training, triggers, guardrails, SSO (MS Entra, Okta), RBAC, dedicated VPC networking, on-prem/private-VPC deployment ("AMP Factory") for AWS/Azure/GCP. [checkthat.ai](https://checkthat.ai/brands/crewai/pricing), [toolworthy.ai](https://www.toolworthy.ai/tool/crewai)

## Memory lifecycle & sharing
- **Capture**: `remember()` saves content with optional explicit scope; if unspecified, an LLM infers placement (scope, categories, metadata). An `extract()` step breaks raw text into atomic facts before storing, avoiding bloated records. [docs.crewai.com/concepts/memory](https://docs.crewai.com/concepts/memory)
- **Consolidate**: on save, checks similarity against existing records; above a `consolidation_threshold` (default 0.85) an LLM decides keep / update / delete / insert_new. Intra-batch dedup at 0.98 cosine similarity uses pure vector math (no LLM cost) for `remember_many()` calls.
- **Retrieve**: `recall()` supports shallow (vector-search-only) and deep (multi-step LLM analysis) modes; short queries (<200 chars) skip LLM analysis by default to save cost/latency. Retrieval uses **composite scoring**: semantic similarity + recency (exponential decay) + importance.
- **Decay/compact**: recency score decays as `0.5^(age_days / half_life_days)`, default half-life 30 days, configurable per use case (e.g., 7 days for sprint work, 180 days for architecture docs) — this is the closest thing to an explicit "decay" knob found in any platform researched so far.
- **Sharing / namespaces / permissions**: hierarchical **scopes** (e.g. `/project/alpha`, `/agent/researcher`) partition memory. `MemoryScope` restricts an agent to a subtree; `MemorySlice` composes disjoint branches (e.g., read-only shared knowledge + a private findings branch) — this is CrewAI's closest analog to Studio's per-tenant/per-agent permissioning. Crews can choose to share memory across all agents or keep it agent-private. Source tracking + private flags support multi-user isolation *within the unified Memory API*.
- **Cross-session token-cost story**: non-blocking saves run on background threads so they don't stall agent turns; `recall()` drains any pending writes before searching, so reads are consistent. No published benchmark of tokens saved per session; the framework's main lever is short-query fast-path (skip LLM analysis) plus the deep/shallow recall split to bound retrieval cost.
- **Known production gap (documented by a third party, not CrewAI itself)**: "CrewAI does not scope memory per user in server environments" — multi-tenant deployments risk context bleed between end-users sharing one API/agent process, since native memory has no `user_id`-style partition at the request layer. Mem0's integration adds a `user_id` parameter to fix this; Mem0 also persists memory externally so it survives container/redeploy churn, since native LanceDB/SQLite storage is machine-bound and wiped on redeploy. This is exactly the "framework memory isn't enough" signal called out in the research brief. [Mem0 blog](https://mem0.ai/blog/crewai-memory-production-setup-with-mem0)
- **Mem0 integration shape**: Mem0 replaces short-term + entity memory as an external provider; CrewAI's native long-term memory (SQLite-backed task outcomes) and the contextual-memory orchestration layer stay native. Two Mem0 deployment paths: Mem0 Cloud (managed) or Mem0 OSS (self-hosted, own vector store/LLM/embedder) for data-residency-sensitive projects. [docs.mem0.ai/integrations/crewai](https://docs.mem0.ai/integrations/crewai)
- **Chat-platform integration**: unknown — no evidence found of memory being surfaced/shared through ChatGPT, Claude, or a chat-portal UI; memory is a backend concept accessed via the Python API or AMP's own dashboards.

## Scale & operational evidence
- 54,242 GitHub stars, 27M+ PyPI downloads (self-reported), "~2B agentic executions in the prior 12 months" as of Jan 2026 — all marketing/traction claims from secondary sources, not independently audited SLAs or benchmark data. [getpanto.ai](https://www.getpanto.ai/blog/crewai-platform-statistics)
- No published multi-tenant scale numbers, memory-store size limits, or latency SLAs for the memory subsystem specifically.
- Real operational friction is visible in CrewAI's own GitHub issues/community forum: hierarchical delegation misrouting bugs (#4783), type errors in hierarchical delegation (#2606), and repeated community confusion threads about how memory (native vs Mem0 vs LTM) actually behaves — evidence the memory/delegation story is not fully battle-tested at the edges. [GitHub #4783](https://github.com/crewAIInc/crewAI/issues/4783), [GitHub #2606](https://github.com/crewAIInc/crewAI/issues/2606), [CrewAI Community](https://community.crewai.com/t/confusion-around-memory-mem0-ltm-openai-and-gemini/5682)

## STEAL — 3-5 concrete ideas for Orvex
1. **Composite recall scoring (similarity + exponential recency decay + importance)** is a clean, explainable ranking formula for surfacing memories — directly reusable for Orvex's cross-agent memory retrieval, with a tunable half-life per memory class (e.g., short for active-incident context, long for architecture decisions). [docs.crewai.com/concepts/memory](https://docs.crewai.com/concepts/memory)
2. **Consolidation-on-write with an LLM arbitration step** (similarity > threshold → LLM decides keep/update/delete/insert_new) plus a cheap non-LLM intra-batch dedup pass (cosine > 0.98) is a good two-tier compaction pattern: cheap vector math handles the common case, LLM judgment only for ambiguous overlaps.
3. **Hierarchical scope + slice model** (`MemoryScope` for subtree restriction, `MemorySlice` for composing a read-only shared branch with a private branch) is a good namespace primitive for Orvex's cross-agent memory: per-tenant / per-project / per-agent scopes with a shared "team knowledge" slice plus private per-agent findings.
4. **Non-blocking background writes with drain-before-read** avoids adding memory-write latency to the agent's turn while still guaranteeing read-your-writes consistency — worth adopting for Orvex's staging/memory writers so agent conversations aren't slowed by memory persistence.
5. **Short-query fast path (skip LLM analysis under N chars)** is a simple, high-leverage cost control for keeping cross-session memory retrieval cheap — apply the same fast/deep recall split to Orvex's memory MCP tool so cheap lookups don't always pay for an LLM-mediated deep search.

## AVOID / where Orvex differs
- **No native per-user/tenant scoping in server deployments** — CrewAI's memory was originally built for single-process, single-user crew runs; multi-tenant SaaS use needs a bolt-on (Mem0's `user_id`) to avoid context bleed. Orvex's memory service must treat per-customer/per-agent isolation as a first-class primitive from day one, not an afterthought — this is the single clearest "framework memory isn't enough" lesson from CrewAI's own ecosystem steering users to Mem0.
- **Machine-bound default storage (LanceDB/SQLite on local disk)** is wiped on redeploy/new container — unsuitable for Orvex's Kafka/Postgres-centric, horizontally-scaled multi-tenant platform; Orvex's memory store must be externalized/durable by design (Postgres core per existing research direction), not a local file store with an external-provider escape hatch bolted on later.
- **Hierarchical delegation lives inside a single crew run**, mediated by a manager LLM making synchronous planning decisions with real reported bugs (misrouting, added latency/cost) — this is agent-to-agent delegation within one orchestration session, not the async, cross-session, discoverable handoff-and-context-passing model beads/Orvex's cross-agent memory track wants (agents in *different* sessions/time discovering prior context). Don't copy CrewAI's synchronous-manager delegation pattern as the model for Orvex's handoff; it solves a different problem (in-crew task routing) than cross-session coordination.
- **No evidence of MCP-exposed memory or chat-platform (ChatGPT/Claude) memory integration** — CrewAI's memory is consumed via its own Python API/AMP dashboards, not exposed as MCP tools itself (CrewAI is an MCP *client*, consuming other servers' tools, not a memory MCP *server*). Orvex's mandate is explicitly to expose memory as MCP tools + CLI + chat-platform prompts — this is unbuilt territory CrewAI doesn't cover, reinforcing that Orvex's MCP-first memory surface is genuine differentiation, not something to copy.
- **No published decay/compaction benchmarks or scale SLAs** — CrewAI's memory story is architecturally interesting but operationally unproven at scale (own community threads show confusion and bugs); don't treat CrewAI's design as validated-at-scale prior art, treat it as a well-documented reference architecture to borrow patterns from, not a scale precedent to cite.
