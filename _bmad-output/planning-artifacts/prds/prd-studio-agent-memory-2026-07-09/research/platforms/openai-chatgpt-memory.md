# OpenAI ChatGPT Memory — Deep Dive (2025-2026)

## What it is
ChatGPT Memory is a consumer-facing personalization feature built into the ChatGPT product (web/mobile/desktop apps), not a standalone product or API. It has two user-facing layers: (1) **Saved Memories** — discrete, explicit facts the user tells ChatGPT to remember (e.g. "I'm vegetarian"), presented as an editable list the user can view, edit, or delete; and (2) **Reference Chat History** — an implicit, non-enumerable synthesis of insights drawn from the user's past conversations, injected into context automatically. Users can toggle each layer independently, or turn memory off entirely. [OpenAI: Memory and new controls for ChatGPT](https://openai.com/index/memory-and-new-controls-for-chatgpt/), [OpenAI Help: Memory FAQ](https://help.openai.com/en/articles/8590148-memory-faq)

## Behind it & traction
Built and operated by OpenAI as a core ChatGPT feature (not a separate company/product). Launched February 2024 (saved memories), expanded with Reference Chat History in April 2025, and most recently overhauled with "Dreaming V3" (rolled out June 4, 2026), an asynchronous background synthesis process. OpenAI reports (its own, unaudited, internal evals) that Dreaming V3 improved factual recall from 67.9% → 82.8%, preference adherence to 71.3%, and time-sensitive accuracy to 75.1% versus the prior system. A ~5x reduction in the compute cost of serving memory allowed OpenAI to extend it from Plus/Pro subscribers down to free-tier users in 2026. [MemX: ChatGPT Memory 2026 guide](https://memx.app/glossary/chatgpt-memory/), [9to5Mac: memory getting smarter, coming to free users](https://9to5mac.com/2026/06/04/openai-says-chatgpts-memory-feature-is-getting-smarter-and-coming-to-free-users/), [Neowin: major upgrade to ChatGPT memory](https://www.neowin.net/news/openai-is-rolling-out-a-major-upgrade-to-chatgpt-memory/)

No independent/third-party benchmark of the recall or preference-adherence numbers was found — treat OpenAI's percentages as marketing claims, not verified evidence.

## Architecture & data model
Publicly documented only at a conceptual level; OpenAI has not published a technical architecture paper or system design doc for memory.
- **Saved Memories**: a structured, discrete list of facts, editable per-item by the user.
- **Reference Chat History**: not a fixed list — described as "more like a dynamic summary" than structured facts, synthesized from many past conversations.
- **Dreaming**: an asynchronous background job that reads across past conversations and maintains a synthesized memory state, injected into the model's context (reportedly into the system prompt) at the start of each new chat — i.e., no per-turn retrieval call visible to the user; the synthesis happens out-of-band on a schedule/trigger unknown to us.
- **Scoping**: for ChatGPT Business/Enterprise (Team-tier), memory is scoped per-project so memories from one shared project don't leak into another.
- **Retention/deletion**: disabling Reference Chat History removes chat-history-derived memories within ~30 days; deletion logs are retained up to 30 days for "safety and debugging"; deleting an individual source conversation does not automatically purge memories already synthesized from it (because synthesis already happened asynchronously and decoupled from the source record).
[OpenAI Help: How does "Reference saved memories" work](https://help.openai.com/en/articles/11146739-how-does-reference-saved-memories-work) (indexed via search; direct fetch returned 403 — see Sources note), [MemX: ChatGPT Memory 2026 guide](https://memx.app/glossary/chatgpt-memory/)

unknown: exact storage substrate (vector DB vs. structured store vs. hybrid), embedding/retrieval mechanism, how "Dreaming" decides what to keep/discard, and whether memory synthesis is itself an LLM call (likely, but not confirmed by OpenAI).

## API & integration surface (MCP? SDK? CLI? chat-platform plugins?)
**Memory is not exposed via the OpenAI API, Assistants API, or any SDK.** It is exclusive to the ChatGPT consumer/business apps. Multiple developers on the OpenAI Developer Community have asked for API-level memory access; OpenAI has not shipped it as of this research (2025-2026). No MCP server, CLI, or chat-platform plugin surface exists for ChatGPT's own memory. Third-party vendors (e.g. Memobase, Mem0, Zep) have built API-accessible "memory" layers specifically to fill this gap for developers building on the OpenAI API. [OpenAI Community: Will Memory capabilities come to the API?](https://community.openai.com/t/will-memory-capabilities-come-to-the-api/934907), [Memobase: ChatGPT Memory Is Not Available via API—But You Can Build Your Own](https://www.memobase.io/blog/openai-memory)

## Memory lifecycle & sharing
- **Capture**: implicit, from conversation content (Reference Chat History) and explicit user statements ("remember that...") for Saved Memories.
- **Consolidate**: the "Dreaming" background process periodically re-reads across many past conversations and re-synthesizes a memory state (this is the consolidation/compaction step — OpenAI does not publish the schedule or trigger conditions).
- **Retrieve**: synthesized memory state is injected into context automatically at the start of each new chat; there is no user-visible retrieval query or ranking step for Reference Chat History (opaque to the user, unlike the enumerable Saved Memories list).
- **Decay/compact/summarize**: handled entirely inside Dreaming; no user control over decay policy beyond an all-or-nothing toggle and per-item delete of Saved Memories. Turning off Reference Chat History purges derived memories within ~30 days.
- **Multi-agent sharing / namespaces / permissions**: memory is scoped to a single ChatGPT user account; for Business/Enterprise, additionally scoped per shared project so memory does not cross projects. There is no concept of one agent/assistant handing off memory to another distinct agent — ChatGPT itself is the only "agent." No cross-user or team-wide shared memory namespace exists.
- **Cross-session token-cost story**: OpenAI does not publish token-cost figures; the entire point of the injected-summary-at-session-start design is to keep per-turn retrieval cost near zero (no live RAG call per user turn) at the expense of transparency and precision.
- **Chat-platform integration**: memory works only inside ChatGPT's own surfaces (web, mobile, desktop). No documented MCP tool, no cross-platform sync into Claude or other chat clients, no public API — i.e., it is a closed, single-vendor, single-app feature, the opposite of the multi-agent/MCP model Orvex is building.

## Scale & operational evidence
- OpenAI states the June 2026 "Dreaming V3" compute optimization cut serving cost ~5x, which is what enabled rollout to free-tier users — implying memory was previously too expensive to serve at free-tier scale. This is the clearest (indirect) operational/cost signal available.
- No published uptime, latency, storage-volume, or per-user memory-size figures were found.
- Regulatory/legal scrutiny is a real operational signal of scale and risk: a coalition of 42 US state attorneys general opened a subpoena/investigation into OpenAI in June 2026 covering data practices and model behavior; Florida sued OpenAI and Sam Altman (filed June 1, 2026); at least seven wrongful-death/psychological-harm lawsuits filed in California name ChatGPT's memory (specifically GPT-4o's expanded memory "stor[ing] and referenc[ing] user information across conversations to create deeper intimacy") as a contributing product-design factor. A federal judge separately compelled OpenAI to produce 20 million anonymized ChatGPT logs in a copyright discovery dispute (Jan 2026), evidencing the scale of retained conversational data underlying memory. ## Pricing & positioning
Memory itself is not separately priced — it is bundled into ChatGPT plans (Free, Plus, Pro, Business, Enterprise) with progressively stronger data-handling guarantees at higher/business tiers rather than more memory capability per se:
- **Free/Plus/Pro (consumer)**: memory on by default (opt-out available); conversations may be used for model training unless the user disables "Improve the model for everyone" in Data Controls.
- **ChatGPT Business** (renamed from "Team" in August 2025; $25/user/mo annual, $30/user/mo monthly): shared workspaces, admin console, and a contractual guarantee that business data (including anything used to form memory) is **not** used for model training by default; memory is scoped per-project.
- **ChatGPT Enterprise**: same no-training guarantee plus stronger admin/compliance controls.
Positioning: memory is marketed as a personalization/stickiness feature for retail users, and as a privacy-safe, project-scoped feature for business users — not as an extensible platform capability for third-party developers. [Data Controls FAQ](https://help.openai.com/en/articles/7730893-data-controls-faq), [How your data is used to improve model performance](https://openai.com/policies/how-your-data-is-used-to-improve-model-performance/), [ChatGPT Enterprise vs Business 2026 comparison](https://www.hungyichen.com/en/insights/chatgpt-enterprise-guide)

## STEAL — 3-5 concrete ideas for Orvex
1. **Two-tier transparency model**: Orvex's cross-agent memory should give humans an explicit, editable, enumerable list (like Saved Memories) for anything load-bearing, while allowing a richer implicit "working context" synthesis underneath for cheap continuity — but unlike OpenAI, Orvex should make the implicit layer inspectable too (a wiki-first audit trail), since opacity is exactly what's drawing lawsuits and regulatory scrutiny against OpenAI.
2. **Async consolidation ("Dreaming") as a cost pattern**: batch/background summarization of accumulated agent logs into a compact injected state at session start, rather than per-turn live retrieval, is a proven way to keep cross-session token cost near flat. Worth adopting for the beads-style cross-agent memory service to keep MCP tool calls cheap.
3. **Per-project/namespace scoping is table stakes**: OpenAI scopes memory per shared project for Business/Enterprise specifically to prevent cross-tenant/cross-project leakage. Orvex's per-customer, per-agent-fleet memory namespaces should be at least this strict from day one, not bolted on later.
4. **Contractual "not used for training" as a trust lever**: OpenAI's Business/Enterprise tiers sell the *absence* of training-on-your-data as a premium feature. Orvex should make an equivalent explicit, contractual guarantee for both the agent staging area and the cross-agent memory store — customers benchmarking against ChatGPT will expect it.
5. **Give users/admins a deletion SLA, and honor it structurally**: OpenAI documents a ~30-day purge window after disabling history, but litigation shows deletion doesn't cleanly cascade to already-synthesized derived memories. Orvex should design deletion/forget as a first-class operation that cascades through consolidated/derived memory artifacts, not just source records — this is a concrete gap Orvex can beat OpenAI on and should call out in the memory-service PRD's NFRs.

## AVOID / where Orvex differs
- **No API/MCP surface for the flagship memory feature**: OpenAI's memory is app-only, unreachable via API or MCP — precisely the anti-pattern for Orvex, whose memory service is designed API/MCP-first for agent-to-agent use. This is OpenAI's biggest weakness relative to what Orvex is building.
- **Single-agent, single-app scope**: ChatGPT memory has no notion of multiple distinct agents handing off work or discovering each other's context — it's one assistant remembering one user. Orvex's memory track is explicitly multi-agent coordination/handoff (beads-inspired), a different problem OpenAI hasn't addressed at all.
- **Opacity of the implicit layer invites exactly the harms now driving lawsuits and a 42-state AG probe** (deepened parasocial "intimacy," inability to fully audit or purge derived memory). Orvex should not replicate a black-box "dreaming" layer without an audit trail, given both the ethical and regulatory exposure.
- **No confirmed technical depth to imitate**: OpenAI has not published real architecture (storage engine, retrieval algorithm, decay policy) — there is nothing concrete to copy technically; only the product-level UX pattern (explicit list + implicit synthesis + toggles) is usefully transferable.
- **Deletion is not truly cascading**: don't copy the current gap where purging a conversation doesn't reliably purge memories already derived from it — treat this as a documented failure mode to design around, not a pattern to follow.
