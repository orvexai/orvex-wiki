# Prior Art Deep-Dive: Anthropic Claude Memory & Projects

## What it is
Two related but distinct surfaces from Anthropic:
1. **Claude.ai consumer/team "Memory"** — an automatic, RAG-backed feature that lets Claude recall context (role, working preferences, ongoing projects) across conversations without the user re-explaining themselves each time. Ships with **per-Project memory isolation**: each Project (Claude.ai's workspace-scoping construct for grouping chats + shared files/instructions) gets its own separate memory space and summary, distinct from the user's global "non-project chat" memory. An "Incognito chat" mode bypasses memory entirely. (support.claude.com, claude.com/blog/memory)
2. **API-level "memory tool"** (`memory_20250818`) — a client-side, file-based tool Claude Code/Agent SDK developers wire up themselves: Claude issues `view/create/str_replace/insert/delete/rename` operations against a virtual `/memories` directory, and the calling application executes them against storage it owns (disk, DB, cloud). This is the building block behind **Claude Managed Agents' persistent memory** (public beta, enterprise product), not the same system as the consumer feature. (platform.claude.com/docs memory-tool, edtechinnovationhub.com)

## Behind it & traction
- Anthropic (foundation-model lab + product company); consumer Memory rolled out to Team/Enterprise Sept 11 2025, then Pro/Max Oct 23 2025, then all users incl. Free ~Mar 2 2026. (claude.com/blog/memory, lumichats.com)
- Managed Agents persistent memory: public beta announced Apr 23 2026, targeted at enterprise/high-growth customers. Named early adopters: **Netflix, Rakuten, Wisedocs, Ando**. Rakuten reports "97% fewer first-pass errors at 27% lower cost and 34% lower latency"; Wisedocs reports 30% faster document-verification workflows. (edtechinnovationhub.com — these are Anthropic/customer-supplied marketing figures, not independently audited)
- Company-level scale (context for how load-bearing this feature is): Anthropic annualized revenue crossed $47B (May 2026, up from $14B in Feb 2026 and $1B Dec 2024); 1,000+ customers spending >$1M/yr (doubled since Feb 2026); ~70% of Fortune 100 use Claude; Deloitte deployment covers 470k+ employees/150 countries. Claude app MAU estimates vary wildly by source (12.48M per AICPB vs. 245M per Sensor Tower) — **treat both as third-party estimates, not Anthropic-disclosed numbers**. (venturebeat.com, getpanto.ai, businessofapps.com — unverified aggregator stats, flagged as marketing/estimate-grade evidence)

## Architecture & data model
- **Consumer Memory**: undocumented internals beyond "RAG technology to locate prior conversations" plus a periodically-refreshed synthesized summary. Memory synthesis updates **every 24 hours** from chat history; deleting a conversation removes it from synthesis within 24 hours. No published schema, vector store, or retention-window technical spec found. (support.claude.com)
- **Per-Project isolation** is the core data-partitioning primitive: each Project = its own memory space + dedicated summary; non-project ("standalone") chats share one separate global memory pool. Splitting work between Projects and plain chats fragments context by design — Anthropic states this explicitly as a limitation, not just a feature. (support.claude.com, xtrace.ai)
- **API memory tool**: architecture is deliberately a thin protocol, not a service — Claude requests file ops against a `/memories` path prefix; the *developer's* application maps that prefix onto real storage (per-user directory, DB keys, encrypted files, etc.). No default backend is provided server-side except optional SDK helpers (`BetaLocalFilesystemMemoryTool`) for local dev. Anthropic supplies no hosted memory store for API use — storage, encryption, expiry, and access control are 100% the caller's responsibility. Eligible for Zero Data Retention (ZDR) — under a ZDR agreement, Anthropic doesn't retain the payload after the API response returns. (platform.claude.com/docs/agents-and-tools/tool-use/memory-tool)
- **Managed Agents memory** (built on the same primitive, hosted): filesystem-mounted store; scoping is **organization- or user-level** with configurable read/write permissions; supports **concurrent multi-agent access** without overwrite collisions; full **audit trail** (which agent/session wrote each update) and **version control / rollback / redaction**; memory updates surface as session events in the Claude Console. This is the closest Anthropic analog to a true "cross-agent memory" product. (edtechinnovationhub.com)

## API & integration surface (MCP? SDK? CLI? chat-platform plugins?)
- **Not MCP-based.** The memory tool is a native Anthropic-provided Messages API tool type (`memory_20250818`), generally available, no beta header required, works on all Claude 4+ models.
- **SDK support**: Python, TypeScript, C#, Java ship helper abstractions (`BetaAbstractMemoryTool` subclass / `betaMemoryTool` / `BetaMemoryToolHandler`) plus a ready-made local-filesystem backend for Python/TS. Go, Ruby, PHP have no helper — devs run the tool-use loop and dispatch by hand.
- **Composable with other context-management primitives**: pairs with **context editing** (client-side clearing of stale tool results) and **compaction** (server-side auto-summarization near the context-window limit) — Anthropic's stated pattern for long-running agents is compaction (cheap, automatic) + memory (durable, selective) together.
- **Chat-platform integration**: consumer Memory is native to claude.ai (web/desktop/mobile), not exposed as a portable/exportable API a third-party chat platform (ChatGPT-style portal, Slack bot, etc.) could consume directly. There is no published MCP server that exposes Claude.ai's consumer memory to external agents — the memory tool is the only externally-programmable surface, and it is bring-your-own-storage. (platform.claude.com docs)

## Memory lifecycle & sharing
- **Capture**: consumer Memory — implicit, from ordinary chat content, no explicit "remember this" API. API memory tool — explicit: Claude decides to write files under `/memories` per a system-injected protocol ("ALWAYS VIEW YOUR MEMORY DIRECTORY BEFORE DOING ANYTHING ELSE... record status/progress/thoughts... ASSUME INTERRUPTION").
- **Consolidate**: consumer Memory resynthesizes its summary on a **24-hour cycle**; API memory tool has no automatic consolidation — Claude/the developer's prompting must instruct it to keep files "organized, coherent, non-cluttered," rename/delete stale files itself.
- **Retrieve**: consumer Memory — RAG search over past conversations, triggered by natural-language recall queries ("what did we discuss about X"), tool calls are shown to the user (Anthropic markets this as more transparent than ChatGPT's silent context injection — a competitor comparison claim from a third-party source, not verified independently). API memory tool — Claude explicitly `view`s the directory/files on each new session before acting.
- **Decay/compact/summarize**: consumer side undocumented beyond the 24h resync + user-triggered "reset" (permanent delete) or "pause" (freeze, keep existing). API side: no built-in decay — Anthropic's docs put the burden on the developer ("periodically delete memory files that haven't been accessed in a long time," cap file size, cap `view` output length) — i.e., **no platform-provided memory-expiry mechanism at all**, only guidance to build one.
- **Multi-agent sharing / namespaces / permissions**: consumer Memory has **no multi-agent sharing** concept — it's single-user, project-scoped. The Managed Agents product is where multi-agent sharing actually exists: org/user-level scoping, configurable read/write ACLs, concurrent-write safety, audit trail of which agent/session wrote what, rollback/redaction. This is the meaningful analog for Orvex's cross-agent memory track, not the consumer feature.
- **Cross-session token-cost story**: the memory tool's explicit selling point is avoiding "keeping everything in the context window" — Claude reads files back **on demand** ("just-in-time context retrieval") rather than the app re-injecting full history every turn, and this is explicitly positioned as the durable complement to compaction for cost/context-window control on long-running agent sessions.
- **Chat-platform integration (ChatGPT/Claude/portals, MCP)**: no MCP server exists for either memory surface as far as findable in official docs; integration is either native-UI-only (consumer) or roll-your-own-via-Messages-API (developer). This is a gap Orvex's MCP-first design does not have to replicate.

## Pricing & positioning
- Consumer Memory and Incognito: bundled at no separate charge across Free/Pro/Max/Team/Enterprise tiers (memory itself free; chat *search* of past conversations gated to paid plans only). (support.claude.com)
- API memory tool: no separate line-item price — billed as ordinary Messages API token usage (tool-result content counts as input tokens like any other tool). Storage cost is entirely the developer's own infrastructure, not an Anthropic-billed line.
- Long-context/base API pricing (context, not memory-specific): standard Sonnet $3/$15 per MTok in/out (intro $2/$10 through Aug 31 2026), Opus $5/$10→$10/$20+ input doubling and output $25→$37.50 once a single request's input exceeds 200K tokens — relevant because memory-file re-reads add to per-request input tokens and could cross these thresholds at scale. (platform.claude.com/docs/about-claude/pricing)
- Positioning: consumer Memory is framed as a trust/safety-forward feature ("fully optional, granular controls," explicit tool-call visibility, org-wide admin kill switch) — a competitive jab at silent-memory competitors. Managed Agents memory is framed as the enterprise-grade, auditable, governable version for production agent fleets — directly adjacent to what Orvex's cross-agent memory track needs to be.

## STEAL — 3–5 concrete ideas for Orvex
1. **Namespace memory by workspace/project, not just by user** — Anthropic's per-Project memory isolation (dedicated space + summary per Project, separate from the global pool) is a clean pattern for Orvex's per-customer/per-workspace partitioning; it also gives a natural place to put a "reset this workspace's memory" admin control mirroring Anthropic's Enterprise org-wide disable/delete.
2. **Just-in-time file-based retrieval over always-injecting full context** — the `/memories` view-on-demand model (vs. dumping everything into every prompt) is directly applicable to keeping agent context windows small in Orvex's cross-agent memory service; pair it with a compaction-like summarization layer the way Anthropic explicitly recommends pairing memory + compaction.
3. **Make writes explicit, structured, and self-organizing via protocol injection** — Anthropic auto-injects a "check memory before acting / record progress as you go / assume interruption" system instruction whenever the tool is present. Orvex's agent-log/handoff prompts (the "MCP tools + CLI + chat-platform instructions" surface) should bake in an equivalent mandatory protocol so agents don't need bespoke prompting per integration.
4. **Own the audit trail and rollback story from day one** — Managed Agents memory's audit log (who/what agent/what session wrote each update) + version rollback + redaction is exactly the governance layer a "librarian reviews staged changes" workflow needs; Orvex's staging-area PRD could borrow this shape directly (memory-store commit log analogous to the wiki staging store).
5. **Concurrent multi-agent write safety without overwrite collisions** is called out explicitly as a Managed Agents memory feature — worth stealing the requirement (not necessarily the implementation) for Orvex's cross-agent memory service, since multiple agents in a fleet will race to log/discover context simultaneously.

## AVOID / where Orvex differs
- **No hosted, opinionated memory service from Anthropic to copy wholesale** — the API memory tool is explicitly "bring your own storage," with expiry/size-capping/PII-stripping left as unimplemented guidance in the docs, not a built platform capability. Orvex's cross-agent memory service is meant to be exactly that missing hosted layer — don't under-scope it to "just a tool contract" the way Anthropic did.
- **Consumer Memory fragments context across Projects vs. non-project chats by design** — Anthropic bills this as a safety feature but their own docs/third-party reviews flag it as a real friction point (context split, must reconstruct). Orvex's cross-agent memory should avoid an equivalent hard wall between "staged wiki content" and "agent coordination memory" unless there's a deliberate security reason, since the PRDs describe these as complementary, not isolated, systems.
- **No cross-platform/MCP exposure of consumer memory** — Anthropic's memory lives inside claude.ai only; it is not portable to other chat surfaces. Orvex's explicit MCP-first, multi-chat-platform requirement (ChatGPT-style portals + Claude + CLI) is a genuine differentiator Anthropic has not built — don't assume there's a reference implementation to imitate here, it must be built from the API memory-tool primitive up.
- **24-hour synthesis latency** on consumer memory is a real lag between "user said X" and "memory reflects X" — unsuitable as-is for Orvex's stated goal of *same-session* handoff/coordination between agents; the Managed Agents (real-time file writes) pattern, not the consumer RAG-resync pattern, is the correct one to model.
- **Metrics discipline**: Rakuten/Wisedocs numbers and most MAU/revenue figures found are vendor-supplied or third-party-estimated, not independently audited — do not cite them in Orvex PRDs as validated benchmarks without that caveat.
