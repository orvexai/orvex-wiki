# Redis Agent Memory Server — Deep Dive

## What it is
An open-source memory layer for AI agents, built and maintained by Redis Inc., providing a two-tier memory model — **working memory** (session-scoped, temporary) and **long-term memory** (persistent, cross-session) — backed by Redis for storage, vector search, and full-text/keyword search. It exposes both a REST API and a Model Context Protocol (MCP) server so agents/chat clients can read and write memory directly. ([GitHub — redis/agent-memory-server](https://github.com/redis/agent-memory-server), [Docs home](https://redis.github.io/agent-memory-server/))

## Architecture & data model
- **Working memory**: session-scoped, holds live conversation messages and structured JSON data, supports TTL-based expiration and automatic summarization of long conversations. ([Docs — Memory Lifecycle](https://redis.github.io/agent-memory-server/memory-lifecycle/), [MCP docs](https://redis.github.io/agent-memory-server/mcp/))
- **Long-term memory**: persistent, cross-session store optimized for semantic (vector), keyword (full-text), and hybrid search, with deduplication and rich metadata (topics, entities, memory type). ([Docs — Long-term Memory](https://redis.github.io/agent-memory-server/long-term-memory/))
- Data model dimensions: namespaces, users, sessions, plus topics/entities/timestamps as filterable metadata — used for multi-tenant isolation. Exact episodic-vs-semantic memory-type taxonomy is not fully detailed in the fetched docs. ([Docs home](https://redis.github.io/agent-memory-server/))
- Storage backend is Redis itself (standalone or Redis Cluster), with a "pluggable vector database backend" abstraction mentioned in project description. ([GitHub](https://github.com/redis/agent-memory-server))
- Background task processing (compaction, deduplication/merging, search-index optimization, forgetting policies) runs via the **Docket** task scheduler as server-side background jobs. ([Docs — Memory Lifecycle](https://redis.github.io/agent-memory-server/memory-lifecycle/))

## API & integration surface (MCP? SDK? CLI? chat-platform plugins?)
- **REST API** for direct HTTP access.
- **MCP server** with two transports: stdio (default, used for Claude Desktop/local setups via `uvx`) and SSE (used e.g. by Cursor via an HTTP URL). No mention found of a streamable-HTTP MCP transport or of official ChatGPT/portal plugins. ([Docs — MCP Server](https://redis.github.io/agent-memory-server/mcp/))
- Key MCP tools: `search_long_term_memory`, `get_long_term_memory`, `memory_prompt`, `set_working_memory`, `create_long_term_memories`, `edit_long_term_memory`, `delete_long_term_memories`. ([Docs — MCP Server](https://redis.github.io/agent-memory-server/mcp/))
- **Python SDK** (`agent-memory-client`) with LangChain integration and "automatic tool conversion" for framework compatibility. ([GitHub](https://github.com/redis/agent-memory-server))
- Multi-provider LLM support for the extraction/summarization pipeline: OpenAI, Anthropic, AWS Bedrock, Ollama, Azure, Gemini — via LiteLLM. ([GitHub](https://github.com/redis/agent-memory-server))
- No CLI tool was surfaced in the fetched pages; no explicit ChatGPT-portal or Slack/Teams chat-platform plugin was found — integrations documented are Claude Desktop, Cursor, and demo repos for Alexa/Google ADK/OpenClaw. ([redis-developer org repos](https://github.com/redis-developer/adk-redis))

## Memory lifecycle & sharing
- **Capture**: three patterns — (1) automatic background LLM extraction of facts from conversation messages, (2) LLM-optimized batch storage where the agent pre-identifies important info alongside a working-memory update, (3) direct long-term-memory API writes. Configurable extraction *strategies*: discrete, summary, preferences, custom. ([Docs — Memory Lifecycle](https://redis.github.io/agent-memory-server/memory-lifecycle/), [GitHub](https://github.com/redis/agent-memory-server))
- **Consolidate**: working memory acts as a staging area that is automatically promoted to long-term storage; background compaction jobs perform deduplication/merging of long-term memories and optimize the search index. ([Docs — Memory Lifecycle](https://redis.github.io/agent-memory-server/memory-lifecycle/))
- **Retrieve**: semantic (vector), keyword (full-text), and hybrid search, with filters by user/session/time/topic/entity; ranking incorporates access-pattern/recency boosting for frequently accessed memories. ([Docs — Long-term Memory](https://redis.github.io/agent-memory-server/long-term-memory/), [Docs — Memory Lifecycle](https://redis.github.io/agent-memory-server/memory-lifecycle/))
- **Decay/compact/summarize**: explicit "forgetting policies" — age-based deletion (`FORGETTING_MAX_AGE_DAYS`), inactivity-based removal (`FORGETTING_MAX_INACTIVE_DAYS`), a combined threshold mode, and budget-based retention (`FORGETTING_BUDGET_KEEP_TOP_N`) — all run as server-controlled background processes via Docket scheduling. Working-memory conversations are auto-summarized/trimmed as they grow. ([Docs — Memory Lifecycle](https://redis.github.io/agent-memory-server/memory-lifecycle/))
- **Multi-agent sharing, namespaces, permissions**: namespace + user + session dimensions provide multi-tenant isolation; auth is OAuth2/JWT or token-based, with `DISABLE_AUTH=true` recommended only for development. The exact cross-agent sharing/permission model (e.g., can Agent B read Agent A's memories within the same tenant, and how that's authorized) is **unknown** — not detailed in the fetched documentation. ([Docs home](https://redis.github.io/agent-memory-server/), [MCP docs](https://redis.github.io/agent-memory-server/mcp/))
- **Cross-session token-cost story**: the pitch is that automatic extraction/summarization/promotion means agents don't need custom promotion/pruning logic and retrieval returns "most relevant context" instead of full history — implying lower per-session token cost — but no quantified token-savings benchmark was found in the fetched sources. **Unknown** (marketing claim, not evidenced with numbers).

## STEAL — 3-5 concrete ideas for Orvex
1. **Named forgetting policies as first-class config, not code.** The `FORGETTING_MAX_AGE_DAYS` / `FORGETTING_MAX_INACTIVE_DAYS` / `FORGETTING_BUDGET_KEEP_TOP_N` knobs are a clean, ops-tunable decay model per tenant — Orvex's cross-agent memory service should expose equivalent per-customer-tweakable retention policies rather than baking decay logic into agent prompts. ([Memory Lifecycle docs](https://redis.github.io/agent-memory-server/memory-lifecycle/))
2. **Two-tier working/long-term split with automatic promotion.** Session-scoped working memory (cheap, TTL'd, auto-summarized) that gets promoted to durable long-term memory only when it clears an extraction bar is a good pattern for keeping Orvex's Dolt-backed store from being flooded by every agent turn.
3. **Multiple extraction strategies (discrete/summary/preferences/custom) as a pluggable pipeline** rather than one fixed summarizer — lets Orvex's librarian-equivalent for cross-agent memory tune extraction per customer/agent type the same way the librarian prompt is per-customer-tweakable in the staging-area PRD.
4. **MCP tool surface shape** (`search_long_term_memory`, `set_working_memory`, `create/edit/delete_long_term_memories`, `memory_prompt`) is a reasonable starting checklist for Orvex's own MCP tool set for cross-agent memory — it separates read/search from write/mutate cleanly, which maps well onto Orvex's "agents never write directly" staging philosophy if adapted (agents propose memory writes, not commit them directly).
5. **Namespace + user + session as the isolation axes** (rather than inventing a bespoke multi-tenancy model) is a simple, provable multi-tenant boundary Orvex could mirror for per-customer/per-agent isolation in the memory service.

## AVOID / where Orvex differs
- **No visible cross-agent sharing/handoff or permission model** in the fetched docs — this is the core of Orvex's beads-inspired cross-agent memory (agents discovering each other's context, handing off work). Redis's product looks built for one-agent-remembers-a-user, not many-agents-coordinate-with-each-other; Orvex should not assume this gap is solved by adopting Redis's model — it needs to be designed fresh.
- **Direct-write model conflicts with Orvex's staging philosophy.** Redis's MCP tools let an agent call `create_long_term_memories`/`edit_long_term_memory` directly against durable storage — this is the opposite of Orvex's wiki staging-area principle (agents never write the customer wiki directly; a librarian reviews/merges). If Orvex's cross-agent memory service borrows Redis's tool shapes, writes should still route through a review/merge boundary, not straight to canonical memory, to stay consistent with Orvex's own no-direct-write architectural stance.
- **Dolt vs Redis storage tradeoff is unresolved and undocumented on Redis's side.** Redis's value prop leans on vector/full-text search speed and TTL semantics, not on versioned/diffable history. If Orvex's Dolt choice is meant to give branchable/auditable memory history (git-like diffs across agent memory changes), that is a capability Redis's model does not appear to offer at all — no versioning/branching/audit-trail features were found in the docs. This is the sharpest architectural fork between the two approaches and should be validated explicitly (fetch Dolt's own docs) before deciding.
- **No hard evidence of production scale** (no latency/throughput numbers specific to the memory server) — don't cite Redis's general case studies as proof this specific product scales; treat scale claims as unverified/marketing until first-party benchmarks are found.
- **Pricing is opaque** — no standalone SKU/pricing was found for the memory server; assume self-hosted OSS unless a future Redis Cloud "Agent Memory" add-on price is confirmed. Do not use Redis Cloud general pricing as a proxy for this product's cost.
