# Temporal — Durable Execution (prior-art deep dive for Orvex Studio memory/staging track)

## What it is
Temporal is an open-source durable execution platform (plus a managed "Temporal Cloud") that lets developers write ordinary application code — orchestration logic ("Workflows") that call out to unreliable operations ("Activities": API calls, LLM calls, DB writes) — and have the platform durably persist every step so execution survives crashes, restarts, and long pauses (minutes to years). Core mechanism: an append-only **Event History** per Workflow Execution that is replayed deterministically to reconstruct in-memory state after any failure, instead of redoing work. [Why Temporal](https://docs.temporal.io/evaluate/why-temporal)

As of 2025-2026 Temporal has explicitly repositioned around AI agents: durable execution is marketed as the reliability substrate under agent frameworks (OpenAI Agents SDK, Vercel AI SDK, MCP), not just microservice orchestration. [Temporal.io homepage](https://temporal.io/) [OpenAI Agents SDK integration blog](https://temporal.io/blog/announcing-openai-agents-sdk-integration)

## Architecture & data model
- **Workflow**: deterministic orchestration code (the "control flow" — loops, branching, parallelism, essentially `main`). Must be replay-safe/deterministic.
- **Activity**: the unit that performs non-deterministic/unreliable work (LLM calls, tool execution, API/DB calls). Activities execute outside the replay path and are automatically retried; their results are what gets checkpointed. [OpenAI SDK integration blog](https://temporal.io/blog/announcing-openai-agents-sdk-integration)
- **Event History**: complete, durable, append-only log of everything that happened in a Workflow Execution (Activity invoked, arguments, completion, return value). On recovery, Temporal restarts the Workflow function from the top and replays History step by step, using recorded events to fast-forward to the exact prior state rather than re-executing side effects. [Event History docs](https://docs.temporal.io/encyclopedia/event-history)
- **Signals / Queries / Updates** (message passing into a running Workflow): Signal mutates a Workflow asynchronously (fire-and-forget, does append to History); Update mutates and can return a response + do validation (also appends to History); Query reads Workflow state without appending to History and can run even against completed Workflows. [Workflow message passing docs](https://docs.temporal.io/encyclopedia/workflow-message-passing)
- **Saga / compensation**: first-class support for defining compensating ("undo") actions per step, run in reverse order on failure — directly aimed at the "tool call succeeded, agent then crashed" class of bug. [Saga pattern blog](https://temporal.io/blog/mastering-saga-patterns-for-distributed-transactions-in-microservices)
- **Hard scale limits that matter for a 100-doc session**: Event History is capped at **51,200 events / 50MB**, with a warning threshold at 10,240 events / 10MB; server/worker performance visibly degrades well before the hard cap (~10K events) because replay downloads and reprocesses the full history on each worker. **Continue-As-New** lets a Workflow atomically restart itself with a fresh History under the same Workflow ID to work around this — but the state snapshot passed through Continue-As-New is capped at **2MB**; larger state must be stored externally (S3/DB) and passed by reference. [Workflow Execution limits](https://docs.temporal.io/workflow-execution/limits) [Very long-running workflows blog](https://temporal.io/blog/very-long-running-workflows) [ContinueAsNew community thread](https://community.temporal.io/t/continue-as-new-when-reaching-5-000-events-limit/4260)
- Backing store: Temporal Server persists Workflow/Event state to a durable DB (Postgres, MySQL, or Cassandra self-hosted; managed in Temporal Cloud). [Saga/system-design overview](https://medium.com/@sanilkhurana7/system-design-series-the-story-and-present-of-durable-execution-and-how-to-use-it-in-your-52509b94d01e)

## API & integration surface (MCP? SDK? CLI? chat-platform plugins?)
- **SDKs**: Go, Java, PHP, Python, TypeScript, .NET — Workflows/Activities are written as ordinary code in-language, not a DSL. [Temporal SDKs docs](https://docs.temporal.io/encyclopedia/temporal-sdks)
- **Agent-framework integrations**: official OpenAI Agents SDK integration (Python, GA) — wraps every agent invocation as a Temporal Activity and orchestration as a Workflow transparently; ships `activity_as_tool` to auto-generate OpenAI tool schemas from Activity function signatures; also a Vercel AI SDK integration ("durable agents with Temporal and AI SDK"), and Pydantic AI's durable-execution integration point. [OpenAI SDK integration blog](https://temporal.io/blog/announcing-openai-agents-sdk-integration) [Vercel AI SDK blog](https://temporal.io/blog/building-durable-agents-with-temporal-and-ai-sdk-by-vercel) [Pydantic Temporal docs](https://pydantic.dev/docs/ai/integrations/durable_execution/temporal/)
- **MCP**: no first-party "Temporal MCP server" product from Temporal itself found. Temporal's own pattern (and the third-party `mcp-agent` project) is the inverse: **wrap each MCP tool call as a Temporal Workflow/Activity** (`MCP Tool → Temporal Workflow → Activities`) so tool execution gets retries, durability, and long-lived state, rather than Temporal exposing MCP tools to consume. Third-party `mcp-agent` ships a CLI (`uvx mcp-agent init`, `uvx mcp-agent deploy`) that scaffolds/deploys MCP+Temporal projects and can target "mcp-agent Cloud" (not Temporal's own product). [Durable MCP blog](https://temporal.io/blog/durable-mcp-how-to-give-agentic-systems-superpowers) [mcp-agent Temporal integration docs](https://docs.mcp-agent.com/advanced/temporal)
- **CLI / Web UI**: `temporal` CLI and a Web UI for inspecting Workflow state, History, and doing "time-travel debugging" (replay to any point). [Why Temporal](https://docs.temporal.io/evaluate/why-temporal)
- **Chat-platform plugins (ChatGPT/Claude/portal-native)**: unknown / not found. No evidence of an official ChatGPT plugin, Claude connector, or portal-embeddable chat widget from Temporal.

## Memory lifecycle & sharing
Temporal is **not** a memory/knowledge product — it is a durability substrate. Mapping the requested lifecycle onto what evidence supports:
- **Capture**: implicit — every Activity call/result and every Signal/Update is appended to Event History automatically as a side effect of execution, not via an explicit "remember this" API.
- **Consolidate**: **Continue-As-New** is the closest analogue to consolidation/compaction — it closes out an Event History and starts a fresh one, carrying forward only what the Workflow author explicitly packs into the new run's input (capped at 2MB, larger state must be externalized by reference). This is manual/architectural, not automatic summarization. [Very long-running workflows blog](https://temporal.io/blog/very-long-running-workflows)
- **Retrieve**: **Query** (no History append, can read even completed Workflows) and the Web UI/CLI history browse are the retrieval paths. There is no semantic/vector search — retrieval is by Workflow ID / structured Query handler, not by meaning.
- **Decay/compact/summarize**: no built-in summarization or decay. Decay is only external: you must bring your own logic to decide what survives across a Continue-As-New boundary.
- **Multi-agent sharing, namespaces, permissions**: Temporal has **Namespaces** as a first-class multi-tenancy/isolation boundary (referenced throughout docs, e.g. pricing/limits pages), and cross-Workflow interaction happens via Signals/Queries/Updates or Child Workflows — i.e., sharing is explicit message-passing between Workflow Executions, not a shared memory pool. No evidence of fine-grained ACL/permission primitives beyond Namespace-level isolation and standard cloud IAM.
- **Cross-session token-cost story**: the one clearly evidenced win — because Activity results are checkpointed, a crash/retry resumes from the last completed Activity rather than re-calling the LLM, explicitly marketed as avoiding "no repeated API calls, no wasted tokens, no lost progress." This is a resume-after-crash story, not a context-compaction-to-save-tokens story. [OpenAI SDK integration blog](https://temporal.io/blog/announcing-openai-agents-sdk-integration)
- **Chat-platform integration**: unknown (see above) — no evidence of native ChatGPT/Claude/portal or MCP-server-exposing-memory-tools product from Temporal itself.

## Scale & operational evidence
- Documented hard limits: 51,200 events / 50MB per Workflow Execution (warn at 10,240/10MB); visible performance degradation approaching ~10K events because replay pulls the full History to the worker. [Workflow Execution limits](https://docs.temporal.io/workflow-execution/limits)
- Continue-As-New state-passing capped at 2MB blob size; the documented workaround is store-by-reference (S3/DB), i.e., Temporal itself does not scale unbounded state — you re-introduce an external store for anything large. [ContinueAsNew community thread](https://community.temporal.io/t/continue-as-new-when-reaching-5-000-events-limit/4260)
- Customer names cited at scale (OpenAI, Replit, Lovable, Snap, Netflix, JPMorgan Chase) are marketing-page claims from Temporal's own funding announcement, not independently sourced case studies with numbers (no public throughput/SLA figures found for the AI-agent workloads specifically). [Temporal news](https://temporal.io/news/temporal-raises-300M-to-make-agentic-ai-real-for-companies)
- Temporal Cloud publishes SLAs: 99.9% standard, 99.99% tier available at a 2x Actions/storage consumption multiplier for replicated (multi-region/multi-cloud) namespaces. [Temporal pricing](https://temporal.io/pricing)

## STEAL — 3-5 concrete ideas for Orvex
1. **Treat "propose 100 doc changes in one session" as a Workflow, not a request loop.** Each proposed change (add/edit/replace/delete) becomes a durable Activity whose result is checkpointed; if the agent session or a librarian-side worker crashes mid-batch, resume from the last completed proposal instead of re-deriving or re-prompting the LLM for the whole batch. This directly targets the stated "survive a single chat session proposing updates to 100 documents" requirement.
2. **Event History as the audit/replay log for staged changes.** Adopt an append-only, replayable event log per staging session (mirroring Event History) so a human reviewer (or the librarian agent) can "time-travel" through exactly what was proposed, in what order, with what inputs — valuable for the librarian's review/merge step and for post-hoc dispute resolution ("why did the wiki change this way?").
3. **Saga/compensation pattern for librarian merges.** When the librarian agent merges/beautifies staged content into the live wiki and a later step fails (e.g., page 47 of 100 conflicts), define compensating actions to roll back or quarantine the partial batch rather than leaving the wiki half-merged — borrow the saga pattern directly rather than reinventing partial-failure handling.
4. **Query vs. Signal/Update distinction for the staging API.** Separate "read current proposal state" (cheap, no log growth) from "mutate/add a proposal" (durable, logged) the way Temporal separates Query from Signal/Update — keeps a chatty agent polling proposal status from bloating the audit log.
5. **Plan for compaction from day one, don't bolt it on later.** Temporal's hard-won lesson is that unbounded History causes real, documented performance cliffs (~10K events) and that its own compaction primitive (Continue-As-New) is capped at 2MB and still requires external storage for large payloads. Orvex's cross-agent memory service should design its own consolidate/decay step (and an explicit size cap plus store-by-reference mechanism) up front rather than discovering the cliff in production.

## AVOID / where Orvex differs
- **Temporal is not a memory or knowledge-retrieval product.** It has no semantic search, no summarization/decay, no concept of "memory" beyond an execution log — Orvex's cross-agent memory service needs to build retrieval/consolidation semantics itself; Temporal only offers the durability substrate underneath, not the memory model on top.
- **No chat-platform or MCP-memory-tool surface.** There's no evidence Temporal exposes anything ChatGPT/Claude/portal agents could call as an MCP memory tool out of the box — Orvex would need to build that layer entirely; Temporal at most inspires the underlying persistence pattern (wrap tool calls as Activities/Workflows).
- **Determinism constraints are a real tax.** Workflow code must be deterministic/replay-safe (no direct randomness, wall-clock reads, or non-deterministic branching inside Workflow code — those must be pushed into Activities). This is a meaningful authoring constraint that Orvex's staging/memory service should not blindly inherit unless it also adopts full replay-based recovery; a simpler durable-queue/outbox pattern (which Orvex already has per ENG-1600 trace-context work) may suffice without full Workflow determinism discipline.
- **Sharing model is message-passing, not shared state.** Cross-agent "sharing" in Temporal is Signals/Queries/Child Workflows between isolated Executions, not a queryable shared memory pool — if Orvex wants agents to *discover* each other's context (per the cross-agent memory PRD goal), that needs an actual searchable store (Postgres/Dolt with indexes), which Temporal does not provide.
- **Cost model doesn't obviously fit a high-volume, low-value-per-event memory log.** Temporal Cloud bills per "Action" (~$0.00005 each); a memory service logging continuous fine-grained agent activity at scale could rack up Action costs fast compared to a plain Postgres table — worth a cost model comparison before considering Temporal Cloud as literal infrastructure for the memory track (vs. just borrowing its patterns).
