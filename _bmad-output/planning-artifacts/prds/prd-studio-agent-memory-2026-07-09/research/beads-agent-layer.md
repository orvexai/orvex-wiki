## 1. The Session-Start Ritual (verbatim)

Beads has **no single** session-start text — the output is templated and branches on git state, config, and mode. All variants share one purpose: re-anchor the agent in the issue graph and forbid parallel tracking systems.

### 1.1 `bd prime` — MCP mode (short, ~50–200 tokens)

Generated by `outputMCPContext()` in `/home/daniel/repos/beads/cmd/bd/prime.go:522-578`. Default (conservative, non-ephemeral, has-remote) branch:

```
[bd prime] If this output is truncated by your host, read the full persisted hook output before continuing; it may contain project memories and session rules not visible in the preview.

# Beads Issue Tracker Active

# 🚨 SESSION CLOSE PROTOCOL 🚨

Before saying "done": bd close <completed-ids>; run checks. Then follow the active profile — conservative reports handoff; team-maintainer may commit/sync/push when explicitly enabled.

## Core Rules
- **Default**: Use beads for ALL task tracking (`bd create`, `bd ready`, `bd close`)
- **Prohibited**: Do NOT use TodoWrite, TaskCreate, or markdown files for task tracking
- **Workflow**: Create beads issue BEFORE writing code, mark in_progress when starting
- **Memory**: Use `bd remember` for persistent knowledge. Do NOT use MEMORY.md files.
- Persistence you don't need beats lost context
- Default: do not commit, push, or run dolt remote sync without explicit authority. Team-maintainer behavior is opt-in and still subordinate to user/orchestrator instructions.

Start: Check `ready` tool for available work.
```

### 1.2 `bd prime` — CLI mode (full, ~1–2k tokens)

Generated by `outputCLIContext()`, same file, lines 581-806. Header (constant across branches):

```
[bd prime] If this output is truncated by your host, read the full persisted hook output before continuing; it may contain project memories and session rules not visible in the preview.

# Beads Workflow Context

> **Context Recovery**: Run `bd prime` after compaction, clear, or new session
> Hooks auto-call this in Claude Code and Codex when a beads workspace is resolved
```

...followed by persistent memories (if any — see §8.4), then the same `🚨 SESSION CLOSE PROTOCOL 🚨` block (checklist form — see §2), Core Rules, and a full command reference (Finding Work / Creating & Updating / Dependencies & Blocking / Sync & Collaboration / Project Health / Quality Tools / Lifecycle & Hygiene / Structured Workflows / Common Workflows).

**Branching logic** (both MCP and CLI modes share this decision tree, `prime.go:522-694`): the text changes based on `stealthMode || localOnly` (no git ops), `isEphemeralBranch()` (no upstream — conservative, no push), `no-push` config, and `agent.profile == team-maintainer`. Each branch swaps in a different `closeProtocol`/`profileRule` string — see §11 for the full profile matrix.

## 2. The Session-End / "Landing the Plane" Ritual (verbatim)

There is no literal string "land the plane" anywhere in code or templates — it's the name Steve Yegge gives this mechanic in his own blog ("Beads Blows Up — the 'land the plane' protocol, session cleanup" — cited from `/home/daniel/repos/beads/ARTICLES.md:11`). Inside the repo it is implemented as two distinct artifacts:

### 2.1 Runtime version — the `bd prime` "SESSION CLOSE PROTOCOL" checklist (CLI mode, conservative/default branch)

`/home/daniel/repos/beads/cmd/bd/prime.go:672-694`:

```
🚨 SESSION CLOSE PROTOCOL 🚨

**CRITICAL**: Before saying "done" or "complete", you MUST run this checklist:

```
[ ] 1. bd close <id1> <id2> ...   (close completed issues)
[ ] 2. run quality gates        (tests, linters, builds when relevant)
[ ] 3. git status               (check what changed)
[ ] 4. follow active profile    (conservative: report handoff; team-maintainer: commit/sync/push if enabled)
```

**Policy:** Conservative is the default. Commit, sync, or push only when the active user, orchestrator, or repository profile grants that authority.
```

This is gated explicitly **on the agent's own intent to declare completion** — "Before saying 'done' or 'complete'" — which is a deliberate hook into the model's natural end-of-turn language, not just a time-based trigger.

### 2.2 Static version — the "Session Completion" block baked into `AGENTS.md`/`CLAUDE.md`

Installed by `bd init` / `bd setup <tool>` via `internal/templates/agents/defaults/beads-section.md` and `beads-section-minimal.md` (identical block, both profiles):

```
## Session Completion

This protocol applies when ending a Beads implementation workflow. It is subordinate to explicit user, repository, and orchestrator instructions.

1. **File issues for remaining work** - Create beads for anything that needs follow-up
2. **Run quality gates** (if code changed) - Tests, linters, builds
3. **Update issue status** - Close finished work, update in-progress items
4. **Handle git/sync by active profile**:
   ```bash
   # Conservative/minimal/default: report status and proposed commands; wait for approval.
   git status

   # Team-maintainer opt-in only, unless current instructions say otherwise:
   git pull --rebase
   bd dolt push
   git push
   git status
   ```
5. **Hand off** - Summarize changes, validation, issue status, and any blocked sync/commit/push step

**Critical rules:**
- Explicit user or orchestrator instructions override this Beads block.
- Do not commit or push without clear authority from the active profile or the current user request.
- If a required sync or push is blocked, stop and report the exact command and error.
```
(`/home/daniel/repos/beads/internal/templates/agents/defaults/beads-section.md:106-131`)

Both rituals converge on the same three moves: **close/update issues → run quality gates → report a git handoff gated by policy profile, never auto-commit by default.**

## 3. Plugin Layer — `/home/daniel/repos/beads/plugins/beads/`

### 3.1 Manifests

Three near-identical plugin manifests (Claude, Codex, Copilot) share one skill tree (`plugins/beads/skills/`), per `plugins/beads/README.md`:
- `.claude-plugin/plugin.json` — declares `skills: "./skills/"`, `commands: "./skills/beads/commands/"`, and registers **both** `SessionStart` and `PreCompact` hooks running bare `bd prime` (v1.1.0, author Steve Yegge, keywords include `"agent-memory"`).
- `.codex-plugin/plugin.json` — same metadata, points `hooks` at `./hooks/hooks.json`, and additionally carries an `interface` block with `defaultPrompt` examples: `"Use $beads to show ready work in this repo."`, `"Use $beads to claim the top ready task and summarize it."`, `"Use $beads to create a discovered-from follow-up."` — i.e. the marketing/onboarding copy itself teaches ready→claim→discovered-from.
- `.codex-plugin/hooks/hooks.json` — the 4-event Codex lifecycle (`SessionStart`/`PreCompact`/`PostCompact`/`UserPromptSubmit`), each with a `statusMessage` shown to the user ("Loading Beads context", "Checking Beads context", "Scheduling Beads context refresh", "Refreshing Beads context").
- `.copilot-plugin/plugin.json` — embedded via `go:embed` in `copilot_manifest.go` (so it ships inside the `bd` binary itself, not just as a file); functionally identical hook wiring to the Claude manifest (`SessionStart`+`PreCompact` → bare `bd prime`).

**Audit note (drift):** `bd setup claude` (`cmd/bd/setup/claude.go`) explicitly *migrates away from* the bare-`bd prime`/`PreCompact` pattern ("Claude Code's current context-injection contract uses the SessionStart JSON envelope; SessionStart also fires after compaction with source=compact, so PreCompact no longer needs a bd prime hook" — comment at line 141-143) and actively strips those legacy hook entries. The checked-in `.claude-plugin/plugin.json` and `.copilot-plugin/plugin.json` have **not** been updated to match — they still register the old bare/`PreCompact` shape. This is a genuine version-skew finding for the hosted-service audit: plugin-installed users and CLI-`bd setup`-installed users are on two different hook contracts.

### 3.2 `agents/task-agent.md` — the autonomous subagent prompt

`/home/daniel/repos/beads/plugins/beads/agents/task-agent.md` is a full 6-step loop prompt ("Find Ready Work → Claim the Task → Execute → Track Discoveries → Complete → Continue") built entirely on MCP tool names (`ready`, `show`, `claim`, `update`, `create`, `dep`, `close`, `blocked`, `stats`). Its explicit rule: *"Always claim before working (MCP: `claim`; CLI: `--claim`) and close when done"* and *"Link discovered work with `discovered-from` dependencies."*

### 3.3 `skills/beads/SKILL.md` (the skill Claude Code loads)

Frontmatter declares `allowed-tools: "Read,Bash(bd:*)"`, `version: "0.60.0"`, `compatible-with: [claude-code, codex]`. Body is deliberately thin (~500 words per ADR-0001, see §3.5) and is structured as: a `bd` vs `TodoWrite` decision test ("Will I need this context in 2 weeks?"), a 6-step **Session Protocol** (`bd ready` → `bd show` → `bd update --claim` → add notes → `bd close --reason` → `bd dolt push`), an error table, and a resource index for progressive disclosure. Critically it tells the agent to treat `bd prime` as the live source of truth rather than duplicating CLI docs: *"Run `bd prime` for AI-optimized workflow context (auto-loaded by hooks)."*

### 3.4 `CLAUDE.md` / `README.md` (maintainer-facing, not agent-facing)

`skills/beads/CLAUDE.md` is a **maintenance guide for humans editing the skill**, not agent-facing prompt text — it enforces the DRY principle: *"NEVER duplicate CLI documentation in SKILL.md or resources"* and gives a checklist for keeping the skill in sync with `bd prime`. `README.md` documents installation (`cp -r beads ~/.claude/skills/`) and the full resource file tree.

### 3.5 ADR-0001: `bd prime` as source of truth

`/home/daniel/repos/beads/plugins/beads/skills/beads/adr/0001-bd-prime-as-source-of-truth.md` records the design decision itself: prior to this, CLI reference was duplicated 2-3× (SKILL.md inline + `CLI_REFERENCE.md` + scattered examples), costing "~3,000+ tokens loaded even for simple operations" and creating "drift risk." The fix: SKILL.md shrank from 3,306→~500 words and now only carries decision frameworks, deferring all CLI syntax to `bd prime`/`bd --help`. This is the architectural root of why `bd prime`'s *output text* (not static docs) is the canonical session-start ritual.

### 3.6 `skills/beads/commands/*.md` — slash-command catalogue (skimmed, 27 files)

All follow one pattern: YAML frontmatter (`description`, `argument-hint`), a one-line purpose statement, then delegate to an MCP tool or `bd` subcommand. Notable ones beyond basic CRUD:

## 4. MCP Server — `/home/daniel/repos/beads/integrations/beads-mcp/`

### 4.1 How it talks to `bd`: subprocess, not a daemon

Confirmed by direct read of `bd_client.py`: every operation shells out via `asyncio.create_subprocess_exec(self.bd_path, *args, ..., stdin=asyncio.subprocess.DEVNULL, ...)` and parses `--json` stdout (`/home/daniel/repos/beads/integrations/beads-mcp/src/beads_mcp/bd_client.py:300-366`). `create_bd_client(prefer_daemon: bool = False, ...)` still accepts a `prefer_daemon` parameter but the docstring says it is **"Deprecated, ignored. Kept for API compatibility"** (`bd_client.py:961-994`) — there is no daemon client implementation left; `BdClient = BdCliClient` is the only concrete class. `stdin` is explicitly set to `DEVNULL` on every subprocess call specifically to prevent the child `bd` process from inheriting the MCP server's own stdio and corrupting the protocol stream. Version compatibility is enforced once per workspace: `_check_version()` parses `bd version` output and raises `BdVersionError` if `< 0.9.0`.

### 4.3 Multi-project / workspace routing

Three coexisting mechanisms (`README.md`, `CONTEXT_MANAGEMENT.md`, `workspace.py`, `tools.py`):
1. **Per-request `workspace_root` param** (recommended) — every tool accepts it; resolved through `contextvars.ContextVar` (`current_workspace`) set/reset around each call by the `@with_workspace` decorator (`server.py:175-205`), so concurrent calls from different projects never cross-contaminate — **except** the code explicitly warns tool implementations must never call `asyncio.create_task()` internally, because `ContextVar` doesn't propagate into spawned tasks (`README.md:189-195`).
2. **`set_context`/`context(action='set')` legacy path** — persists into a module-level dict `_workspace_context` (since raw `os.environ` doesn't reliably persist across MCP tool calls) plus mirrors into `os.environ` for backward compat.
3. **Auto-discovery fallback** — if neither is set, `_find_beads_db_in_tree()` walks up from CWD exactly like the Go CLI, including following `.beads/redirect` files (agent/worker dirs sharing a canonical DB) and stopping at git worktree boundaries so a linked worktree doesn't silently fall through to the main repo's `.beads/` unless it deliberately has none (`tools.py:146-243`, mirrored in `workspace.py:46-59`).

A single MCP server instance is recommended for *all* projects — it maintains a **connection pool keyed by canonicalized workspace path** (symlinks resolved via `realpath`, git-toplevel detected), performs a health-check-then-reconnect-with-backoff dance (`_health_check_client`/`_reconnect_client`, 3 attempts, 0.1/0.2/0.4s backoff) before reusing a pooled client, and never evicts (LRU eviction explicitly deferred to a "Phase 2"). Per-project MCP server instances via `BEADS_WORKING_DIR` env are explicitly discouraged: *"⚠️ Problem: AI may select the wrong MCP server for your workspace"* (`README.md:132`).

### 4.4 Context-budget / compaction logic (`CONTEXT_ENGINEERING.md`)

Three independent levers, claimed ~80-90% aggregate context reduction:
1. **Lazy schema loading** — `discover_tools()`/`get_tool_info()` (~500 bytes vs "10-50k tokens" for full schemas up front).
2. **`IssueMinimal`/`BriefIssue`/`BriefDep` models** — list endpoints return an ~80-byte-per-issue shape (`id/title/status/priority/issue_type/assignee/labels/dependency_count/dependent_count`) instead of the ~400-byte full `Issue`; `brief=True` shrinks further to just 4 fields.
3. **`CompactedResult` auto-compaction** — when a `list`/`ready` result exceeds `COMPACTION_THRESHOLD` (default 20), the server returns `{compacted: true, total_count, preview: [...first N], preview_count, hint}` instead of the full set. Tunable via env: `BEADS_MCP_COMPACTION_THRESHOLD` (default 20, must be ≥1) and `BEADS_MCP_PREVIEW_COUNT` (default 5, must be ≥1 and ≤ threshold) — validated at import time in `_get_compaction_settings()` (`server.py:94-113`), raising `ValueError` on invalid combos.

Separately, `/home/daniel/repos/beads/internal/compact/haiku.go` implements **issue-body** compaction (distinct from MCP result compaction): closed issues 30+ days old get summarized by an LLM (`ai.model` config default `claude-haiku-4-5-20251001`, `internal/config/config.go:268`) using this exact prompt template (`internal/compact/haiku.go:264-291`):
```
You are summarizing a closed software issue for long-term storage. Your goal is to COMPRESS the content - the output MUST be significantly shorter than the input while preserving key technical decisions and outcomes.
...
IMPORTANT: Your summary must be shorter than the original. Be concise and eliminate redundancy.

Provide a summary in this exact format:

**Summary:** [2-3 concise sentences covering what was done and why]
**Key Decisions:** [Brief bullet points of only the most important technical choices]
**Resolution:** [One sentence on final outcome and lasting impact]
```
This requires `ANTHROPIC_API_KEY` and is invoked via `bd admin compact` (see `commands/compact.md` and `examples/compaction/`, §6.4).

### 8.4 Memory injection mechanics (`bd remember`)

`/home/daniel/repos/beads/cmd/bd/memory.go` implements `bd remember`/`bd memories`/`bd forget`/`bd recall` as a KV store (`kvPrefix + memoryPrefix + key` → value) with two anti-footgun guards worth flagging as prompt-engineering-for-a-CLI-tool: (1) a bare single-word insight matching a known `bd` subcommand name is refused unless `--key` is explicit (protects against `bd remember recall` accidentally storing the literal word "recall"), and (2) if the positional arg round-trips through `slugify()` unchanged (i.e., it *looks* like a key, not prose) and that key already exists, the command **silently becomes a read** (`bd recall` semantics) instead of a write — deliberately paving the "did you mean to ask, not tell" desire path. Memories are injected into `bd prime` output via `formatMemoriesForPrime()`/`renderPrimeMemories()` (`prime.go:358-508`), sorted alphabetically (no timestamps kept), with hard caps (`--max-memories`/`--max-memory-chars` flags or `prime.max-memories`/`prime.max-memory-chars` config, env-overridable) that always emit at least one memory and prepend an elision banner (*"N more memories are not shown here (capped by max-memories=N). Browse the full set with `bd memories <keyword>`..."*) rather than silently truncating.

## 10. Prompting Patterns That Keep the Graph Current

Distilled from the above, in rough order of how central each is to the `bd prime` runtime text vs. static docs.

1. **Issue-before-code, not after.** `bd prime` Core Rules: *"Workflow: Create beads issue BEFORE writing code, mark in_progress when starting"* (`cmd/bd/prime.go:722`). Same rule appears in the installed `AGENTS.md`/`CLAUDE.md` block's "Workflow for AI Agents" step 2 (`internal/templates/agents/defaults/beads-section.md:59-64`). This front-loads graph creation instead of relying on retrospective logging.

2. **Claim-before-work as an atomic, named verb.** Not "set status to in_progress" but a dedicated CAS operation — `bd update <id> --claim` / MCP `claim` tool, described as *"Atomically claim an issue for work (assignee + in_progress in one CAS-style operation)"* (`server.py:1127`). `task-agent.md`: *"Always claim before working (MCP: `claim`; CLI: `--claim`) and close when done."* Giving the state transition a single named tool (rather than a generic `update`) makes it much harder for an agent to "forget" the second half of set-assignee+set-status.

3. **`discovered-from` as mandatory provenance on every side-quest.** Universal across every layer read: `task-agent.md` step 4 ("Track Discoveries"), `beads-section.md` Quick Start example (`--deps discovered-from:bd-123`), `PATTERNS.md` "Side Quest Handling," `DEPENDENCIES.md`'s entire fourth section, `WORKFLOWS.md` "Discovery and Issue Creation" checklist. The MCP `create`/`dep` tool docstrings repeat it: *"Use this when you discover new work during your session. Link it back with beads_add_dependency using 'discovered-from' type"* (`tools.py:478-480`). This is the single most-repeated instruction in the corpus — it converts ephemeral "oh, and also—" discoveries into permanent, attributable graph edges instead of being lost in scrollback.

4. **Notes as a zero-context handoff artifact, explicitly gated on compaction survival.** `RESUMABILITY.md`'s test: *"Would a fresh Claude instance (or you after 2 weeks) struggle to resume this work from the description alone? If yes, add implementation details."* `WORKFLOWS.md` "Compaction Survival" gives a template (`COMPLETED:`/`IN PROGRESS:`/`NEXT:`/`KEY DECISION:`/`BLOCKER:`) and explicitly contrasts a good vs. bad note. `bd prime`'s own Core Rules: *"Memory: Use `bd remember \"insight\"` for persistent knowledge across sessions. Do NOT use MEMORY.md files — they fragment across accounts."* — this pattern treats the notes/memory fields as the durable substrate and conversational context as disposable by design, not as a failure mode to work around.

5. **Session-close gated on the agent's own completion language.** `bd prime`: *"Before saying 'done' or 'complete', you MUST run this checklist"* (`cmd/bd/prime.go:711`). This is a deliberate hook into the model's natural end-of-turn phrasing rather than a time- or tool-call-based trigger — it fires exactly when the agent is about to stop updating the graph.

6. **Hard prohibition on shadow tracking systems, repeated at every layer.** `bd prime`: *"Prohibited: Do NOT use TodoWrite, TaskCreate, or markdown files for task tracking."* `beads-section.md`: *"❌ Do NOT create markdown TODO lists / ❌ Do NOT use external issue trackers / ❌ Do NOT duplicate tracking systems."* By eliminating alternative places state could live, the system removes the agent's ability to accidentally maintain a competing, non-durable model of "what's left to do."

7. **The dependency graph itself, not agent judgment, decides "what's next."** `bd ready` only surfaces issues with zero open `blocks` edges — `DEPENDENCIES.md`: *"Only `blocks` dependencies affect what work is ready... bd maintains ready state automatically."* `WORKFLOWS.md`'s "Ready Front" model explicitly reframes planning language to avoid a *"cognitive trap"*: *"'Phase 1 before Phase 2' → brain thinks 'Phase 1 blocks Phase 2' → WRONG... Correct: 'Phase 2 needs Phase 1' → `bd dep add phase2 phase1`."* This converts freeform "what should I work on" reasoning into a structural graph query, which only stays correct if agents keep the graph honest.

8. **A crisp decision test that pre-empts under-use of the graph.** `BOUNDARIES.md`/`SKILL.md`: *"Decision test: 'Will I need this context in 2 weeks?' YES = bd, NO = TodoWrite"* — by giving agents a cheap one-line heuristic (rather than a long policy to reason through), it lowers the activation energy for creating an issue in the first place, which is the precondition for everything else on this list.

9. **`bd prime` as a mandatory re-anchoring ritual at every context-loss boundary, engineered around each host's actual hook limitations.** Not just "run this sometimes" — SessionStart, PreCompact-as-pre-check, PostCompact-marker + next-UserPromptSubmit-refresh (Codex, because *"PreCompact cannot preserve Beads context by printing text"*, `docs/CODEX_INTEGRATION.md:31`), and a truncation-defense banner prepended to every `bd prime` output (*"If this output is truncated by your host, read the full persisted hook output before continuing"*, `cmd/bd/prime.go:346`) so that even a host that clips hook previews doesn't silently drop memories/rules.

10. **Self-updating instructions (DRY via `bd prime` as SSOT) as an indirect graph-currency mechanism.** ADR-0001's stated rationale — collapsing 3 duplicated CLI-reference copies into one live-generated source — isn't about the graph directly, but it prevents the failure mode where an agent follows stale, hand-copied instructions (wrong flag names, deprecated `bd sync` instead of `bd dolt push`, etc.) that would otherwise cause writes to silently miss the real graph-of-record. `commands/sync.md`/`commands/import.md` explicitly document their own deprecation for exactly this reason.

## 11. Summary of Notable Audit Findings (for the hosted-service context)

- **MCP is explicitly positioned as the fallback, not the primary integration** (`docs/CLAUDE_INTEGRATION.md`), on pure token-economy grounds (10-50k vs 1-2k tokens). A hosted MCP-first service inherits none of the CLI+hooks efficiency story and should expect to re-derive its own equivalent of `bd prime`'s compact/full mode split and truncation-defense banner.
- **No daemon exists in the MCP server** — it is subprocess-per-call against the `bd` binary, with a connection *concept* (pool) that's really just a health-checked cache key, not a persistent RPC channel. `prefer_daemon` is vestigial/ignored.
- **Two same-named-but-different "minimal" profiles** (template verbosity vs. policy authority) are easy to conflate; the codebase itself keeps them in separate Go packages/types (`agents.Profile` vs `config.AgentProfile`) for exactly this reason.
- **Plugin manifests (Claude/Copilot) are behind the CLI installer's current hook contract** (bare `bd prime` + `PreCompact` still present in `plugin.json`, already migrated away from in `cmd/bd/setup/claude.go`) — a live drift example, useful for illustrating why a hosted service needs a single source of truth for hook wiring rather than N copies.
- **`commands/version.md` hardcodes a stale plugin version ("0.9.2")** against an actual manifest version of `1.1.0` — a small but concrete illustration of the doc-drift risk ADR-0001 was written to solve elsewhere but that clearly still recurs in adjacent files.
- **The "land the plane" ritual is a product/marketing name for a mechanism, not a code string** — the actual implementation is the two-part SESSION CLOSE PROTOCOL (runtime, `bd prime`) / Session Completion (static, installed file) pairing in §2, both explicitly subordinate to *"explicit user, repository, and orchestrator instructions"* and gated by the tri-state policy profile — i.e. Beads never lets its own prompting override an explicit human/orchestrator instruction, which is a reasonable default stance to carry into a hosted-memory-service prompt design.
