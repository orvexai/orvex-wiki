# PRD Quality Review — Cross-Agent Memory Service (hosted, beads-inspired)

## Overall verdict
This is a strong, unusually rigorous PRD: decision-dense, discovery-grounded, with clean IDs, a load-bearing glossary, and a complete Assumptions Index roundtrip — it is build-ready as a chain-top platform spec and its thesis (coordination memory, not personalization memory) genuinely holds the features together. Two things are worth fixing before architecture and epics lean on it: the headline success metric (SM-1) names a target but not a measurement method, so the PRD's central value claim risks being unfalsifiable; and the supersession of a *built, shipping* `/v1/memory` route + live `memory` table (§9) under-scopes the data migration/cutover — the single largest execution risk in the program. Everything else is minor polish against a high bar.

## Decision-readiness — strong
A decision-maker can act on this. Choices are stated as decisions, not buried: Dolt is dropped for the core store with tiered rationale (§10; addendum A1), coordination writes are "direct and ungated" (§4.1), the beads CLI is reimplemented rather than wrapped (§5, A2), and three existing memory implementations are superseded (§9). Trade-offs name what was given up — A1 lists what survives from Dolt/beads (cell-conflict keying, hash-ID collision math, bi-temporal model) versus what is dropped, and A2 "Rejected alternatives" is a real give-up ledger (gating writes, LLM-KG extraction, per-item ACLs, building on Mem0/Zep/Letta) each with a reason. The seven Open Questions (§11) are genuinely open architecture-refinement items (bi-temporal depth, role registry, adapter scope), not rhetorical. The scarcity of `[NOTE FOR PM]` callouts (one, at §6.2) is not dodging here — the tensions were resolved into the addendum as made decisions rather than parked, which is the right posture for a green-light PRD. No findings.

## Substance over theater — strong
The content is earned, not furniture. The four UJ protagonists (Nova, Atlas/Bolt, Rhea) read at first like anthropomorphized theater but are load-bearing: each journey carries coordination semantics that cannot be specified without a journey — the atomic claim race between twin agents (UJ-2), crash-and-reap recovery (UJ-2 edge case), the honest-empty degraded prime (UJ-1 edge case). The novelty claim ("Nobody in the profiled market combines a queryable multi-agent work-graph with curated knowledge routing", "In short") is specific and falsifiable, backed by the 18-platform sweep and the A2A-has-no-memory datapoint (A5) — earned differentiation, not a template section. NFRs carry product-specific thresholds (p95 ≤ 300ms, 100k items/500k edges, ≤ 300 tokens) rather than "scalable/secure/reliable" boilerplate. The Vision (§1) could not swap into another PRD. No findings.

## Strategic coherence — strong
The PRD has a clear thesis and bets on it: coordination memory occupying the unoccupied ground between a multi-agent work-graph and curated knowledge routing. Features serve a unified arc — work-graph core (coordination substrate) → remembered facts + prime (cheap sessions) → consolidation/compaction ("the field's weakest area, our differentiator", §4.3) → MCP-first surfaces (because agents live in ChatGPT/Claude) → prompt pack (the chat-platform substitute for hooks). MVP scope kind is coherently **platform** ("a memory substrate every product and surface shares", §2.1) and the scope logic matches (breadth of surfaces, supersession, contracts reservation). Success Metrics validate the thesis rather than measuring activity, and — notably — the counter-metrics are real: SM-C1 ("More stored items is not success", countering Mem0's pricing anti-pattern) and SM-C2 (prime-size creep) each name a paired SM they guard. This is the strongest dimension.

### Findings
- **medium** SM-1 names a target but no measurement method (§8) — SM-1 reads "Median tokens spent reconstructing context in instrumented sessions drops ≥ 40% with prime+recall enabled vs. baseline." "Tokens spent reconstructing context" is the numerator of the PRD's headline value claim, but nothing defines how those tokens are isolated from the rest of a session's spend, nor how the no-prime baseline is captured on a platform where prime is installed by prompt pack. As written the KPI is not cleanly falsifiable. *Fix:* add one line to SM-1 (or NFR-MEM3) defining the measurement — e.g. prime/recall injected-token count as the instrumented proxy, compared against a control cohort or a pre-install baseline window — so downstream instrumentation has a target it can actually build.

## Done-ness clarity — adequate
The FRs are mostly strong on testable consequence: FR-MEM1 (create p95 ≤ 150ms + CloudEvents emission + adaptive short-ID rule), FR-MEM4 (atomic CAS claim, TTL lease, reaper revert — bounded by SM-2's <1% double-claims / ≥95% recovery), FR-MEM5 (explicit `submitted → claimed → accepted/returned` state machine), FR-MEM11 (0.98-cosine dedup, ADD/UPDATE/DELETE/NOOP, soft-delete-with-trail), FR-MEM13 (immediate cascading delete of derived artifacts, no TTL floor). Adjective-scrubbing comes back mostly clean — NFR-MEM6's "honest-empty prime/recall with status" is specific, not "handles gracefully." Two gaps keep this from strong. FR-MEM18 (the ritual pack) is a content deliverable whose "done" is inherently fuzzy, but the PRD answers that well by making FR-MEM19 adherence telemetry + SM-3 thresholds the measurable proxy — treat that as a strength, not a gap. The genuine soft spot is anomaly detection.

### Findings
- **low** "Anomaly" invoked three times, never bounded (FR-MEM17, UJ-3) — FR-MEM17 lists "queue anomalies" as a rendered console signal and UJ-3 says "the anomaly is flagged", but the only concrete detection rules anywhere are stale-claims and re-opening-rejected-work (UJ-3). An engineer building the console read view would not know what else counts. *Fix:* either enumerate the v1 anomaly set (stale claim, re-open loop, orphaned blocker) or scope FR-MEM17 anomalies to exactly those two and defer the rest.

## Scope honesty — adequate
Assumption and non-goal discipline is excellent and would carry this dimension to strong on its own: §5 Non-Goals does real work (orchestration boundary, no personalization, no LLM-KG pipeline, no P2P federation, no per-item ACLs, coordination never gated, no beads CLI), the `[NON-GOAL for MVP]` callout at §2.2 and `[NOTE FOR PM]` at §6.2 land where omissions could be silently assumed, and every inline `[ASSUMPTION]` roundtrips cleanly to the §12 index (verified below). Open-items density (7 OQs + 8 assumptions + 1 note) is acceptable for a green-light platform PRD because none block the core coordination substrate — the two that matter most (service topology, quota dimensions) are legitimately architecture/billing's to close. The one real gap is that the riskiest scope item is under-surfaced.

### Findings
- **medium** Supersession of a *live* route + table under-scopes the data migration/cutover (§9, §6.1) — §9 supersedes "orvex-studio-api FR-SA5–11 Memory FormSpec + its built `/v1/memory` route and tenant-partitioned `memory` table (migrates to the platform service; studio-api becomes a client)", and §6.1 puts "supersession of studio-api Memory routes + hidden MCP memory tools" in scope. But the *data* migration of existing rows, the cutover sequencing, and the client-swap risk are never scoped or risk-flagged. The PRD applies explicit hard-cut treatment to the MCP tools ("hard cut to the new section", FR-MEM14) and to Dolt, and even cites the house no-fallbacks rule for FR-MEM13 — yet the live table, the biggest execution risk, gets none of that same rigor. *Fix:* add a migration/cutover line to §6.1 or §10 that names the existing-data path under the no-fallbacks hard-cut pattern (required new path + loud migration error, no legacy shim) and flags the studio-api client swap as a sequenced risk.

## Downstream usability — strong
As a chain-top PRD feeding architecture and epics, this sources cleanly. The glossary (§3) anchors every domain noun and the terms are used identically downstream — spot-checked Ready Work/`ready`, Prime/`memory_prime`/prime-compact, Handoff, Reaper, Close Protocol all hold across FRs, UJs, SMs, and NFRs. IDs are contiguous, unique, and resolve: FR-MEM1–20, NFR-MEM1–7, UJ-1–4, SM-1–5 + SM-C1–C2, and every SM's "Validates FR-…" cross-reference points at a real FR. Each FR references glossary terms rather than "see above", so sections stand alone. All four UJs have named protagonists carrying context inline — no floating UJs. The only frictions are minor and routed to Mechanical below (beads jargon in one out-of-scope line; an unpinned external contract reference). Strong.

## Shape fit — strong
The PRD is not forced into a wrong shape. This is a multi-stakeholder platform capability spec (agents, fleet operators, finance/eng leads, the platform itself), so the UJ formality is appropriate rather than overhead — a single-operator tool would not need journeys, but a coordination product must show the claim/handoff/recovery flows, and it does. The heavy formality (20 FRs, addendum) matches genuine stakes: a new platform service superseding three implementations. On the brownfield axis the rubric cares about, existing-code references are specific and sourced (the built `/v1/memory` route, the `memory` table, FR-AI11 `ai_memories` as planned-only, the hidden `studio_memory_get/save` tools; grounded via A6 → librarian-portal-scale-audit.md), and all UJs are new so there is no existing/new confusion. No findings.

## Mechanical notes
- **Prime compact token-budget drift (low).** The glossary (§3, Prime) sets compact at "~50–200 tokens", NFR-MEM3 sets "prime-compact ≤ 300 tokens p95", UJ-1 says "~200 tokens", and "In short"/§1 say "a few hundred tokens". Not contradictory (200 ≤ 300) but two ceilings for one budgeted quantity; downstream will implement one number. Reconcile the glossary ceiling and NFR-MEM3.
- **beads jargon undefined (low).** §6.2 defers "formulas/molecules/gates/swarm analogs" — these are beads concepts with no glossary entry, so a reader without beads context cannot evaluate the deferral. Low stakes (explicitly out of scope); a parenthetical gloss would close it.
- **Unpinned external contract (low).** FR-MEM16 targets a shape "compatible with the MCP KG reference contract" without pinning which contract/version; OQ #6 already flags the adapter scope as open, so this is a pointer to tighten when #6 closes.
- **Assumptions Index roundtrip — clean.** All eight inline `[ASSUMPTION]` tags (v1 type set; lease TTL/reaper grace; compaction windows; read-only adapter; pack-revision approval; NOTICE attribution; scale envelope; service name + quota dimensions) appear in §12, and every §12 entry maps back to an inline tag. No orphans either direction.
- **ID continuity — clean.** No gaps or duplicates across FR/NFR/UJ/SM series; external cross-refs to the sibling Staging PRD (FR-STG1, its UJ-1) are named targets that cannot be resolved standalone, which is inherent to the two-PRD program rather than a defect.
- **Required sections — all present** for a chain-top platform PRD at this stake level (Vision, Target User + JTBD + Non-Users + UJs, Glossary, Features, Non-Goals, MVP Scope, NFRs, Success Metrics + counter-metrics, Integration/Dependencies, Constraints, Open Questions, Assumptions Index, plus a depth addendum).
