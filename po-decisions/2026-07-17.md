## PO-D1 — MCP ⊇ CLI: the MCP must be able to do everything the CLI can (PO, in-chat, 2026-07-17)

**Directive (verbatim intent).** "MCP should do everything CLI can." Capability parity:
every capability exposed through orvex-cli must be reachable through the MCP surface.

**Interpretation applied (logged, not asked — full-autonomy doctrine).** This is read as
*capability-surface parity under identical server-side governance*, not as granting the
MCP autonomous governance powers:

- The CLI itself has no autonomous governance power today — ratify/force-new/supersede
  are server-gated behind human-minted tokens (ratify-token guard, workspace force
  gates). Parity therefore means the MCP carries the SAME verbs with the SAME
  server-enforced human-token gates: given a human-minted token it can transport it
  (exactly as the CLI does); absent a token it returns `needs_human_publish`.
- The invariant "AI never self-promotes" (P6, MCP canon `8nngxUpOXY`/`tQcQ01U2iy`,
  re-baseline `ZGjLctEnGH`) is preserved server-side and is NOT weakened by this
  directive.
- This SUPERSEDES the narrower posture "governance verbs are human/CLI-only and absent
  from the MCP surface" wherever the canon states it: the verbs now appear in the MCP
  surface, token-gated, rather than being omitted.
- Progressive disclosure still applies: parity means *reachable* (e.g. behind toolsets /
  tool-search gating past the hero set), not "all verbs in the default hero toolset."

**Consequence for the redo.** The Track-1 parity deliverable gains a third list: the
MCP-vs-CLI delta (every CLI capability not yet reachable via MCP). The MCP PRD redo
inherits the CLI capability inventory as a requirements floor. Flag for Daniel's review
at PRD ratification: confirm the token-gated reading of governance verbs matches his
intent.

## PO-D4 — Compose the family: exploit knowledge/ai/identity/audit fully; never route AI work to the engine (PO, in-chat, 2026-07-17)

**Directive (verbatim intent).** "Remember we have orvex-studio-knowledge,
orvex-studio-ai, orvex-studio-identity etc. and other microservices such as audit — we
should be integrating and taking full advantage of them — so not to route AI to
orvex-wiki — maybe reads can come from knowledge etc."

**Interpretation applied.**

- The wiki AI surfaces (wiki-api /v1, the wiki domain of the MCP, orvex-cli) are
  COMPOSITIONS of the family, not fronts for the engine. Every capability lands on the
  service best positioned to serve it: search/related/RAG → knowledge (already settled
  D-M8/PO-ruling-9); ask/synthesis → ai; authn/session → identity; compliance events →
  audit (canonical PRD `vhC5XXCYkC`); entitlements → billing. The AGPL engine is the
  system of record for document writes + collab — it is NOT the default read path for
  agents and NEVER the AI compute path.
- NEW design question opened by the PO (a first-class Track-2 topic, not settled canon):
  serve agent DOCUMENT READS from the knowledge service's index (read-replica posture)
  instead of engine round-trips — evaluate for the 10k-customer/cell scale win, against
  freshness/indexing-lag, lossless fidelity (does the index hold full DfM content?),
  per-page ACL enforcement point, and provenance/trust metadata. "Maybe" = design it
  and bring the trade-off back, not adopt blindly (PO-D2 applies to this too).
- Parity dispositions must name the SERVING service per capability, not just
  "covered/missing" — a capability parked on the wrong service is a gap.
